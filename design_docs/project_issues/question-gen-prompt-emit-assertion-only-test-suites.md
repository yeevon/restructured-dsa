# Should `question_gen`'s prompt change to emit assertion-only test suites (reference the implementation target by name, do not define it) so the in-app runner's concatenate splice works on real generated Questions?

**Status:** Open
**Surfaced:** 2026-05-12 (TASK-017 / ADR-042) — ADR-042 builds the in-app code-execution sandbox whose splice is "concatenate the learner's `response` followed by the Question's `test_suite` into one translation unit". That splice is correct for an **assertion-only** test suite (one that *calls* the implementation target but does *not* define it). ADR-040's `question_gen` prompt currently steers the LLM to emit test suites that *embed a reference implementation of the target inline* (the TASK-016 real-engine gate confirmed this — each `quiz_id=6` `test_suite` is a self-contained C++ file that defines `void append(...)` itself and then `assert`s against it). Concatenating the learner's code with such a test suite double-defines the target → an honest `compile_error` from `g++` (never a fabricated pass/fail — MC-5's spirit is intact), but **not** the responsive pass/fail loop §6 wants on a real generated Question. ADR-042 deliberately did not bolt a fragile body-replacement splice into the runner (matching `void append(...) {` to its closing `}` is a real C++ parse — overloads, templates, nested braces, string-literal braces — and a wrong strip silently corrupts the test, the exact failure MC-5's spirit forbids); the right fix is the *generator*, and that's a separate, well-scoped change captured here. (TASK-017's ADRs explicitly defer it: ADR-042 §The splice files this issue and notes the dependency; TASK-017 §Out of scope says "If `/design` concludes `question_gen`'s prompt should change to emit assertion-only test files … file a project_issue, don't fold it in".)

**Decide when:** before TASK-017's "real-engine end-to-end run-tests check" verification gate can be filled `pass` against a real generated Quiz (e.g. `quiz_id=6`-style) — until the prompt change lands, that gate must be filled against a hand-written assertion-only test suite, or the gate's expectation is "a reference-impl-embedding `test_suite` reports `compile_error`, honestly". The human decides the ordering: schedule the `question_gen`-prompt-change TASK before / alongside / after TASK-017's gate. Forced no later than the first time someone wants the in-app runner to produce pass/fail on a real generated Question.

## Question

ADR-040's `question_gen` `LLMStep` prompt instructs the model to "emit a runnable test suite per Question and to name the implementation target in the `prompt`" (exact wording the implementer's, tunable per ADR-036's precedent). Currently the model interprets "runnable test suite" as "a self-contained file" — so it embeds a reference implementation. Should the prompt instead instruct the model to emit a test suite that:

1. *references* the implementation target by its named signature (calls `append(...)` / instantiates the class / etc.) but does **not** define it (no reference implementation in the test file);
2. is paired with the expectation that the learner's code provides the definition (the runner concatenates `learner_response + test_suite` into one translation unit, or compiles them as two TUs — ADR-042's concatenate-into-one-TU splice);
3. for C++, `#include`s only what the assertions need (`<cassert>`, any data-structure headers the test uses) and forward-declares / relies-on-the-learner-providing the target's struct/class and function signatures (or the prompt fixes a small shared "preamble" — the struct/class definitions — that *both* the learner's code and the test suite assume, emitted in the `prompt` so the learner knows the exact shape to implement against);
4. for Python, defines no implementation, imports/references only the named function/class the learner implements (the test module `unittest.main()`s against names the runner's spliced source provides).

This is a `question_gen`-prompt question — arguably a future TASK (a prompt-wording change + a re-run of the real-engine generation gate to confirm the new output shape + a confirmation that the in-app runner now produces pass/fail on the regenerated Quiz). It does **not** require a schema change (`questions.test_suite` is still a string), a `GeneratedQuestion` shape change, or a runner change — just the prompt and a re-gate.

## Options known

- **(a) Change `question_gen`'s prompt to emit assertion-only test suites that reference (do not define) the implementation target, with a shared preamble (the struct/class definitions) emitted in the `prompt`.** The runner's concatenate splice (ADR-042) then works on real generated Questions: `learner_response` (the learner's implementation, written against the preamble's struct/class shapes) + `test_suite` (the assertions, referencing the named functions) → one TU → compiles and runs. Cleanest end-to-end; needs a prompt-wording change and a re-gate; the `prompt` now also has to carry the preamble (the data-structure definitions) so the learner implements against the exact shapes the test assumes — which is consistent with ADR-040's "name the implementation target in the `prompt`" (it extends it: name *and shape* the target).
- **(b) Change the runner's splice to a two-translation-unit compile (`g++ solution.cpp test.cpp -o runner`) and change the prompt so the test suite `#include`s a header the learner's TU provides** (the test suite `#include "solution.h"` and `extern`-declares the target; the learner's code is `solution.cpp`). More machinery (a header convention, two TUs); the prompt has to fix the header name and the declarations. ADR-042 picked single-TU concatenate as simpler; this option revisits that *and* the prompt together. Heavier than (a).
- **(c) Leave `question_gen` as-is; accept that the in-app runner reports `compile_error` on real generated Questions until the learner is told to write a self-contained file too** — i.e. the take page instructs the learner to write a complete program (their own struct/class defs + their impl + a `main` that does nothing, or no `main`) and the runner... no, this doesn't work — the test suite has its own `main` with the asserts; two `main`s is also a conflict. (c) is not actually viable; recorded to show why (a) or (b) is needed.
- **(d) A runner-side textual strip of the reference implementation from the test suite** — rejected in ADR-042 (too fragile; a wrong strip silently corrupts the test — MC-5's spirit). Not re-litigating it here; recorded so a future `/design` doesn't propose it again without new evidence.

## Constraints

- **Manifest §6** ("Code is written, run, and tested within the application … run tests against it") — the in-app runner is the consumer; this issue is about making the *generator* produce test suites the runner can splice the learner's code into.
- **Manifest §5 / §7** — the test suite holds only test *source code* (never a choice/recall/describe artifact); the prompt change must not weaken ADR-040's non-coding-format forbiddances; the `min_length=1` validator on `GeneratedQuestion.test_suite` stays.
- **MC-1** — the `question_gen` module's imports stay `ai_workflows.*` + `pydantic` (+ stdlib) only; the prompt is a string; no SDK change. ADR-036's `aiw run` CLI-subprocess invocation is unchanged. MC-5 — a Question whose LLM-supplied `test_suite` is missing/blank still makes the whole Quiz `generation_failed` (ADR-040); the prompt change doesn't touch that path.
- **MC-6** — unchanged; the test suite goes to `data/notes.db`'s `questions.test_suite`, never under `content/latex/`.
- **ADR-040 / ADR-041 / ADR-036 / ADR-037** — the representation (a string), the storage (`questions.test_suite TEXT`), the processor wiring (read it from the `aiw run` artefact, persist verbatim), the failure handling — all unchanged; only the prompt wording (and possibly the `prompt`-carries-a-preamble convention) changes.
- **ADR-042** — the runner's concatenate-into-one-TU splice; the `RunResult` shape; the C++/Python language sniff — all unchanged; this issue is about feeding the runner test suites that splice cleanly.
- **TASK-016 audit** (`design_docs/audit/TASK-016-questions-carry-a-test-suite.md`, the `ai-workflows generation sanity (with test suites)` gate row, "Observation for the in-app-test-runner slice") — the empirical evidence that the current output embeds reference implementations inline; a re-run of that gate after the prompt change confirms the new shape.

## Resolution

When the `question_gen`-prompt-change TASK's `/design` runs, mark this issue `Resolved by ADR-NNN` (the ADR that changes the prompt — likely citing ADR-040, consuming ADR-042's runner-splice expectation unchanged; the prompt change is additive to ADR-040's "name the implementation target" instruction — "name *and shape* it; emit assertions only, not a reference implementation"), and note: which option was chosen ((a) assertion-only + preamble-in-prompt, or (b) two-TU + header-convention); whether the `prompt` now carries a preamble (the struct/class defs); the re-run real-engine generation gate's result (does the regenerated Quiz's `test_suite` splice cleanly with a hand-written correct implementation → `RunResult(status="ran", passed=True)`); and whether TASK-017's real-engine end-to-end gate was (re-)filled against the regenerated Quiz.
