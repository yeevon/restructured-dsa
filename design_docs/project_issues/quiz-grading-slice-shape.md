# The grading slice's shape — the out-of-band grading processor, the Grade aggregate's schema, the Weak-Topic-identification path, and the `notifications` table + surface

**Status:** Open
**Surfaced:** 2026-05-12 (TASK-015 / ADR-038 / ADR-039) — a tracking breadcrumb consolidating decisions already deferred by ADR-033 ("the Grade schema … to the grading task"), ADR-036 ("the relational Topic vocabulary … to the Weak-Topic-identification slice"), and ADR-037 ("the active Notification entity (a `notifications` table + a learner-visible surface) ships with the grading slice"). Not a standing question awaiting `/design` *now* — the grading slice is the next Quiz-pillar task, and these decisions are squarely *that* task's `/design` to make. Filed so the next `/next` cycle sees the slice and its deferred sub-decisions in one place.

**Decide when:** the grading-slice task's `/design` (the next Quiz-pillar task after TASK-015). Forced then; not before.

## Question

The Quiz pillar's planned sequence (from TASK-014's forecast, restated in TASK-015): schema + read surface (TASK-013) → generation (TASK-014) → the Quiz-taking surface (TASK-015) → **the grading workflow** → the replay-+-fresh composition. TASK-015 ships a `submitted` Attempt — a `quiz_attempts` row `submitted` + one `attempt_questions` row per Question carrying the learner's `response` code (`is_correct` / `explanation` NULL) — and the submit route is written so the grading slice's processor requires *no change* to it (it already does "record and return", mirroring the generation processor / `requested`-row pattern, ADR-037). The grading slice has to decide:

1. **The out-of-band grading processor.** Mirroring ADR-037's `python -m app.workflows.process_quiz_requests` (poll-and-process `requested` Quizzes): a `python -m app.workflows.process_quiz_attempts` (or similar) that polls `submitted` Attempts, sets `grading`, invokes a new `ai-workflows` grading workflow via the `aiw run` CLI subprocess (ADR-036's downstream surface — the workflow gets the Quiz's Questions + the learner's code per Question; returns per-Question correctness + explanation + the Grade aggregate's components), persists the result via `app.persistence.*` (the grading-slice persistence functions — `mark_attempt_grading` / `mark_attempt_graded` / `mark_attempt_grading_failed` / `save_attempt_grades` or similar, mirroring ADR-037's `mark_quiz_*` and ADR-039's `start_attempt`/`submit_attempt`), sets `graded`, and (with the Notification entity) notifies the learner. On failure: `grading_failed`, no fabricated Grade (MC-5), the detail recorded (a `quiz_attempts.grading_error TEXT` column, additive — mirroring ADR-037's `quizzes.generation_error`). The submit route does **not** invoke it (MC-4 — ADR-038 §The route pair).
2. **The Grade aggregate's schema** (manifest §8: "per-Question correctness, per-Question explanation, an aggregate score, identified Weak Topics, recommended Sections to re-read"). ADR-033 deferred it "to the grading task" and noted both shapes are additive: a `grades` table (PK `attempt_id`, FK to `quiz_attempts`, `score`, `weak_topics TEXT`, `recommended_sections TEXT`) **or** nullable columns on `quiz_attempts` (`score`, `weak_topics`, `recommended_sections`). The per-Question portion (correctness + explanation) already has a home — `attempt_questions.is_correct` / `.explanation` (ADR-033; the rows created at Attempt start by ADR-039) — which the grading slice `UPDATE`s.
3. **The Weak-Topic-identification path** and the relational Topic vocabulary. ADR-033 / ADR-036 deferred the `topics` table + `question_topics` join + the Topic-vocabulary-source decision "to the Weak-Topic-identification slice" (the first thing that *queries* across Topics — which is this slice: "identified Weak Topics" comes from aggregating which Topics the learner got wrong, which needs Topics as first-class queryable entities). The migration from `questions.topics` (the `'|'`-delimited TEXT column ADR-033 ships) to the relational form is additive (ADR-036): `CREATE TABLE topics`, `CREATE TABLE question_topics`, backfill from the delimited column — a data migration, not a schema change. The Topic-vocabulary *source* (a `content/`-derived file? a Python module? a `topics` table seeded from a source?) is part of this decision.
4. **The `notifications` table + a learner-visible Notification surface** (manifest §8: "a learner-visible indication that an async AI result has become available — most commonly a Grade or a newly-generated Quiz"). ADR-037 records it "ships with the grading slice" (for the generation slice, the "notified" obligation was met by the per-Section Quiz status flipping to "Ready"; for grading, a real Notification entity is wanted — a Grade is a more substantial result than "the Quiz is ready"). The decision: the `notifications` table's schema (a `notifications` row per available result — `notification_id`, a type (`quiz_ready` | `grade_ready`), a reference to the Quiz/Attempt, `created_at`, a `seen_at` nullable?); where the Notification surface lives (a header/badge in `base.html.j2`'s chrome — ADR-008 §Future-surfaces forecast "most natural shape is a header/badge inside `base.html.j2`'s chrome region; styling lives in `base.css` because it is page chrome"); whether generation also writes a `notifications` row retroactively (ADR-037's per-Section status flip stays as the in-context view; the Notification is the out-of-context alert).

## Options known

These are sketches, not commitments — the grading slice's `/design` decides:

- **The processor:** mirror ADR-037's `process_quiz_requests` exactly (a `process_quiz_attempts` module, `aiw run` subprocess, the documented downstream surface, not the framework's internal `_dispatch.run_workflow`) — vs. a combined "process all pending AI work" command — vs. an in-request synchronous grade (**ruled out by MC-4 / manifest §6** — not an option).
- **The grading workflow's authoring surface:** a CS-300-owned declarative `WorkflowSpec` module at `app/workflows/grade_attempt.py` (mirroring ADR-036's `question_gen.py` — one `LLMStep` + a `response_format` validator; `GradeAttemptInput` carrying the Questions + the learner's code per Question; `GradeAttemptOutput` carrying per-Question correctness + explanation + the aggregate's components), loaded via `AIW_EXTRA_WORKFLOW_MODULES` — vs. extending `question_gen.py` (rejected — a different workflow).
- **The Grade aggregate's home:** a `grades` table (PK `attempt_id`) — vs. nullable columns on `quiz_attempts` — both additive (ADR-033); the architect's forecast leans `grades` table (the aggregate is a distinct entity with a clean 1:1 to the Attempt, and a separate table keeps `quiz_attempts` lean).
- **The Topic vocabulary:** a `topics` table + `question_topics` join, backfilled from `questions.topics`, the vocabulary seeded from a `content/`-side source (consistent with manifest §8's "Topics form a project-wide vocabulary maintained alongside Chapter content" — i.e. a curriculum artifact, sourced like the LaTeX corpus) — vs. a Python module like `app/designation.py` — `/design`'s call; the migration is additive either way.
- **The Notification entity:** a `notifications` table + a header/badge in `base.html.j2` (ADR-008's forecast) — vs. a rail-resident Notification panel — `/design`'s call.

## Constraints

- **Manifest §6 / §7 / MC-4:** AI work is asynchronous from the learner's perspective — grading runs out-of-band, never in the submit request; the learner is notified when results are ready (Notification entity).
- **Manifest §6 / MC-5:** AI failures are visible, never fabricated — `grading_failed` is a real state (ADR-033's enum already names it); no placeholder Grade, no fabricated correctness; the detail recorded (additive `grading_error` column).
- **Manifest §4 / MC-1 / ADR-036:** AI invocation goes through `ai-workflows` (the `aiw run` CLI subprocess — the documented downstream surface); no direct LLM SDK use; the grading workflow is a CS-300-owned `WorkflowSpec` module under `app/workflows/`; CS-300 code persists the result via `app.persistence.*`.
- **Manifest §7 / MC-8:** "every post-first Quiz contains both replayed wrong-answer Questions and freshly-generated Questions" — the grading slice produces the *Weak Topics* (for the fresh portion) and fills `attempt_questions.is_correct` (the wrong-answer-replay history for the replay portion); the composition slice (the one after) reads both.
- **ADR-022 migration story:** additive only (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, nullable `ALTER TABLE ADD COLUMN`); the `grades` table / `score` columns / `grading_error` column / `topics` + `question_topics` tables are all additive — none fires the migration trigger. (The per-module schema-fragment refactor ADR-033 flagged "when the n-tables threshold makes the monolithic `_SCHEMA_SQL` block awkward" may finally be worth doing when the grading slice adds 2–3 more tables — the grading slice's `/design` weighs it.)
- **ADR-033 / ADR-039:** the `quiz_attempts` / `attempt_questions` schema and the `attempt_questions` rows created at Attempt start are consumed unchanged; the grading slice `UPDATE`s `is_correct` / `explanation` on the existing rows and transitions `quiz_attempts.status` `submitted` → `grading` → `graded`/`grading_failed`.
- **ADR-038:** the submit route is "record and return"; the grading processor builds on the `submitted` Attempts it leaves; no change to the submit route is needed.
- **MC-7:** single-user — no `user_id` on the Grade / Notification / Topic tables.
- **ADR-008:** the Notification surface, if it lives in `base.html.j2`'s chrome, is `base.css` (page chrome); the prefix posture extends.
