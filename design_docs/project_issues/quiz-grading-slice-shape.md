# The grading slice's shape ‚Äî the out-of-band grading processor, the Grade aggregate's schema, the Weak-Topic-identification path, and the `notifications` table + surface

**Status:** Resolved by ADR-048 + ADR-049 + ADR-050 + ADR-051 (Proposed 2026-05-13, TASK-019)

## Resolution note (2026-05-13, TASK-019)

Resolved by four ADRs together (mirroring how TASK-014 split into ADR-036 + ADR-037, TASK-017 into ADR-042 + ADR-043 + ADR-044, and TASK-018 into ADR-045 + ADR-046 + ADR-047):

- **ADR-048** owns the `grade_attempt` `ai-workflows` workflow ‚Äî a CS-300-owned `WorkflowSpec` module at `app/workflows/grade_attempt.py` (mirroring ADR-036 / ADR-040's `question_gen.py`); one `LLMStep` + `prompt_fn` + `response_format=GradeAttemptOutput`; the prompt commits architecturally to "read the runner's verdict, produce *explanation* + *aggregate*, do **not** re-judge correctness" ‚Äî the ¬ß8 amendment's "correctness determined by whether the learner's code passed the Question's tests" is realized by the schema itself, which has no `is_correct` field on `QuestionGrade` (the LLM cannot disagree with the runner because it has no field through which to disagree); `min_length=1` on `QuestionGrade.explanation`; `model_config = ConfigDict(extra="forbid")` on every Pydantic model (so a non-coding / chat / tutor-style field is unexpressible ‚Äî manifest ¬ß5 / ¬ß7); a tier registry `grade_attempt_tier_registry()` mirroring `question_gen_tier_registry()`; `RetryPolicy(max_attempts=3, exponential backoff)`; loaded via the extended `AIW_EXTRA_WORKFLOW_MODULES=app.workflows.question_gen,app.workflows.grade_attempt`; MC-1 preserved (imports stay `ai_workflows.*` + `pydantic` + stdlib only). The architect's tightening of the project_issue's option set: the workflow output's per-Question shape is `QuestionGrade(question_id, explanation)` ‚Äî no `is_correct` field, because the ¬ß8 commitment "correctness = test result" must be an architectural property, not a hope about LLM behavior. The persistence (ADR-050) writes `is_correct` from `test_passed`.

- **ADR-049** owns the out-of-band `python -m app.workflows.process_quiz_attempts` grading processor (option **the manual command mirroring ADR-037's `process_quiz_requests`** from ¬ßOptions known) ‚Äî polls `submitted` Attempts, transitions `submitted ‚Üí grading ‚Üí graded` / `grading_failed`, invokes the workflow via ADR-036's `aiw run` CLI subprocess (`aiw run grade_attempt --input ... --run-id attempt-{attempt_id}`), parses the artefact, validates it (Pydantic + CS-300 sanity check on the `question_id` set + bounds-check on `score`), persists transactionally via ADR-050's `save_attempt_grade`, sets `graded`. On any failure (non-zero exit, malformed artefact, Pydantic-validation failure, `question_id` mismatch, persistence transaction raise): `grading_failed` + detail in the new nullable `quiz_attempts.grading_error TEXT` column (ADR-050; mirroring ADR-037's `quizzes.generation_error`); zero partial Grade ever persists (the transactional discipline); no silent unbounded retry (the framework's bounded `RetryPolicy(max_attempts=3)` inside the workflow run; the CS-300 processor does not re-process `grading_failed` rows); the learner-facing signal stays the honest "Grading failed" (ADR-051's render). MC-5 honored architecturally end-to-end. **The "notified" obligation for this slice:** the take-page state flip on next `GET .../take` from `submitted` to `graded` *is* the ¬ß8 Notification mechanism (learner-visible, indicates the Grade is available, not real-time ‚Äî ¬ß5); **the active Notification entity (a `notifications` table + chrome badge + `seen_at` lifecycle, ADR-008's "Future-surfaces" forecast) is deferred to a follow-on slice** that ships the unified surface for *all* async-result kinds (Quiz `ready`, Grade `graded`, future Lecture audio). The deferral is conformant per ADR-035 ("describes what's built" ‚Äî the take-page state flip is what's built; the active Notification entity is what a follow-on slice's `/design` builds) and explicitly **departs from ADR-037's literal "ships with the grading slice" forecast** with the reasoning recorded (the take-page state flip honestly satisfies ¬ß8 Notification for this slice; the active Notification entity is a separable design surface better designed with multiple result kinds in hand). **The architectural commitment: "the persisted `score` is recomputed from `SUM(is_correct)`, not taken verbatim from the workflow"** ‚Äî ADR-049 records it, ADR-050 implements it inside `save_attempt_grade`. The same `aiw run` CLI invocation pattern, no enqueue from the submit route (ADR-038 unchanged), the CLAUDE.md "Commands" line flagged for the human to add.

- **ADR-050** owns the Grade aggregate's persistence (option **(i) a separate `grades` table** from ¬ßOptions known) ‚Äî a new `grades` table (PK `attempt_id` FK ‚Üí `quiz_attempts`; `score INTEGER NOT NULL`, `weak_topics TEXT NOT NULL DEFAULT ''`, `recommended_sections TEXT NOT NULL DEFAULT ''`, `graded_at TEXT NOT NULL`); a new nullable `quiz_attempts.grading_error TEXT` column (additive `ALTER TABLE ADD COLUMN` via `_apply_additive_migrations`'s `PRAGMA table_info` check + the `CREATE TABLE quiz_attempts` block in `_SCHEMA_SQL`, mirroring ADR-037's `generation_error` recipe / ADR-041's `test_suite` / ADR-044's `test_*` / ADR-046's `preamble`); `attempt_questions.is_correct` and `.explanation` (ADR-033's NULL-until-graded columns) now writeable by the grading slice; a new `Grade` dataclass; `QuizAttempt` gains `grading_error: str | None`; `AttemptQuestion` gains `is_correct: bool | None` and `explanation: str | None`; six new persistence functions (`list_submitted_attempts`, `mark_attempt_grading`, `mark_attempt_graded`, `mark_attempt_grading_failed`, `save_attempt_grade`, `get_grade_for_attempt`) under `app/persistence/quizzes.py`, re-exported from `app/persistence/__init__.py`; the **`test_passed` ‚Üí `is_correct` mapping** (`True`‚Üí1, `False`‚Üí0, NULL/non-`'ran'`‚Üí0 ‚Äî failure-to-pass = not correct, with the explanation surfacing the failure mode honestly per MC-5's spirit); the **score cross-check** (recomputed from `SUM(is_correct)` inside `save_attempt_grade`'s transaction, not taken verbatim from the workflow ‚Äî the ¬ß8 truth is the runner's verdict); `weak_topics` and `recommended_sections` persist as `'|'`-delimited TEXT mirroring `questions.topics` (ADR-033); **the relational `topics` table + `question_topics` join stays deferred to the composition slice** (the first thing that *queries* across Topics ‚Äî MC-8's fresh-portion-driver needs the cross-Section / cross-Attempt query, which is one slice later; the migration is additive per ADR-022); no `user_id` (MC-7); SQL stays under `app/persistence/` (MC-10); additive migration per ADR-022 (no migration-trigger fires; ADR-033's "n-tables threshold ‚Üí per-module schema-fragment refactor" forecast is *closer* ‚Äî sixth Quiz-domain table ‚Äî but not yet triggered, and the composition slice may trigger it).

- **ADR-051** owns the graded-state and grading-failed-state rendering on the take page ‚Äî `quiz_take.html.j2` gains `{% elif attempt.status == 'graded' %}` and `{% elif attempt.status == 'grading_failed' %}` branches (the existing `in_progress` and `submitted` renders unchanged); the `graded` branch renders an aggregate `<section class="quiz-take-grade">` block (score, Weak Topics list, recommended Sections list with `#section-{n-m}-end` anchor links per ADR-031), per-Question the existing read-only blocks unchanged plus a new `<div class="quiz-take-explanation">` and a correctness indicator via `.quiz-take-question-correct` / `.quiz-take-question-incorrect` modifier classes; the `grading_failed` branch renders an honest `<section class="quiz-take-grading-failed">` block with the primary failure message + a collapsible `<details>` exposing `grading_error` (the author *is* the learner; honesty over hiding); no fabricated Grade anywhere (MC-5); no submit form / no "Run tests" button on graded/failed Attempts (these are `in_progress`-only); new `.quiz-take-grade-*` / `.quiz-take-explanation` / `.quiz-take-question-correct` / `.quiz-take-question-incorrect` / `.quiz-take-grading-failed` rules in `app/static/quiz.css` (reusing the `quiz-take-*` namespace per ADR-008; no new file; no `base.css` change); **the visual vocabulary across runner and Grade is unified** ‚Äî the runner's `.quiz-take-results-pass` / `.quiz-take-results-fail` green/red palette extends to the Grade's correct/incorrect indicators (the runner's verdict and the Grade's correctness are the same truth per ADR-050's mapping; identical color vocabulary is the architecturally honest render); the existing take-page route reads `get_grade_for_attempt(attempt.attempt_id)` for a `graded` Attempt and passes the `grade` context var to the template; no new route, no signature change; no JavaScript needed (the `<details>` toggle is native HTML5). Realizes ADR-049's "notified-by-state-flip" call ‚Äî the take page's render branching from `submitted` to `graded` on next `GET .../take` is the ¬ß8 Notification mechanism for this slice.

**Shapes chosen vs the ¬ßOptions known sketches:**

- **The processor:** mirror ADR-037 exactly (a `process_quiz_attempts` module, `aiw run` subprocess, the documented downstream surface) ‚Äî **chosen** as forecast.
- **The grading workflow's authoring surface:** a CS-300-owned `WorkflowSpec` module at `app/workflows/grade_attempt.py` (mirroring `question_gen.py`) ‚Äî **chosen** as forecast, with the schema tightening (no `is_correct` field on `QuestionGrade`).
- **The Grade aggregate's home:** a `grades` table ‚Äî **chosen** as forecast (over the nullable-columns-on-`quiz_attempts` alternative; clean PK 1:1, keeps `quiz_attempts` lean, forward-flexible).
- **The Topic vocabulary:** stays as `questions.topics` (delimited TEXT) plus `grades.weak_topics` (delimited TEXT) for now ‚Äî **the relational `topics` table is deferred to the composition slice**. The Topic-vocabulary-source decision (curriculum-side artifact, Python module, seeded `topics` table) stays unmade until the consumer's query needs are concrete.
- **The Notification entity:** **deferred to a follow-on slice** (departing from ADR-037's literal forecast that the active Notification entity "ships with the grading slice"); the take-page state flip is the ¬ß8 Notification mechanism for this slice per ADR-049's reasoning. The follow-on slice's `/design` owns the schema, the surface placement (chrome badge in `base.html.j2` per ADR-008's forecast vs rail panel vs page), and the `seen_at` lifecycle. Recorded as a tracking note for the next `/next` cycle; not filed as a separate project_issue at this gate (the deferral is conformant per ADR-035 ‚Äî describing what's built, not what won't be built ‚Äî so a new project_issue isn't necessary; if the next `/next` cycle picks up the Notification slice, it'll see ADR-049's deferral and proceed).

## Question

The Quiz pillar's planned sequence (from TASK-014's forecast, restated in TASK-015): schema + read surface (TASK-013) ‚Üí generation (TASK-014) ‚Üí the Quiz-taking surface (TASK-015) ‚Üí **the grading workflow** ‚Üí the replay-+-fresh composition. TASK-015 ships a `submitted` Attempt ‚Äî a `quiz_attempts` row `submitted` + one `attempt_questions` row per Question carrying the learner's `response` code (`is_correct` / `explanation` NULL) ‚Äî and the submit route is written so the grading slice's processor requires *no change* to it (it already does "record and return", mirroring the generation processor / `requested`-row pattern, ADR-037). The grading slice has to decide:

1. **The out-of-band grading processor.** Mirroring ADR-037's `python -m app.workflows.process_quiz_requests` (poll-and-process `requested` Quizzes): a `python -m app.workflows.process_quiz_attempts` (or similar) that polls `submitted` Attempts, sets `grading`, invokes a new `ai-workflows` grading workflow via the `aiw run` CLI subprocess (ADR-036's downstream surface ‚Äî the workflow gets the Quiz's Questions + the learner's code per Question; returns per-Question correctness + explanation + the Grade aggregate's components), persists the result via `app.persistence.*` (the grading-slice persistence functions ‚Äî `mark_attempt_grading` / `mark_attempt_graded` / `mark_attempt_grading_failed` / `save_attempt_grades` or similar, mirroring ADR-037's `mark_quiz_*` and ADR-039's `start_attempt`/`submit_attempt`), sets `graded`, and (with the Notification entity) notifies the learner. On failure: `grading_failed`, no fabricated Grade (MC-5), the detail recorded (a `quiz_attempts.grading_error TEXT` column, additive ‚Äî mirroring ADR-037's `quizzes.generation_error`). The submit route does **not** invoke it (MC-4 ‚Äî ADR-038 ¬ßThe route pair).
2. **The Grade aggregate's schema** (manifest ¬ß8: "per-Question correctness, per-Question explanation, an aggregate score, identified Weak Topics, recommended Sections to re-read"). ADR-033 deferred it "to the grading task" and noted both shapes are additive: a `grades` table (PK `attempt_id`, FK to `quiz_attempts`, `score`, `weak_topics TEXT`, `recommended_sections TEXT`) **or** nullable columns on `quiz_attempts` (`score`, `weak_topics`, `recommended_sections`). The per-Question portion (correctness + explanation) already has a home ‚Äî `attempt_questions.is_correct` / `.explanation` (ADR-033; the rows created at Attempt start by ADR-039) ‚Äî which the grading slice `UPDATE`s.
3. **The Weak-Topic-identification path** and the relational Topic vocabulary. ADR-033 / ADR-036 deferred the `topics` table + `question_topics` join + the Topic-vocabulary-source decision "to the Weak-Topic-identification slice" (the first thing that *queries* across Topics ‚Äî which is this slice: "identified Weak Topics" comes from aggregating which Topics the learner got wrong, which needs Topics as first-class queryable entities). The migration from `questions.topics` (the `'|'`-delimited TEXT column ADR-033 ships) to the relational form is additive (ADR-036): `CREATE TABLE topics`, `CREATE TABLE question_topics`, backfill from the delimited column ‚Äî a data migration, not a schema change. The Topic-vocabulary *source* (a `content/`-derived file? a Python module? a `topics` table seeded from a source?) is part of this decision.
4. **The `notifications` table + a learner-visible Notification surface** (manifest ¬ß8: "a learner-visible indication that an async AI result has become available ‚Äî most commonly a Grade or a newly-generated Quiz"). ADR-037 records it "ships with the grading slice" (for the generation slice, the "notified" obligation was met by the per-Section Quiz status flipping to "Ready"; for grading, a real Notification entity is wanted ‚Äî a Grade is a more substantial result than "the Quiz is ready"). The decision: the `notifications` table's schema (a `notifications` row per available result ‚Äî `notification_id`, a type (`quiz_ready` | `grade_ready`), a reference to the Quiz/Attempt, `created_at`, a `seen_at` nullable?); where the Notification surface lives (a header/badge in `base.html.j2`'s chrome ‚Äî ADR-008 ¬ßFuture-surfaces forecast "most natural shape is a header/badge inside `base.html.j2`'s chrome region; styling lives in `base.css` because it is page chrome"); whether generation also writes a `notifications` row retroactively (ADR-037's per-Section status flip stays as the in-context view; the Notification is the out-of-context alert).

## Options known

These are sketches, not commitments ‚Äî the grading slice's `/design` decides:

- **The processor:** mirror ADR-037's `process_quiz_requests` exactly (a `process_quiz_attempts` module, `aiw run` subprocess, the documented downstream surface, not the framework's internal `_dispatch.run_workflow`) ‚Äî vs. a combined "process all pending AI work" command ‚Äî vs. an in-request synchronous grade (**ruled out by MC-4 / manifest ¬ß6** ‚Äî not an option).
- **The grading workflow's authoring surface:** a CS-300-owned declarative `WorkflowSpec` module at `app/workflows/grade_attempt.py` (mirroring ADR-036's `question_gen.py` ‚Äî one `LLMStep` + a `response_format` validator; `GradeAttemptInput` carrying the Questions + the learner's code per Question; `GradeAttemptOutput` carrying per-Question correctness + explanation + the aggregate's components), loaded via `AIW_EXTRA_WORKFLOW_MODULES` ‚Äî vs. extending `question_gen.py` (rejected ‚Äî a different workflow).
- **The Grade aggregate's home:** a `grades` table (PK `attempt_id`) ‚Äî vs. nullable columns on `quiz_attempts` ‚Äî both additive (ADR-033); the architect's forecast leans `grades` table (the aggregate is a distinct entity with a clean 1:1 to the Attempt, and a separate table keeps `quiz_attempts` lean).
- **The Topic vocabulary:** a `topics` table + `question_topics` join, backfilled from `questions.topics`, the vocabulary seeded from a `content/`-side source (consistent with manifest ¬ß8's "Topics form a project-wide vocabulary maintained alongside Chapter content" ‚Äî i.e. a curriculum artifact, sourced like the LaTeX corpus) ‚Äî vs. a Python module like `app/designation.py` ‚Äî `/design`'s call; the migration is additive either way.
- **The Notification entity:** a `notifications` table + a header/badge in `base.html.j2` (ADR-008's forecast) ‚Äî vs. a rail-resident Notification panel ‚Äî `/design`'s call.

## Constraints

- **Manifest ¬ß6 / ¬ß7 / MC-4:** AI work is asynchronous from the learner's perspective ‚Äî grading runs out-of-band, never in the submit request; the learner is notified when results are ready (Notification entity).
- **Manifest ¬ß6 / MC-5:** AI failures are visible, never fabricated ‚Äî `grading_failed` is a real state (ADR-033's enum already names it); no placeholder Grade, no fabricated correctness; the detail recorded (additive `grading_error` column).
- **Manifest ¬ß4 / MC-1 / ADR-036:** AI invocation goes through `ai-workflows` (the `aiw run` CLI subprocess ‚Äî the documented downstream surface); no direct LLM SDK use; the grading workflow is a CS-300-owned `WorkflowSpec` module under `app/workflows/`; CS-300 code persists the result via `app.persistence.*`.
- **Manifest ¬ß7 / MC-8:** "every post-first Quiz contains both replayed wrong-answer Questions and freshly-generated Questions" ‚Äî the grading slice produces the *Weak Topics* (for the fresh portion) and fills `attempt_questions.is_correct` (the wrong-answer-replay history for the replay portion); the composition slice (the one after) reads both.
- **ADR-022 migration story:** additive only (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, nullable `ALTER TABLE ADD COLUMN`); the `grades` table / `score` columns / `grading_error` column / `topics` + `question_topics` tables are all additive ‚Äî none fires the migration trigger. (The per-module schema-fragment refactor ADR-033 flagged "when the n-tables threshold makes the monolithic `_SCHEMA_SQL` block awkward" may finally be worth doing when the grading slice adds 2‚Äì3 more tables ‚Äî the grading slice's `/design` weighs it.)
- **ADR-033 / ADR-039:** the `quiz_attempts` / `attempt_questions` schema and the `attempt_questions` rows created at Attempt start are consumed unchanged; the grading slice `UPDATE`s `is_correct` / `explanation` on the existing rows and transitions `quiz_attempts.status` `submitted` ‚Üí `grading` ‚Üí `graded`/`grading_failed`.
- **ADR-038:** the submit route is "record and return"; the grading processor builds on the `submitted` Attempts it leaves; no change to the submit route is needed.
- **MC-7:** single-user ‚Äî no `user_id` on the Grade / Notification / Topic tables.
- **ADR-008:** the Notification surface, if it lives in `base.html.j2`'s chrome, is `base.css` (page chrome); the prefix posture extends.
