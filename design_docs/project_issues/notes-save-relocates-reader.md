# Saving a Note relocates the reader to the chapter top — the Notes-form PRG round-trip has the same snap problem ADR-031 fixed for the completion toggle

**Status:** Open — but the decision is recorded in **ADR-032 (`Proposed`, deferred)**; this file is a tracking breadcrumb only, not an open question awaiting `/design`.
**Surfaced:** 2026-05-11 (post-TASK-012 `966b824` — the TASK-012 Playwright finding that Chromium resets `scrollY` to `0` on a fragment-less `POST → 303 → GET /lecture/{chapter_id}` navigation applies to the Notes-form save, which is exactly that navigation shape).
**Decide when:** a future task that touches the Notes form (`_notes_rail.html.j2` / the `POST /lecture/{chapter_id}/notes` route) or the page layout — bundle with that task per the standalone-Notes-task-is-the-wrong-shape precedent (TASK-009/010/011 rejections; `notes-surface-rhs-rail-supersedure-of-adr028.md` "Decide when" guidance). Not a standalone task.

**Primary artifact: `design_docs/decisions/ADR-032-notes-save-must-not-relocate-the-reader.md` (`Proposed`).** The human asked for an ADR specifically (*"make new adr for notes forcing anchor on save"*) and chose to defer (*"log it and we'll deal with it in the future not now"*). ADR-032 records: (1) the requirement — saving a Note must not relocate the reader from their reading position; (2) the forecast mechanism — client-side JavaScript (the project's first), because no clean no-JS / CSS-only mechanism exists for the general case (the ADR-031 `#section-{n}-end` + `scroll-margin-top` trick worked only because `.section-end` is a static element at a known reading position; the Notes form is in an always-visible sticky rail used from an arbitrary scroll position, so there is no static anchor for "where the reader was"); (3) two candidate JS shapes left for a future task's `/design` — (a) async `fetch()` submit, no navigation, list updated in place (cleanest; could extend to the completion toggle for symmetry), or (b) JS scroll-restore (`sessionStorage` + `window.scrollTo`) around the existing PRG round-trip (minimal; still flickers); (4) that the no-JS-posture supersedure this implies (superseding the no-JS commitments in ADR-023/025/027/028/029/030/031) is **not effected by ADR-032** — it is a separate human-accepted act at the moment a task implements the JS, so the no-JS posture stays in force until then.

ADR-032 also corrects a mistaken assumption ADR-023 / ADR-028 / ADR-029 / ADR-030 / ADR-031 carried — "the Notes form's PRG round-trip does not have the snap problem because the form is in the rail, not at the bottom of a tall Section." That reasoning confused *where the form is* (a sticky rail near the top) with *where the reader is* (anywhere in the main column — the rail is sticky precisely so the reader need not scroll back). The redirect *shape* is the same as the completion toggle's; the *user-perceived effect* (reader snapped to chapter top) is the same too. ADR-032 does **not** supersede ADR-023 / ADR-028 / ADR-029 — their decisions (route, validation, PRG, multiple-Note order, RHS-rail placement, the `rail_notes_context` variable, no-edit/delete/Markdown, textarea sizing, the "visibility follows scroll-position-cost" principle) all stand; only the factual side-note is corrected, and that correction lives in ADR-032's context section (not as an in-place edit to those ADRs, per `AS-1`).

## Why this is a breadcrumb, not an open `/design` question

The *requirement* is decided (ADR-032 §Decision). Only the *mechanism between candidates (a) and (b)* is open, and that is a future task's `/design`, not a standing project_issue — together with the build/serve-story decision (flat static `.js`, consistent with ADR-003's "no build step"?), the `<script>`-loading decision (`base.html.j2`?), the progressive-enhancement-fallback decision (retain the no-JS PRG path?), the Playwright-JS-on/JS-off-test-variant decision, the no-JS-posture supersedure ADR, and the "extend to the completion toggle for symmetry?" decision. This file exists so the parked item shows up in `ls design_docs/project_issues/`; the work is in ADR-032 and in the future task's `/design`.

## Resolution

When a future task implements ADR-032, that task's `/design` resolves this file — mark it `Resolved by ADR-NNN` (the no-JS-posture supersedure ADR the task writes) and note which mechanism shape ((a) or (b)) was chosen. ADR-032 itself moves from `Proposed` to `Accepted` (or is superseded) at that time.
