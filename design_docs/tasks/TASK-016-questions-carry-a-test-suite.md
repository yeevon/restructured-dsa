## What and why

The amended manifest (human edit, 2026-05-12) tightened the Quiz pillar: §8 **Question** now reads "A Question is a hands-on coding task: it asks the learner to implement a concept from the Section in code, **and it carries a test suite the learner runs in-app to verify the implementation**"; §6 added "**Code is written, run, and tested within the application**"; §8 **Grade** now defines per-Question correctness as "**determined by whether the learner's code passed the Question's tests**". But the Questions the generation slice produces (TASK-014, ADR-036) carry only a `prompt` and `topics` — **no test suite**. That is a gap relative to the amended manifest, and it is *load-bearing*: until Questions carry a test suite, the learner can't run tests in-app (§6), the Attempt can't carry test results (§8), and the Grade can't be computed from test outcomes (§8). The next loop step — grading — is *defined* in terms of "the Question's tests"; those tests don't exist yet.

This task makes them exist. It is the smallest useful step toward closing the gap: **the `question_gen` workflow emits a test suite per Question; the Question Bank persists it; the per-Section surface reflects it (minor or not at all).** Specifically:

1. **The workflow output schema gains a test suite per Question.** ADR-036's `GeneratedQuestion` (currently `prompt: str` + `topics: list[str]`) gains a test-suite field — a representation of runnable test code that verifies an implementation of the Question's coding task. The `LLMStep`'s prompt is extended to instruct the LLM to emit that test suite alongside each Question's prompt; the `response_format` validator validates it. The test-suite field's type must be such that it can **only** hold test code — never an option list, a true/false key, a "describe" prompt, or a recall answer (manifest §5/§7; mirrors ADR-036/ADR-033's no-non-coding-field posture). `app/workflows/question_gen.py`'s imports stay `ai_workflows.*` + `pydantic` only (MC-1); AI invocation is still the existing `aiw run` CLI subprocess (ADR-036) — **no new workflow, no new AI surface**.
2. **The Question Bank persists the test suite.** A `questions.test_suite` column (additive `ALTER TABLE ADD COLUMN`, nullable, per ADR-022) — or a `question_tests` table if `/design` judges the representation warrants rows-per-test-case (additive `CREATE TABLE IF NOT EXISTS`; `/design`'s call). The processor's persistence call (`add_questions_to_quiz`/successor, ADR-037) gains the test-suite payload per Question; the `Question` dataclass (or a new one) carries it; an accessor returns it. `import sqlite3` + SQL literals stay under `app/persistence/` (MC-10); no `user_id` (MC-7).
3. **The generation-failure path covers a bad test suite.** An LLM that returns a missing/empty/malformed test suite for a Question causes the *existing* `generation_failed` path (ADR-037) to fire — the processor **does not** synthesize a placeholder test suite to ship the Question anyway (MC-5: AI failures are surfaced, never fabricated; a Question without a real test suite is not a shippable Question now that §8 says it carries one). `/design` decides the "what counts as a valid test suite from the LLM" sanity check; the failure on a bad one is `generation_failed`, zero `questions` rows persisted, no placeholder anywhere.
4. **The per-Section surface — minor or unchanged.** The `.section-quiz` block's `ready` entry *may* gain a small honest caption (e.g. "Ready — N Questions, each with tests") — or stay unchanged, treating "the Question carries a test suite" as a structural fact not yet surfaced. `/design`'s call; any caption must **not** imply the learner can *run* the tests yet (that is the next slice). If a surface change is made it reuses `section-quiz-*` in `lecture.css` (ADR-008); no new CSS file.
5. **Tests** — pytest for the workflow-schema change (validates a well-formed test suite; rejects a malformed/missing one), the persistence change (test suites are written and round-trip), the processor wiring (test suites persist from a stubbed `aiw run` artefact; a Question missing/empty in its test-suite field → `generation_failed`, zero rows, no placeholder — MC-5); the MC-10 boundary grep extended; the MC-1 boundary grep over `app/workflows/` still clean; no `user_id` anywhere new (MC-7); the new column/table additive (ADR-022). Playwright only if the per-Section caption changes.

It deliberately does **not** ship: the **in-app code-execution / test-runner surface** (the "Run tests" affordance, the sandbox for running arbitrary learner code, the `attempt_questions` test-result columns, displaying test outcomes to the learner) — that is the **next** slice (and it pulls in a substantial sandboxing/security ADR); the **grading workflow / Grade aggregate / Weak Topics / `notifications`** (later still — and now keyed off test results, per the amended §8); the **replay-+-fresh composition** for post-first Quizzes (later); any change to the take surface beyond what it *already* renders (it renders `attempt_questions`, which this task does not change — the test *suite* lives on the `Question`); changes to the Notes surface, the completion toggle, or the parser (all settled); the parked ADR-032 (not touched); the two Open parser-fidelity issues; lint/type-check tooling. **`/design` MUST NOT smuggle in a code-runner, the `attempt_questions` test-result columns, the grading workflow, or the Grade aggregate — those are later slices.**

It advances the primary objective by **making the manifest's "the Question carries a test suite" true** — the precondition for the in-app test-runner (§6), which is the precondition for a §8-conformant Grade (correctness = code passed tests), which is the precondition for the Weak-Topic feedback the composition slice closes the loop with. It is sized as a focused slice — a workflow output-schema extension + an additive persistence change + the processor wiring + tests + maybe a one-line caption — smaller than TASK-014 (no new workflow, no new processor, no new async machinery; an extension of the existing one).
