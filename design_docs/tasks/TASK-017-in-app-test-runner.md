# TASK-017: In-app test runner — the learner runs a Question's test suite against their submitted code, in-app, in a sandboxed subprocess, and sees pass/fail (slice 4 of the Quiz pillar; the precondition for a §8-conformant Grade)

**Inputs read:**

- **Manifest** — §3 (Primary Objective: per-Section Quizzes whose results feed back into next-Quiz generation; this slice makes the *result* signal — "did the learner's code pass the Question's tests" — actually computable, which the grading slice (next) consumes); §4 (Secondary Objective: `ai-workflows` is the AI engine — **running a test suite is not AI work**, so this slice introduces no new `ai-workflows` workflow and no LLM call; MC-1 / MC-4 do not bind the test-run path; the secondary objective "never justifies adding a feature" — this is not a new feature, it is the second half of "Code is written, run, and tested within the application" the manifest now mandates, whose first half (the tests *exist*) shipped in TASK-016); §5 (Non-Goals — **"No live / synchronous AI results"** binds *AI* output (grading, generation, lecture audio); a *test run* is not AI output, so the run-and-see-results loop here may be synchronous — `/design` reasons this explicitly and does not reflexively copy ADR-037's async pattern; **"No remote deployment / hosted product"** bounds the sandbox's complexity ceiling — a *local* sandbox, no cloud-execution service; **"No multi-user features"** — no `user_id` on the new test-result columns or any new construct; **"No in-app authoring of lecture content"** / MC-6 — the sandbox must never be able to write under `content/latex/` (the learner's code runs in a temp working dir away from it) and must not corrupt `data/notes.db`; **"No coverage of CS-300 Chapters 1–2"** / **"No LMS features"** — orthogonal); §6 (**"Code is written, run, and tested within the application. The learner answers a Question by writing code, running it, and running tests against it — all inside the application's own interface. No external editor, no separate test runner, no copy-paste-into-another-tool workflow"** — *this is the §6 entry this task implements*: TASK-016 made the tests exist; this task lets the learner *run* them, in-app; reads as wanting a responsive write→run→see loop; **"AI work is asynchronous from the learner's perspective"** — unchanged and not eroded: this slice adds no AI; the run path is not AI; **"AI failures are visible … never fabricates a result"** — orthogonal to the *runner* (no AI in the run path), but the *spirit* applies: a runner failure — a timeout, a crash, a compile error in the test suite, a sandbox-setup failure — must be surfaced to the learner honestly ("the test run timed out" / "the test suite failed to compile" — never a fabricated "all tests passed" or "all tests failed"); "Single-user" / "A Lecture has a single source" / "Quizzes scope to Sections" / "Mandatory and Optional are honored everywhere" — orthogonal/unchanged); §7 (**"The reinforcement loop is the reason this project exists. Quiz performance MUST drive next-Quiz generation"** — loop performance is, per the amended §8, "determined by whether the learner's code passed the Question's tests"; the *raw* pass/fail signal this slice produces is what the grading slice reads to set per-Question correctness and identify Weak Topics; **"Every Question is a hands-on coding task. The learner writes code that implements a concept … Questions never ask the learner to describe, explain, recall, or choose among options"** — unchanged; the runner runs *test code* against the learner's *implementation code*; no non-coding surface introduced; **"Every Quiz Attempt, Note, and completion mark persists across sessions"** — the latest test-run result persists on the Attempt's `attempt_questions` rows (or a new table — `/design`'s call); **"Mandatory and Optional are separable in every learner-facing surface"** — unchanged; the take page already carries the designation badge per ADR-038); §8 (**Quiz Attempt** — "Carries the learner's code response for each Question, **the in-app test results for each response**, a progress status through grading, and — once graded — a Grade" — *this is the §8 entry this task implements*: the Attempt currently carries `response` (the learner's code, ADR-039) but **no test results**; this task adds the test-result storage; **Grade** — "per-Question correctness (**determined by whether the learner's code passed the Question's tests**)" — the Grade is the *next* slice; this task produces the raw test result the Grade is *defined in terms of* — whether `attempt_questions.is_correct` is set *by the runner* ("tests passed ≡ correct") or *by the grading slice reading the test result* is `/design`'s call, both defensible; **Question** — "carries a test suite the learner runs in-app to verify the implementation" — TASK-016 made `questions.test_suite` carry it; this task is the *"runs in-app"* clause; **Quiz** / **Question Bank** / **Weak Topic** / **Notification** — consumed/orthogonal; no new glossary terms forced — Quiz Attempt and Question are already in §8 with the in-app-test-results / runs-in-app language present).

## What and why

TASK-016 made a Question carry a test suite (`questions.test_suite` — a runnable test-code string, ADR-040/ADR-041). But nothing *runs* it. Manifest §6 says "**Code is written, run, and tested within the application** — The learner answers a Question by writing code, running it, and running tests against it — all inside the application's own interface", and §8 Quiz Attempt says it carries "**the in-app test results for each response**". The take surface (ADR-038) lets the learner write and submit code — but there is no "Run tests", no test-result storage, and no way for the §8-conformant Grade ("correctness **determined by whether the learner's code passed the Question's tests**") to ever be computed, because that signal doesn't exist yet.

This task makes it exist. **The learner, on the take page, sees the Question's test suite read-only (so they know the signature and edge cases), clicks "Run tests", and the application runs that test suite against their submitted code in a sandboxed subprocess — with a CPU/memory limit and a timeout — and shows pass/fail plus the test output; the latest result persists on the Attempt.** Specifically:

1. **A code-execution sandbox primitive.** A new `app/sandbox.py` (or `app/runner.py` — `/design` names it; *not* under `app/workflows/`, which is the AI path): given a Question's `test_suite` string and the learner's `response` code string, it splices them (`/design` decides how — the TASK-016 audit note flags that the generated test suites currently embed a reference implementation inline, so the splice is "replace the placeholder body" or "compile against a harness"), writes to a temp working dir well away from `content/latex/`, runs under `resource` limits (CPU time, memory cap, no fork bomb) plus a `subprocess.run(timeout=...)` (or `/design`'s chosen sandbox — a bare subprocess with `resource` limits is the forecast; `bubblewrap`/`nsjail` a reasonable upgrade; a container only if `/design` makes the case against the "no remote deployment / locally-run" simplicity ceiling), and returns a structured result (passed, output, a status — ran / timed_out / compile_error / setup_error). It **never writes under `content/latex/`** (MC-6) and does not touch `data/notes.db` (MC-7-orthogonal but: the route writes the result, not the sandbox). It imports no LLM SDK (MC-1 stays vacuously clean — this is not `app/workflows/` and there is no AI here).
2. **Test-result storage on the Attempt.** `attempt_questions` gains test-result columns — `test_passed INTEGER` (1/0/NULL), `test_output TEXT`, `test_run_at TEXT` (additive nullable `ALTER TABLE ADD COLUMN`s per ADR-022, mirroring ADR-037's `generation_error` and ADR-041's `questions.test_suite`; in the `CREATE TABLE attempt_questions` block for fresh DBs too) — or, `/design`'s call, an `attempt_question_runs` table if run-history is wanted (the forecast: denormalized columns, no new table — the "latest run" is what the surface and the grading slice need). The `AttemptQuestion` dataclass gains the fields (or a new dataclass carries them); a `save_attempt_test_result(attempt_id, question_id, ...)` function `UPDATE`s them; `list_attempt_questions` carries them through. `import sqlite3` + SQL literals stay under `app/persistence/` (MC-10); no `user_id` (MC-7).
3. **A "Run tests" route.** A new route under ADR-038's per-Quiz-take URL space (`POST .../take/run-tests` form-POST + PRG re-render, or a `fetch()`-target JSON endpoint + a small JS handler — `/design`'s call, both ADR-035-permitted; the forecast leans synchronous with a timeout — the responsive write→run→see loop §6 wants; **MC-4 does not bind it — running tests is not AI work**, and `/design` reasons that explicitly): same path-param validation as the take routes; resolves the latest `in_progress` Attempt; reads the `question_id` + the submitted code; fetches the Question's `test_suite` via a new accessor (ADR-041 §No new accessor — "the runner slice adds one"); calls the sandbox; persists the result; re-renders the take page (or returns JSON) showing pass/fail + the output. **Does not invoke grading (MC-4); does not generate a Quiz (MC-9); does not write under `content/latex/` (MC-6); a runner failure (timeout, crash, compile error) is surfaced honestly (MC-5's spirit) — never a fabricated pass/fail.**
4. **The take-surface changes.** `quiz_take.html.j2`, per `in_progress` Question: a read-only test-suite display (a `<pre class="quiz-take-test-suite">` showing `questions.test_suite`), a "Run tests" affordance (a button), and a results panel (`<div class="quiz-take-results">` showing pass/fail + the test output, empty before a run). The submit form (ADR-038) is otherwise unchanged; the `submitted`-Attempt state is unchanged unless `/design` decides to show the last test result there (forecast: leave as-is or show read-only). New `quiz-take-*` CSS in `app/static/quiz.css` (ADR-008/ADR-039 — `quiz.css` already exists; no new file, no `base.css` change). If `/design` adds JS, that's the project's first JS asset — decided on its merits in the ADR (ADR-035).
5. **Tests** — pytest for the sandbox primitive (passing code → pass; failing code → fail; infinite loop → timeout surfaced, not a hang; write attempt under `content/latex/` → blocked / corpus untouched — MC-6; uncompilable test suite → compile_error surfaced honestly), the test-result persistence round-trip, the `POST .../run-tests` route (happy path; validation; `submitted`-Attempt behavior), MC-6 / MC-7 / MC-10 boundary checks, the additive migration (ADR-022); Playwright for the rendered "Run tests" affordance, the read-only test-suite block, and the results panel after a run.

It deliberately does **not** ship: the **grading workflow / Grade aggregate / Weak Topics / `notifications`** (the *next* slice — now keyed off the test results this task produces; tracked in `quiz-grading-slice-shape.md`, which this task does **not** edit — no decision here forces it); the **replay-+-fresh composition** for post-first Quizzes (later — needs the grading slice's Weak Topics and the `attempt_questions.is_correct` wrong-answer history, which the grading slice sets from these test results); a **full in-browser IDE** ("VS Code online" was a roadmap *reference point*, not a literal spec — this slice ships a "Run tests" affordance on the existing take page, not an editor; a richer editing surface is a later, separate consideration); any change to the **per-Section `ready`-entry caption** on the Lecture page (out of scope this task by forecast — the runner-existing changes the calculus; a follow-on can revisit); the **Quiz-prompt-rendering bug** (`{{ aq.prompt }}` autoescaped — explicitly deprioritized by the user); changes to the Notes surface, the completion toggle, the parser, or the generation/take routes beyond what this slice needs (all settled); the parked ADR-032 (not touched — no Notes-form / page-grid change); lint/type-check tooling. **`/design` MUST NOT smuggle in the grading workflow, the Grade aggregate, the `notifications` entity, the replay-+-fresh composition, or a full in-browser IDE — those are later slices.**

It advances the primary objective by **making "Code is written, run, and tested within the application" true** (§6) — the learner can now write code *and run the Question's tests against it, in-app* — and by producing the raw test-result signal the §8-conformant Grade is *defined in terms of* ("correctness determined by whether the learner's code passed the Question's tests"), which the grading slice (next) reads to set per-Question correctness and identify the Weak Topics the composition slice closes the loop with. It is sized as a focused-but-substantial slice — a sandbox primitive + an additive persistence change + one route + a take-surface change + tests; the sandbox is the central decision (running arbitrary learner code is a real isolation/robustness problem, *bounded* by the single-user/single-machine/locally-run threat model — manifest §5). The next slice is **the grading slice** (the out-of-band grading processor + a new `ai-workflows` grading workflow that reads the in-app test results and produces per-Question explanations + the Grade aggregate with Weak Topics + a `notifications` entity), then **the replay-+-fresh composition** for post-first Quizzes (MC-8 in full).

## Acceptance criteria

- [ ] Given the new code-execution sandbox primitive (`app/sandbox.py` or `app/runner.py` — `/design`'s name; **not** under `app/workflows/`), when it is invoked with a Question's `test_suite` string and a `response` code string that genuinely **passes** that test suite, then it returns a structured result indicating **passed** (a `passed: True` / equivalent flag) and the test output, and the run completes within the timeout; when invoked with a `response` that genuinely **fails** the test suite, it returns **passed=False** plus the failing test output; in **neither** case does it write any file under `content/latex/` (MC-6) and in neither case is the result fabricated (the pass/fail reflects the actual test execution).

- **ADR-NNN: The in-app code-execution sandbox — the isolation mechanism, the test-suite/learner-code splice, the supported languages, and the structured result shape.** Decides:
