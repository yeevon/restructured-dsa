# TASK-002: Chapter navigation grouped by Mandatory/Optional designation

**Manifest sections this touches:** §3 Primary Objective (drives consumption — learner needs a way to *discover* what to consume), §5 Non-Goals (no LMS, no remote deployment, no in-app authoring, no mobile-first), §6 Behaviors and Absolutes ("Mandatory and Optional content are honored everywhere"; "Lectures and Notes scope to Chapters"), §7 Invariants ("Mandatory and Optional are separable in every learner-facing surface"), §8 Glossary (Chapter, Lecture, Mandatory, Optional).

**Architecture state assumed:**
- ADR-001 (Lecture source layout — `content/latex/`) — Accepted.
- ADR-002 (Chapter and Section identity — Chapter ID = file basename) — Accepted.
- ADR-003 (Rendering pipeline — FastAPI + Jinja2 + pylatexenc) — Accepted; the existing FastAPI app and Jinja2 template surface are extended (not replaced) by this task.
- ADR-004 (Mandatory/Optional designation source — `chapter_designation()` function) — Accepted; this task's grouping reads that function rather than reimplementing the rule.

**Project issues this forces:** `design_docs/project_issues/multi-chapter-source-naming.md` — the existing chapter files (`ch-01-cpp-refresher.tex` vs `ch2.tex`, …, `ch13.tex`, with `ch8.tex` absent) use two distinct naming conventions. The "Decide when:" condition for that issue is "the first task that … requires an index page covering multiple Chapters … whichever comes first." This task triggers it. The architect must resolve this issue during `/design TASK-002`.

## Goal

Add a learner-facing navigation surface that exposes every Chapter present in `content/latex/` grouped into two sections — Mandatory and Optional — with each Chapter rendered as a link to its existing Lecture page. The grouping is the separation: each designation appears as its own labeled section containing only the Chapters that belong to it. No filter mechanism, no toggle, no query parameter. The Chapter 1 Lecture page that shipped in TASK-001 becomes reachable from this surface rather than requiring URL guessing, and Chapters 2–13 become discoverable for the first time.

## Why now

- Manifest §3 names "consumption" as the primary objective. Without navigation, the only way to consume Chapter 1 is to type its URL by hand, and Chapters 2–13 (which already exist on disk) are entirely undiscoverable. The Lecture surface from TASK-001 has zero discovery affordance.
- Manifest §7 ("Mandatory and Optional are separable in every learner-facing surface") is binding for any surface that exposes more than one Chapter. Grouping Chapters under their designation labels satisfies the invariant by *construction* — there is no view in which the split is hidden, because the layout itself is the split.
- Manifest §6 ("Mandatory and Optional content are honored everywhere") is currently honored only at the per-Lecture badge. A navigation surface without designation grouping would violate it the moment it lists the second Chapter.
- The open project issue `multi-chapter-source-naming.md` is ripe: its "Decide when:" condition triggers exactly here, and resolving it now (while no persistence references Chapter IDs as foreign keys) is far cheaper than resolving it later. Manifest §7 forecasts that cost: "Every Quiz Attempt, Note, and completion mark persists across sessions" — once persistence lands and IDs become foreign keys, a rename is a data migration.
- This task is a strict prerequisite for *validating* Chapters 2–13 as Lecture pages (a future task). Validation requires reaching each Chapter's URL; navigation lets the human reach them without URL guessing. The navigation does not require the Chapters to render successfully — it only links to them, and a broken Chapter fails loudly under ADR-003's existing contract. So this task does not depend on Chapter 2–13 readability being established first; the dependency runs the other way.

## Acceptance criteria

- [ ] Given the FastAPI app is running, when the human navigates to a documented landing route (e.g., `GET /`, or — at the architect's discretion in `/design` — a navigation rail rendered alongside every Lecture page), then a navigation surface is reachable that exposes every Chapter present in `content/latex/` grouped into two visibly-labeled sections: Mandatory and Optional.
- [ ] Given the navigation surface, when the human inspects the two sections, then every Chapter under "Mandatory" has its designation derived from `chapter_designation()` and is in fact Mandatory; every Chapter under "Optional" has its designation derived from `chapter_designation()` and is in fact Optional. The grouping is computed at render time, not hardcoded as chapter-number ranges or as a static list of IDs (MC-3).
- [ ] Given the navigation surface, when the human clicks the link for a listed Chapter, then the existing Lecture route from TASK-001 (`GET /lecture/{chapter_id}`) is reached. The link target is computed from the Chapter ID, not hand-coded per Chapter.
- [ ] Given a Chapter's Lecture page (e.g., the Chapter 1 page that shipped in TASK-001), when the human is on that page, then the navigation surface is reachable from it (via a link, a rail, or whichever shape the design ADR fixes). The Lecture page is no longer a navigation dead end.
- [ ] Given the human runs the dev command twice in a row without editing source, then both runs of the navigation surface produce equivalent rendered output (determinism, consistent with ADR-003's determinism rule for the Lecture pipeline). Chapter ordering within each designation group is deterministic and documented in the design ADR (e.g., by Chapter number ascending).
- [ ] Given the existing chapter files in `content/latex/` use inconsistent naming (`ch-01-cpp-refresher.tex` vs `ch2.tex`, … `ch13.tex`, with `ch8.tex` absent), when the navigation surface runs, then the resolution decided in `/design TASK-002` is honored: either (a) all files have been renamed to a single canonical convention as a content-management step the human performs outside the application, or (b) the discovery code tolerates both conventions per a parser/discovery rule fixed in an ADR. The implementation does not silently pick one; the architect chooses and the ADR records.
- [ ] Given the human inspects the navigation surface, when a Chapter file is malformed in a way the discovery layer cannot derive a display label from (e.g., no `\title{…}` macro, or a Chapter ID that does not match the canonical pattern decided by ADR), then that Chapter row fails loudly — either the surface shows the error explicitly for that row, or the surface refuses to render with a structured message — but never silently fabricates a title or designation. (Honors ADR-002's and ADR-004's fail-loudly rules.)
- [ ] Given a learner-facing surface now exists across multiple Chapters, when the manifest-conformance skill runs, then MC-3's architecture-portion check (no chapter-number literals outside `app/designation.py`) passes against the navigation implementation. (MC-3 became active when ADR-004 was Accepted; this task is the first place a multi-Chapter-surface violation could land.)
- [ ] Given the source root is `content/latex/`, when discovery enumerates Chapters, then no path under `content/latex/` is opened for write. (MC-6, already active per ADR-001.)

## Mandatory / Optional impact

YES — this task introduces a learner-facing surface that displays Chapter content metadata across multiple Chapters. The Mandatory-only view is preserved by the layout itself: the navigation surface renders two labeled sections, "Mandatory" and "Optional," each containing only the Chapters whose designation matches. The learner reads only the Mandatory section to see Mandatory-only content; the Optional section is visually separated and labeled. The split is structural, not behind a control. The rule for separating Mandatory from Optional is: every Chapter discovered under `content/latex/` is routed through `chapter_designation()` (ADR-004), and the result places the Chapter in exactly one of the two sections. No Chapter appears in both sections; no Chapter appears in neither. If `chapter_designation()` raises (per ADR-004's fail-loudly rule), the navigation fails loudly per the acceptance criterion above.

## Async / run contract

NO — this task involves no AI-service work. There is no grading, no Question generation, no TTS, no Weak Topic extraction, and no `ai-workflows` integration. The navigation surface is rendered synchronously from filesystem state via the same Python parsing pipeline ADR-003 already established. The async/run contract checks do not apply.

## Architectural decisions expected

- **Navigation surface shape.** Whether the grouped-by-designation surface is a dedicated landing page (`GET /`), a persistent left-hand rail rendered alongside every Lecture page, both, or some other shape. The architect picks one shape and records it. The shape is design-ADR territory; the task fixes the goal (grouped, navigable, both designations visible as sections with per-Chapter links from a single surface), not the chrome.
- **Chapter discovery mechanism.** How the application enumerates Chapters under `content/latex/` (filesystem scan at request time, scan at startup with cache, explicit registry). Whatever the architect picks must honor MC-6 (read-only against `content/latex/`) and ADR-003's determinism rule.
- **Display-label source.** Whether the per-Chapter label shown in the navigation comes from each `.tex` file's `\title{…}` macro (already partially extracted per ADR-003 for the Lecture page header) or from the Chapter ID itself, with the fallback rule when `\title{…}` is missing or unreadable.
- **Within-group ordering.** What deterministic order the Chapters appear in within each designation group (e.g., by extracted Chapter number ascending). Documented so the determinism acceptance criterion is enforceable.
- **Resolves `design_docs/project_issues/multi-chapter-source-naming.md`.** The architect must produce an ADR resolving this open issue during `/design TASK-002`, since enumerating Chapters across two naming conventions is exactly the case the issue exists to settle.
- (These become ADRs during Mode 2; listing them here forecasts work, not commits to it.)

## Out of scope (this task)

- A Mandatory-only filter, toggle, query parameter, or any other state-bearing affordance for hiding Optional Chapters. The grouping is the separation; an additional filter would be redundant and was explicitly rejected by the human (see "Alternatives considered (task direction)" below).
- Rendering Chapters other than Chapter 1 successfully. The existing renderer was validated against `ch-01-cpp-refresher.tex` only. Whether Chapters 2–13 actually render cleanly under the same pipeline is a *separate* problem (file-by-file parser robustness against the rest of the corpus). This task only requires that the navigation surface *lists* and *links to* every Chapter present on disk; clicking a link for a Chapter whose source the parser cannot handle should fail loudly per the existing ADR-003 contract, but fixing that failure is not this task's responsibility. (A natural follow-up task: per-Chapter Lecture validation across Chapters 2–13.)
- Notes (entity, UI, storage). Manifest §8 names Notes; they are deferred until a task forces them.
- Quizzes (entity, UI, generation, attachment). The reinforcement loop is the manifest's center of gravity but cannot be tackled before persistence and AI integration ADRs land.
- Persistence layer of any kind (no DB, no on-disk state across runs). MC-10 stays dormant.
- AI integration of any kind. `ai-workflows` is not yet wired in.
- Audio / listening consumption mode for the Lecture (manifest §7 contemplates it but no task forces it yet).
- Search, full-text indexing, or cross-Chapter linking beyond the per-row Lecture link.
- Section-level entries on the navigation surface. The surface is per-Chapter; manifest §8 anchors Lectures and Notes at the Chapter level. Section-level navigation continues to live inside each Chapter's Lecture page (via ADR-002 anchors).
- Completion state, progress indicators, "next" affordances. Completion lives at the Section level (manifest §7) and depends on persistence; out of scope until those land.
- Styling polish, dark mode, mobile layout (manifest §5: no mobile-first).
- Auth, multi-user features (manifest §5).
- "Recently viewed," history, breadcrumbs, or any session-bearing affordance.

## Alternatives considered (task direction)

Four materially different directions were considered. The first was rejected on human pushback after the original TASK-002 proposal; the others are alternative *next tasks*.

1. **(Originally proposed; REJECTED on human pushback) Chapter index page with a Mandatory-only filter (toggle / checkbox / query param).** The original TASK-002 proposal exposed Mandatory/Optional via a learner-controlled filter affordance — flip a control, hide Optional Chapters. Rejected by the human: "mandatory filter is overkill … we can just have a LHS rail with Mandatory then links to chapters 1-6, and Options and links to chapters 7+ and that will cover this and gives easy chapter navigation … if the core of this task is filter its a no go." The reasoning is sound: at the current corpus size, a stable Chapter-level attribute does not need a control to expose; the layout itself is the exposure. A filter adds state (the filter's on/off value), a mechanism (toggle/query param), and surface area for bugs (the default view, the "filter applied but no Chapters in this group" empty state, the URL-shareability question), all without any clearer learner outcome than grouping already provides. A grouped layout satisfies §7 by construction. **The chosen direction (this task) drops the filter entirely.**

2. **(Chosen) Add a navigation surface that groups Chapters by designation.** Smallest vertical slice that produces a new learner-facing surface and unblocks discovery for every Chapter on disk. Forces resolution of the ripe `multi-chapter-source-naming` project issue at the cheapest possible moment (no persistence yet → no foreign-key migration cost). Exercises ADR-004's M/O mapping at multi-Chapter scope, where manifest §7's "separable in every learner-facing surface" invariant most strongly bites. Self-contained: doesn't require persistence, AI, or `ai-workflows` integration. Strictly smaller than the rejected filter version: removes the filter mechanism (state + control + default-view rule + toggle behavior), keeps everything else.

3. **Render Chapters 2 through 13 as Lecture pages (multi-Chapter Lecture coverage), and defer the navigation surface.** Bigger primary-objective win in absolute content-volume terms. Rejected as the next task because: (a) it forces the same naming issue without delivering a new *kind* of surface — it only widens an existing one; (b) the Chapters 2–13 source files have not been validated against the existing parser, and touching all 11 files in one task risks the task's "single session" budget; (c) without navigation, the additional Chapters would still be undiscoverable, so the marginal consumption gain per chapter is gated by URL guessing; (d) the human's pushback explicitly contemplated parking the navigation task — but only if validation came first — and the dependency runs the other way: validation needs navigation to be reachable, navigation does not need validation to succeed (broken Chapters fail loudly per ADR-003). Better as a follow-up: do the navigation surface first, then per-Chapter validation as smaller tasks.

4. **Park this task entirely until Chapters 2–13 are individually validated as renderable.** The human flagged this as a possibility ("not sure if that should be done instead this task 2 or if should be parked till other chapters are includinged"). Rejected as the next task because the navigation surface does not depend on Chapter 2–13 renderability — it depends only on file enumeration plus per-Chapter designation, both of which already work today. Parking the navigation surface would mean continuing to URL-guess for every Chapter 2–13 validation attempt, which is friction that compounds across 11 future validation passes. The cost of building navigation now is bounded; the cost of validating 11 Chapters without navigation is paid once per Chapter. Building navigation first amortizes that cost.

Two additional directions considered and rejected for the same reasons recorded in the original TASK-002 (kept here so the rejection record survives the reshape):

5. **Build the persistence layer (define and stand up a database for forthcoming Quiz / Notes / Attempt entities).** Highest leverage for future work but is a horizontal layer with no learner-facing surface — explicitly the wrong shape per the architect's "vertical slices, not horizontal layers" rule. Persistence ADRs should land in service of a feature task that *needs* persistence (Notes is the natural first such task), not as preemptive infrastructure.

6. **Notes-first (introduce persistence by adding a Notes feature).** Notes are a real manifest-anchored entity (§8) and would naturally drag in a persistence ADR. Rejected as the *next* task because: (a) navigation is a strict prerequisite for Notes-with-Chapter-binding to be navigable, since Notes scope to Chapters and the user needs to *find* the Chapter to write a Note against it; (b) doing Notes before navigation would still leave us URL-guessing for Chapters 2–13. Navigation unblocks Notes; Notes does not unblock navigation.

A seventh idea — start the Quiz pipeline by defining Question, Quiz, Attempt, Grade schemas and one happy-path AI-generated Question — was rejected as too large for one task: it forces persistence + `ai-workflows` integration + Quiz UI + grading + reinforcement-loop composition simultaneously. Manifest §7 (the reinforcement loop) is sacred but is reached through smaller, separately-gated steps.
