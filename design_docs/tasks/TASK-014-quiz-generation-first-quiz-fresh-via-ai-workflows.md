## What and why

TASK-013 shipped slice 1 of the Quiz pillar — the Quiz domain schema (ADR-033) + the per-Section Quiz read surface + a `POST .../quiz` route that records a `requested`-status Quiz row "with no processor — explicitly the next task's job." Manifest §7 calls the reinforcement loop "the reason this project exists" and §4 names `ai-workflows` "the only architectural commitment made at the manifest level"; this task finally exercises both: it adds the **processor** — the `ai-workflows` integration that walks a `requested` Quiz through `generating` to `ready`, generating a fresh set of hands-on coding-task Questions for the Section, asynchronously (the click already returned; the result arrives later; the learner is notified). It is slice 2 of the Quiz pillar's planned 3 (TASK-013 forecast: "this task (schema + read surface, no AI) first, then generation (the `ai-workflows` integration + async delivery + Notification) next, then composition (replay-+-fresh) after generation exists").

But "Quiz generation in full" — fresh-Question generation + the replay-+-fresh composition for post-first Quizzes + Weak-Topic targeting — is multi-session by construction and presupposes Attempt history (which needs the Quiz-taking surface and the grading workflow, both later slices). So this task ships the **first-Quiz, fresh-only** case, which is *exactly what the manifest mandates for a first Quiz* ("contains only fresh Questions (the bank is empty)" — §7) — not a violation of MC-8 (which forbids fresh-only Quizzes *after the first*). The slice it ships:

1. **The `ai-workflows` integration.** The AI-engine ADR — what `ai-workflows` is (a pip package? a local path dependency? a vendored module?), where the workflow modules live, the integration shape (how application code invokes a workflow), and the forbidden-SDK list the conformance skill will then enforce (MC-1's architecture portion activates). This task is `ai-workflows`'s first real consumer in this project (the manifest's secondary objective: "demonstrate that `ai-workflows` … is a viable engine for AI-driven features in a real consumer application beyond its own examples").
2. **The async-delivery mechanism.** The processing runs out-of-band — never inside a request handler (MC-4 / §6). `/design` decides the out-of-band shape (a background worker? a separate process? a manual `python -m …` command for the single-user dev setup?) — bounded by "no request blocks on the AI call" (MC-4) and "the processor processes existing `requested` rows, it never *creates* one" (MC-9 — only the user creates a request).
3. **The fresh-Question generation workflow.** Given a `requested` Quiz for a Section: set `quizzes.status = 'generating'`; read the Section's LaTeX content (`content/latex/{chapter_id}.tex`, read-only — MC-6 — via the existing parse path); invoke the `ai-workflows` generation workflow with the Section's content; receive a set of hands-on coding-task Question candidates; persist them to the Section's Question Bank (`questions` rows, never deleted — §8) and link them to the Quiz (`quiz_questions` rows); set `quizzes.status = 'ready'`. On failure: `quizzes.status = 'generation_failed'`, no Questions synthesized, no silent infinite retry (MC-5); the per-Section surface renders "Generation failed" (already wired, ADR-034). The generated Questions are coding tasks only — never describe/recall/choose (§5 / §7); the workflow's prompt-to-the-AI constrains the output and the persistence write rejects (or the workflow does not produce) anything else. Topic tags on the generated Questions land in `questions.topics` (ADR-033's `'|'`-delimited column) *unless* `/design` lands the relational `topics` table + `question_topics` join that ADR-033 deferred "to the generation task" — see "Architectural decisions expected" §3.
4. **The "notified when ready" mechanism.** `/design` decides whether this slice ships the Notification entity (the `notifications` table ADR-022 forecast + ADR-033 deferred, plus a learner-visible Notification surface) or whether the per-Section Quiz status-list display flipping to "Ready" (already shipped, ADR-034) is the sufficient "notified when ready" mechanism for this slice — bounded by MC-4 / §6 ("the learner submits, continues working, and is notified when results are ready"; "a learner-visible indication that an async AI result has become available" — §8). The architect must reason about whether MC-4's "notified" obligation is met by a passive status-flip the learner sees on next visit, or whether it requires an active Notification surface — and decide, naming the call.
5. **Tests** — pytest for the generation workflow with a *fake/stubbed AI engine* (no real `ai-workflows`/LLM call in a test — MC-1; `/design` decides the test seam): the `requested` → `generating` → `ready` happy path (≥1 coding-task Question persisted to the Bank, `quiz_questions` rows created); the failure path → `generation_failed` (no Questions synthesized — MC-5); the no-`user_id` check; the MC-10 boundary grep extended to the new workflow module; the MC-1 boundary check (no raw LLM SDK import anywhere); the first-Quiz-only guard (MC-8 — a `requested` row for a Section that already has a non-failed Quiz is handled per `/design`'s decision and is *not* turned into a fresh-only post-first Quiz); the out-of-band-ness check (the `POST .../quiz` route returns 303 immediately, the Quiz is still `requested` until the processor runs). Playwright for the rendered per-Section surface in its `generating` / `ready` / `generation_failed` states (driven via the test seam, no real AI call) and the Notification surface if one ships; M/O inheritance still holds.

It deliberately does **not** ship: the replay-+-fresh composition for post-first Quizzes (the next-after-grading slice — needs Attempt history); Weak-Topic targeting (same — no Weak Topics until grading exists; the first Quiz needs no targeting, the bank is empty); the Quiz-taking surface (where the learner writes code against a Question, response persistence, the "submit Attempt" route — a later slice; ADR-034 already says "no takeable affordance ships … even for a `ready` Quiz"); the grading workflow (`quiz_attempts.status` `grading` → `graded`/`grading_failed`, the Grade aggregate — a later slice); the Grade entity's schema (ADR-033 deferred it to the grading task — still deferred); changes to the Notes surface, the completion toggle, or the parser (all settled); the parked ADR-032 (Notes-save scroll — not touched unless `/design`'s Notification surface unexpectedly touches the page layout, which the forecast says it does not); the two Open parser-fidelity issues; lint/type-check tooling. **`/design` MUST NOT smuggle in the grading workflow or the replay-+-fresh composition — those are later slices.**

It advances the primary objective by **making the reinforcement loop's generation half real** — the first Quiz for a Section is now actually generated (by AI-driven processing, on the user's manual trigger, asynchronously, with failures surfaced honestly), which is the §7 "reason this project exists" moving from "schema + empty-state" to "the user clicks Generate and gets a Quiz." It also fulfills the secondary objective (§4): `ai-workflows` gets its first real consumer in a consumer application. It is sized as a vertical slice (the AI-engine ADR + the async-delivery mechanism + the workflow + tests, with the surface mostly already in place from TASK-013), though it is the largest slice the project has attempted — see "Architectural concerns" for the size risk and the `> NEEDS HUMAN`-likely points (the `ai-workflows` integration shape is the main one — if the architect cannot determine what `ai-workflows` is and how to depend on it without the human, `/design` ends with `> NEEDS HUMAN`). The next slice is **the Quiz-taking surface** (the learner writes code against a `ready` Quiz's Questions, the Attempt persists), then **the grading workflow** (`grading` → `graded`, the Grade aggregate, Weak Topics), then **the replay-+-fresh composition** (post-first Quizzes — MC-8 in full).

## Acceptance criteria

- [ ] Given a Section with a `requested`-status Quiz row (created by the existing `POST /lecture/{chapter_id}/sections/{n}/quiz` route, TASK-013) and an empty Question Bank for that Section, when the out-of-band Quiz-request processor runs (the mechanism — a command, a worker, etc. — is `/design`'s call), then the Quiz transitions `requested` → `generating` → `ready`, **at least one `questions` row is persisted to that Section's Question Bank** (each with a non-empty `prompt`, the correct `section_id`, and Topic tags in whatever shape `/design` chose — `questions.topics` delimited column or `question_topics` join rows), **`quiz_questions` rows link those Questions to the Quiz** (with `position` set), and the per-Section Quiz surface on `GET /lecture/{chapter_id}` shows the Quiz's status as "Ready".
- [ ] Given the generation workflow, when it invokes the AI, then it does so **exclusively through `ai-workflows`** (the AI-engine ADR names the integration shape and the workflow module path) — and the codebase, when grepped, contains **no direct LLM/agent SDK import** (`import openai`, `import anthropic`, `from openai import …`, `import google.generativeai`, a raw `httpx`/`requests` call to an LLM provider's API endpoint, etc. — the exact forbidden set is enumerated by the AI-engine ADR and reflected into the conformance skill in the same commit) anywhere outside whatever `ai-workflows` itself is (MC-1).
- [ ] Given the `POST /lecture/{chapter_id}/sections/{n}/quiz` route is hit, when the response is returned, then it is the existing 303 PRG redirect **returned immediately** — the AI generation call has **not** run inside the request handler (the Quiz is still `requested` at the moment the route returns; the processing happens out-of-band) — i.e., no code path completes AI processing synchronously inside the request that submits it (MC-4 / manifest §6).
- [ ] Given the generation workflow's AI call fails (the test seam injects a failure), when the processor handles it, then the Quiz transitions to `generation_failed`, **no `questions` rows are synthesized for that Quiz** (no placeholder/fabricated Question), the failure is **not** masked by an unbounded silent retry (a bounded retry-then-fail is acceptable; an infinite retry that never surfaces the failure is not), and the per-Section Quiz surface shows the Quiz's status as "Generation failed" (MC-5 / manifest §6). `/design` decides whether the failure detail is persisted (e.g., a `quizzes.generation_error TEXT` column — additive) but the learner-facing signal is the honest "Generation failed".
- [ ] Given a generated Quiz, when its Questions are inspected, then **every Question is a hands-on coding task** — the `prompt` is an instruction to implement a concept from the Section in code; **no Question is a multiple-choice / true-false / short-answer / describe-the-concept / recall item**, and the `questions` table has no column that could hold one (the schema from ADR-033 is unchanged on this point — no `option_*` / `correct_choice` / `answer_text` / `describe_*` / `recall_*`); the workflow's prompt-to-the-AI constrains the output to coding tasks and the persistence write rejects (or the workflow does not produce) anything else (manifest §5 / §7).
- [ ] Given a generated Quiz reaches `ready`, when its `section_id` and the `section_id` of every Question linked to it are inspected, then **they all reference exactly one Section** — no Question of a different Section is linked into the Quiz; no Quiz row aggregates across Sections; no route or query in the codebase composes a Quiz from Questions of more than one `section_id` (MC-2 / manifest §6 / §7).
- [ ] Given a Section that already has a non-failed Quiz (`ready` or `generating`), when a second `requested`-status Quiz row is created and the processor runs, then the processor handles it per `/design`'s decision (forecast options: reject the second request at the route level so the row is never created; or leave the second `requested` row unprocessed for the future composition slice; or process it through the *composition* path — **but the composition path is not built this task**, so the only acceptable outcomes are "rejected at request" or "left in `requested`") — and in **no case** is a fresh-only post-first Quiz produced (a post-first Quiz that contains only freshly-generated Questions and no replayed wrong-answer Questions is a bug per MC-8 / manifest §7; this task ships only the first-Quiz case). `/design` records the chosen guard shape.
- [ ] Given the out-of-band processor, when its trigger path is inspected, then it **processes existing `requested` rows; it never creates one** — there is no background job, scheduled task, or auto-trigger that generates a Quiz request; the only way a `requested` row comes into existence is the user's explicit click on the existing "Generate a Quiz for this Section" button (MC-9 / manifest §7). `/design` decides whether the `POST .../quiz` route also signals the processor (an enqueue) or whether the processor polls independently — either is acceptable so long as the route does not block on the AI call (MC-4).

- **The async-delivery mechanism for a single-user, locally-run, no-CI project is a genuine design choice — don't over-build it.** The manifest forbids real-time AI (§5) and requires async (§6), but it does not require a production-grade job queue.
