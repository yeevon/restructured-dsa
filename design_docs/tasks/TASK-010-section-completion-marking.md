## What and why

The manifest's §7 invariant **"Completion state lives at the Section level. Chapter-level progress is derived from Section state."** has been a paper commitment for the project's entire ten-task history; no Section has ever been markable as complete. The Notes pillar is now bootstrapped (TASK-009; ADR-022/ADR-023). The Lecture pillar is substantially polished. The next manifest-anchored vertical slice that is **not** Quiz-bootstrap (which forces ~5 ADRs in one task and an `ai-workflows` integration) is Section completion marking.

This task ships the **minimum viable per-Section completion toggle**: a learner can mark any Section as complete from the Lecture page, the completion state persists across server restarts, and re-visiting the Section's Lecture page shows the Section as complete. It deliberately **does not** ship: Chapter-level derived progress display, Mandatory-only progress views, rail-side progress indicators, completion timestamps surfaced in UI, undo confirmation, completion-driven navigation cues, or any Quiz integration. Those are follow-ups.

It advances the primary objective by activating an unbuilt manifest invariant, validates ADR-022's single-shared-DB cohabitation commitment in concrete code (a second entity-table joining the same `app/persistence/` boundary), and establishes the per-Section affordance surface pattern that Quiz-bootstrap will reuse. It deliberately defers the larger Quiz-bootstrap question for one more cycle so the persistence-cohabitation pattern is exercised first on a non-AI surface.

## Acceptance criteria

- [ ] Given the dev server is running and the user navigates to `GET /lecture/{chapter_id}` for any of the 12 corpus Chapters, when the page renders, then each Section has a visible per-Section completion affordance (form + state indicator) — exact shape decided in `/design`.
- [ ] Given the user marks Section `{chapter_id}#section-{n-m}` as complete via the completion affordance, when the form submission completes, then the completion record persists to the project's persistence store (path/shape decided in `/design`) and the user is returned to (or remains on) the Lecture page with the Section now visibly marked complete.
- [ ] Given the user has previously marked Section `{chapter_id}#section-{n-m}` as complete and the dev server is restarted, when the user re-visits `GET /lecture/{chapter_id}`, then that Section is shown as complete (manifest §7: "Completion state lives at the Section level"; the state outlives the FastAPI process).
- [ ] Given the user has marked one Section complete in Chapter X and visits Chapter Y, when Chapter Y's Lecture page renders, then only Chapter Y's Sections are reflected — no cross-Chapter leak, no stale "complete" state on Chapter Y's Sections.
- [ ] Given a Section is currently marked complete, when the user clicks the affordance again, then the Section is **unmarked** (toggle, not append-only) — incomplete is the default; complete is the toggle-on state; users can change their mind. (Architect may decide between toggle-semantics vs append-history-row semantics during `/design`; this AC fixes the user-visible behavior, not the schema.)

- [ ] Given the existing test suite (619 passing as of TASK-009), when `python3 -m pytest tests/` runs, then all existing tests pass — no regressions on Lecture rendering, navigation, or Notes.
- [ ] Given the new completion surface, when Playwright tests run, then at least one Playwright test exercises the round-trip: load Lecture → mark a Section complete → reload Lecture → assert the Section is shown as complete. ## Architectural decisions expected

The architect should anticipate **one or two ADRs** during `/design TASK-010`. The shape mirrors TASK-009: one persistence-layer ADR (now an *amendment*-style addition to ADR-022's cohabitation commitment) plus possibly one surface-shape ADR. Whether to split or merge is the architect's call.

- **ADR-NNN: Section completion schema and persistence module.** Decides:
  - **Schema shape.** Candidate options: (1) `section_completions(section_id PK, completed_at)` — one row per completed Section; unmarking deletes the row; presence ≡ complete. (2) `section_completions(section_id PK, is_complete BOOLEAN, completed_at)` — one row per ever-touched Section; toggle flips the boolean. (3) `section_completion_events(event_id, section_id, action TEXT, at TEXT)` — append-only event log; complete = latest event is `mark`. Architect's forecast: (1) for simplicity unless audit-history is a real requirement (manifest §5 forbids LMS features; audit history of completion has no current consumer).
  - **Whether `chapter_id` is a separate column or derivable from `section_id`.** Section IDs per ADR-002 are `{chapter_id}#section-{n-m}` — the chapter_id is recoverable by string-split. Storing it redundantly enables per-chapter queries without a `LIKE` scan. Architect picks.
  - **Whether the section_id is validated against the discovered Section set on write.** Architect picks; precedent (ADR-023 §Validation) suggests yes — write-time validation prevents orphan rows from typos or stale URLs.
  - **Module path and public API.** Forecast: `app/persistence/section_completions.py` with `mark_section_complete(section_id) -> None`, `unmark_section_complete(section_id) -> None`, `is_section_complete(section_id) -> bool`, `list_complete_sections_for_chapter(chapter_id) -> list[str]` (returns Section IDs). ADR-022's package-boundary rule extends naturally.
  - **Migration story.** ADR-022 committed to `CREATE TABLE IF NOT EXISTS` at first connection until the first non-additive change. This is the first non-Notes additive change — well within the existing migration story; no new migration ADR forced.
  - **Cohabitation validation.** This is the first concrete test of ADR-022's "single shared SQLite DB; new entity = new module under `app/persistence/`" commitment. The architect should explicitly note in the ADR whether the experience validates ADR-022's pattern, surfaces a need to amend it, or is silent on the question.

- **ADR-NNN (possibly separate): Section completion UI surface.** Decides:
  - **Route shape.** Candidates: (1) `POST /lecture/{chapter_id}/sections/{n-m}/complete` form-encoded + PRG redirect (mirrors ADR-023's per-Chapter Notes shape, scaled down to per-Section). (2) `POST /lecture/{chapter_id}/complete` with `section_number` in the body. (3) A single toggle route that flips state based on the current value. Architect's forecast: (1) with explicit `mark` vs `unmark` distinction (e.g., separate routes or a hidden form field `action=mark|unmark`).
  - **Template placement.** Where the affordance lives within each `<section id="...">` block in `lecture.html.j2`. Candidates: (a) inline in the Section heading (next to `<h2 class="section-heading">`), (b) as a button at the end of each Section's body, (c) floating in the right margin. Architect's forecast: (a) for visibility.
  - **State indicator shape.** How "this Section is complete" is signaled visually. Candidates: (a) checkbox checked, (b) heading suffix "✓ Complete", (c) section-level CSS class (`.section-complete` adds a green left-border or muted heading color), (d) all of the above. Architect picks.
  - **CSS class names and file location.** Per ADR-008, completion-related styles go in `app/static/lecture.css` (Section is Lecture-body content). Class names: `.section-complete`, `.section-completion-form`, `.section-completion-button`, etc. — exact set decided in `/design`.
  - **Form-handling pattern.** Reuse ADR-023's synchronous PRG + no-JS commitment, scoped per-Section.

These become Proposed ADRs during `/design TASK-010`. Nothing substantive lands in `architecture.md` until the human gates them. **Whether one ADR or two depends on whether the architect finds the schema and surface decisions cleanly separable.** ADR-022/ADR-023 split because the sites diverged; here the surface ADR is small and may fold naturally into the schema ADR. Architect's call.

## Architectural concerns I want to raise

- **This task is the first concrete validation of ADR-022's single-shared-DB cohabitation commitment.** ADR-022 §Future-cohabitation explicitly anticipated multiple entity-modules under `app/persistence/` sharing one `connection.py` and one SQLite file. Section completion is the first non-Notes case. The architect should treat the `/design` cycle as **an empirical test of ADR-022's pattern**: if the new module's `CREATE TABLE IF NOT EXISTS` block, public API shape, and route-handler integration compose cleanly against the existing pattern with no friction, the ADR is validated. If anything surfaces (e.g., the schema-bootstrap function in `connection.py` becomes awkward as more tables are added; the boundary rule needs sharpening for cross-table queries; the test-fixture pattern for per-table isolation breaks down), the architect should record it in the ADR and consider whether a small supersedure or amendment is warranted. **Specifically: do not silently route around any friction; surface it.**

- **Section IDs are URL-fragment-shaped (`{chapter_id}#section-{n-m}`).** The `#` character is a real character in URL fragments but is the **path-vs-fragment separator** in URLs. The route handler cannot use the full Section ID as a path segment (`/lecture/{chapter_id}#section-{n-m}/complete` is not a valid path; the `#` terminates the path). The architect needs to commit to a route shape that decomposes the Section ID — likely `/lecture/{chapter_id}/sections/{n-m}/complete` or equivalent, with the persistence layer composing the full Section ID internally. Surfaced because the existing Section ID scheme (ADR-002) was designed for HTML anchor use, not for path-segment use, and this task is the first time a Section is the *subject* of an HTTP route. **Not a bug — just a translation step the `/design` cycle must explicitly commit to.**

- **Manifest §6 "Mandatory/Optional honored everywhere" applies to completion state but the task explicitly defers the derived view.** The architect should design the schema and accessor functions to *not foreclose* a future "Mandatory-only progress" view (the AC names this), but the ADR should also be honest that no UI surface in this task consumes the designation. **Surfaced so the architect does not over-design** — the natural consumption point is the future derived-progress view (a follow-up task); building it preemptively into this task is scope creep.

- **Section completion is *not* a Quiz precondition under the manifest.** Manifest §7 separates the reinforcement loop from completion state: a learner can mark a Section complete without taking a Quiz, and can take a Quiz on a Section without marking it complete. The architect should NOT couple these concepts in the schema, the API, or the UI. Surfaced because pattern-matching might suggest "Section is complete → it's been Quizzed → wire them" — but the manifest does not commit to that semantics, and coupling them would force superseding this ADR when Quiz-bootstrap lands.
