## What and why

The Quiz pillar so far: TASK-013 shipped the schema (ADR-033) + the per-Section read surface + a `POST .../quiz` route that records a `requested` Quiz row; TASK-014 shipped the generation processor (ADR-036, ADR-037) that walks a `requested` Quiz to `ready`, generating fresh hands-on coding-task Questions. So a Section can now have a `ready` Quiz with Questions — but **nothing can be done with it**: ADR-034 explicitly shipped the `ready` state with "**no takeable affordance** — the Quiz-taking surface is a later slice." This task is that slice. It makes a generated Quiz *takeable*: the learner opens it, sees each Question's coding-task prompt with a code-entry field, writes their implementation, and submits — and the submission persists as a Quiz Attempt (a `quiz_attempts` row + one `attempt_questions` row per Question carrying the learner's `response`), which is the §7 "Every Quiz Attempt … persists across sessions" invariant, and the raw material the loop's later slices consume.

It is slice 3 of the Quiz pillar's planned sequence (TASK-014 forecast: schema + read surface → generation → **the Quiz-taking surface** → the grading workflow → the replay-+-fresh composition). The slice it ships:

1. **The Quiz-taking surface.** A new HTML surface (a page, or section of one — `/design`'s call) reached from the `.section-quiz` block's `ready` entry (which gains a "Take this Quiz" affordance — the thing ADR-034 deferred). It renders the Quiz's Questions in order (`quiz_questions.position`), each with its coding-task `prompt` and a code-entry `<textarea>` (never a non-code input — manifest §5/§7); a submit button; and a way back to the Section. The take affordance appears **only** on `ready` Quizzes — a `requested`/`generating`/`generation_failed` Quiz is not takeable, and the surface says so honestly if reached for one.
2. **The Quiz Attempt lifecycle (up to `submitted`).** Starting the Quiz creates a `quiz_attempts` row in `in_progress` (plus `attempt_questions` rows — at start or at submit, `/design`'s call). Submitting records the learner's `response` for each Question and flips `quiz_attempts.status` → `submitted`, setting `submitted_at`. The Attempt persists across sessions (§7). **No grading**: `attempt_questions.is_correct` and `.explanation` stay NULL; no Grade aggregate is computed, displayed, or stored; the submit route does **not** trigger grading synchronously (MC-4 — grading is a later out-of-band slice). A `submitted` Attempt is shown honestly as "submitted — not yet graded" (or similar), never with a fabricated score (MC-5's spirit applied to the read surface — even though no AI ran, the surface must not invent a result).
3. **The persistence layer additions.** New typed public functions in `app/persistence/quizzes.py` (forecast: `start_attempt`, `get_attempt`, `list_questions_for_quiz`, `save_attempt_responses`, `submit_attempt` — `/design`'s exact set/names) and the `QuizAttempt` (and maybe `AttemptQuestion`) dataclass(es); SQL stays under `app/persistence/` (MC-10). No new table — `quiz_attempts` and `attempt_questions` already exist in `connection.py`'s `_SCHEMA_SQL` (ADR-033); a `CREATE INDEX IF NOT EXISTS` may be added (additive — `/design`'s call). No `user_id` (MC-7).
4. **Styling.** This is a UI task; styling is in scope. `/design` scopes the CSS per ADR-008's class-name-prefix rule: the take affordance on the `.section-quiz` block reuses `section-quiz-*` in `lecture.css`; the take-page surface's classes (`quiz-*` / `quiz-take-*`) go in `lecture.css` or a new `app/static/quiz.css` — a Quiz-taking page is neither Lecture-body content nor page-chrome/rail, so a third flat CSS file is defensible (ADR-008's "non-overlapping by prefix, loaded flat, no preprocessor" posture extends to it); `/design` decides, citing ADR-008.
5. **Tests** — pytest for the Attempt-lifecycle persistence functions (start → save responses → submit; `is_correct`/`explanation` stay NULL; no `user_id`), the take route(s) (404/422 for unknown Quiz/Section; honest error for a non-`ready` Quiz; **submit returns its redirect with the Attempt in `submitted`, not `graded` — no grading call inside the request**, MC-4; M/O context holds, MC-3; one Quiz → one Attempt referencing one Section, MC-2), and the MC-10 boundary grep. Playwright for the rendered take surface (the take affordance on `ready` Quizzes only; the surface renders prompts + code fields; entering code + submitting persists the Attempt and the surface reflects it honestly — no fabricated Grade; M/O context holds).

It deliberately does **not** ship: the grading workflow (`quiz_attempts.status` `grading` → `graded`/`grading_failed`; the AI-driven grading of the learner's code; this is the next slice — a new `ai-workflows` workflow run out-of-band, like the generation processor); the Grade aggregate's schema or surface (ADR-033 deferred the Grade schema "to the grading task" — still deferred; no `grades` table, no per-Question correctness display, no aggregate score, no Weak Topics, no recommended-Sections-to-re-read this task); the `notifications` table + a learner-visible Notification surface (ADR-037 records it ships with the grading slice — still deferred); the replay-+-fresh composition for post-first Quizzes (a later slice — needs the grading slice's Weak Topics and the `attempt_questions.is_correct` wrong-answer history this task starts recording); a code-execution / test-runner surface (the learner writes code; no code is *run* — running it is the grading slice's concern, or a future one; this task just records the text); changes to the Notes surface, the completion toggle, the generation processor, or the parser (all settled); the parked ADR-032 (Notes-save scroll — not touched unless `/design`'s take page unexpectedly touches the Lecture-page grid, which the forecast says it does not); the two Open parser-fidelity issues; lint/type-check tooling. **`/design` MUST NOT smuggle in grading, the Grade aggregate, the Notification entity, or a code-runner — those are later slices.**

It advances the primary objective by **closing the gap between "a Quiz exists" and "the loop has data to feed back"**: a generated Quiz is now takeable, an Attempt persists, and that Attempt is the §7 raw material — once the grading slice turns `attempt_questions.response` into `is_correct` + a Grade with Weak Topics, the composition slice turns that into the next Quiz's replay-+-fresh content, and the loop the manifest calls "the reason this project exists" is closed. It is sized as a vertical slice (one or two routes + a template + the Attempt-lifecycle persistence functions + the take affordance on an existing block + tests) — smaller than TASK-014 (no AI integration, no async machinery — just a synchronous form-post surface and a few persistence functions, mirroring the patterns already established by the completion toggle and the Notes form). The next slice is **the grading workflow** (the learner's code → `is_correct` per Question + a Grade with Weak Topics, via a new `ai-workflows` workflow run out-of-band), then **the replay-+-fresh composition** (post-first Quizzes — MC-8 in full).

## Alternatives considered (task direction)

- **(Chosen) The Quiz-taking surface — take a `ready` Quiz, write code per Question, submit an Attempt; no grading.** Slice 3 of the Quiz pillar's planned sequence. It is now a true vertical slice (a `ready` Quiz with Questions can exist, post-TASK-014); it ships a self-contained, demonstrable thing (the learner clicks "Take this Quiz", writes code, submits, the Attempt persists); it is the prerequisite for the grading slice (no Attempt → nothing to grade) and thus for the loop. Sized smaller than TASK-014 (no AI, no async — a synchronous form-post surface + a few persistence functions, mirroring the completion toggle and the Notes form).
- **The grading workflow first (skip the take surface) — generate `is_correct`/`explanation`/Grade for an Attempt.** Rejected as out of order: grading presupposes an Attempt with the learner's `response` in each `attempt_questions` row — which is exactly what *this* task creates. There are no Attempts to grade until the take surface ships. Grading is slice 4.
- **The take surface *plus* grading in one task.** Rejected as over-scoped and not a clean slice: grading is a second `ai-workflows` workflow run out-of-band (a whole processor, a Grade aggregate schema, the Weak-Topic-identification path, the `notifications` table + surface) — TASK-014-sized on its own; combining it with the take surface would be a multi-session task that doesn't deliver a demonstrable thing at the halfway point. The take surface alone is a clean, demonstrable slice; grading is the next one.
- **A "preview / read-only" Quiz view (show the Questions, no code entry, no Attempt) as an intermediate step.** Rejected as a non-slice: showing the Questions read-only delivers nothing the per-Section Quiz block doesn't already imply ("this Quiz has N Questions"), produces no Attempt, and doesn't advance the loop. If the learner can see the Questions, they should be able to answer them — that's the same slice.
- **A code-execution / test-runner surface bundled in (the learner writes code and it's *run* against tests on submit).** Rejected as out of scope and premature: running the learner's code is the *grading* concern (or a future one) — the manifest's "every Question is a hands-on coding task" requires the learner to *write* code, not that the system *runs* it in this surface; recording the text and grading it later (the §6 async decoupling) is the manifest-aligned shape. A code-runner would also pull in a sandboxing question (running arbitrary learner code) that is a significant decision in its own right — out of scope for the take surface.
- **Defer the take affordance; ship only the persistence functions + a `GET` route with no UI.** Rejected — half-implementation by construction (a route that returns a page no affordance links to, with no styling): this is a UI task; the take surface and the take affordance are the deliverable. UI-1 forbids shipping layout-bearing DOM (or a route that returns it) without the surface being a real, reachable, styled thing.
