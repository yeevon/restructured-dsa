## What and why

The manifest's **primary objective (§3)** names three learner-facing pillars: per-Chapter Lectures, per-Chapter Notes, and per-Section Quizzes with reinforcement. Eight completed tasks (TASK-001 → TASK-008) have all been Lecture-rendering. The Lecture pillar is now substantially polished (12/12 chapters render at HTTP 200; 562/562 tests green; tabular, callout-title, texttt-math, unhandled-env, and macro-bleed defects eliminated). **Notes and Quizzes remain at zero.**

Notes is the lighter of the two unbuilt pillars: per the glossary (§8), Notes are *user-authored*, *never auto-generated*, and have **no AI surface**. Quizzes are heavier (manifest §4 forces `ai-workflows` integration; §6 forces async; §7 forces the reinforcement loop machinery + Question Bank + Notification). Bootstrapping Notes first establishes the persistence layer that Quizzes will later reuse, without dragging AI/async questions into the same task.

This task ships the **minimum viable Note surface**: a learner can write a Note bound to a Chapter, the Note persists across server restarts, and the Note is visible on subsequent visits to that Chapter's Lecture page. It deliberately **does not** ship: edit, delete, optional Section reference, multiple Notes per Chapter, formatting/Markdown, search, or any cross-Chapter Notes view. Those are follow-ups that build on the persistence-layer ADR this task forces.

It advances the primary objective directly (manifest §3 — Notes are the pillar this task adds) and ends MC-10 dormancy (currently three tasks of inherited "persistence-layer ADR pending" observations — TASK-005 audit Run 007, TASK-007 task file, TASK-008 audit Run 001/005).

## Acceptance criteria

- [ ] Given the dev server is running and the user navigates to `GET /lecture/{chapter_id}` for any of the 12 corpus Chapters, when the page renders, then a Notes UI affordance is visible on the page (form + any existing Note for that Chapter) — placement decided in `/design`.
- [ ] Given the user submits a non-empty Note body via the Notes UI for Chapter X, when the form submission completes, then the Note persists to the project's persistence store (path/shape decided in `/design`) and the user is returned to (or remains on) Chapter X's Lecture page with the new Note now visible.
- [ ] Given the user has previously written a Note on Chapter X and the dev server is restarted, when the user re-visits `GET /lecture/{chapter_id=X}`, then the previously-written Note is visible on the page (manifest §7: "every Note … persists across sessions").
- [ ] Given the user visits `GET /lecture/{chapter_id=Y}` where Y has no Note, when the page renders, then the Notes UI shows an empty-state (no Notes yet) and the form is available — never an error, never a stale Note from another Chapter.
- [ ] Given the same user (single-user system; manifest §6, §7), when any Note is created, it is owned by the single user implicitly — there is no `user_id` column, no auth check, no per-user partition (MC-7 architecture portion is now enforced once the persistence schema is Accepted).
- [ ] Given a Note exists for Chapter X, when *any* code path under `app/` reads or writes that Note, then it does so exclusively through the persistence package defined by the new persistence-layer ADR — no DB driver imports outside that package, no SQL string literals outside that package (MC-10 architecture portion now active).
- [ ] Given the persistence store path/file/connection target chosen in `/design`, when the user runs the app, then nothing under `content/latex/` is touched for write (MC-6 preserved by construction; persistence store lives outside `content/latex/`).
- [ ] Given the existing 562-test suite (TASK-001 → TASK-008), when `python3 -m pytest tests/` runs, then all existing tests pass — no regressions on Lecture rendering or navigation.
- [ ] Given the new Notes surface, when Playwright tests run, then at least one Playwright test exercises the round-trip: load Lecture → submit Note → reload Lecture → assert Note visible. (ADR-013 split-harness is reused: HTTP-protocol pytest tests cover store/read shape; Playwright covers the rendered surface.)
- [ ] Given the staged diff, when manifest-conformance runs, then MC-6 (read-only Lecture source), MC-7 (single user — manifest *and* architecture portions), MC-10 (persistence boundary — architecture portion now enforceable) all PASS.
- [ ] Given the rendered Notes surface, when the human reviews fresh last-run Playwright screenshots per ADR-010, then the Notes affordance is visually present, legible, and stylistically consistent with the existing Lecture surface (re-use of `app/static/lecture.css` palette, no broken layout). Recorded as `rendered-surface verification — pass (TASK-009 Notes UI)` in the audit Human-gates table.

## Architectural decisions expected

The architect should anticipate **two ADRs** during `/design TASK-009`. Splitting them follows the ADR-019 / ADR-020 precedent: when the architectural sites differ and the supersedure paths diverge, two ADRs are clearer than one.

- **ADR-NNN: Persistence layer — store choice and package boundary.** Decides:
  - **Store technology.** Candidate options: (1) SQLite via `sqlite3` standard library (zero new dependency, file-based, single-writer-fine for single-user), (2) SQLite via SQLAlchemy Core or ORM (heavier dependency, gives a migration story), (3) flat JSON file under a project data directory (simplest possible; no DB at all), (4) SQLite via `aiosqlite` for async (FastAPI is async; matches ADR-003's forward-looking framework choice).
  - **Where the store file lives.** Candidate paths: `data/notes.db`, `var/notes.db`, `~/.local/share/restructured-cs300/notes.db`, an env-overridable path. Must be outside `content/latex/` (MC-6).
  - **Package boundary.** The new persistence package's path (e.g., `app/persistence/`) and the rule that DB driver imports / SQL string literals live exclusively inside it (MC-10 architecture portion).
  - **Schema for the Note entity.** At minimum: `note_id` (primary key — UUID or autoincrement), `chapter_id` (foreign-key-conceptual to a Chapter; ADR-002 defines Chapter ID), `body` (the user-authored text), `created_at` / `updated_at` (timestamps). **Explicitly deferred** by this task: `section_id` (optional Section reference per §7; not needed for minimum viable), edit/delete behavior, soft-delete, history.
  - **Migration story.** Candidates: (a) raw `CREATE TABLE IF NOT EXISTS` at startup (simplest), (b) Alembic or similar migration tool (heavier; defers to a future ADR if needed). Architect's forecast: (a) for now; supersede when a schema change forces (b).
  - **Single-user enforcement.** No `user_id`. The architecture portion of MC-7 activates: any future contributor cannot add a `user_id` without superseding this ADR.

- **ADR-NNN: Note creation/read surface — route shape, template surface, and form-handling pattern.** Decides:
  - **Routes.** Candidates: (1) `POST /lecture/{chapter_id}/notes` form-encoded body + `GET /lecture/{chapter_id}` already shows the Note, (2) `POST /notes` with `chapter_id` in the body, (3) HTMX-style partial reload, (4) full-page-reload form submission. Architect's forecast: (1) — form-encoded `POST /lecture/{chapter_id}/notes` returning a redirect to `GET /lecture/{chapter_id}`. Simple, no JS dependency, preserves the FastAPI route/template shape established by ADR-003.
  - **Template surface.** Where the Notes form + display lives in `lecture.html.j2`. Candidates: (a) bottom of the page below all Sections, (b) top of the page near the Mandatory/Optional badge, (c) collapsible side panel. Architect picks based on UX simplicity and ADR-008 layout compatibility.
  - **Empty-state behavior.** Candidates: (a) form alone with placeholder text, (b) form + "No notes yet" caption, (c) form + onboarding hint. Architect picks; whichever is chosen, must be visible in the rendered HTML for a Chapter with no Notes.
  - **Submit-feedback behavior.** Candidates: (a) full-page reload (simplest; works without JS), (b) flash message in URL fragment, (c) inline status. Architect's forecast: (a).

These become Proposed ADRs during `/design TASK-009`. Nothing substantive lands in `architecture.md` until the human gates them. The persistence-layer ADR is the load-bearing one — **its acceptance ends the project's three-task-old MC-10 dormancy.**
