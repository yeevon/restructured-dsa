## What and why

TASK-011 shipped the chapter-progress rail decoration and two placement supersedures (completion to bottom-of-Section per ADR-027; Notes to a rail-resident panel per ADR-028). The human's post-commit review found two follow-on UX problems with what shipped, and filed both as project_issues whose Decide-when text explicitly says to bundle them into one follow-up task and one `/design` cycle:

1. **Notes is in the wrong rail.** ADR-028 put the Notes panel in the **left-hand** rail (the chapter rail), which is now visually crowded (chapter list + M/O headings + per-Chapter progress + Notes panel stacked at `minmax(220px, 18rem)` width). Meanwhile the entire region **right of the centered 860px main column is empty** on any wide desktop — the manifest §5 "no mobile-first" stance specifically licenses using that horizontal space. The fix: a **three-column layout** — LHS chapter rail | centered main column | **RHS Notes rail** — moving the Notes panel out of `_nav_rail.html.j2` into its own partial. This un-crowds the LHS rail, puts Notes adjacent to the reading column (right-side annotation matches handwritten margin notes), and consumes the empty real estate. The principle is the one ADR-028 itself encoded — *visibility follows scroll-position-cost* — applied with a corrected understanding of which rail has the real estate.

2. **Marking a Section complete snaps the user 30 viewports up.** ADR-025 committed the completion toggle to a POST → 303 → GET round-trip whose redirect target included a `#section-{n}` fragment (intended to land the user back at the Section heading). When the affordance was at the *top* of the Section (ADR-025), that anchor mostly held position. ADR-027 moved the affordance to the *bottom* of the Section but did not revisit the fragment — so now the user clicks "mark complete" at the bottom of a 30-viewport-tall Section and the redirect snaps them all the way back up to the heading. The human's principle: **the click at the bottom-of-Section affordance is a natural-flow reading-flow act; the response should not move the user.** The fix: drop the fragment from the 303 redirect (or, if Chromium's scroll-preservation on POST→303→GET-same-URL proves unreliable in testing, introduce a minimal client-side JS async toggle — a real architectural threshold the project has deliberately avoided, so `/design` must decide and the human must gate it).

This task ships:

1. **A three-column page layout** — `base.html.j2`'s `<div class="page-layout">` grid becomes three columns (LHS chapter rail | main | RHS Notes rail); the RHS rail is `position: sticky` like the LHS rail; the new RHS rail styling lives in `base.css` per ADR-008's prefix convention. Exact `grid-template-columns` values and breakpoint behavior decided in `/design`.
2. **The Notes panel extracted to a new RHS rail partial** — the `<section class="rail-notes">` block moves out of `_nav_rail.html.j2` into a new partial (e.g., `_notes_rail.html.j2`), included by `base.html.j2` in the new RHS column. Per-Chapter scoping is unchanged from ADR-028 (panel renders only when a Chapter context exists; omitted from `GET /`). The textarea sizing approach (`rows="3"` + `field-sizing: content` + `resize: vertical` fallback + no JS) is unchanged from ADR-028. CSS class names either keep the `rail-notes-*` / `rail-note-*` prefix (still rail-resident) or rename to `notes-rail-*` for clarity — `/design` picks; either way they stay in `base.css`.
3. **A no-snap completion redirect** — the `POST .../complete` route handler stops emitting the `#section-{n}` fragment in its `303 Location` header (Option 1), OR — if `/design` finds Chromium's same-URL scroll-preservation unreliable under Playwright — introduces a minimal progressive-enhancement JS handler (Option 2: form intercepted by `fetch()`, DOM updated in place, no-JS PRG fallback retained). `/design` picks; if Option 2, the JS commitment is a new ADR the human must gate (it supersedes the no-JS posture ADR-028 §Synchronous form posture inherited from ADR-023/025).

It deliberately does **not** ship: any change to which Chapter's Notes the panel shows or the Notes route/validation/persistence (all retained from ADR-028); any change to where the completion affordance lives within a Section (retained from ADR-027); any change to the LHS chapter rail's contents, M/O grouping, or per-Chapter progress decoration (retained from ADR-006/007/026); Notes edit/delete/Markdown/Section-ref (still deferred from TASK-009); cross-Chapter aggregation; mobile breakpoints beyond "usable in a desktop browser" (manifest §5); Quiz machinery of any kind; `ai-workflows` integration; the two remaining parser-fidelity issues; lint/type-check tooling.

It advances the primary objective by restoring Notes' visibility-and-usability to a surface the user actually wants (the empty RHS real estate, reading-adjacent) and by removing a jarring UX side-effect of the completion toggle — both are post-commit-identified quality gaps blocking shippable polish on the Lecture page. It defers Quiz-bootstrap by **one more cycle** so Quiz-bootstrap arrives with the Lecture page surface (both rails + per-Section affordance zone) fully settled — Quiz-bootstrap will introduce its own per-Section affordance and possibly a per-Chapter Quiz-status surface, both of which should inherit a stable layout rather than landing into placement noise the human is still correcting.
