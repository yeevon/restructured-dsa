## What and why

TASK-017 shipped an in-app test runner that runs a Question's `test_suite` against the learner's code in a sandboxed subprocess. The runner's splice is `learner_response + test_suite` concatenated into one translation unit (ADR-042) — which is correct for **assertion-only** test suites (a test file that *calls* the implementation target by name but does *not* define it). But ADR-040's `question_gen` prompt currently steers the LLM to emit test suites that *embed a reference implementation of the target inline*: each generated `test_suite` is a self-contained C++ file that defines (say) `void append(...)` itself and then `assert`s against it. Concatenating the learner's code with such a test suite double-defines the target → the runner reports an honest `compile_error` (MC-5's spirit is intact — never a fabricated pass/fail), but **on a real generated Question the learner can never see `passed=True`** no matter how correct their implementation is. The "Run tests" loop §6 ships, but its happy path is unreachable on real Questions.

This task makes the happy path reachable. **`question_gen`'s prompt changes** to instruct the LLM to:

1. emit **assertion-only test suites** that reference the implementation target by its named signature but do not define it (no reference implementation inside the test file);
2. emit, alongside `prompt` / `topics` / `test_suite`, a new **`preamble: str`** field carrying any struct/class/header shapes the test suite *and* the learner's implementation must share (e.g. the `struct Node { int data; Node* next; };` definition that a "Implement `void append(LinkedList&, int)`" Question needs both the test's assertions and the learner's `append` body to agree on);

and the supporting machinery to:

3. **Persist `preamble` on the Question** — `GeneratedQuestion.preamble: str` (additive Pydantic field), `questions.preamble TEXT` (additive nullable column per ADR-022's migration story, mirroring ADR-041 / ADR-037 / ADR-044), `Question.preamble: str | None` (dataclass field), `add_questions_to_quiz` payload dict + the row→dataclass converter + `list_questions_for_quiz` `SELECT` + `get_question` all carry it through;
4. **Splice the preamble in the runner** — `app/sandbox.py`'s `run_test_suite` gains a `preamble: str = ""` parameter; the splice becomes `preamble + learner_response + test_suite` (one TU). The runner's `RunResult` shape, `resource` limits, language sniff, and honest-failure surfacing are unchanged (ADR-042 §The splice extends the splice; ADR-042's other decisions consumed unchanged).
5. **Show the preamble on the take page** — `quiz_take.html.j2` per `in_progress` Question, if `preamble` is non-empty, render a read-only `<pre class="quiz-take-preamble">` above the response textarea, captioned "Shared code — your implementation and the test suite both depend on this". The existing read-only `<pre class="quiz-take-test-suite">` block (ADR-043) stays; both render. The `submitted` state similarly. New `.quiz-take-preamble` rule in `app/static/quiz.css` (reuses the `quiz-take-*` namespace per ADR-008 — no new file, no `base.css` change).
6. **Re-run the real-engine generation gate** (the empirical confirmation that the regenerated Quiz's `test_suite` is assertion-only and splices cleanly: a hand-written correct implementation + the regenerated `test_suite` + the regenerated `preamble` → `RunResult(status="ran", passed=True)`; a deliberately-wrong implementation → `passed=False`; a deliberately-non-terminating implementation → `timed_out`). **Re-run TASK-017's parked pass-path** — the human writes a real correct implementation of one Question's named signature on a regenerated Quiz's take page, clicks "Run tests", and sees a green `.quiz-take-results-pass` block. This closes the loop TASK-017 verified structurally and TASK-016 verified the generation side of.

It deliberately does **not** ship: any change to the **runner's isolation mechanism** (ADR-042's bare-subprocess + rlimits + timeout + temp `cwd` consumed unchanged — only the splice signature extends, additively, with a `preamble` parameter that defaults to `""` so pre-task call-sites are source-compatible); any change to the **`RunResult` shape**, the **language sniff**, the **`compile_error` honesty path**, or the **MC-6 corpus-protection check**; any change to the **`attempt_questions` test-result persistence** (ADR-044) — `is_correct` is still set by the grading slice (next); any change to the **submit route**, the **generation route**, the **per-Section Quiz surface caption**, the **`first-Quiz-only` guard** (ADR-037), or the **Notes / completion / parser / Lecture surfaces**; the **grading workflow / Grade aggregate / Weak Topics / `notifications` entity / composition** (the *next* slices); the **deprioritized Quiz-prompt-rendering bug** (`{{ aq.prompt }}` autoescaped — explicitly out of scope per the user's deprioritization). **`/design` MUST NOT smuggle in the grading workflow, the Grade aggregate, the `notifications` entity, the replay-+-fresh composition, or fold in the autoescape bug.**

It advances the primary objective by **closing the §6 loop on real generated Questions**: the manifest amendment that opened with "Code is written, run, and tested within the application" has, since TASK-017, been satisfied *structurally* (the runner exists, the route exists, the surface exists, the persistence exists) but not *empirically* (on a real generated Question, the runner reports `compile_error` 100% of the time, no matter what the learner writes). After this task, the learner can write a correct implementation against a real generated Question's `prompt` + `preamble` and see `passed=True`. That is the §6 loop reaching its happy path on the real path, which the grading slice (next) then reads as the per-Question correctness signal §8 Grade is *defined in terms of*. It is sized as a **small focused refinement task** — one prompt-wording change + one additive Pydantic field + one additive DB column + one persistence-dataclass field-add + one sandbox-signature extension + one template change + one CSS rule + a re-run of two existing real-engine gates; implementable in one session.

The next slice is **the grading slice** (the out-of-band `process_quiz_attempts` processor + a new `ai-workflows` `grade_attempt` workflow that reads the in-app test results + the Grade aggregate's home + Weak Topics + the `notifications` entity + the Notification surface). After grading, the **replay-+-fresh composition** for post-first Quizzes (MC-8 in full).
