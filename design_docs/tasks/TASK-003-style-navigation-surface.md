## What and why

The TASK-002 working tree is `Blocked — half-implemented`: the rail and landing page are correctly wired into the DOM (158 tests pass, ADR fidelity preserved) but render as an unstyled wall of text in a real browser because no CSS rules exist for any of the rail's class names (`.lecture-rail`, `.nav-rail-inner`, `.nav-section-label`, `.nav-chapter-list`, `.nav-chapter-item`, `.nav-chapter-error`, `.nav-chapter-empty`, `.page-layout`, `.page-main`). This task ships the styling deliverable so TASK-002 can commit as a usable surface: the LHS rail visually presents as a left-hand rail, the Mandatory and Optional sections are visually distinct (manifest §7), Chapter rows are clickable affordances, the page-level layout (`.page-layout` / `.page-main`) lays the rail and main content side-by-side rather than stacking vertically, and the landing page (`GET /`) is browseable as the bookmarkable home. Manifest §3 (consumption) is the load-bearing motivation: a wall of unstyled text is a worse consumption surface than the curl-style Lecture page TASK-001 shipped.

The task is also forced to **close `design_docs/project_issues/adr006-rail-half-implemented-no-css.md`** by producing the architectural decision the project_issue exists to settle — either by amending ADR-006 to scope CSS, or by drafting ADR-008 for the styling layer (the architect picks one in `/design TASK-003`).

## Architectural decisions expected

- **Resolution path for the project_issue.** Either (1) amend ADR-006 (revert to `Proposed`, add a "Styling scope" subsection, re-Accept — `authority-state-check` AS-1 allowed amendment cycle) or (2) draft a new ADR-008 dedicated to navigation/landing styling that cites ADR-006 as the mechanism it styles. Path 3 (ship unstyled) is a no-go per the project_issue and per UI-1/UI-3. The architect picks Path 1 or Path 2 in `/design TASK-003`.
- **Which CSS file holds the new rules.** Three options materially differ: (a) extend the existing `app/static/lecture.css` with rail / landing rules (single CSS asset); (b) introduce a new file (`app/static/nav.css`, `app/static/base.css`, or similar) loaded alongside or instead of `lecture.css` from `base.html.j2`; (c) split into a `base.css` (page chrome + rail) and a `lecture.css` (Lecture-body content), both loaded from the base template. The architect picks one and records it in the design ADR. UI-2 forces this to be scoped, not left to the implementer.
- **Class-name namespace and visual treatment for the M/O distinction.** The structural class names already exist (`.nav-section-label`, `.nav-chapter-item`, `.nav-chapter-error`, `.nav-chapter-empty`, `.page-layout`, `.page-main`, `.lecture-rail`, `.nav-rail-inner`, `.nav-chapter-list`). The design ADR fixes (i) whether Mandatory/Optional sections use the existing `designation-mandatory` / `designation-optional` color palette from `lecture.css` or introduce a new rail-specific palette, (ii) the visual mechanism that makes the per-row error state legible (color, strikethrough, icon, dedicated `nav-chapter-error` rule), and (iii) the page-layout mechanism (`flexbox` vs `grid` vs floats — though all three are unconstrained by the manifest, the choice is recorded so the implementer doesn't pick silently).
- **Rendered-behavior verification mechanism.** Settled by **ADR-010 (Proposed)**: Playwright via `pytest-playwright` is the project's UI test framework; existing rendered-DOM-content tests are migrated to `tests/playwright/`; HTTP-protocol / source-static / runtime-side-effect tests stay in pytest; last-run screenshots are saved under a gitignored artifact directory; the human reviews screenshots as the visual confirmation portion of the gate. The architect's prior recommendation (manual-browser gate, drafted as ADR-009) was **Rejected by the human**; the human directed UI-4 option 1 instead. ADR-010 implements that direction.
- **Whether the adjacent `\\` linebreak rendering issue (Run 009 finding) is in scope or a follow-up.** Each Chapter title in the rail and the Lecture page header carries a literal `\\` substring (e.g., `CS 300 -- Chapter 1 Lectures\\ C++ Refresher: ...`). This is a TASK-001-era fidelity bug that became more visible at scale; arguably a styling task surfaces this prominently and should fold it in. The architect decides scope.

## Alternatives considered (task direction)

Five materially different next-task directions were considered.

1. **(Chosen) Style the navigation surface so TASK-002 ships as a usable browser experience.** Smallest vertical slice that unblocks the open `adr006-rail-half-implemented-no-css.md` project_issue and lets TASK-002 commit. Forces UI-1/UI-2/UI-3/UI-4/UI-5/UI-6 to be honored on the first UI deliverable that has tripped over the silent-ship pattern. Manifest §3 (consumption) is the load-bearing motivation: an unstyled wall of text is a worse consumption surface than no surface, and committing TASK-002 in its current state would record that as the project's idea of "done." Cost is bounded (CSS rules for ~9 class names + a flexbox/grid layout); benefit is releasing the parked working tree and establishing the styling-ADR pattern that future surfaces (Notes UI, Quiz UI, Notification chrome) will compose against.

2. **Validate Chapters 2 through 13 as renderable Lecture pages (the per-Chapter parser-robustness pass anticipated by TASK-002's "Out of scope" section).** Bigger primary-objective win in absolute content-volume terms (11 more Chapters consumable). Rejected as the *next* task because: (a) it doesn't unblock TASK-002's open project_issue, leaving the navigation surface half-implemented; (b) per the project_issue itself, "TASK-002 cannot commit as 'complete' until project_issue resolves" — going broader before closing the existing blocker is the wrong order; (c) any visual regressions found while validating Chapter 2-13 rendering would be tangled with the rail's missing CSS, making bisection harder. Better as a follow-up after this task ships and TASK-002 commits.

3. **Fix the adjacent `\\` linebreak fidelity issue across all 12 surfaced titles.** A real bug (Run 009 adjacent finding): `\\` LaTeX linebreak macros are passed through to HTML without conversion, so titles render with literal `\\` substrings in both the rail labels and the Lecture page headers. Smaller in scope than this task. Rejected as the next task because: (a) it does not unblock the open project_issue (the half-implementation is the load-bearing blocker, not the linebreak bug); (b) styling work surfaces the issue more prominently, so folding the fix into the design ADR for this task is the natural moment to address it (and is offered as an in-scope-or-defer architect decision above). Standalone, this would be a TASK-004-or-later fidelity task.

4. **Park TASK-003, propose a Notes-feature task that drags the persistence-ADR commitment.** Notes are the next natural manifest entity after Lectures (manifest §8). A Notes feature would force a persistence-layer ADR (currently dormant in MC-10), which is necessary for the manifest's reinforcement loop (§7) to ever land. Rejected as the next task because: (a) Notes-on-a-Chapter requires the Chapter to be findable, and the navigation surface is currently a wall of unstyled text, making Notes UX worse than necessary on the first deliverable; (b) Notes is a substantial vertical slice (UI + persistence ADR + storage shape + Chapter binding) that is not "small enough for one session" while the current project tree carries an open blocker; (c) per the orchestrator's read in the project_issue, the half-implementation should be closed before broadening surface area further. Notes is the right task *after* TASK-003 + Chapters-2-13 validation.

5. **Drop the rail (revert ADR-006's "rail on every Lecture page" provision); ship landing-page-only.** This is Path 3-adjacent: instead of styling the rail, remove it. Rejected because: (a) it reverses an Accepted ADR for a reason that is not architectural — "I didn't ship CSS yet" is not evidence the rail mechanism is wrong; (b) Lecture pages would become navigation dead ends again, violating TASK-002's acceptance criterion the human accepted; (c) the rail is amortized infrastructure for future surfaces (Notes UI, Quiz UI) per ADR-006's Consequences section. Reversing ADR-006 to escape a styling task trades a real architectural commitment for a 30-minute CSS write — wrong tradeoff.

- Mobile / responsive layout. Manifest §5 ("No mobile-first product"); usable-on-desktop is sufficient.
- Dark mode, theming, user-customizable styles. Manifest §5 (single user) makes user-preference state pointless; stylistic decisions are committed in CSS.
- Accessibility audit beyond "the rail is reachable on a desktop browser." Real a11y work (semantic landmarks, ARIA, color-contrast WCAG audit) is a real concern but a separate, larger task. - Rendering Chapters 2-13 successfully. The styling work covers their rail rows; whether their Lecture pages render is a separate validation task.
