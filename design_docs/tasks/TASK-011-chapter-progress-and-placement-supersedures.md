## What and why

Manifest §7 commits to a two-part invariant: **"Completion state lives at the Section level. Chapter-level progress is derived from Section state."** TASK-010 shipped the first half (the Section primitive). The second half — a **derived per-Chapter progress display** that consumes the persisted Section state — is the next unbuilt manifest commitment, and it is the natural surface to bundle two outstanding placement supersedures the human filed at the previous two tasks' commit gates.

This task ships:

1. **A per-Chapter progress display in the navigation rail** — for each Chapter row in `_nav_rail.html.j2`, render a small "X / Y" progress decoration (X = complete Sections, Y = total Sections in the Chapter), respecting the existing Mandatory/Optional grouping. The display is read-only; the data source is `app/persistence/section_completions` (existing) plus Section discovery (existing).
2. **Supersedure of ADR-025 §Template-placement** (per `section-completion-affordance-placement.md` Option 1 forecast) — relocate the per-Section completion form from inline-next-to-`<h2>` to the **end of each `<section>` block**, with a visual end-of-section break before the next Section's heading. The top-of-Section heading row may retain a *read-only status indicator* if `/design` finds it useful (Option 2 in the issue), but the *action* affordance moves to the bottom of the Section.
3. **Supersedure of ADR-023 §Template-surface** (per `notes-surface-placement-visibility.md` Option 1 forecast) — relocate the Notes section from bottom-of-page to a **rail-resident Notes panel** below the chapter list, sharing the rail's `position: sticky` real estate. Per-Chapter Notes become visible from any scroll position.

It deliberately does **not** ship: cross-Chapter progress aggregation surfaces (e.g. "you are 27% through Mandatory"), Mandatory-only filtered progress views (the rail already exposes the Mandatory/Optional split, so a filtered view is a separate consumer), completion-history surfacing or hover-tooltips, completion-driven navigation cues (e.g. "next incomplete Section"), per-Chapter completion timestamps, Notes edit/delete (still deferred from TASK-009), Notes Markdown rendering, optional Section reference on Notes, Quiz machinery of any kind, or `ai-workflows` integration.

It advances the primary objective by activating the second half of the manifest §7 completion invariant and by resolving two human-filed placement-quality issues that have been blocking shippable polish on the Lecture page. It defers Quiz-bootstrap by **exactly one cycle** so Quiz-bootstrap arrives with the per-Section affordance placement already settled (Quiz-bootstrap will introduce a third per-Section affordance whose placement must be jointly designed with completion's; doing the placement supersedure first means Quiz-bootstrap inherits a stable surface).

- **The supersedures' citation discipline is load-bearing.** Per the architect-prompt rule "To propose superseding an `Accepted` ADR, write a new ADR that explicitly cites the prior decision and explains what evidence justifies the supersedure," each supersedure ADR must (a) cite the prior ADR explicitly (`Supersedes: ADR-023` / `Supersedes: ADR-025`), (b) quote the human's post-commit review framing as the evidence (both quotes are recorded verbatim in the project_issue files; lift them), (c) explain why the prior ADR's reasoning was correct given what was known then (it was — both reviewers and `/auto` accepted them) but is no longer correct given post-commit empirical evidence. The reading-flow lesson (action affordances follow the cognitive sequence; visibility follows scroll-position-cost) is the load-bearing architectural principle the supersedures encode — both ADRs should name it explicitly so future placement decisions inherit it.

- **`architecture.md` maintenance protocol fires on each supersedure.** Per the architect prompt "ADR `Accepted` → `Superseded`: move row to 'Superseded'; add the replacement to 'Accepted'; regenerate the project-structure summary." The `/design` cycle should either (a) reduce default `rows`, (b) commit to a CSS rule that allows the textarea to grow vertically as the user types (`min-height` + `field-sizing: content` for browsers that support it; manual JS resize for those that don't — but JS introduction would itself be an ADR-worthy commitment given ADR-023's explicit no-JS posture), or (c) accept the cramped default and commit that Notes-as-rail is a "quick capture" surface, with longer-form Notes deferred to a future "expand to dialog" feature. - **Rail-resident Notes panel forces a decision on the landing page's Notes rendering.** `GET /` has no Chapter context; the rail still renders. The `/design` cycle MUST decide what the Notes panel shows on the landing page (hide / "select a Chapter" placeholder / omit). Surfaced because it's easy to miss until Playwright catches it.

- **Existing tests will fail by design.** TASK-010 added Playwright assertions that the completion form lives next to `<h2 class="section-heading">` (top-of-Section). TASK-009 added pytest assertions that Notes appears in `lecture.html.j2`'s `{% block main %}` after the last Section. - **No coupling to Quiz machinery.** This task touches the Lecture page surface (rail + Section blocks) which Quiz-bootstrap will also touch. The architect should NOT introduce any Quiz-shaped scaffolding "while we're here" (e.g., "leave a slot in the rail for per-Chapter Quiz status"). - **Section-progress denominator stability.** The denominator (Y = total Sections in Chapter) comes from `extract_sections(chapter_id)`.
