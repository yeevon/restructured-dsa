## What and why

TASK-017 shipped the in-app test runner; TASK-018 made the runner's signal **meaningful** on real generated Questions (`passed=True` / `passed=False` instead of universal `compile_error`). The §6 loop now reaches its happy path on real Questions — *structurally* (a learner can write code, click "Run tests", see honest pass/fail) and *empirically* (verified end-to-end by TASK-018 Gate 2). But the §8 Grade — "per-Question correctness (determined by whether the learner's code passed the Question's tests), per-Question explanation, an aggregate score, identified Weak Topics, and recommended Sections to re-read" — does **not exist yet**: a submitted Attempt sits in `quiz_attempts.status = 'submitted'` indefinitely; `attempt_questions.is_correct` and `.explanation` are NULL; there is no Grade aggregate; the take page renders "Submitted — grading not yet available" forever. The reinforcement loop (manifest §3 / §7 / MC-8) cannot close — there is nothing for the composition slice to consume.

This task makes the §8 Grade exist. It ships:

1. **A second CS-300-owned `WorkflowSpec` module at `app/workflows/grade_attempt.py`** (mirroring `question_gen.py` exactly in shape: one `LLMStep` + a `prompt_fn` + `response_format`; `GradeAttemptInput` carrying the Quiz's Questions (each with its `prompt` + `preamble` + `test_suite`) + the learner's per-Question `response` + the persisted per-Question test results (`test_passed` / `test_status` / `test_output`); `GradeAttemptOutput` carrying per-Question `explanation: str` + the aggregate's components — `score: int` / `weak_topics: list[str]` / `recommended_sections: list[str]` — with `ConfigDict(extra="forbid")`; the prompt frames "the Question's *correctness* signal is the persisted test result — do not re-judge whether the code is correct; produce the *explanation* of why it worked / why it didn't, identify the Weak Topics from the failing Questions' Topic tags, recommend the Sections those Topics tie to"; **MC-1 preserved** — imports stay `ai_workflows.*` + `pydantic` + stdlib only; the workflow is registered via `AIW_EXTRA_WORKFLOW_MODULES=app.workflows.grade_attempt` alongside `app.workflows.question_gen`).
2. **A second out-of-band processor `python -m app.workflows.process_quiz_attempts`** (mirroring `process_quiz_requests.py` exactly: poll `submitted` Attempts, set `grading`, invoke `aiw run grade_attempt --input ... --run-id attempt-{attempt_id}`, parse the artefact, persist via `app.persistence.*`, set `graded`; on failure → `grading_failed` + detail in the new `quiz_attempts.grading_error TEXT` column; the manual command is added to `CLAUDE.md`'s Commands section per ADR-037's precedent — flagged for the human, the architect does **not** edit `CLAUDE.md`).
3. **The Grade aggregate's persistence**. `/design`'s call between (i) a separate `grades` table (PK `attempt_id`, FK to `quiz_attempts`, columns `score INTEGER`, `weak_topics TEXT`, `recommended_sections TEXT`) and (ii) nullable columns on `quiz_attempts` (`score INTEGER`, `weak_topics TEXT`, `recommended_sections TEXT`); ADR-033 records both shapes as additive. Forecast: **(i) a separate `grades` table** — the Grade aggregate is a distinct entity with a clean 1:1 to the Attempt, and a separate table keeps `quiz_attempts` lean as more lifecycle metadata accrues. Either way: the per-Question portion (`is_correct` + `explanation`) goes on `attempt_questions` (the existing NULL-until-graded columns ADR-033 created precisely for this slice). `weak_topics` and `recommended_sections` persist as `'|'`-delimited TEXT (mirroring `questions.topics` per ADR-033) — the relational `topics` table migration stays deferred to the composition slice.
4. **The graded-state rendering on the take page**. `quiz_take.html.j2` for a `graded` Attempt renders: an aggregate-score block (`<section class="quiz-take-grade">` — "X / Y" plus the weak-topics + recommended-sections lists); per-Question, the read-only `response` and `test_*` panel (unchanged), plus a new read-only `<div class="quiz-take-explanation">` block carrying the per-Question explanation and an `is_correct`-derived visual indicator (a green / red badge keyed to `attempt_questions.is_correct`). For a `grading_failed` Attempt: a `<section class="quiz-take-grading-failed">` honest-failure block ("Grading failed — try again" + the `grading_error` detail if useful, `/design`'s call); no fabricated Grade, MC-5. New `.quiz-take-grade-*` / `.quiz-take-explanation` / `.quiz-take-grading-failed` rules in `app/static/quiz.css`, reusing the `quiz-take-*` namespace per ADR-008. **No new CSS file. No `base.css` change.**
5. **Validating MC-5's spirit and §8 Grade-correctness**. `is_correct` is set from the persisted `attempt_questions.test_passed` — the test runner's verdict (`True` → `is_correct=1`, `False` → `is_correct=0`, NULL / `test_status != 'ran'` → the Attempt is `grading_failed` for that Question (or `/design`'s call: treat a non-run test as `is_correct=0` with an explanation noting the test never ran — leans the second, since the learner having not clicked "Run tests" before submitting should not block grading; the *explanation* surfaces it honestly). The grading workflow **does not free-form-judge correctness**; it reads the runner's verdict and produces the *explanation* and the *aggregate* — §8 Grade's "correctness determined by whether the learner's code passed the Question's tests" is honored architecturally, not just by convention.

It deliberately does **not** ship:

- **The active Notification entity** — a `notifications` table + a chrome badge in `base.html.j2` is a distinct slice. The §8 Notification obligation for this slice is met by the take page flipping from "Submitted — grading not yet available" to the graded state on the next `GET .../take`, mirroring ADR-037's per-Section status-flip pattern for the generation slice. Forecast follow-on task: "The active Notification entity — a `notifications` table + a chrome badge + the surface that lists pending Notifications + the `seen_at` lifecycle". `/design` should record the deferral explicitly; **MC-5 is satisfied** by the take-page state flip; **§8 Notification** is satisfied by the same; the active surface is a separate slice's call.
- **The relational Topic-vocabulary migration** — the `topics` table + `question_topics` join + backfill from `questions.topics` ADR-036 deferred "to the Weak-Topic-identification slice (the first thing that *queries* across Topics)" stays deferred to the *composition* slice (the slice after this one), which needs to query "all Questions tagged with Topic X across this Section's Question Bank" to drive the fresh portion of the next Quiz. **This task computes Weak Topics for one Attempt** (which is a `list[str]` from the Questions' delimited `topics` column the failing Questions tag) — that does **not** require a relational vocabulary. The migration becomes load-bearing only when the *composition* path needs a cross-Section / cross-Attempt query, which is one slice later.
- **The replay-+-fresh composition for the second Quiz of a Section** (MC-8 in full) — the slice **after** this one; this slice writes the inputs (`attempt_questions.is_correct` + the Grade's `weak_topics`), the composition slice reads them.
- **A grade-failure retry surface** — `grading_failed` is a real state; re-grading is a manual operation (re-running the processor will skip `grading_failed` rows, mirroring ADR-037; a future learner-facing "Try grading again" affordance is a separate slice's call).
- **Lecture-content / parser-fidelity / Notes / completion / rail / landing-page / per-Section Quiz surface changes** — all settled; out of scope.
- **The deprioritized Quiz-prompt-rendering bug** (`{{ aq.prompt }}` autoescaped → literal backticks) — user-deprioritized; *adjacent* (the take page renders prompts again on the graded surface) but **explicitly out of scope**; flagged so `/design` doesn't quietly fold it in.
- **The TASK-018 follow-up `question-gen-prompt-text-redundant-with-preamble.md`** — editorial; doesn't block grading; out of scope.
- **Lint/type-check tooling** — 13th+ recurrence; out of scope; re-flagged in "Architectural concerns".

It advances the primary objective by **producing the §8 Grade aggregate the manifest's reinforcement loop is defined in terms of**. After this task: a submitted Attempt becomes a graded Attempt with per-Question correctness + per-Question explanation + an aggregate score + identified Weak Topics + recommended Sections; the §6 loop's "the learner answers, runs, sees results" extends to "the learner answers, runs, submits, sees a Grade with a learning signal". The *next* slice (replay-+-fresh composition) consumes the wrong-answer history + Weak Topics this slice writes to drive the second Quiz of a Section per MC-8 — and only at that point does the manifest's §3 reinforcement loop close end-to-end on real Questions. This is the larger of the project's remaining vertical slices but is still a single-session task with strong precedent (TASK-014 shipped the analogous slice — workflow + processor + lifecycle + failure-handling — for *generation*; this slice ships the analogous shape for *grading*; the same architectural pattern, mirrored).

## Acceptance criteria

- [ ] Given a `submitted` Attempt (`quiz_attempts.status = 'submitted'`), when `python -m app.workflows.process_quiz_attempts` is run, then the processor selects the Attempt, sets `quiz_attempts.status = 'grading'`, invokes the `grade_attempt` workflow via the `aiw run grade_attempt --input ... --run-id attempt-{attempt_id}` CLI subprocess (mirroring ADR-037; `AIW_EXTRA_WORKFLOW_MODULES=app.workflows.question_gen,app.workflows.grade_attempt` in the subprocess env), parses the terminal artefact from the CLI's machine-parseable stdout (`json.dumps(artifact, indent=2)` + the `total cost: $X.XXXX` trailer), persists the Grade via `app.persistence.*` (per-Question `is_correct` derived from `attempt_questions.test_passed` + per-Question `explanation` from the artefact; the aggregate `score` + `weak_topics` + `recommended_sections` to the Grade store of `/design`'s choosing), sets `quiz_attempts.status = 'graded'` and `graded_at = <now>`, and exits cleanly. ## Architectural decisions expected

The architect should anticipate **three to four ADRs** during `/design TASK-019` (mirroring how TASK-014 split into ADR-036 + ADR-037 and TASK-017 split into ADR-042 + ADR-043 + ADR-044):

- **ADR-NNN: The `grade_attempt` `ai-workflows` workflow.** Decides:
  - **The Pydantic input/output schemas.** `GradeAttemptInput` (the Attempt's Questions + responses + test results + Section grounding); `GradeAttemptOutput` (per-Question `explanation` + aggregate `score` / `weak_topics` / `recommended_sections`); `extra="forbid"` on every model; `min_length=1` on `explanation` (a Question's explanation cannot be the empty string — a missing explanation is a generation-shaped failure, MC-5); whether `weak_topics` / `recommended_sections` are `list[str]` or a more structured `list[WeakTopic]` (forecast: plain `list[str]`, consistent with `questions.topics`'s delimited-TEXT shape; the relational vocabulary stays deferred to the composition slice).

  - **The `LLMStep`'s tier.** Forecast: one tier named via `grade_attempt_tier_registry()`, mirroring `question_gen_tier_registry()` per ADR-036; `/design`'s call on whether grading uses the same model as generation or a different one.

## Architectural concerns I want to raise

1. **The slice is larger than recent slices.** TASK-019 is forecast at 3–4 ADRs and one session, which is at the upper end of what has shipped in single sessions (TASK-013 / TASK-017 / TASK-018 each shipped 3 ADRs in one session). The grading slice's full §8 surface area is wider — workflow + processor + lifecycle + failure-handling + persistence + rendering. **Deferring the active Notification entity and the relational Topic-vocabulary migration to follow-on slices keeps it tractable.** The architect's `/design` should confirm those deferrals up front and *not* let scope creep in. If during `/design` the architect concludes the slice is genuinely two-session-shaped, the right move is to split the task (TASK-019a = workflow + processor + Grade-storage + lifecycle; TASK-019b = the graded-state rendering on the take page) and pause Mode 2 with `> NEEDS HUMAN: TASK-019 should split into 019a + 019b — confirm`. The forecast is one session, but the architect makes the final call.

2. **§8 Grade-correctness is an architectural property, not a hope about LLM behavior.** §8 Grade says "correctness determined by whether the learner's code passed the Question's tests". The architecturally honest implementation is: `attempt_questions.is_correct` is derived from `attempt_questions.test_passed` (the runner's verdict), **not** free-form-judged by the grading LLM. The grading workflow's job is the *explanation* + the *aggregate*, not the correctness signal. The architect's `/design` must record this commitment in the workflow ADR explicitly — otherwise the LLM may re-judge correctness and disagree with the runner, which is the §8 violation MC-5's spirit forbids.

3. **The `grading_failed` honesty path is load-bearing.** Today, a `grading_failed` Attempt has no grading_error column to record the reason — adding one (additive nullable, mirroring ADR-037) is the cheap path to MC-5 conformance. The take page's `grading_failed`-state render must not fabricate a Grade, and **no partial Grade may persist** even if the workflow produced *some* of the per-Question explanations before failing. The test-writer should write the "partial-failure path persists nothing" test explicitly.

4. **The Notification deferral is conformant, but the take-page state flip is the *only* notification surface this slice ships.** The architect's `/design` should record this explicitly per ADR-035 ("describes what's built"). The follow-on slice that adds the active `notifications` table + chrome badge is a separate `/design` cycle.

5. **The relational Topic-vocabulary deferral keeps the door open.** This slice persists Weak Topics per Attempt as a `'|'`-delimited TEXT list (consistent with `questions.topics`). The composition slice (next) will need to query across Topics — that's when the migration becomes load-bearing. The architect's `/design` should confirm the deferral and **not** start the migration here.

6. **The 13th-recurrence lint/type-check tooling gap** (`tooling-lint-and-type-check.md`). Re-flagged, not actioned. After grading + composition close the loop, a non-feature cycle is the right time.

7. **The roadmap memory note** is still stale relative to ADR-040's reordering (runner → grading → composition → polish; this task is grading). Updated by TASK-018 informally, still worth a refresh. Not the architect's edit to make — flagged for the human.

8. **No `> MANIFEST TENSION:`, no `> NEEDS HUMAN:`, no `> PRIMARY OBJECTIVE COMPLETE:`.** The manifest is internally consistent for this slice; the primary objective is *not* complete (the composition slice still needs to ship after grading); this task advances it (the §8 Grade exists; the §3 reinforcement loop's outputs — `is_correct` + `weak_topics` — are written for the composition slice to read).
