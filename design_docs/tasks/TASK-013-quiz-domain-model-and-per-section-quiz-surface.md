## What and why

Three consecutive tasks (TASK-010 section-completion, TASK-011 chapter-progress + placement supersedures, TASK-012 RHS Notes rail + no-snap redirect) polished the Lecture page while deferring the Quiz pillar "by one more cycle" each time. The Lecture page is now settled (TASK-012 committed; only a human-deferred parked ADR and two low-priority parser-fidelity issues remain). Manifest §7 says the reinforcement loop "is the reason this project exists" — continuing to defer it is the wrong call. But Quiz-bootstrap as a single task forces ~5 ADRs (Quiz/Question/Attempt/Grade schema; `ai-workflows` integration per §4; async result delivery per §6; Notification scaffolding per §8; reinforcement-loop composition — replay + fresh — per §7) — multi-session by construction, and the prior task files themselves flagged this.

This task is **the first vertical slice of Quiz-bootstrap**: the **Quiz domain model + the per-Section Quiz read surface**, with Quiz *generation* (the `ai-workflows` integration, async delivery, Notification, and the replay-+-fresh composition) explicitly deferred to the next task. The slice it ships:

1. **The Quiz domain schema** — new tables for the Quiz, Question, Question Bank, and Quiz Attempt entities (manifest §8), landing in `data/notes.db` per ADR-022's "single shared SQLite database" cohabitation commitment, under a new `app/persistence/quizzes.py` (and possibly `app/persistence/topics.py` if `/design` ships the Topic vocabulary now — see "Architectural decisions expected"). Every Quiz `section_id NOT NULL` (MC-2 / §6 / §7: bound to exactly one Section; no Chapter-bound Quiz row; no aggregation row). The Question schema carries a coding-task prompt and Topic tags — no choice/recall/describe columns (§5 / §7: every Question is a hands-on coding task). The Question Bank table allows a Question to belong to multiple Quizzes for its Section over time and is never deleted (§8). The QuizAttempt table records per-Question correctness (so wrong-answer-replay history exists) and carries a status enum reserving the async-grading states (`requested` / `generating` / `awaiting_submission` / `submitted` / `grading` / `graded` / `failed` — exact set is `/design`'s call) so the generation/grading tasks add states without a non-additive migration. No `user_id` anywhere (§6 / MC-7). The migration story is ADR-022's (`CREATE TABLE IF NOT EXISTS`).
2. **The per-Section Quiz read surface** — `GET /lecture/{chapter_id}` gains a per-Section Quiz-data lookup (mirroring `complete_section_ids` from ADR-024 and `rail_notes_context` from ADR-028), and `lecture.html.j2` renders a per-Section Quiz block: with no Quizzes yet, an empty-state ("No quizzes yet for this Section"); the structure is in place to list past Quiz Attempts and (later) their Grades once the generation/grading tasks land. `/design` decides *where* in the reading flow this block goes (forecast: in or near the `.section-end` wrapper, per ADR-027's §Future-surfaces forecast — but the architect decides afresh applying the placement-quality principles ADR-027/030/028 encoded), the class namespace, and whether a *placeholder* per-Section "Generate a Quiz for this Section" affordance is shipped now (a real route creating a `requested`-status row whose downstream processing is the next task's job; or a disabled "coming next" affordance; or nothing yet) — bounded by MC-9 / §7 ("Quiz generation is always explicitly user-triggered" — whatever ships, nothing auto-generates).
3. **Tests** — pytest for the Quiz schema/persistence (create/list round-trips, the MC-10 boundary, `section_id` validation against the parsed Section set, migration idempotency, the no-`user_id` check, the Question-can-be-in-multiple-Quizzes check, the Question-Bank-never-deleted semantics); HTTP-protocol pytest for the per-Section Quiz surface render and (if shipped) the placeholder trigger route; Playwright for the rendered per-Section Quiz surface (empty-state present, M/O inheritance visible if relevant, placeholder trigger present/disabled if shipped).

It deliberately does **not** ship: any `ai-workflows` integration or AI call of any kind (the next task); any Quiz *generation* — fresh or replayed (the next task); the replay-+-fresh composition logic (manifest §7 — a later task, after generation exists); async result delivery / the Notification entity surface (a later task; the schema may *name* a `notifications` table per ADR-022's forecast but does not have to — `/design` decides whether to ship it now or defer it to the task that has a consumer); the Grade entity's full schema (`/design` decides whether to ship a minimal `grades`-or-`attempt_questions` shape now or defer it — it has no consumer until grading exists, so the architecture-on-spec test applies); any Quiz-status decoration in a rail (no consumer yet; likely deferred); a Quiz-taking surface (where the learner writes code against a Question — a later task); cross-Section anything (forbidden, §5); a per-Chapter Quiz aggregation entity (forbidden, §6); changes to the Notes surface, the completion toggle, or the rails (all settled by TASK-009/010/011/012); the parked ADR-032 (Notes-save scroll — not touched, this task does not touch the Notes form); the two Open parser-fidelity issues; lint/type-check tooling.

It advances the primary objective by **finally starting the Quiz pillar** — the third and largest of the three named consumption surfaces, and the home of the reinforcement loop manifest §7 calls "the reason this project exists" — after three deferrals, by laying the schema + read surface the loop is built on. It is sized as a vertical slice (schema + route + template + tests, mirroring exactly how Notes was bootstrapped in TASK-009 and section-completion in TASK-010), forcing 1–3 ADRs (the Quiz schema; the per-Section Quiz surface placement; possibly the Topic vocabulary), none of which is `ai-workflows`. The next task is **Quiz generation via `ai-workflows`** — which then has a settled schema and surface to write into, and can focus purely on the AI integration + async delivery + Notification rather than also inventing the schema and the surface mid-flight.

## Acceptance criteria

- [ ] Given the dev server starts against a fresh (empty) `data/notes.db`, when the persistence layer bootstraps the schema, then the Quiz domain tables exist (the exact table set — `quizzes`, `questions`, `quiz_attempts`, and whatever join/history table the Question Bank ↔ Quiz Attempt relationship needs — is decided in `/design`'s schema ADR), every one created via `CREATE TABLE IF NOT EXISTS` (ADR-022's migration story), and re-running the bootstrap on the same DB is idempotent (no error, no duplicate-table failure).
- [ ] Given the Quiz schema, when the tables are inspected, then **the `quizzes` table has a `section_id TEXT NOT NULL` column and no Chapter-bound-Quiz column / no cross-Section-aggregation column** — every Quiz row references exactly one Section (MC-2 / manifest §6 / §7). There is no `chapter_id`-only Quiz row that aggregates Sections, and no route or query in the codebase composes a Quiz from Questions of more than one `section_id`.
- [ ] Given the Quiz schema, when the tables are inspected, then **no Quiz-domain table has a `user_id` column** and no auth / per-user-partitioning code is introduced (MC-7 / manifest §6).
- [ ] Given the Question schema, when the `questions` table is inspected, then it carries a coding-task prompt column and a Topic-tag relationship (the shape — a TEXT column with a delimited tag list, a `question_topics` join table, or a `topics` table + join — is `/design`'s call), and it carries **no `option_*` / `correct_choice` / `answer_text` / `describe_*` / `recall_*`-style column** — the schema does not admit a non-coding Question format (manifest §5 / §7: every Question is a hands-on coding task).
- [ ] Given a Question persisted to a Section's Question Bank, when it is associated with a second Quiz for the same Section, then the schema permits it (a Question may appear in multiple Quizzes for its Section over time — manifest §8); and given a Quiz is deleted or a Quiz Attempt is discarded (no such path is shipped this task, but the schema is checked), then the schema does not cascade-delete the Question from the Bank (the Question Bank is never deleted — manifest §8; `/design` decides whether this is enforced by `ON DELETE` clauses, by the absence of any delete path, or both).
- [ ] Given a Quiz Attempt row, when it is inspected, then it carries a status column whose enum reserves the async-grading lifecycle states (the exact set — e.g., `requested`, `generating`, `awaiting_submission`, `submitted`, `grading`, `graded`, `failed` — is `/design`'s call), **including a failure state** (so the future grading task surfaces an AI failure as a failure rather than fabricating a Grade — MC-5 / manifest §6), and it carries the structure to record per-Question correctness (so wrong-answer-replay history exists for the future composition logic — MC-8 / manifest §7); `/design` decides whether the per-Question-correctness structure is shipped as columns now or as a separate `attempt_questions`-style table now or stubbed for the grading task — but the schema must not require a non-additive migration to add it.
- [ ] Given the new persistence module(s) (`app/persistence/quizzes.py` and any others `/design` adds), when the codebase is grepped, then `import sqlite3` and SQL string literals (`SELECT` / `INSERT` / `UPDATE` / `DELETE` / `CREATE TABLE` / `BEGIN` / `COMMIT` / `ROLLBACK` in a string literal) appear **only** under `app/persistence/` (MC-10), and routes/templates call only the typed public functions exposed by `app/persistence/__init__.py` — they never receive a `sqlite3.Connection` or a raw row tuple (they receive the Quiz/Question/QuizAttempt dataclass/`TypedDict` instances ADR-022's pattern uses for Notes).
- [ ] Given the dev server is running and the user navigates to `GET /lecture/{chapter_id}` (any of the 12 corpus Chapters), when the page renders, then **a per-Section Quiz surface is present for each Section** — with no Quizzes yet for that Section, it shows an empty-state (the exact copy — e.g., "No quizzes yet for this Section" — is `/design`'s call), and the surface's placement in the reading flow (forecast: in or near the `.section-end` wrapper per ADR-027 §Future-surfaces — but `/design` decides afresh) is recorded in the placement ADR.
- [ ] Given the per-Section Quiz surface, when the user is on a Lecture page, then the surface respects the parent Chapter's Mandatory/Optional designation (inherited via ADR-004's function — Mandatory Chapters' Sections and Optional Chapters' Sections both show the surface; nothing about the surface hides the M/O split — MC-3 / manifest §6).
- [ ] Given `/design` decides to ship a placeholder per-Section "Generate a Quiz for this Section" affordance, when the user interacts with it, then **either** a real route (`/design` decides the shape) creates a Quiz-or-QuizAttempt row in a `requested`/`pending` status whose downstream processing is explicitly the next task's job (no AI call is made this task — the row sits in `requested` status, which is not a fabricated result, it is a recorded user request), **or** the affordance is rendered disabled / as a "coming next" caption with no route — in **no case** does anything auto-generate a Quiz, run a background job, or call an LLM SDK (MC-9 / MC-1 / manifest §7).
