## What and why

The corpus is 12 Chapters. Only **Chapter 1** has been visually validated as a renderable Lecture page; the other **11 Chapters have never been rendered through `GET /lecture/{chapter_id}` and confirmed via the ADR-010 screenshot-review gate.** TASK-001 shipped the pipeline; TASK-002/003 shipped navigation; TASK-004 fixed two parser bugs surfaced by Chapter 1's screenshot review. Each prior screenshot-review pass on Chapter 1 surfaced new parser bugs (tabular column-spec passthrough, callout-title passthrough, `\\` linebreak passthrough). It is overwhelmingly likely the other 11 Chapters contain additional latent parser bugs that the project has not yet seen, because the project has never looked.

This task is the first **triage pass** across the whole corpus. The goal is **not** to fix every bug found — it is to (a) confirm every Chapter renders without crashing, (b) capture screenshots of every Chapter for the human's review, and (c) catalog any latent parser/rendering bugs as new project_issues so they can be triaged into focused follow-up tasks. Manifest §3 (drive consumption) is the load-bearing motivation: a corpus where 11/12 Chapters have unknown rendering quality is not a consumption surface, it is a consumption hypothesis.

This task is also the natural moment to resolve the `latex-linebreak-macro-passthrough-in-titles.md` project_issue. The architect decides during `/design` whether to fold the `\\` linebreak fix into this task's scope (since the validation pass will surface it across all 12 titles) or to keep it as a separate follow-up task.

## Architectural decisions expected

- **Scope and shape of the validation harness.** Three options materially differ:
  - **(a) Pure Playwright batch test:** one parameterized Playwright test that walks all 12 Chapter IDs, asserts HTTP 200 + screenshot capture, and lets the human review the `tests/playwright/artifacts/` set. Reuses ADR-010's gate; no new mechanism.
  - **(b) Playwright batch + HTTP-protocol pytest layer:** a parameterized HTTP-only pytest that asserts 200 on every `GET /lecture/{chapter_id}` (fast, no browser), plus a Playwright pass for screenshots. Splits "does it not crash" from "does it look right."
  - **(c) Add a CLI tool that renders all 12 Chapters to disk for offline human review:** a separate `python -m app.render_all` command that writes 12 HTML files to a tmp dir. Adds a new code surface (CLI), trades against the in-test workflow ADR-010 commits to. Probably wrong shape but worth recording.
  The architect picks one in `/design`. Recommend (b) for separation of concerns + speed of feedback.
- **`\\` linebreak macro handling.** Either (a) resolve the existing project_issue with a new ADR — choosing Option 1 (regex strip + small macro whitelist) or Option 2 (route titles through pylatexenc) — and fold the fix into this task's implementation; or (b) explicitly defer with a recorded justification and let the project_issue stay Open. The architect picks one. The recommended path is to resolve now (Option 1, regex strip + whitelist for `\\`, `\large`, and similar pure-formatting macros), because the validation pass will make this bug visible across all 12 Chapter rows in screenshots and the fix is bounded.
- **Bug-triage discipline.** This task is a triage pass, not a fix-everything pass. The `/design` ADR (or this task's framing alone if no ADR is forced beyond the harness shape and the linebreak fix) records that **new latent bugs are filed as project_issues, not folded in.** This is the inverse of TASK-004's "fix two parser bugs in one task" discipline; the difference is scale: TASK-004 fixed two known bugs surfaced from one Chapter's review; this task expects to surface an unknown number of bugs from 11 unvisited Chapters, and bundling them all would explode scope.
- **Whether the audit-file `Status: Implemented` (TASK-004 final state) and the reviewer's `CHANGES REQUESTED` need reconciliation before this task implements.** TASK-004 was committed (`ad52ab2`) despite the reviewer's CHANGES REQUESTED on findings that include an ARCHITECTURE LEAK in `CLAUDE.md` and stale-text in two project_issue files. The leak is human-owned (Tier-2 file). The stale-text is architect-owned. The architect mechanically reconciles the stale-text in `/design TASK-005` as a pre-flight cleanup; the leak is re-flagged for the human in this task's "Architectural concerns" section. No formal reconciliation ADR is anticipated; this is index-maintenance and human-flagging.

## Alternatives considered (task direction)

Five materially different next-task directions were considered.

1. **(Chosen) Validate Chapters 2-13 as a triage pass; surface latent bugs as new project_issues.** Smallest vertical slice that answers the question manifest §3 forces: "is the consumption surface actually consumable across the whole corpus?" Re-uses ADR-010's gate at scale for the first time. Cost is bounded (one parameterized Playwright test + 12 screenshots + human review). Surfaces the latent-bug surface for prioritization without committing to fix everything in one cycle. Most importantly, it converts the project's current state ("we hope the other 11 Chapters render") into known state ("here is the list of bugs we have"). The follow-up tasks then operate on real evidence, not speculation.

2. **Fix the `\\` linebreak passthrough in titles (resolves the only Open project_issue).** Smaller scope than this task. Rejected as the *primary* direction because: (a) it addresses one known bug while leaving the larger validation question unanswered; (b) the validation pass would surface this bug across all 12 titles anyway, so the fix is more naturally scoped as part of the validation pass (the architect can fold it in during `/design`); (c) doing the linebreak fix first means going through a multi-chapter validation later anyway and re-finding the linebreak alongside whatever else is found. Better as an in-scope or just-after side effect of this task than as a standalone next task.

3. **Begin the persistence layer (force the persistence-layer ADR currently dormant in MC-10).** A real architectural commitment that the manifest §7 invariants ("Quiz Attempt, Note, completion mark persists across sessions") will eventually demand. Rejected as the *next* task because: (a) building persistence on top of a rendering surface where 11/12 Chapters have unknown quality is "building on a cracked foundation" — the same argument TASK-004's framing made about Notes; (b) persistence is a substantial vertical slice (storage choice, schema shape, migration story, transaction boundary) that does not fit "small enough for one session"; (c) the right time to commit to persistence is when the consumption surface is stable AND the first persistence-bearing entity (Note, Quiz Attempt, completion mark) has a concrete task forcing it. None of those preconditions hold today.

4. **Begin the Notes vertical slice (first user-authored content surface; manifest §8 entity).** Forces the persistence-layer ADR. Rejected as the *next* task for the same reasons as Option 3: builds on unverified rendering, drags a heavyweight ADR before the consumption surface is stable, exceeds "one session" scope.

5. **Address the CLAUDE.md ARCHITECTURE LEAK and stale-text findings from TASK-004's reviewer as a focused cleanup task.** Smaller than even Option 2. Rejected as a *task* because: (a) the leak is human-owned (Tier-2; only the human edits CLAUDE.md) — agents cannot fix it; (b) the stale-text in the two project_issue files is architect-owned and is a routine mechanical fixup that fits inside any next architect run as a pre-flight, not a standalone task. Both findings are re-surfaced in this task's "Architectural concerns" section; the architect handles the stale-text during `/design TASK-005` and the leak is the human's call.
