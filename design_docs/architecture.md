## Project structure (high level)

*Derived from the current Accepted-ADR set (001–008, 010–022, 024, 026, 027, 028, 029, 030, 031, 033, 034, 035, 036, 037, 038, 039, 040, 041, 042, 043, 044; ADR-009 was never assigned; ADR-023 and ADR-025 are Superseded — ADR-023 §Template-surface by ADR-028, ADR-025 §Template-placement by ADR-027 and §Round-trip-return-point by ADR-030; ADR-028's §Rail-integration / §Template-surface portion is superseded by ADR-029, its remaining decisions still in force; ADR-030's §Decision / no-fragment mechanism is superseded by ADR-031, but ADR-030's load-bearing principle remains in force; ADR-032 is `Proposed` and deferred — not in force).*

**Source corpus and rendering pipeline.** Lecture source is a read-only LaTeX corpus under `content/latex/`, one `.tex` file per Chapter, basenames in the canonical form `ch-{NN}-{slug}` (ADR-001, ADR-005); each Chapter ID is the file basename, and Section IDs are `{chapter_id}#section-{n-m}` derived from `\section{N.M ...}` numbering (ADR-002). A local FastAPI application parses one Chapter `.tex` at request time using `pylatexenc`, renders the structured intermediate representation through Jinja2, and serves it on `127.0.0.1` (ADR-003); each Section dict carries a derived `section_number` field (the part after `section-` in the fragment) for route-URL composition (ADR-025). The Mandatory/Optional designation for any Chapter is computed by a single Python function whose threshold cites manifest §8, with no config file or sidecar (ADR-004).

**Parser fidelity rules.** Tabular column-spec arguments are stripped from rendered output via balanced-brace consumption, with warn-per-node for complex spec features (ADR-011, ADR-017); callout `[Title]` arguments are extracted and emitted as `<div class="callout-title">` inside the callout div (ADR-012); the `\\` linebreak macro is stripped from title extraction (ADR-014); `\texttt{}` renders as `<span class="texttt">` so MathJax processes embedded inline math (ADR-018); unhandled LaTeX environments use a generic consume-and-skip fallback with no per-environment editorial commitment (ADR-019); raw-text fallback paths run a defensive macro-stripping pass so text-formatting macros do not leak (ADR-020). Multi-chapter validation splits Playwright (visual) and HTTP-protocol pytest (smoke) (ADR-013); validation-pass bugs are routed by class — LaTeX/parser folded in, rail/CSS escalated to its own task, else filed as a project_issue (ADR-015).

**Navigation and page layout.** Navigation lives on a `GET /` landing page and on a left-hand rail included in every Lecture page via a shared `base.html.j2`, both surfaces rendering from one `discover_chapters()` helper (ADR-006); that helper discovers Chapters by scanning `content/latex/` at request time, labels each row from the Chapter's `\title{...}` macro using the same extractor the Lecture page uses, and orders within each Mandatory/Optional group by parsed chapter number ascending (ADR-007). Page-chrome and rail styling live in `app/static/base.css`, while Lecture-body styling stays in `app/static/lecture.css`; the two files are non-overlapping by class-name prefix (`page-*`, `nav-*`, `index-*`, `rail-*`, `lecture-rail` → `base.css`; `lecture-*`, `section-*`, `callout-*`, `designation-*` → `lecture.css`), both loaded flat from the base template (no preprocessor, no build step), the rail's Mandatory/Optional headings reuse the `designation-mandatory`/`designation-optional` palette the Lecture badge uses, and the page-level layout is a CSS Grid (ADR-008). The left rail carries a per-Chapter derived "X / Y" progress decoration on each chapter row, fed by a single bulk persistence accessor and clamped so X never exceeds Y (ADR-026).

**Persistence.** SQLite via the stdlib `sqlite3` module is the persistence layer; `app/persistence/` is the only package that talks to the database; the Note schema, the section-completion schema, and the Quiz domain schema all live there in one shared `data/notes.db` (ADR-022, ADR-024, ADR-033). Notes are bound to a Chapter and stored most-recent-first; section completion uses presence-as-complete semantics in a `section_completions` table under `app/persistence/section_completions.py` with write-time validation at the route handler (ADR-022, ADR-024).

**Quiz domain schema.** The Quiz pillar's persisted entities live in `app/persistence/quizzes.py` as five tables created via `CREATE TABLE IF NOT EXISTS` in the same bootstrap block as the Note/section-completion schema (ADR-033, under ADR-022's umbrella): `quizzes` (a `section_id TEXT NOT NULL` column holding the `{chapter_id}#section-{n-m}` form per ADR-002, validated at the route handler; a `status` generation-lifecycle enum stored as TEXT — `requested` → `generating` → `ready` → `generation_failed`); `questions` (a Section's Question Bank — a coding-task `prompt` column, a `'|'`-delimited `topics` TEXT column, and a nullable `test_suite TEXT` column holding one runnable test-code string per Question, no choice/recall/describe columns); `quiz_questions` (a many-to-many join so a Question may appear in multiple Quizzes for its Section over time); `quiz_attempts` (an attempt/grading-lifecycle `status` enum — `in_progress` → `submitted` → `grading` → `graded` → `grading_failed`); and `attempt_questions` (one row per Question per Attempt carrying `response` / `is_correct` / `explanation`, the latter two NULL until grading, so per-Question wrong-answer-replay history is recordable without a non-additive migration; plus four nullable test-result columns added by ADR-044 — `test_passed INTEGER` (1/0/NULL — meaningful only when `test_status = 'ran'`), `test_status TEXT` (`ran` | `timed_out` | `compile_error` | `setup_error`, mirroring ADR-042's `RunResult.status`), `test_output TEXT` (the run's combined stdout+stderr / compiler diagnostic / setup-failure message, truncated to a sane cap), `test_run_at TEXT` (the ISO-8601 UTC timestamp of the latest run) — holding the latest in-app test run for that response, all NULL until the learner runs the tests, written by `save_attempt_test_result`; the structured `test_status` column carries the failure mode so a timeout / compile error / setup error is never conflated with pass/fail (MC-5's spirit); `is_correct` stays untouched by the runner — the grading slice (next) reads the persisted test result and sets it). No `user_id` on any table. `quizzes` also carries a nullable `generation_error TEXT` column (added by ADR-037 via additive `ALTER TABLE ADD COLUMN`) holding the failure detail when generation fails — a debugging aid for the author; the learner-facing signal stays the honest "Generation failed". `questions` carries a nullable `test_suite TEXT` column (added by ADR-041 via the same additive `ALTER TABLE ADD COLUMN` / `_apply_additive_migrations` mechanic, also declared in the `CREATE TABLE questions` block for fresh databases) holding the runnable test-code string ADR-040's `question_gen` workflow emits per Question; NULL only for a Question that predates TASK-016 — a Question persisted from TASK-016 forward always has a non-empty `test_suite`, since ADR-040's generation processor fails the whole Quiz (`generation_failed`, zero `questions` rows) rather than persist a Question without one or synthesize a placeholder (MC-5); the `Question` dataclass gains a `test_suite: str | None` field that the existing `Question`-returning accessors (e.g. `list_questions_for_quiz`) carry through, and `add_questions_to_quiz`'s per-Question payload dict gains a `"test_suite"` key; the in-app test-runner slice (ADR-042–ADR-044) added the first runtime consumer — `get_question(question_id) -> Question | None` in `app/persistence/quizzes.py`, which the "Run tests" route uses to fetch the target Question's `test_suite` to feed the sandbox. The relational Topic vocabulary (a `topics` table + `question_topics` join), the Grade aggregate (score / weak-topics / recommended-sections), and a `notifications` table are deferred to their consuming tasks — all additive. `app/persistence/__init__.py` exposes the typed public functions; `import sqlite3` and SQL literals stay under `app/persistence/`.

**Per-Section Quiz surface.** Each `<section>` on the Lecture page carries a Quiz block inside the existing `<div class="section-end">` wrapper, after the completion form (ADR-034, the cognitive-sequence action zone ADR-027 forecast): an empty-state caption ("No quizzes yet for this Section.") when the Section has no Quizzes, otherwise a plain-language Quiz-status list that never presents a `requested`/`generating` Quiz as finished — `requested` → "Requested — generation pending"; `generating` → "Generating…"; `ready` → "Ready" plus a "Take this Quiz" button-styled link to the Quiz-taking surface (`section-quiz-take-link` in `lecture.css`, added by ADR-038); `generation_failed` → "Generation failed". The `requested`/`generating`/`generation_failed` entries get no takeable affordance; the empty-state caption and the "Generate a Quiz" button are unchanged by ADR-038. A user-triggered `POST /lecture/{chapter_id}/sections/{section_number}/quiz` route validates the Section ID at the route handler, applies the **first-Quiz-only guard** — if the Section already has a non-failed Quiz (`requested`, `generating`, or `ready`; a `generation_failed` Quiz does not count, so re-clicking after a failure is allowed) the request is rejected with HTTP 409, so a fresh-only post-first Quiz can never be produced this slice (ADR-037; MC-8 stays clean by construction; the replay-+-fresh composition path that would handle a second request is a later slice) — otherwise calls `app.persistence.request_quiz(section_id)` (inserts one `status='requested'` Quiz row — no AI call, no background job, nothing fabricated) and returns a `303 See Other` PRG redirect to `GET /lecture/{chapter_id}#section-{section_number}-end`, reusing ADR-031's `#section-{n-m}-end` + `scroll-margin-top` no-relocate recipe unchanged; `render_chapter` gains one bulk `list_quizzes_for_chapter` query passed to the template as `section_quizzes_by_id` (mirroring `complete_section_ids` and `rail_notes_context`). **Quiz-taking surface.** A `ready` Quiz is taken via a `GET/POST /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take` route pair in `app/main.py`, keyed under the existing per-Section URL space (ADR-038): `GET` validates `chapter_id` (corpus `.tex` exists / well-formed — `404`/`422`), `section_number` (a known Section — `404`), `quiz_id` (an existing Quiz whose `section_id` matches the URL — `404`), and the Quiz being `ready` (a `requested`/`generating`/`generation_failed` Quiz renders an honest "not ready to take" state, no takeable form — exact status code implementer's call), then checks `get_latest_attempt_for_quiz(quiz_id)` — if the latest Attempt for the Quiz is `submitted` it renders the submitted state on that Attempt; the take surface re-renders in the submitted state). The submit route does **not** invoke grading — no `ai-workflows` call, no LLM SDK, no out-of-band job kicked off from the request; the Attempt sits in `submitted` until the (later) grading slice's out-of-band processor picks it up, mirroring ADR-037's generation processor, and the submit route is written so that adding that processor later needs no change to it (MC-4). A third route, `POST .../take/run-tests` (same path-param validation; ADR-043), is the "Run tests" affordance's target: it parses the whole take form, calls `save_attempt_responses(attempt_id, responses)` to persist every Question's in-flight code first, fetches the target Question's `test_suite` via `get_question(question_id)`, runs it against that Question's code through ADR-042's `run_test_suite(test_suite, response) -> RunResult`, persists the structured result via `save_attempt_test_result(attempt_id, question_id, passed=…, status=…, output=…, run_at=now)` (ADR-044), and returns `303 See Other` → `GET .../take#question-{question_id}` (the GET re-render shows every textarea repopulated and that Question's results panel updated; the `.quiz-take-question` block's `scroll-margin-top` keeps the learner on the Question they ran — ADR-031's no-relocate recipe pattern). Running tests is a within-`in_progress` action — it does not flip the Attempt's status, invoke grading (MC-4), generate a Quiz (MC-9), or write under `content/latex/` (MC-6); a `timed_out` / `compile_error` / `setup_error` `RunResult` is persisted and rendered as that status, never a fabricated pass/fail (MC-5's spirit). The take-page CSS is a new flat `app/static/quiz.css` file (the `quiz-take-*` namespace — `.quiz-take`, `.quiz-take-header`, `.quiz-take-form`, `.quiz-take-question`, `.quiz-take-prompt`, `.quiz-take-response`, `.quiz-take-submit`, `.quiz-take-submitted`, `.quiz-take-response-readonly`, `.quiz-take-not-ready`, `.quiz-take-back-link`, plus ADR-043's `.quiz-take-test-suite`, `.quiz-take-run-tests`, `.quiz-take-results`, `.quiz-take-results-pass`, `.quiz-take-results-fail`, `.quiz-take-results-status`, `.quiz-take-results-output` and a `scroll-margin-top` on `.quiz-take-question`), loaded flat from `base.html.j2` alongside `base.css` and `lecture.css` (no preprocessor, no build step — ADR-008's posture extended to a third per-surface file, which ADR-008 §Future-surfaces forecast); the "Take this Quiz" affordance on the `.section-quiz` block's `ready` entry reuses the `section-quiz-*` namespace in `lecture.css` (a `.section-quiz-take-link` button-styled-link rule — no new file for that part, no `base.css` change) (ADR-038, ADR-008). INSERTs the `quiz_attempts` row and one `attempt_questions` row per Question — `response`/`is_correct`/`explanation` NULL — at Attempt start, in one transaction, so `save_attempt_responses` is a clean `UPDATE` and the per-Question structure exists from the moment the Attempt does); `get_attempt(attempt_id) -> QuizAttempt | None`; `get_latest_attempt_for_quiz(quiz_id) -> QuizAttempt | None` (the most recent Attempt for a Quiz, any status — by `created_at`/`attempt_id` — or `None`; lets `GET .../take` show the `submitted` state after the PRG redirect without `start_attempt` spawning a fresh `in_progress` Attempt; added in-flight during TASK-015 implementation and human-gated — see ADR-039's Amendment note); `list_questions_for_quiz(quiz_id) -> list[Question]` (a `quiz_questions ⨝ questions` join ordered by `position`); `list_attempt_questions(attempt_id) -> list[AttemptQuestion]` (one per `attempt_questions` row, joined with the Question's `prompt` + `test_suite` and `quiz_questions.position` and carrying the row's `test_passed`/`test_status`/`test_output`/`test_run_at`, ordered by `position` — ADR-044); `save_attempt_responses(attempt_id, responses: dict[int, str]) -> None` (per-row `UPDATE attempt_questions SET response = ?`); `submit_attempt(attempt_id) -> None` (`UPDATE quiz_attempts SET status='submitted', submitted_at=<now>` — touches no `attempt_questions` row, no `is_correct`/`explanation`, invokes no grading — MC-4); ADD COLUMN` block in `_apply_additive_migrations` (mirroring ADR-037's `generation_error` / ADR-041's `test_suite` — additive, no migration trigger). `import sqlite3` and the SQL literals stay in `app/persistence/`; the take route, the submit route, the `run-tests` route, `app/sandbox.py`, and `quiz_take.html.j2` call only the typed public functions re-exported by `app/persistence/__init__.py`, receiving dataclasses (MC-10) — and `app/sandbox.py` calls none of them (the route, not the sandbox, persists the result). The grading workflow, the Grade aggregate's schema and surface, the `notifications` table and a Notification surface, and the replay-+-fresh composition for post-first Quizzes are all later slices and explicitly out of scope of ADR-038/ADR-039 (the in-app code-execution/test-runner surface ADR-038/ADR-039 deferred has since been added by ADR-042–ADR-044 — see "In-app test runner (code-execution sandbox)" below; the grading slice's shape — the out-of-band grading processor, the Grade schema, the Weak-Topic-identification path and the relational Topic vocabulary, the `notifications` table — is tracked in `design_docs/project_issues/quiz-grading-slice-shape.md`).

**In-app test runner (code-execution sandbox).** The §6-mandated "Code is written, run, and tested within the application" loop is realized by an in-app test runner that executes a Question's `test_suite` against the learner's submitted code (ADR-042 the engine, ADR-043 the route + surface, ADR-044 the persistence). The engine is a new module **`app/sandbox.py`** (under `app/` but **not** under `app/workflows/` — MC-1 stays vacuously clean: no LLM, no `ai-workflows` call, no forbidden SDK; the sandbox runs *test code* against *implementation code*) exposing one public entry point — `run_test_suite(test_suite: str, response: str) -> RunResult` — that: creates a fresh throwaway working directory via `tempfile.mkdtemp()` (under the OS temp root, never under `content/latex/` or `data/` — MC-6 is load-bearing here; removed in a `finally`); writes the spliced source — **the learner's `response` concatenated with the Question's `test_suite`** into one translation unit (correct for an assertion-only test suite that *references* the implementation target by name; ADR-040's *current* generated output embeds a reference implementation inline, so the concatenation double-defines the target and the run honestly reports `compile_error` — the fix is a small `question_gen`-prompt change tracked in `design_docs/project_issues/question-gen-prompt-emit-assertion-only-test-suites.md`, a future TASK; a heuristic body-replacement splice was considered and rejected as too fragile / dishonest-on-failure); sniffs the language — **C++ via `g++`** when the source contains `#include` (the language ADR-040's prompt steers toward), **Python via `python3`** when it has `import`/`def test`/`unittest`/`pytest` markers and no `#include`, anything else → a `setup_error`; compiles (for C++) and runs the program as a subprocess with `cwd=` the temp dir (so any file the learner's code writes goes there, not under `content/latex/`), a fresh minimal env, `stdin` closed, a `subprocess.run(timeout=...)` wall-clock cap (a few seconds — tunable, like ADR-036's tier model string), and a POSIX `preexec_fn` that calls `resource.setrlimit` for `RLIMIT_CPU` / `RLIMIT_AS` / `RLIMIT_NPROC` / `RLIMIT_FSIZE` / `RLIMIT_CORE = 0` and `os.setsid()` (so a timeout-kill reaps the whole process group); and returns a `RunResult` dataclass — `status` (`'ran'` | `'timed_out'` | `'compile_error'` | `'setup_error'`), `passed: bool | None` (meaningful only when `status == 'ran'`: `True` on exit 0, `False` on non-zero exit / aborted `assert`; `None` otherwise), `output: str` (combined stdout+stderr, or the compiler diagnostic, or the failure message — truncated to a sane cap, e.g. 16 KiB, to bound the size persisted). `status` is the load-bearing field for honesty — `passed` alone cannot distinguish "ran and failed" from "timed out" from "didn't compile", and conflating those would be the MC-5-spirit violation. `RunResult` is the sandbox module's transient return type, **not** a persistence dataclass — the persisted view is ADR-044's `AttemptQuestion` extension; two dataclasses at two layers. No container, no `bubblewrap`/`nsjail`/`firejail`, no WASM/`pyodide`: the bare-subprocess-under-rlimits approach is the lean fit for the single-user / single-machine / locally-run threat model (manifest §5 — "the attacker is the author, who can already run anything on their own machine"; the requirement is robustness, not adversarial isolation); a Linux-native jail is the documented future upgrade behind the same `run_test_suite` interface if the bare subprocess ever proves leaky. `app/sandbox.py` uses only the stdlib (`subprocess`, `resource`, `tempfile`, `shutil`, `os`, `pathlib`) and is **POSIX/Linux-only** (consistent with the project being a locally-run personal project); **no `pyproject.toml` dependency** is added (`g++` / `python3` are system binaries invoked via `subprocess`, not pip packages). MC-4 does **not** bind the test-run path — running a test suite is executing code, not AI work; the manifest's async-AI rule is about LLM-driven processing (generation, grading, lecture audio), which this isn't — so the run is synchronous-with-a-timeout (ADR-042 records *why* MC-4 is not in play; ADR-043 makes the synchronous-vs-async route call on its merits and chose synchronous + no JS). The route, surface, and persistence wiring are described under "Quiz-taking surface" above (the `POST .../take/run-tests` route; the read-only test-suite block, "Run tests" button, and results panel in `quiz_take.html.j2`; the `attempt_questions` test-result columns + `save_attempt_test_result` + `get_question` in `app/persistence/quizzes.py`).

**AI integration and Quiz generation.** `ai-workflows` is the project's AI engine (manifest §4) — CS-300 depends on the `jmdl-ai-workflows` PyPI package (a version pin `>= 0.4` in `pyproject.toml` `[project] dependencies`; `requires-python` is `>= 3.12`; the co-located `/home/papa-jochy/prj/ai-workflows` clone is read-only reference only, not a path dependency) (ADR-036). The first-Quiz, fresh-only generation workflow is a CS-300-owned declarative `WorkflowSpec` module at `app/workflows/question_gen.py` — one `LLMStep` (with a `prompt_fn` so LaTeX `{}` in the Section content cannot collide with `str.format` placeholders) plus the required `response_format` validator, a `question_gen_tier_registry()` helper naming one LLM tier, `QuestionGenInput` carrying the Section's content + title, and `QuestionGenOutput.questions: list[GeneratedQuestion]` where `GeneratedQuestion` has a `prompt: str` (a hands-on coding-task instruction), `topics: list[str]`, and a `test_suite: str` (`min_length=1` — one runnable test-code string verifying an implementation of the Question's coding task; ADR-040 — the `LLMStep` prompt instructs the model to emit it naming the implementation target so the learner's code has a stable signature to satisfy) — no field anywhere that could hold a multiple-choice / true-false / short-answer / describe / recall item (a non-coding Question is unexpressible, and `test_suite` holds only test *source code*, never an option list / true-false key / describe prompt / recall answer — manifest §5 / §7); the module is loaded into the framework's registry via `AIW_EXTRA_WORKFLOW_MODULES=app.workflows.question_gen` (ADR-036). Processing runs out-of-band — never inside a request handler (MC-4): the manual command `python -m app.workflows.process_quiz_requests` polls `requested`-status Quiz rows and, for each, sets `generating`, reads the Section's `.tex` content (read-only — MC-6, via the existing parse path), invokes the workflow by **shelling out to `aiw run question_gen --input ... --run-id quiz-{quiz_id}`** as a subprocess (with `AIW_EXTRA_WORKFLOW_MODULES` in the subprocess env — the documented `ai-workflows` downstream surface, not the framework's internal `_dispatch.run_workflow`), reads the terminal artefact from the CLI's machine-parseable stdout (`json.dumps(artifact, indent=2)` then a `total cost: $X.XXXX` trailer; structured logs on stderr; a non-zero exit / `error: …` on stderr is a failure), persists the generated Questions to the Section's Question Bank (`questions` rows — never deleted; each row also carries the Question's `test_suite` read from the artefact and persisted verbatim — never transformed, never regenerated — ADR-040/ADR-041) and links them to the Quiz (`quiz_questions` rows) via `app.persistence.*` only (the workflow returns data; CS-300 code persists it — MC-10), and sets `ready` (ADR-037). On failure: `generation_failed`, no `questions` synthesized, the detail recorded in `quizzes.generation_error`, no unbounded silent retry (the framework's `RetryingEdge` does bounded retry inside the workflow run; the command does not re-process `generation_failed` rows; a fresh attempt is a new user-triggered `requested` row — MC-5 / MC-9) (ADR-037); a Question whose LLM-supplied `test_suite` is missing / empty / whitespace-only is itself a generation failure — the whole Quiz becomes `generation_failed` with zero `questions` rows persisted and no placeholder test suite synthesized anywhere (now that §8 says a Question "carries a test suite", a Question without a real one is not shippable — ADR-040 / MC-5). For this slice the learner is "notified" the Quiz is ready by the per-Section Quiz surface flipping to "Ready" on the next `GET /lecture/{chapter_id}` (a learner-visible, non-real-time indication — manifest §5/§6/§8); the active Notification entity (a `notifications` table + a learner-visible surface) ships with the grading slice (ADR-037). No `app/` code imports any LLM/agent SDK directly — MC-1's architecture portion enumerates the forbidden set (`openai`/`anthropic`/`google.generativeai`/`google.genai`/`cohere`/`mistralai`/`groq`/`together`/`replicate`/`litellm`/`langchain*`/`langgraph*`/a raw provider-API HTTP call) and names the workflow path `app/workflows/`; the only AI import in `app/` is `ai_workflows.*` (the `WorkflowSpec` authoring API + `ai_workflows.primitives.tiers` types); AI *invocation* goes through the `aiw` CLI subprocess (ADR-036). `ai-workflows` keeps its own run/checkpoint store (`~/.ai-workflows/*.sqlite3`, env-overridable); CS-300's `data/notes.db` carries no `ai-workflows` tables (ADR-036, ADR-022 §`ai-workflows`-cohabitation option (c)). Tests use CS-300's own processor boundary as the seam — the `aiw run` subprocess call is mocked to return a canned artefact or raise a failure; no test makes a real LLM/provider call (MC-1's spirit); the real-engine generation is a human-only verification gate (the framework's in-process `StubLLMAdapter` monkeypatch is not usable across a subprocess) (ADR-036). The relational Topic vocabulary (`topics` table + `question_topics` join) ADR-033 deferred "to the generation task" stays deferred — to the Weak-Topic-identification slice (the first thing that *queries* across Topics); first-Quiz generation only *stores* tags, which `questions.topics` (the `'|'`-delimited TEXT column) holds; the migration to the relational form is additive (ADR-036).

**Notes surface.** Notes are created via `POST /lecture/{chapter_id}/notes` (form-encoded body, PRG 303 redirect to `GET /lecture/{chapter_id}`, no fragment), with route-handler validation (trim; reject empty/whitespace-only; reject unknown Chapter; reject bodies > 64 KiB); the Notes panel is rail-resident in a right-hand rail — `<section class="rail-notes">` lives in `_notes_rail.html.j2`, included by `base.html.j2` in the right-hand column of a three-column CSS Grid layout (LHS chapter rail | centered main | RHS Notes rail), `position: sticky`, with the grid degrading to two columns on `GET /` where no Chapter context exists; the panel is rendered only when a Chapter context exists (omitted from `GET /`), with a `rows="3"` textarea using CSS `field-sizing: content` plus `resize: vertical` fallback (a clean CSS-only auto-resize, so no JavaScript is needed here); multiple Notes display most-recent-first; submit feedback is a full-page PRG reload; no edit/delete, no Markdown (ADR-023 §non-template-surface decisions, ADR-028, ADR-029). The bottom-of-page Notes section that ADR-023 originally placed in `lecture.html.j2` was removed by ADR-028; ADR-028 originally placed the rail-resident panel in `_nav_rail.html.j2` (the left-hand rail), which ADR-029 superseded. (Parked: ADR-032 — `Proposed`, deferred, no task scheduled — records that the Notes-form PRG round-trip relocates the reader to the chapter top on save and that fixing it needs the project's first client-side JavaScript; not yet in force. Adopting that JS is a normal architectural step, not the breach of a no-JS rule — there is no binding no-JS posture; see ADR-035.)

**Section completion surface.** A per-Section completion toggle posts to `POST /lecture/{chapter_id}/sections/{section_number}/complete` with a form-encoded `action` field (`mark` | `unmark`), idempotent, with route-handler validation of `chapter_id`, `section_number`, and `action`, returning a `303 See Other` PRG redirect to `GET /lecture/{chapter_id}#section-{section_number}-end` — the fragment points at the `.section-end` wrapper (which carries `id="section-{n-m}-end"`), and `.section-end` carries a large viewport-relative `scroll-margin-top` in `lecture.css` so fragment navigation lands the wrapper near the bottom of the viewport ≈ where the user clicked, leaving the user where they were rather than relocating them (ADR-031, superseding ADR-030 §Decision; ADR-030 superseded ADR-025 §Round-trip-return-point's `#section-{n}` heading anchor; a Playwright regression test locks the post-toggle scroll-delta within ≤ 200px of the pre-click position); the affordance is a `<form>` inside that `<div class="section-end">` wrapper at the bottom of each `<section>` (ADR-027, superseding ADR-025 §Template-placement); a complete Section carries three layered indicators — button text, button-color modifier, and a `.section-complete` class on the `<section>` element; `GET /lecture/{chapter_id}` passes a `complete_section_ids` set to the template (ADR-024, ADR-025, ADR-027, ADR-030, ADR-031). This surface uses a synchronous form post + PRG redirect and needs no JavaScript — a description of what was clean and sufficient here, not a project-wide constraint (see ADR-035).

**Placement-quality principles** (encoded by the post-TASK-010/011/012 supersedures, binding on future surface placement): action affordances follow the cognitive sequence, not the template scope (ADR-027); visibility follows scroll-position-cost — a surface a user must scroll past the entire content to reach is, at scale, invisible (ADR-028; ADR-029 retains this principle and corrects its application to the rail with the real estate); and the response to a reading-flow action should not relocate the user — completion is an annotation on what was just read, not a navigation event (ADR-030's principle, implemented by ADR-031's `#section-{n}-end` anchor + `scroll-margin-top` mechanism after ADR-030's no-fragment mechanism was empirically refuted in Chromium).

**ADR scope** (ADR-035): an ADR records the architecture the project *uses* — the decision made and why — not a restriction of the available toolkit. JavaScript (and asset infrastructure) is available; no-JS / CSS-only solutions are preferred where clean and sufficient (and have been, for every surface shipped to date — those decisions stand) but are never mandatory, and choosing JavaScript when a surface genuinely needs it is a normal architectural step decided in its own ADR. There is no project-wide "no JavaScript" invariant — no manifest, CLAUDE.md, or conformance-skill entry establishes one — so reviewers do not enforce one; the "no-JS posture" framing that had accreted across ADR-006/023/025/027/028/029/030/031 (and ADR-034) was softened in place to descriptions of what each surface does, decisions unchanged.

**UI verification.** Rendered-behavior verification for any UI surface uses Playwright through the `pytest-playwright` plugin so all tests run under the single `python3 -m pytest tests/` command; rendered-DOM-content tests live under `tests/playwright/` while HTTP-protocol, source-static, and runtime-side-effect tests stay in pytest under `tests/`; the default browser is Chromium driven by a session-scoped `live_server` fixture against the local FastAPI app; per-test screenshots and on-failure traces from the last run are written to a single artifact directory under `tests/` that is excluded from version control; the verification gate is satisfied when the pytest run is green and the human reviews the last-run screenshots, recorded as a `rendered-surface verification — pass` row in the audit Human-gates table (ADR-010).

**Workflow infrastructure.** The orchestrator verifies each subagent's expected file outputs via `git diff` after every phase and remediates mechanical gaps directly, except test-assertion files which require a test-writer re-invocation (ADR-016, ADR-021).
