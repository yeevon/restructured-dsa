## Context

TASK-019 ships the *processor* for the Quiz pillar's grading half. ADR-048 decided the *workflow* (the CS-300-owned `WorkflowSpec` module at `app/workflows/grade_attempt.py`, with one `LLMStep`, the `GradeAttemptInput` / `GradeAttemptOutput` schemas, the prompt's architectural commitment "read the runner's verdict, produce explanation + aggregate, do not re-judge correctness", the `RetryPolicy(max_attempts=3, exponential)`, the tier registry). This ADR decides the *CS-300 side*: where the grading processing runs (out-of-band, never inside a request handler — MC-4 / §6), how it is triggered, how the learner is notified the result is ready, and the failure-handling discipline. The decision space mirrors ADR-037's exactly — *because the grading slice's out-of-band shape mirrors the generation slice's exactly*. The architectural novelty is small; the ADR's job is to record that the existing pattern extends cleanly to grading and to be explicit about the two non-trivial deltas: (a) which lifecycle states this processor walks (`submitted → grading → graded` / `grading_failed`, not `requested → generating → ready` / `generation_failed`), and (b) whether the take-page state flip is sufficient "notified" for this slice (the architect's call, with reasoning).

The relevant established context: ADR-038 ships the submit route, which records the Attempt as `submitted` and returns 303 — no AI call, no enqueue, no background job. ADR-039 created `attempt_questions` rows at Attempt start with `is_correct` / `explanation` NULL (ADR-033's NULL-until-graded columns); ADR-044's runner persisted `test_passed` / `test_status` / `test_output` / `test_run_at` per Question. The processor reads all of those, marshals them through `GradeAttemptInput`, invokes the workflow, parses `GradeAttemptOutput`, and persists the Grade.

The decision space:

- **Where the processing runs.** A manual command mirroring `process_quiz_requests` (the precedent ADR-037 set) — vs a background worker (rejected by ADR-037's reasoning, which still holds: concurrency on the SQLite connection, app-lifetime coupling, worker-crash blast radius), vs a separate daemon, vs a heavyweight job queue. The precedent settles all of these; this ADR records the call without re-litigating.
- **How the trigger handoff works.** Same call as ADR-037: the submit route does **not** enqueue; the processor polls `submitted` Attempts independently when the author runs it. Keeping the route's "record and return" guarantee trivially true (MC-4) is worth keeping.
- **How the learner is notified the result is ready.** This is the substantive call this ADR makes. Two options:
  - **(a)** The take-page state flip — when `GET .../take` is rendered for an Attempt whose status is now `graded`, the page renders the graded state (the aggregate score + per-Question explanations + Weak Topics + recommended Sections — ADR-051). This is the same notification shape ADR-037 used for the generation slice (the per-Section status-flip), extended one step further inward (from the per-Section Quiz list, where ADR-037's status-flip lives, to the take page itself, where the grading slice's state-flip lives). It is learner-visible, it indicates the result is available, it is not real-time (§5).
  - **(b)** The active Notification entity — a `notifications` table + a chrome badge in `base.html.j2` + a `seen_at` lifecycle (ADR-008's "Future-surfaces" forecast). ADR-037 deferred this to the grading slice ("the active Notification entity ships with the grading slice"). The architect must reason whether shipping it *now* in this slice is the right call, or whether the take-page state flip is sufficient for this slice with the Notification entity deferred to a follow-on slice.
- **Failure-detail persistence.** Mirror ADR-037 — add a nullable `quiz_attempts.grading_error TEXT` column (additive `ALTER TABLE ADD COLUMN`); persist the failure detail; the learner-facing signal stays "Grading failed" (ADR-051's render); the column is a debugging aid for the author. Or don't — and rely on the processor's stderr log. The precedent (`generation_error`) is clean and cheap; mirroring it here is the obvious call.
- **Partial-Grade discipline.** When the workflow produces *some* of the per-Question explanations but fails before completing (e.g. the `GradeAttemptOutput`'s `per_question` list is missing entries for some `question_id`s, or the aggregate fields are missing/malformed) — does the processor persist what came back and mark the Attempt `graded`, persist what came back and mark `grading_failed`, or persist *nothing* and mark `grading_failed`? MC-5 ("never fabricates a result") cuts toward persist-nothing — a partial Grade with NULL aggregate / missing per-Question explanations is a fabricated complete Grade, and the learner cannot distinguish "the workflow succeeded with these explanations" from "the workflow failed partway and these are what survived".
- **Re-processing `grading_failed` rows.** Same call as ADR-037 — the processor does **not** re-process `grading_failed` Attempts on a later run (no silent unbounded retry, MC-5). A re-grade requires a user-triggered new `requested` Quiz row → a new Attempt → a new `submitted` Attempt; a future "Try grading again" affordance on a `grading_failed` Attempt would be a *user-triggered* path (MC-9-compliant) — out of scope this slice.

The manifest constrains the decision through **§5** ("No live / synchronous AI results … arrives when ready and is surfaced via Notification — not in real time"; "no remote deployment / no hosted product" — bounds the simplicity ceiling), **§6** ("AI work is asynchronous from the learner's perspective. Submission, processing, and result delivery are decoupled in time. The learner submits, continues working, and is notified when results are ready"; "AI failures are visible … never fabricates a result"; "single-user"), **§7** ("Every Quiz Attempt … persists across sessions" — the `submitted` Attempt the processor reads persists across sessions; the `graded` / `grading_failed` Attempt the processor writes persists across sessions), and **§8** (the Grade glossary the processor realizes; the Notification glossary the architect must reason about).

## Decision

### The out-of-band mechanism — a manual command, `python -m app.workflows.process_quiz_attempts` (mirroring ADR-037 exactly)

The grading processing runs in a **manual command**: `python -m app.workflows.process_quiz_attempts` — a new module under `app/workflows/` (mirroring `process_quiz_requests.py`'s shape section-for-section) with a `__main__` entry point. The author runs it after exercising the take page on a Quiz they submitted, or periodically. It is **not** a background thread/task in the FastAPI process; **not** a separate daemon; **not** a job queue. (ADR-037's "Why a manual command, not a background worker" arguments apply unchanged — they are about the project's posture as a single-user, no-CI, locally-run dev project, not about which AI workflow is being processed; the manual-command shape is the right level for a second processor as much as for the first one. The arguments are not re-stated here in full; see ADR-037 §The out-of-band mechanism.)

What the command does, each invocation:

1. `app.persistence.list_submitted_attempts()` (new in ADR-050 — the read accessor) — the `quiz_attempts` rows with `status = 'submitted'`. (Optionally bounded — "process up to N this run" — implementer's call; for a single-user dev project the set is small.)
2. For each such Attempt:
   1. `app.persistence.mark_attempt_grading(attempt_id)` — `status` → `'grading'` (ADR-050). One transactional update.
   2. **Marshal the workflow's input.** Read the Attempt's Questions + responses + persisted test results from `attempt_questions ⨝ questions` (via `app.persistence.list_attempt_questions(attempt_id)` — ADR-039 / ADR-044, which already carries `prompt` / `preamble` / `test_suite` / `response` / `test_passed` / `test_status` / `test_output`). Map each `AttemptQuestion` to a `QuestionForGrading` dict (the ADR-048 `GradeAttemptInput`-compatible shape), translating `test_status = NULL` (the learner submitted without running) to the synthetic `'not_run'` enum value and `test_output = NULL` to `""`. Read the Section's parsed content (read-only via the existing `app.parser` path — MC-6; the `chapter_id` is the prefix of the Attempt's Quiz's `section_id` per ADR-002; mirroring `process_quiz_requests`'s `_get_section_content` helper).
   3. **Invoke the workflow via `aiw run`** (ADR-036's CLI-subprocess invocation pattern; mirrors `process_quiz_requests`'s `_invoke_question_gen` shape):
      ```
      subprocess.run(
          ["aiw", "run", "grade_attempt",
           "--input", f"section_title={...}",
           "--input", f"section_content={...}",
           "--input", f"questions={json.dumps([qfg.model_dump() for qfg in questions_for_grading])}",
           "--run-id", f"attempt-{attempt_id}"],
          env={**os.environ,
               "AIW_EXTRA_WORKFLOW_MODULES":
                  "app.workflows.question_gen,app.workflows.grade_attempt",
               "PYTHONPATH": prepended_repo_root_pythonpath},
          cwd=_REPO_ROOT,
          capture_output=True, text=True,
      )
      ```
      (The `--input questions=` is a JSON-serialized list — the `aiw run` CLI accepts complex inputs as JSON strings; the framework parses them into the `GradeAttemptInput.questions: list[QuestionForGrading]` field. The implementer chooses the exact CLI form per the framework's documented surface. Not an in-process `_dispatch.run_workflow` import — ADR-036 rejected that.)
      On exit-0, parse the terminal artefact from stdout (same shape as `process_quiz_requests`: `json.dumps(artifact, indent=2)` then a `total cost: $X.XXXX` trailer; structured logs on stderr) → `{"per_question": [...], "score": int, "weak_topics": [...], "recommended_sections": [...]}`. On non-zero exit / `error: ...` line on stderr → failure (step e).
   4. **Validate the artefact** — CS-300 sanity check on the workflow's output (the framework's `response_format=GradeAttemptOutput` validator already enforces shape inside the subprocess, but the artefact crosses a JSON serialization boundary and CS-300 should not trust the wire blindly):
      - The artefact must be a dict with the four required keys (`per_question`, `score`, `weak_topics`, `recommended_sections`); a missing key → failure.
      - `per_question` must be a list whose `question_id` set exactly matches the input's `QuestionForGrading.question_id` set (no missing entries, no extra entries) — otherwise the per-Question explanations cannot be mapped back to the right rows; mismatch → failure.
      - Each `per_question[i].explanation` must be a non-empty string (the schema enforces `min_length=1`, but the validation is defensive at the CS-300 boundary — an artefact that survived in-framework validation but somehow has an empty string here is a failure).
      - `score` must be `>= 0` and `<= len(per_question)` (the schema enforces `>= 0`; the upper bound is CS-300's sanity check — see §The score cross-check below).
      - `weak_topics` and `recommended_sections` must be lists of strings (possibly empty).
   5. **Persist transactionally** — one call to `app.persistence.save_attempt_grade(attempt_id, *, per_question_explanations, weak_topics, recommended_sections, score_from_workflow, test_results)` (ADR-050; exact signature ADR-050's call) that, in **one transaction**:
      - `UPDATE`s each `attempt_questions` row in this Attempt with `is_correct` (derived from `test_passed`: `True` → 1, `False` → 0, `None`/`not_run` → 0 per ADR-050's mapping) and `explanation` (from `per_question[i].explanation`);
      - INSERTs one `grades` row (PK `attempt_id`) with `score` (recomputed from the persisted `is_correct` values per ADR-050's score cross-check, *not* taken verbatim from the workflow — see below), `weak_topics` (`'|'`-delimited TEXT), `recommended_sections` (`'|'`-delimited TEXT), `graded_at = <now>`;
      - `UPDATE`s `quiz_attempts.status = 'graded'`, `graded_at = <now>`, leaves `grading_error` NULL.
      If the persistence function raises, the whole transaction rolls back and the processor falls through to step f (failure path). No half-Grade can persist (the transactional discipline is ADR-050's concern; the processor calls one function and gets atomicity for free).
   6. **Log a one-line summary** per Attempt to stderr — "Processed attempt_id=N → graded (score=K/M)" on success; "Processed attempt_id=N → grading_failed: <reason>" on failure.

On failure at any point in 2.c – 2.e (the `aiw run` subprocess exits non-zero, stderr has `error: …`, the artefact fails JSON parsing, the artefact fails the CS-300 sanity check, the Pydantic re-validation on the wire fails, the persistence transaction raises, the per-Question `question_id` set mismatches the input, the workflow returns a `score` outside `[0, len(per_question)]`):

f. `app.persistence.mark_attempt_grading_failed(attempt_id, error=<str>)` — sets `quiz_attempts.status = 'grading_failed'`, writes the error string into the new nullable `quiz_attempts.grading_error TEXT` column (ADR-050; mirroring ADR-037's `quizzes.generation_error`), **persists no Grade** (no `grades` row inserted; `attempt_questions.is_correct` / `.explanation` stay NULL — same value they had before this run). Log the failure summary; continue to the next Attempt. **Zero fabricated Grade.** **No silent unbounded retry** — the framework's `RetryingEdge` already does *bounded* retry (`max_attempts=3`) inside the workflow run (the subprocess); the CS-300 processor does **not** wrap the `aiw run` call in a retry loop, and does **not** re-process `grading_failed` Attempts on a later run. A re-grade requires a user-triggered new Attempt (a learner re-clicks Generate / re-takes the Quiz — out of scope this slice; a future "Try grading again" affordance on a `grading_failed` Attempt would be user-triggered, MC-9-compliant, and additive). MC-5.

### The score cross-check — the persisted `score` is recomputed from `is_correct`, not taken verbatim from the workflow

The §8 amendment makes correctness a function of the test result. ADR-048's `GradeAttemptOutput.score` is the LLM's *claim* about the count of Questions whose `test_passed = True`; the persistence layer (ADR-050) recomputes it from the actual `is_correct` writes (`SUM(is_correct)` over the Attempt's `attempt_questions` rows after the per-Question update) and stores *that* in `grades.score`. If the LLM's claimed `score` differs from the recomputed value by more than zero, the processor logs a warning to stderr (debugging aid) but **the persistence proceeds with the recomputed value** — the §8 truth is the test result, not the LLM's count. If the LLM's `score` is wildly out of bounds (negative, or `> len(per_question)`) the artefact is rejected (step e); within bounds, the recompute silently wins.

This is the architectural realization of ADR-048's commitment that the workflow does not re-judge correctness. The workflow's `score` field exists because the LLM benefits from being prompted to reason about the count (it produces better explanations when it explicitly counts what worked); the *persisted* `score` is the runner's truth.

### The trigger handoff — the submit route (ADR-038) is unchanged; no enqueue; the processor polls `submitted` Attempts independently

ADR-038's `POST .../take` route is **unchanged**: it records the responses, flips `quiz_attempts.status = 'submitted'`, sets `submitted_at`, and returns 303 to `GET .../take`. It does **not** enqueue the workflow, signal the processor, start a thread, or fire-and-forget an `asyncio` task. The processor polls `submitted` Attempts independently when the author runs it. (Same call as ADR-037 §The trigger handoff — same reasoning, same outcome.)

### How the learner is notified — for this slice, the take-page state flip is the "notified" mechanism; the active Notification entity ships with a follow-on slice

**For TASK-019, the take page's render branching on `quiz_attempts.status` (ADR-051) — flipping from "Submitted — grading not yet available" to the graded state on the next `GET .../take` — is the "notified when results are ready" mechanism. No `notifications` table ships this task.**

The reasoning extends ADR-037's per-Section status-flip argument inward to the take page. When the learner submitted the Attempt, they were on the take page; when they next return to that page (refresh, or come back via the per-Section Quiz list / a deep link), they see the graded state with their score, explanations, Weak Topics, and recommended Sections — which *is* "a learner-visible indication that an async AI result has become available" (§8 Notification glossary): it's learner-visible, it indicates the Grade is available, it's not real-time (§5).

Was ADR-037's "the Notification entity ships with the grading slice" promise this slice's obligation? Re-reading ADR-037 §How the learner is notified carefully: ADR-037 deferred the active Notification entity to "the grading slice, where there's a Grade to notify about too and a real 'the learner is elsewhere' case". The "real 'the learner is elsewhere' case" is the substantive condition — and TASK-019's framing acknowledges this: the take-page state flip works when the learner returns *to that take page*, but the learner is plausibly elsewhere (browsing a different Chapter, taking another Quiz, reading a Lecture) when the grading completes. A Notification surface (a chrome badge in `base.html.j2`) would catch them wherever they are.

The architect's call: **defer the active Notification entity to a follow-on slice, with the deferral recorded explicitly per ADR-035 ("describes what's built, not what won't be built").** The reasoning:

- **The slice's scope is already large** — three to four ADRs (ADR-048 + ADR-049 + ADR-050 + ADR-051) covering the workflow, the processor, the persistence (new table + new column + new functions + new dataclass), and the take-page rendering. Adding the Notification entity (a `notifications` table + a chrome badge + a `seen_at` lifecycle + a CSS surface + a route to mark seen) is genuinely a separable concern with its own design surface (which surfaces produce Notifications? Quiz `ready` + Grade `graded` retroactively? When do they clear — on next-`GET` of the resource, or only on explicit dismiss? Where does the badge live — header, rail, a dedicated `/notifications` page? Per ADR-008 the natural shape is a header/badge inside `base.html.j2`'s chrome region, but the surface design is still real work).
- **The take-page state flip honestly satisfies §8 for this slice.** §8 Notification is "a learner-visible indication that an async AI result has become available". The take page now shows the Grade when it's ready; that is a learner-visible indication of an available async result. The §8 glossary does not say "the indication must reach the learner wherever they are"; it says "learner-visible". The take page is learner-visible.
- **The Notification entity earns its keep when it's a unified surface for multiple result kinds** — Quiz `ready` *and* Grade `graded` *and* (future) Lecture-audio-ready *and* whatever else lands. Shipping it now with one kind of Notification (Grade) is shipping a single-purpose badge; shipping it with *all* the kinds in one slice is a cleaner design.
- **The deferral is conformant per MC-5 and §8.** No fabricated Notification — the take-page state flip shows the graded state *only* when `quiz_attempts.status = 'graded'` actually holds (which the processor sets only after `save_attempt_grade` succeeds). The §8 obligation is honestly met by the take-page state flip. ADR-035 makes this an architecturally clean call ("describes what's built" — the take-page state flip is what's built; the active Notification entity is what a follow-on slice builds).

ADR-037's "ships with the grading slice" framing was a forecast — read in context, the substantive obligation was "when there's a Grade to notify about *and* a real 'the learner is elsewhere' case". The architect's call: the take-page state flip honestly satisfies the §8 Notification obligation for this slice (it is a learner-visible indication), and the active Notification entity is a separable follow-on slice's call. **This is the call this ADR makes**; the human reviews at the acceptance gate and can disagree (if so, the human directs the architect to file ADR-049a or to amend this ADR to ship the Notification entity now — at which point the implementer adds a `notifications` table, a chrome badge, and the new persistence-side rows produced by the processor when `quiz_attempts.status` transitions to `graded`).

### Failure-handling discipline — `grading_failed` on any failure; the detail persisted in `quiz_attempts.grading_error TEXT` (new nullable column, additive)

On any failure (the `aiw run` subprocess exits non-zero, stderr has `error: …`, the artefact fails parsing/validation, the persistence transaction raises): `app.persistence.mark_attempt_grading_failed(attempt_id, error=<str>)` — sets `quiz_attempts.status = 'grading_failed'`, writes the error string into a new nullable column **`quiz_attempts.grading_error TEXT`** (added by ADR-050 via the `_apply_additive_migrations` `PRAGMA table_info` check + the `CREATE TABLE quiz_attempts` block in `_SCHEMA_SQL` — additive per ADR-022, mirroring ADR-037's `quizzes.generation_error`). **Zero `grades` rows inserted on the failure path. Zero `attempt_questions.is_correct` / `.explanation` writes on the failure path.** No silent unbounded retry — the framework's `RetryingEdge` (ADR-048's `RetryPolicy(max_attempts=3)`) does bounded retry inside the workflow run; the CS-300 processor does not wrap `aiw run` in a retry loop and does not re-process `grading_failed` Attempts on a later run. The learner-facing signal stays the honest "Grading failed — try again" (ADR-051's render); the `grading_error` detail is a debugging aid for the author, exactly mirroring how ADR-037 used `generation_error`. Whether to expose the `grading_error` detail to the learner is ADR-051's call (forecast: yes, in a small details expander — the author *is* the learner here, and exposing the failure mode is more honest than hiding it; but the architectural commitment is "the learner-facing signal is the honest failure"; the expander is a render call).

### Partial-Grade discipline — no partial Grade persists; one transaction or nothing

If the workflow's `aiw run` exits 0 but the artefact is malformed (missing keys / `question_id` mismatch / out-of-bounds `score` / a `per_question[i]` missing `explanation` / the persistence transaction raises mid-update): **the Attempt is `grading_failed`, no `grades` row is inserted, no `attempt_questions.is_correct` / `.explanation` write persists.** This is enforced by ADR-050's transactional `save_attempt_grade(attempt_id, ...)` function — either every per-Question update + the `grades` INSERT + the `quiz_attempts.status` flip all commit, or the transaction rolls back and the row stays `grading` (which the failure handler then flips to `grading_failed`). The processor does **not** persist "what came back" on partial failure — a partial Grade with NULL aggregate or missing per-Question explanations is a fabricated complete Grade (the learner cannot tell "the workflow succeeded with these explanations" apart from "the workflow failed partway and these are what survived"), and MC-5 forbids it.

### Scope of this ADR

This ADR fixes only:

1. **The out-of-band mechanism:** a manual command `python -m app.workflows.process_quiz_attempts` (a `__main__` module under `app/workflows/`, mirroring `process_quiz_requests.py`'s shape section-for-section) — poll `submitted` Attempts, walk each `submitted` → `grading` → run the workflow via ADR-036's `aiw run` CLI subprocess → `graded` (persist Grade in one transaction) or `grading_failed`; log a per-Attempt summary to stderr. **Not** a background worker; **not** a daemon; **not** a job queue. **No enqueue from the submit route.**
2. **The workflow invocation:** `aiw run grade_attempt --input section_title=... --input section_content=... --input questions=<json> --run-id attempt-{attempt_id}`, with `env={..., "AIW_EXTRA_WORKFLOW_MODULES": "app.workflows.question_gen,app.workflows.grade_attempt", "PYTHONPATH": prepended_repo_root}` and `cwd=_REPO_ROOT` (mirroring `process_quiz_requests`). The exact CLI form for the JSON-serialized `questions` input is the implementer's call per the documented `aiw run` surface; the architecturally-binding parts are the workflow name (`grade_attempt`), the run-id format (`attempt-{attempt_id}`, mirroring `process_quiz_requests`'s `quiz-{quiz_id}`), and the env-var extension. The `AIW_EXTRA_WORKFLOW_MODULES` extension is the architectural delta — both processors set it to the same comma-separated value so either workflow is loadable from either subprocess (low cost; consistent; the implementer changes the existing `process_quiz_requests` env value in the same commit, with a test confirming it).
3. **Artefact parsing + CS-300 sanity check:** parse `json.dumps(artifact, indent=2)` + `total cost: $X.XXXX` trailer from stdout (the existing `_parse_artefact_from_stdout` helper from `process_quiz_requests` is the model — extract it to a shared helper if useful; implementer's call, not architecture); validate the four required keys, the `question_id` set match, the explanation non-emptiness, and the `score` bounds; any failure → step f.
4. **Persistence call:** one transactional call to `app.persistence.save_attempt_grade(attempt_id, ...)` (ADR-050's function — see ADR-050 for exact signature and atomicity contract) that maps `test_passed` → `is_correct`, writes per-Question `explanation`, INSERTs the `grades` row with the **recomputed `score` from the persisted `is_correct` set** (not the workflow's claimed `score`), persists `weak_topics` / `recommended_sections` as `'|'`-delimited TEXT, flips `quiz_attempts.status = 'graded'` and `graded_at`. If the transaction raises, fall to step f.
5. **Failure handling:** on any failure, `app.persistence.mark_attempt_grading_failed(attempt_id, error=<str>)` — `quiz_attempts.status = 'grading_failed'`, writes the error string into the new nullable `quiz_attempts.grading_error TEXT` column (ADR-050); **zero `grades` rows, zero `attempt_questions.is_correct` / `.explanation` writes** on the failure path; no silent unbounded retry; the processor does not re-process `grading_failed` rows on a later run. The learner-facing signal is the honest "Grading failed" (ADR-051's render); `grading_error` is a debugging aid.
6. **The "notified" obligation for this slice:** the take-page state flip on next `GET .../take` from `submitted` to `graded` (ADR-051's render). **No active Notification entity ships this task** — deferred to a follow-on slice that ships the unified `notifications` table + chrome badge + `seen_at` lifecycle for all async-result kinds (Quiz ready, Grade graded, future Lecture audio, …). The deferral is recorded explicitly per ADR-035 — describing what's built (the take-page state flip), not what won't be built (the chrome badge isn't a posture, just a follow-on slice's design).
7. **The score cross-check** (architectural):** the persisted `grades.score` is recomputed from `SUM(attempt_questions.is_correct)` over the Attempt's rows after the per-Question update — not taken verbatim from the workflow. A divergent LLM `score` is logged (debugging aid) but the persisted truth is the runner's verdict, mirroring ADR-048's architectural commitment "correctness = test result, not LLM judgment". (The score is the count of correct Questions; the test result is the source of "correct"; therefore the score is computed from the test result.)
8. **The flag (not an edit — the architect does not own CLAUDE.md):** CLAUDE.md's "Commands" section should gain "Process pending Quiz attempts (out-of-band Quiz-grading processor, ADR-049): `python -m app.workflows.process_quiz_attempts`" — the human owns that edit; flagged here so it's not missed. Mirrors ADR-037's same flag, which the human honored.

This ADR does **not** decide:

- **The `grade_attempt` workflow's authoring surface** — owned by **ADR-048**.
- **The Grade aggregate's persistence — the `grades` table, the `quiz_attempts.grading_error` column, the new persistence functions (`mark_attempt_grading`, `save_attempt_grade`, `mark_attempt_grading_failed`, `mark_attempt_graded`, `list_submitted_attempts`, `get_grade_for_attempt`), the dataclass shapes, the `test_passed` → `is_correct` mapping** — owned by **ADR-050**.
- **The graded-state and grading-failed-state rendering on the take page** — owned by **ADR-051**.
- **The active Notification entity** (a `notifications` table + chrome badge + `seen_at` lifecycle) — explicitly deferred to a follow-on slice; this ADR records the deferral and the reasoning.
- **A "Try grading again" affordance on a `grading_failed` Attempt** — a future user-triggered surface; MC-9-compliant; additive; not this slice.
- **A "show the `grading_error` detail to the learner" expander** — ADR-051's call on what the failure render shows.
- **The composition slice's relaxation of the first-Quiz-only guard** — the slice after this one's `/design` concern.
- **The relational Topic vocabulary** — deferred to the composition slice.

## Alternatives considered

**A. A background worker started at app launch (a thread or `asyncio` task in the FastAPI process that polls `submitted` Attempts).**
Rejected for the same reasons ADR-037 rejected the same alternative for the generation processor: a second writer on the SQLite connection (ADR-022 explicitly chose stdlib `sqlite3` with no concurrency story); an app-lifetime coupling (grading only happens if the FastAPI process is running); a worker that must not crash the app; a shutdown-handling question. The arguments are not project-specific to the generation slice; they apply identically to the grading slice. The manual command sidesteps all of them. If post-TASK-019 the manual-command UX proves too friction-heavy, a future ADR can add a thin worker; building it now is premature.

**B. The submit route enqueues the workflow (fire-and-forget an `asyncio` task / start a thread on `POST .../take`).**
Rejected. An enqueue from the route is the seed of a background worker (Alternative A's complexity, introduced via the route); the route's "record and return" guarantee (MC-4) becomes less obviously true (a fire-and-forget task started from the request is AI processing kicked off by the request, even if the request doesn't await it); a hung or crashing fire-and-forget task is a request-handler reliability question. Keeping the submit route purely "record and return" and the processor purely "poll and process, on the author's trigger" is the clean separation that makes MC-4 trivially satisfied. The benefit of an enqueue ("the processor doesn't have to poll") is nil at single-user scale.

**C. A combined "process all pending AI work" command (one command that polls both `requested` Quizzes and `submitted` Attempts).**
Considered. Rejected mildly: the two processors handle different lifecycle states, different workflows, different artefact shapes, and different persistence transitions; combining them into one command means more code that handles two concerns at once, makes a per-workflow failure mode harder to debug (the author wants "what failed when grading attempt 42" — a focused command is easier), and conflates two operational concerns (the author can run generation without running grading, or vice versa, when debugging). Two parallel commands mirror the two parallel workflows; the architectural shape is consistent. Adding a "process everything" wrapper later is trivial (a 3-line shell script or a `python -m app.workflows.process_all` that calls both) — building it now bundles concerns.

**D. Ship the active Notification entity this slice — the `notifications` table + chrome badge + `seen_at` lifecycle.**
Considered (this is what ADR-037's "the active Notification entity ships with the grading slice" promise reads as if taken literally). Rejected by this ADR, with the reasoning recorded in §How the learner is notified above. Summary: the take-page state flip honestly satisfies §8 Notification for the Grade case; the active Notification entity's design surface (which result kinds produce Notifications? where does the badge live? what's the `seen_at` lifecycle?) is genuinely a separable concern with its own `/design`; deferring it to a follow-on slice keeps TASK-019's scope tractable and waits for the right moment (when there's a unified surface for multiple result kinds, not a single-purpose badge for one). The deferral is conformant per ADR-035 (describing what's built — the take-page state flip — not what won't be built). The architect's call; the human reviews at the acceptance gate.

**E. Don't persist the failure detail (no `quiz_attempts.grading_error` column).**
Rejected mildly, same as ADR-037 §Alternative E: a nullable additive column costs nothing (no migration-trigger, NULL for every non-failed Attempt) and saves the author from having to correlate a failure with a stderr log; the learner-facing message stays "Grading failed"; the `grading_error` detail is a debugging aid; MC-5 honored (no fabrication). Cheap, useful, additive — ship it.

**F. Persist a partial Grade on workflow partial-failure (what came back goes in; the rest stays NULL; status flips to `graded`).**
Rejected outright on MC-5 grounds. A partial Grade with NULL aggregate or missing per-Question explanations is indistinguishable from "the workflow succeeded with these explanations" — the learner cannot tell partial-failure from honest-incomplete. The §8 Grade definition is *all four facets* (correctness + explanation + score + weak topics + recommended sections); a Grade missing the aggregate or some explanations is not a Grade by §8's terms. The atomic-transaction discipline (ADR-050's `save_attempt_grade`) makes partial-Grade physically impossible.

**G. Persist a partial Grade with a status `graded_partially` (a new lifecycle state).**
Considered as a softer alternative to F. Rejected: a new lifecycle state for "the LLM gave us some of the Grade but not all" is over-engineering for an edge case (the workflow's bounded retry already covers the transient case; a permanent partial-failure means the workflow itself is broken or the LLM is misbehaving, both of which are debugging targets for the author, not learner-facing states); the take-page render would have to branch on a third state with unclear semantics ("partial Grade — score N/M for the K questions that came back, ungraded for the rest"); the composition slice's MC-8 read would have to decide whether a `graded_partially` Attempt's Weak Topics count. Cleaner to fail honestly (`grading_failed`) and let the author re-trigger; cheaper to keep the lifecycle two-outcome.

**H. Re-process `grading_failed` Attempts on subsequent processor runs (silent retry of permanent failures).**
Rejected on MC-5 grounds — same call as ADR-037 §No silent unbounded retry. The framework's `RetryingEdge` does *bounded* retry inside the workflow; the CS-300 processor does not re-process `grading_failed` rows. A re-grade requires a new user-triggered Attempt; a future "Try grading again" affordance would be a *user-triggered* path, MC-9-compliant, additive — out of scope this slice.

**I. Skip the score cross-check (trust the LLM's `score` verbatim).**
Rejected on §8 grounds: §8 makes correctness a function of the test result; the score is the count of correct Questions; therefore the score must be derived from the test result, not from the LLM's count. The LLM may miscount (especially with many Questions) and the persisted `grades.score` must be honest; recomputing from `SUM(is_correct)` is cheap (it's already inside the same transaction that wrote each `is_correct`) and architecturally clean. The LLM's `score` is retained as a workflow output (it improves prompt reasoning) but is not the persisted truth.

## Manifest reading

Read as binding for this decision:

- **§5 Non-Goals.** "No live / synchronous AI results … not in real time at the moment of request" — the manual command is the out-of-band mechanism; the submit POST returns 303 immediately; the Grade arrives when the author runs the processor. "Single-user" — no `user_id` on the new `grading_error` column or the `grades` table (ADR-050's concern); the processor has no notion of "whose Attempt". - **§7 Invariants.** "The reinforcement loop is the reason this project exists. Quiz performance MUST drive next-Quiz generation" — this processor produces `attempt_questions.is_correct` writes (the wrong-answer-replay history) and the Grade's `weak_topics` (the fresh-portion driver) — both inputs the composition slice (next) reads. "Quiz generation is always explicitly user-triggered. "Every Quiz Attempt … persists across sessions" — the `submitted` Attempts the processor reads persist across sessions; the `graded` / `grading_failed` Attempts the processor writes persist across sessions; ADR-033's "never deleted" posture is preserved.
- **§8 Glossary.** **Grade** — the processor realizes the four-facet Grade via the persistence-layer call (ADR-050). **Notification** — "a learner-visible indication that an async AI result has become available (most commonly a Grade or a newly-generated Quiz)" — for this slice, the take-page state flip from `submitted` to `graded` *is* that indication (learner-visible; indicates the Grade is available; not real-time, per §5). Carries the learner's code response for each Question, the in-app test results for each response, a progress status through grading, and — once graded — a Grade" — the processor walks the "progress status through grading" (`submitted → grading → graded` / `grading_failed`) and produces the "once graded — a Grade" via ADR-050's persistence.

No manifest entries flagged as architecture-in-disguise for this decision. ## Consequences

**Becomes possible:**

- A `submitted` Attempt actually gets graded: `python -m app.workflows.process_quiz_attempts` walks it `grading` → `graded` (with the §8 Grade persisted: per-Question `is_correct` from the runner, per-Question `explanation` from the LLM, aggregate `score` recomputed from `is_correct`, `weak_topics`, `recommended_sections`); the take page now renders the graded state on next `GET .../take`.
- The §6 loop's "the learner answers, runs, submits, sees a Grade" closes end-to-end on real Questions for the first time — the §3 reinforcement loop's *downstream half* (the per-Question correctness signal and the per-Attempt Weak Topics) is produced; the composition slice (next) reads them to drive the *upstream half* (the second Quiz of a Section).
- The same clean separation between route and processor as the generation slice — the submit route (ADR-038) records and returns; the processor polls and processes, on the author's trigger; MC-4 trivially satisfied.
- An honest failure path: `grading_failed` on any failure, the detail persisted in `quiz_attempts.grading_error` for the author to read; zero fabricated Grade; no silent unbounded retry; MC-5 honored architecturally. The take page renders "Grading failed — try again" (ADR-051) honestly.
- The score cross-check makes the persisted `grades.score` the architecturally-binding truth (the runner's verdict); a divergent LLM `score` is logged but does not propagate to the user. §8 Grade-correctness is honored end-to-end (workflow schema + persistence cross-check).

**Becomes more expensive:**

- `app/workflows/` grows a second processor module (`process_quiz_attempts.py`). Mitigation: mirrors `process_quiz_requests.py`'s shape section-for-section; the input-marshalling step (reading `attempt_questions`, mapping to `QuestionForGrading` dicts) is the one architectural delta and is straightforward.

**Becomes impossible (under this ADR):**

- The `POST .../take` submit route blocking on the AI call (the route does no AI processing — ADR-038's "record and return" is unchanged).
- A fabricated Grade on workflow failure (`grading_failed` on any failure path; zero `grades` rows; zero `is_correct` / `explanation` writes; the take page renders the honest failure state).

- An LLM-fabricated `score` overriding the runner's truth (the persisted `grades.score` is recomputed from `SUM(is_correct)`; a divergent LLM `score` is logged but does not propagate).
- A background event auto-grading an Attempt (the command processes existing user-created `submitted` Attempts; it never creates one; nothing scheduled).

- **The active Notification entity** — a `notifications` table + a chrome badge in `base.html.j2` + a `seen_at` lifecycle (ADR-008's "Future-surfaces" forecast) — a follow-on slice that ships the unified surface for all async-result kinds (Quiz `ready`, Grade `graded`, future Lecture audio).
