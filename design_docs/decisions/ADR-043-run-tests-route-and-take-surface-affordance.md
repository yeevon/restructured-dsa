## Context

ADR-042 builds the code-execution engine (`app/sandbox.py`'s `run_test_suite(test_suite, response) -> RunResult`). This ADR decides how the learner *invokes* it from the Quiz-taking page and how the page presents the test suite, the affordance, and the result. ADR-044 owns the persistence layer (`save_attempt_test_result`, the new accessor, the `attempt_questions` columns, the `AttemptQuestion` extension); this ADR is the route + the template + the CSS.

Constraints:

- **ADR-038's take surface is the canvas.** `GET/POST /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take` already render `quiz_take.html.j2` (the three-column shell, the designation badge, the section title, and — for an `in_progress` Attempt — a `<form method="post">` with, per Question, the `prompt` as text and a code-entry `<textarea name="response_{question_id}">`, plus a "Submit Quiz" button; for a `submitted` Attempt, the prompts + the learner's `response` read-only). ADR-038 / ADR-039 explicitly defer "the in-app code-execution/test-runner surface … the 'Run tests' route + affordance, the `attempt_questions` test-result columns, displaying test outcomes, whether the take surface displays the test suite read-only" to this slice.
- **§6 reads as wanting a responsive write→run→see loop.** "The learner answers a Question by writing code, running it, and running tests against it — all inside the application's own interface" — the natural reading is: write code, click Run, see pass/fail; not "click Run, get notified later."
- **MC-4 does not bind the test-run path** — running a test suite is executing code, not AI work; the manifest's async-AI rule is about LLM-driven processing (generation, grading, lecture audio), which this isn't. So a synchronous, timeout-bounded run is permitted. (ADR-042 records the same reasoning from the engine side; this ADR makes the synchronous-vs-async route call on its merits.)
- **ADR-035** settled that there is no binding no-JS posture — JavaScript is part of the toolkit; no-JS / CSS-only is *preferred where clean and sufficient*, never mandatory; and choosing JS when a surface genuinely needs it is a normal architectural step decided in its own ADR. So this ADR must affirmatively decide: does the "Run tests" loop genuinely need JS, or is a no-JS shape clean and sufficient here?
- **ADR-008** namespaces the take-page CSS as `quiz-take-*` in `app/static/quiz.css` (which already exists, loaded flat from `base.html.j2`); ADR-008 §Future-surfaces forecast `quiz.css` as the per-surface file for the take page. Any take-surface CSS this slice adds reuses that namespace and that file — no `base.css` change, no new CSS file.
- **MC-6**: the route must never write under `content/latex/` (the sandbox already runs in a temp dir away from it; the route reads the corpus `.tex` read-only for validation, exactly as ADR-038's take routes already do, and never writes it). **MC-5's spirit**: a sandbox failure (timeout, crash, compile error, setup error) is reflected honestly in the response — never a fabricated pass/fail. **MC-9**: the "Run tests" action generates no Quiz. **MC-10**: the route calls only typed `app.persistence.*` functions; no `sqlite3`, no SQL literal in `app/main.py`.

## Decision

### The route shape — a synchronous `POST .../take/run-tests` form post that submits the whole take form, with a PRG redirect back to `GET .../take`; no JavaScript

A new route in `app/main.py`:

```
POST /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take/run-tests
```

- **Same path-parameter validation as ADR-038's take routes** — `chapter_id` (corpus `.tex` exists / well-formed → 404 / 422), `section_number` (a known Section → 404), `quiz_id` (an existing Quiz whose `section_id` matches the URL → 404), the Quiz being `ready` (a non-`ready` Quiz → the same error code ADR-038's GET uses for that case — implementer's call, consistent with ADR-038). A `question_id` form field identifies which Question to run; a `question_id` not in the Quiz/Attempt is ignored (no sandbox invoked for it), consistent with ADR-039's `save_attempt_responses` "ignore a `question_id` not in the Attempt" posture. Invalid request → the appropriate error code; **nothing persisted; no sandbox invoked.**
- **The form the button submits is the whole take form.** The "Run tests" button lives *inside* `quiz_take.html.j2`'s `<form>` (the same form the "Submit Quiz" button is in) but submits to the `run-tests` action with a `question_id` hidden/named field identifying the target Question. So the POST body carries every Question's `<textarea>` content plus the target `question_id`.
- **Handler flow** (Quiz is `ready`, an `in_progress` Attempt resolved via `start_attempt(quiz_id)` — ADR-039's reuse-the-latest semantics, same as the submit route):
  1. Parse the form → a `{question_id: code}` dict from the `response_{id}` fields (exactly as the submit route already does) + the target `question_id`.
  2. `save_attempt_responses(attempt_id, responses)` — persist **all** of the learner's in-flight code first (so the upcoming PRG-redirect's GET re-render shows what they typed, in *every* textarea, not stale persisted state; this is why the button submits the whole form, not just one textarea).
  3. Fetch the target Question's `test_suite` via ADR-044's new accessor (`get_question(question_id)` → `Question` carrying `.test_suite`). If the Question has no `test_suite` (a legacy NULL — impossible for a Question persisted from TASK-016 forward, but defensive), the run is a `setup_error` ("this Question has no test suite") — surfaced honestly, persisted as such, no sandbox invoked.
  4. `run_test_suite(test_suite, learner_code_for_that_question)` → `RunResult` (ADR-042).
  5. `save_attempt_test_result(attempt_id, question_id, passed=result.passed, status=result.status, output=result.output, run_at=<now>)` (ADR-044).
  6. **PRG redirect — `303 See Other` → `GET .../take#question-{question_id}`** (the take page's per-Question block carries `id="question-{question_id}"`; the take page's CSS gives `.quiz-take-question` a generous `scroll-margin-top`, reusing ADR-031's no-relocate recipe pattern so the GET re-render lands the learner on the Question they just ran rather than at the page top — the take page is its own page so the relocation cost is small, but the recipe is free and consistent). The GET re-render reads the now-persisted responses and the now-persisted test result and renders each Question with its `<textarea>` repopulated and its results panel showing the latest result. (Proper PRG — a refresh after the redirect does not re-run anything.)
- **The route does not invoke grading** — no `ai-workflows` call, no LLM SDK, no out-of-band grading job kicked off; the Attempt stays `in_progress` (running tests is a *within-`in_progress`* action — it does not flip the Attempt's status; only "Submit Quiz" / `submit_attempt` does that). **The route does not generate a Quiz.** **The route does not write under `content/latex/`.** A `RunResult` whose `status` is `timed_out`/`compile_error`/`setup_error` is persisted and rendered as that status — never a fabricated pass/fail.

**No JavaScript.** A synchronous form-POST + PRG that submits the *whole* form (so all textareas' content is saved before the redirect, so the GET re-render loses nothing) + a `#question-{id}` anchor with `scroll-margin-top` (so the learner isn't relocated to the page top) is **clean and sufficient** here — it reuses `save_attempt_responses` (already exists), reuses the take-page render path (already exists), reuses ADR-031's no-relocate recipe (already exists), and adds zero front-end infrastructure. The async-`fetch()`-+-JS shape (see Alternatives) buys a marginally smoother in-place panel update at the cost of the project's first JS asset, a `<script>` tag, a JSON endpoint variant, and DOM-update code — not worth it when the no-JS shape loses nothing. (Per ADR-035 this is a *description of what's clean and sufficient here*, not a no-JS rule; a future surface that genuinely needs JS gets it in its own ADR; the parked ADR-032's "first client-side JavaScript" forecast is for the *Notes-save* surface and is not activated by this — different surface, different decision; whichever ships first, the other still gets its own ADR.)

### The `quiz_take.html.j2` changes — per `in_progress` Question: a read-only test-suite block, a "Run tests" button, a results panel; the `submitted` state additionally shows the last test result read-only

In the `in_progress` branch of `quiz_take.html.j2`, inside each `.quiz-take-question` block (after the `prompt`, around the `<textarea>`):

- An `id="question-{{ aq.question_id }}"` on the `.quiz-take-question` block (the PRG anchor target).
- A **read-only test-suite display** — a `<pre class="quiz-take-test-suite">{{ aq.test_suite }}</pre>` (monospace, scrollable, visually distinct from the response textarea) showing the Question's `test_suite` (ADR-044's `AttemptQuestion` extension carries it through `list_attempt_questions`). Rationale: §6's "running tests against it" reads as the learner *knowing* what the tests check — and ADR-040's prompt already names the signature in the `prompt`, so the test suite isn't a surprise, it's the spec. (If a Question's `test_suite` is NULL — a legacy row — the block is omitted.) This block is autoescaped by Jinja (it's source code, displayed as text — escaping is correct and intentional; no `| safe`).
- A **"Run tests" affordance** — a `<button type="submit" formaction="/lecture/{{ chapter_id }}/sections/{{ section_number }}/quiz/{{ quiz.quiz_id }}/take/run-tests" name="question_id" value="{{ aq.question_id }}" class="quiz-take-run-tests">Run tests</button>` inside the existing take `<form>`. The `formaction` overrides the form's `action` so this button posts to `run-tests` (the "Submit Quiz" button still posts to `take`); `name="question_id" value="..."` puts the target Question in the POST body. (No JS — `formaction` and `name`/`value` on a `<button type="submit">` are plain HTML.)
- A **results panel** — `<div class="quiz-take-results">…</div>`, rendered from ADR-044's `AttemptQuestion` test-result fields: empty (or a muted "not run yet" hint) when `test_run_at` is NULL; when present, a `.quiz-take-results-pass` block ("Tests passed") if `test_status == 'ran'` and `test_passed`; a `.quiz-take-results-fail` block ("Tests failed") if `test_status == 'ran'` and not `test_passed`; an honest status line for the failure statuses — "The test run timed out" (`timed_out`), "The test suite failed to compile" (`compile_error`), "Could not run the tests: <message>" (`setup_error`) — never a fabricated pass/fail; and a `<pre class="quiz-take-results-output">{{ aq.test_output }}</pre>` showing the run's output / diagnostic.

The "Submit Quiz" button and the take `<form>` are otherwise **unchanged** (ADR-038's submit route is unchanged). The take page's three-column shell + the rails + the designation badge are **unchanged in structure** — the new content lives inside the main column (`.quiz-take`), not in the grid; the parked ADR-032 (Notes-save scroll) is not activated.

In the **`submitted` branch**: the prompts + the learner's `response` read-only stay (ADR-038), and **each Question additionally shows its last test result read-only** — the same results-panel markup as the `in_progress` branch (a `.quiz-take-results` block from the persisted `test_*` fields), shown read-only (no "Run tests" button after submit — running tests is a `in_progress`-only action). This is the honest test-execution record, not a fabricated Grade: `is_correct`/`explanation` are still NULL and nothing is rendered in their place (the Grade is the next slice — MC-5's spirit; ADR-038's posture preserved). (The architect chose the "additionally shows the last test result read-only" branch over "unchanged" because the test result is genuinely part of the Attempt's record per §8 — "the in-app test results for each response" — and showing it read-only is consistent with showing `response` read-only; it fabricates nothing.)

The `not_ready` state is **unchanged** (no takeable form, no test-runner affordance — the Quiz isn't `ready`).

### The CSS — new `quiz-take-*` rules in `app/static/quiz.css`

New rules in the existing `app/static/quiz.css` (the `quiz-take-*` namespace per ADR-008 / ADR-039):

- `.quiz-take-test-suite` — a monospace `<pre>`, a distinct background/border from `.quiz-take-response` so the learner can tell "the tests" from "your code at a glance", `max-height` + `overflow: auto` so a long test file scrolls within its block.
- `.quiz-take-run-tests` — a button, styled in the established take-page button vocabulary (a secondary/outline variant of `.quiz-take-submit-button`'s palette so it reads as a lighter-weight action than "Submit Quiz").
- `.quiz-take-results`, `.quiz-take-results-pass`, `.quiz-take-results-fail`, `.quiz-take-results-status` (the honest failure-status line), `.quiz-take-results-output` (a monospace `<pre>` for the run output) — the panel, with the pass/fail modifiers carrying the visual pass-vs-fail distinction (a green-ish vs amber/red-ish accent in the established palette; the failure-status line for timeouts/compile errors reads as a neutral "couldn't run", not a "you failed").
- `.quiz-take-question` gains a generous `scroll-margin-top` (reusing ADR-031's no-relocate pattern) so the PRG `#question-{id}` anchor lands the learner on the Question they ran.

**No `base.css` change. No new CSS file.** `quiz.css` already exists and is already loaded flat from `base.html.j2` (ADR-039); `base.html.j2` is unchanged (no `<script>` tag added — no JS).

## Alternatives considered

**A. An async `fetch()` to a `POST .../take/run-tests` JSON endpoint + a small client-side JS handler that updates only the target Question's results panel.** Preserves the learner's in-progress edits in *other* textareas without a round-trip, and updates the panel in place (no page reload, no scroll jump). Rejected: the no-JS shape *also* loses nothing — because the "Run tests" button submits the *whole* form, `save_attempt_responses` persists every textarea before the redirect, and the GET re-render repopulates every textarea from persisted state; the `#question-{id}` + `scroll-margin-top` recipe keeps the learner where they were. The JS variant buys a marginally smoother panel update (no full-page repaint) at the cost of the project's first JS asset, a `<script>`, a JSON-response code path *and* a no-JS fallback (or no fallback — a regression from the project's current "everything works without JS" property), and DOM-mutation code to maintain. ADR-035 prefers no-JS *where clean and sufficient* — and this is. (It would have made the in-place update marginally smoother and made the surface the project's first JS-dependent one — a real cost, not a free upgrade.) If a future iteration of the take page grows to *need* in-place updates (e.g. a long multi-Question Quiz where the round-trip becomes annoying), that JS gets its own ADR then.

**B. Out-of-band test running — a "Run tests" click records a request row, an out-of-band `python -m app.workflows.process_test_runs`-style processor picks it up, the result shows on the next page load (mirroring ADR-037's generation processor).** Rejected as the wrong pattern to copy: ADR-037's generation is out-of-band *because of MC-4* — generation is AI-driven processing, and the manifest mandates AI work be async from the learner's perspective. **Running a test suite is not AI work** — it's executing code, bounded by a few-second timeout — so MC-4 does not apply, and §6's "the learner answers a Question by … running tests against it" reads as a *responsive* loop, not a "submit, wait, get notified" loop. Copying ADR-037's async shape here would (a) make the learner wait for a poll cycle to see whether their code passes, which §6 reads against, and (b) misapply a guardrail (MC-4) that doesn't bind this path. The architect's call: synchronous, with the ADR stating *why* MC-4 doesn't apply (which it does, above and in ADR-042's Conformance check). (It would have made the route handler trivially fast and made the learner's experience worse and the architecture muddled.)

**C. The "Run tests" button submits only the target Question's textarea (a tiny per-Question `<form>` nested in the page, not the take form).** Smaller POST body. Rejected: nested `<form>`s are invalid HTML, and a separate non-nested per-Question form would mean the "Run tests" POST doesn't carry the *other* Questions' in-flight code — so the PRG-redirect's GET re-render would show stale (persisted-at-Attempt-start, i.e. empty) textareas for every Question except the one run, silently discarding the learner's typing. Submitting the whole form (one `<form>`, the "Run tests" button using `formaction` to redirect to the `run-tests` action) is the clean HTML-only way to carry everything. (It would have made the POST body smaller and made the no-JS round-trip lossy — unacceptable.)

**D. The `submitted` state stays unchanged (no last-test-result display) — option per ADR-038's posture.** Defensible (the task forecast "unchanged, or last-result-read-only"). Rejected in favor of showing the last result read-only: §8 says the Attempt "carries … the in-app test results for each response" — so the test result is part of the Attempt's record, and a `submitted` Attempt that *hides* it is less honest than one that shows it (read-only, with nothing fabricated — no Grade, no score, no correctness judgment; `is_correct`/`explanation` stay NULL and render as nothing). The cost is one reused markup block in the `submitted` branch. (It would have made the `submitted` branch a one-line non-change and made the Attempt's persisted record invisible to the learner — a small loss.)

**E. The take-page CSS in a new file / in `base.css`.** Rejected mechanically — ADR-008's prefix rule routes `quiz-take-*` to `quiz.css`, which already exists; ADR-039 forecast it as the take-page's per-surface file; there is no reason to add a fourth flat CSS file or to put take-page rules in `base.css` (page-chrome). (No real alternative — recorded for completeness.)

## Consequences

**Becomes possible / easier:**

- The learner can run a Question's test suite against their code from the take page and see pass/fail (or an honest failure status) — §6's "run … and run tests against it … all inside the application's own interface" becomes true on a real surface.
- The take page now shows the test suite (the spec), the "Run tests" affordance, and a results panel — and persists the latest result on the Attempt (ADR-044), visible across reloads and in the `submitted` state.
- The route is shaped so a JSON-endpoint variant (if a future ADR wants in-place JS updates) is a thin addition, not a rewrite.

**Becomes more expensive:**

- A new route in `app/main.py` (`POST .../take/run-tests`) with the same path-param validation as the take routes (shared with — or factored from — the existing take-route validation; implementer's call). Mitigation: the validation is copy-of-the-take-route; the handler body is parse-form → `save_attempt_responses` → `get_question` → `run_test_suite` → `save_attempt_test_result` → 303-redirect; ~50 lines; pinned by pytest (happy path / validation / `submitted`-behavior).
- `quiz_take.html.j2` grows a test-suite `<pre>`, a "Run tests" button, and a results panel per Question, plus the `submitted`-branch results display. Mitigation: the markup is small and reuses the established `quiz-take-*` vocabulary; no new template.
- `app/static/quiz.css` grows ~6 new rules. Mitigation: `quiz.css` already exists; no new file; no `base.css` change.

**Becomes impossible (under this ADR):**

- A "Run tests" click silently discarding the learner's typing in other textareas (the whole form is submitted; `save_attempt_responses` persists all of it before the redirect).
- A fabricated pass/fail on the take page (the results panel renders from `RunResult.status` via ADR-044's persisted `test_status` — a timeout/compile-error/setup-error shows as that, never as "passed"/"failed"; MC-5's spirit).
- Grading being kicked off from the take page (the route invokes no `ai-workflows`, no LLM SDK, no grading job; the Attempt stays `in_progress`; MC-4).
- A Quiz being generated by the "Run tests" action (MC-9).
- The route writing under `content/latex/` (it reads the `.tex` read-only for validation, exactly as ADR-038's take routes do; the sandbox runs in a temp dir; MC-6).
- The take page becoming JS-dependent (no `<script>` added; the loop is plain HTML + a 303 redirect).

**Supersedure path if it proves wrong:** if the round-trip proves annoying in practice, a new ADR adds an async `fetch()` + a small JS handler (the route returning JSON instead of a redirect — a thin addition; the handler logic is unchanged); ADR-035's posture explicitly permits that, decided in its own ADR. If the `#question-{id}` + `scroll-margin-top` anchor doesn't land the learner well (it landed the section-completion toggle within ≤200px per ADR-031's Playwright regression, so it should), a new ADR revisits the anchor — the same supersedure shape as ADR-030 → ADR-031.

## Manifest reading

- **§6 "Code is written, run, and tested within the application … all inside the application's own interface. No external editor, no separate test runner, no copy-paste-into-another-tool workflow"** — read as **binding product behavior**; this ADR (with ADR-042 / ADR-044) implements it. Its "responsive write→run→see" reading drives the synchronous-route call. Not architecture-in-disguise — *what* the product does, not *how*.
- **§6 "AI work is asynchronous from the learner's perspective"** — read as **binding** and **untouched**: this slice adds no AI; the "Run tests" route does no AI work; the submit route is unchanged; grading is still invoked from no request. The route is correctly identified as outside MC-4's scope. Not architecture-in-disguise.
- **§6 "AI failures are visible … never fabricates a result"** — read as **binding**; orthogonal to the runner (no AI in the run path), but its *spirit* binds the results panel: a sandbox failure is shown as that failure, never a fabricated pass/fail. Not architecture-in-disguise.
- **§5 "No non-coding Question formats" / §7 "Every Question is a hands-on coding task. Questions never ask the learner to describe, explain, recall, or choose among options"** — read as **binding**; the take page's per-Question controls stay code-entry only (a `<textarea>` for code, a "Run tests" button, a read-only test-suite `<pre>`, a results panel) — no option radios, no true/false toggles, no non-code input introduced. Not architecture-in-disguise.
- **§8 Quiz Attempt "the in-app test results for each response"** — read as **binding glossary/behavior**; it's why the `submitted` state shows the last test result read-only and why the result persists on the Attempt (ADR-044). Not architecture-in-disguise.
- **§7 "Every Quiz Attempt … persists across sessions"** — read as **binding**; the latest test result persists on the Attempt's `attempt_questions` rows (ADR-044), so it survives a reload and a new session. Not architecture-in-disguise.
- No manifest entry read as architecture-in-disguise for this decision. No manifest entry flagged. No manifest tension.

## Conformance check

- **MC-1 (No direct LLM/agent SDK use).** The route imports no AI SDK and no `ai_workflows.*`; it calls `app.sandbox.run_test_suite` (stdlib-only, no AI — ADR-042) and `app.persistence.*`. `app/workflows/` is unchanged by this slice. **PASS** (vacuously — no AI surface added).
- **MC-4 (AI work asynchronous from the learner's perspective).** **Reasoned around explicitly, not violated.** Running a test suite is executing code, not AI-driven processing — so the synchronous, timeout-bounded "Run tests" route is permitted; MC-4's scope is generation/grading/lecture-audio, all of which remain out-of-band; the submit route is unchanged; grading is invoked from no request handler. **PASS** (the manifest principle is untouched; the test-run path is correctly outside it).
- **MC-5 (AI failures surfaced, never fabricated).** Orthogonal in the letter (no AI in the run path); binding in spirit: the results panel renders from `RunResult.status` (persisted as `test_status` — ADR-044) — `timed_out`/`compile_error`/`setup_error` show as honest "couldn't run" messages, never as a fabricated pass/fail; and the `submitted` state shows the persisted test result, never a fabricated Grade/score/correctness (`is_correct`/`explanation` stay NULL and render as nothing). **PASS** (spirit honored).
- **MC-6 (Lecture source read-only).** The route reads `content/latex/{chapter_id}.tex` read-only for path-param validation (exactly as ADR-038's take routes already do — same code path); it never writes under `content/latex/`; the sandbox runs in a temp dir away from it (ADR-042). A pytest case (in ADR-042's pre-flag) asserts the corpus is byte-for-byte unchanged after a run that attempts a corpus write. **PASS.**
- **MC-7 (Single user).** No `user_id` on the route, the form, the template, or anything this ADR introduces; no auth, no session. **PASS.**
- **MC-8 (Reinforcement loop preserved).** This slice produces the raw correctness signal the grading slice (next) reads to set per-Question correctness and identify Weak Topics; no fresh-only-post-first-Quiz path is created; the first-Quiz-only guard (ADR-037) is unchanged. **PASS.**
- **MC-9 (Quiz generation is user-triggered).** The "Run tests" action generates no Quiz — it runs a test suite and persists a result. (The action is itself user-triggered, which is fine — it generates nothing.) **PASS.**
- **MC-10 (Persistence boundary).** No `import sqlite3`, no SQL literal in `app/main.py`; the route calls only the typed public functions from `app/persistence/__init__.py` (`get_quiz`, `start_attempt`, `save_attempt_responses`, `get_question`, `save_attempt_test_result`, `list_attempt_questions`) and receives dataclasses (`Quiz`, `QuizAttempt`, `Question`, `AttemptQuestion`); the SQL for the new column lives in `app/persistence/` (ADR-044). **PASS.**
- **MC-2 / MC-3** — orthogonal (a test run is for one Question in one Quiz in one Section; no designation logic). **PASS.**

## Test-writer pre-flag

New pytest (HTTP-protocol / route tests under `tests/`):

- `POST .../take/run-tests` happy path — a `ready` Quiz with an `in_progress` Attempt; the form carries the textareas + a `question_id`; after the request: `save_attempt_responses` persisted all responses, `save_attempt_test_result` persisted the result, the response is a `303` to `GET .../take#question-{id}`; following the redirect, the take page shows the repopulated textareas and the results panel reflecting the run.
- Validation — unknown `chapter_id` (404/422), unknown `section_number` (404), `quiz_id` whose `section_id` doesn't match the URL (404), a `question_id` not in the Quiz/Attempt (no sandbox invoked, nothing persisted for it — consistent with ADR-039's ignore-unknown-`question_id` posture; the response is the appropriate code or a no-op redirect — implementer's call), a non-`ready` Quiz (the ADR-038-consistent error code; nothing persisted; no sandbox invoked).
- `submitted`-Attempt behavior — `POST .../take/run-tests` against a Quiz whose latest Attempt is `submitted` resolves no `in_progress` Attempt → the run is not performed (the action is `in_progress`-only); the response is the appropriate code or a redirect to `GET .../take` (which shows the `submitted` state with the last test results read-only) — implementer's call; nothing fabricated.
- The route invokes no `ai-workflows` / no LLM SDK / kicks off no grading job (a boundary grep + a behavioral check that `submit_attempt` / a grading function is not called); the route writes nothing under `content/latex/` (the corpus is byte-for-byte unchanged after a run).
- A boundary grep: `app/main.py` has no new `import sqlite3` and no new SQL literal; the route's persistence access is via `app.persistence.*` only.
