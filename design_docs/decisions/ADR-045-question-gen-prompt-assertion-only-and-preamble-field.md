## Context

ADR-042 §The splice records: "the splice = concatenate `learner_response + test_suite` into one translation unit (correct for assertion-only test suites; ADR-040's current reference-impl-embedding output → an honest `compile_error`; `design_docs/project_issues/question-gen-prompt-emit-assertion-only-test-suites.md` tracks fixing the generator)". ADR-042 deliberately did **not** bolt a fragile body-replacement splice into the runner (matching `void append(...) {` to its closing `}` is a real C++ parse — overloads, templates, nested braces, string-literal braces — and a wrong strip silently corrupts the test, the exact failure MC-5's spirit forbids); the right fix is the *generator*, and that's this slice. TASK-016's real-engine generation gate (the `quiz_id=6` artefact) empirically confirmed the current shape: each `test_suite` is a self-contained C++ file that *defines* (e.g.) `void append(LinkedList&, int)` itself and then `assert`s against it. Concatenating the learner's code with such a test suite double-defines the target → `g++` reports a redefinition / `main` ambiguity → ADR-042 reports `compile_error` honestly (MC-5's spirit intact) — but **on a real generated Question the learner can never see `passed=True`**, no matter how correct their implementation. The §6 loop ("Code is written, run, and tested within the application") ships *structurally* (the runner exists, the route exists, the surface exists, the persistence exists — TASK-017 / ADR-042 / ADR-043 / ADR-044) but not *empirically* — its happy path is unreachable on real Questions. TASK-017's "real-engine end-to-end run-tests check" verification gate is parked on exactly this: the gate has been filled `pass` only against the *honest-compile_error* path; the pass-path is what this slice unlocks.

The shape of the fix is non-trivial because a clean assertion-only test suite needs the learner's implementation and the test suite to **agree on shared data-structure shapes**: if the prompt asks "Implement `void append(LinkedList&, int)`" and the test suite says `LinkedList l; append(l, 4); assert(l.head->data == 4);`, then *something* has to define `LinkedList` and `Node` — and that something must be visible to both the learner's `append` body and the test's assertions. There are several ways to surface this shared piece, materially different in editorial framing and in mechanism:

- **Where does the shared piece live?**
  - **A first-class `GeneratedQuestion.preamble: str` field** alongside `prompt` / `topics` / `test_suite` — the LLM emits four pieces, the storage carries four pieces, the runner splices three of them (preamble + response + test_suite), the take page renders three of them (prompt, preamble read-only, test_suite read-only).
  - **Inline inside `prompt`** — the prompt text contains the struct/class definitions in a code block ("Implement `void append(...)` against the following struct: ```cpp\nstruct Node { ... };\n```"). The runner would have to text-parse the prompt to extract the preamble for the splice, which is exactly the fragile pattern ADR-042 rejected for the body-replacement splice.
  - **Inline at the top of `test_suite`** — the test suite starts with the struct definitions, then the assertions. The learner's implementation would then need to *not* define those structs (or `g++` redefines on the concatenation). This pushes the problem one layer down (the struct defs are *visible to the assertions* but the runner can't separate them from "the assertions" cleanly), and it means the take page's read-only `<pre class="quiz-take-test-suite">` block shows the structs mixed with the asserts — editorially worse for the learner trying to understand "what the test checks".
  - **A two-translation-unit compile** (option (b) from the project_issue) — `g++ solution.cpp test.cpp -o runner` + a header convention. The test suite `#include "solution.h"` and `extern`-declares the target; the learner's code is `solution.cpp`. This is more machinery (a header convention, two TUs, an `extern`-declaration convention the LLM must produce), and ADR-042's `run_test_suite` signature would change *shape* (two TUs, not one) rather than gain an additive default argument.

- **What does the `preamble` look like at the schema layer?** A required `str = Field(min_length=1)` (every Question must have shared shapes — wrong: many Questions need none) or a `str = Field(default="")` empty-string-allowed (literally-empty preamble is valid for a Question that needs no shared shapes — e.g. "Implement `int two_sum(int*, int, int)`" needs no struct).

- **What does the processor do with a missing `preamble`?** A whole-Quiz `generation_failed` (mirroring ADR-040's `test_suite` policy) — wrong, because empty `preamble` is a real and valid semantic; or a defensive `preamble = ""` default when the artefact's JSON lacks the key (consistent with empty-string-allowed at the schema), passed through verbatim to the persistence call.

- **What does the take page do with a non-empty `preamble`?** Render it read-only alongside the read-only test-suite block — the learner sees the shared shapes their code must implement against — or hide it in the splice and never show it (the learner sees a test suite asserting against fields it doesn't define, and has to guess the shapes). The latter violates §6's *spirit* — "running tests against it" reads as the learner *knowing* what the tests assume.

Manifest constraints binding the decision: §5 (Non-Goals — "No non-coding Question formats" — the `preamble` field's type is `str` (source code, the struct/class shapes the test depends on); it cannot hold an option list / true-false key / describe / recall artifact, exactly as `test_suite` and `prompt` cannot; the schema's `extra="forbid"` admits no choice/correct/answer field; "No live / synchronous AI results" — this task does no live AI; the existing out-of-band processor consumes the new shape; "No in-app authoring of lecture content" — the preamble is generated by `question_gen` and persisted to `data/notes.db`'s `questions` row, nothing written under `content/latex/`; "No multi-user features" — no `user_id` on the new field), §6 ("Code is written, run, and tested within the application" — this task makes the pass-path *reach* on real generated Questions, which the current reference-impl-embedding shape prevents; "AI failures are visible … never fabricates a result" — the `min_length=1` validator on `test_suite` stays (a Question with a missing/blank `test_suite` still makes the whole Quiz `generation_failed` per ADR-040); the runner's `compile_error` honesty path (ADR-042 / MC-5's spirit) is preserved — what changes is that the generator output now splices *cleanly* so the runner *reaches* `passed=True/False` on a real Question, but the path to honest failure is unchanged), §7 ("Every Question is a hands-on coding task" — preserved; the four-field tuple (`prompt` + `topics` + `test_suite` + `preamble`) all hold coding-task artifacts; no non-coding Question is expressible; "Every Quiz Attempt, Note, and completion mark persists across sessions" — the new field persists with the Question, which is part of the Bank, "never deleted" per §8), §8 (**Question** — "carries a test suite the learner runs in-app to verify the implementation" — preserved; the `test_suite` is still a runnable test-code string, the `preamble` is the shared-shapes header it depends on, the test suite per se is the assertion-only form that *uses* the preamble's shapes; **Question Bank** — "Never deleted" — pre-task Questions in the Bank stay valid as-is with `preamble = NULL`, and they continue to splice as `learner_response + test_suite` (the runner's default-`""` `preamble` path is byte-equivalent to ADR-042's pre-task splice — ADR-047 records this)).

## Decision

### The prompt change — emit assertion-only test suites that reference but do not define the implementation target; emit shared struct/class shapes in a new `preamble` field, not inside `test_suite` or `prompt`

`_question_gen_prompt_fn` in `app/workflows/question_gen.py` is extended. The existing STRICT REQUIREMENTs 1–6 (every Question is a hands-on coding task; no multiple-choice / true-false / describe / recall; 3–6 Questions; 1–3 topic tags; `prompt` names the function/class signature; `test_suite` is runnable test source code, never an option list / true-false key / description / recall answer) are **preserved unchanged**. ADR-040's STRICT REQUIREMENT 7 (the `test_suite` requirement, which currently steers toward "a self-contained runnable test file") is **rewritten** to commit to an assertion-only test suite that references but does not define the implementation target. A **new** STRICT REQUIREMENT 8 commits to the `preamble` field carrying any shared struct/class/header shapes the test suite (and the learner's implementation) depend on, and forbids embedding those shapes inside `prompt` or `test_suite`.

The architectural commitment is the *substance* — not the exact wording — and is exactly:

1. **The `test_suite` field is assertion-only.** The string the LLM emits as `test_suite` is runnable test source code that *calls* the named function / instantiates the named class the `prompt` asks the learner to implement, and *asserts* against the result, but it does **not** define or implement that function/class. No reference implementation body lives inside `test_suite`. (For C++, this is a `<cassert>`-based test that calls e.g. `append(l, 4)` and `assert(l.head->data == 4)` but does not define `void append(...)`.)
2. **The `preamble` field carries shared shapes.** Any struct definitions / class definitions / typedefs / `#include` directives / `using` declarations / other header-style source that *both* the learner's implementation *and* the `test_suite`'s assertions depend on, the LLM emits as the `preamble` string — not inside `test_suite`, not inside `prompt`. (For the "Implement `void append(LinkedList&, int)`" example: the `struct Node { int data; Node* next; };` and `struct LinkedList { Node* head; };` definitions go in `preamble`; the `assert(append(l, 4), l.head->data == 4)`-style assertions go in `test_suite`.)
3. **The `preamble` field is empty when not needed.** A Question that needs no shared shapes (e.g. "Implement `int two_sum(const std::vector<int>&, int)`" — both the learner's implementation and the test suite can `#include <vector>` independently and agree on the standard-library shape) carries `preamble = ""`. An empty preamble is a real and valid semantic, not a failure.
4. **The `prompt` field continues to name the function/class signature.** The `prompt` text describes the coding task and names the signature the learner implements; it does NOT carry the struct/class definitions inline (those go in `preamble`).

The exact phrasing of STRICT REQUIREMENTs 7 and 8 is implementer-followable (tunable, like ADR-036's tier model string and ADR-040's existing prompt wording) — the architectural commitment is the substance above. The implementer's first cut, after this ADR lands, should land committed phrasing that names "assertion-only test suite", "does not define the implementation target", and "preamble field for shared shapes"; the empirical-confirmation gate (TASK-018 §Verification gates' "real-engine end-to-end" check) is the only way to know whether the LLM actually follows the new framing — see §Consequences and §Conformance check.

### The `GeneratedQuestion` schema — gains one additive field `preamble: str = Field(default="")`; `model_config = ConfigDict(extra="forbid")` preserved; `test_suite` `min_length=1` preserved

`GeneratedQuestion` in `app/workflows/question_gen.py` gains a field **`preamble: str = Field(default="")`** alongside the existing `prompt: str`, `topics: list[str]`, and `test_suite: str = Field(min_length=1)`. The schema's `model_config = ConfigDict(extra="forbid")` is **preserved** (the schema still admits only `prompt` + `topics` + `test_suite` + `preamble`, never a `choices` / `correct_choice` / `answer_text` / `describe_*` / `recall_*` / non-coding field — §5 / §7 / MC-1's architectural spirit). The existing `test_suite` `min_length=1` validator is **preserved** (a literally-empty `test_suite` still raises at validation time and still makes the whole Quiz `generation_failed` per ADR-040 / ADR-037 / MC-5 — this ADR does not weaken that path). `QuestionGenOutput`'s shape is **unchanged** — its first field is still `questions: list[GeneratedQuestion]` (the framework's `FINAL_STATE_KEY` convention from ADR-036).

**Why `Field(default="")` and not `Field(min_length=1)` or required-no-default:**

- **Empty preamble is a real and valid semantic.** Many Questions need no shared shapes (a Question that asks the learner to implement a function over standard-library types). Requiring `preamble` non-empty would force the LLM to invent shared shapes for Questions that don't need them, degrading prompt quality and adding cognitive noise on the take page (the learner sees a "Shared code" block that contains nothing meaningful).
- **`default=""` round-trips through `aiw run`'s JSON serialization predictably.** Old artefacts (replayed pre-prompt-change saved runs) and fresh artefacts from an LLM that didn't follow the new framing both serialize to the same shape — an absent or empty `preamble` key — and the processor's defensive `q.get("preamble", "")` reads them identically.
- **It mirrors how `questions.topics` defaulted in `add_questions_to_quiz`** — the field is present, the empty case is encoded as the natural empty value, no special "missing vs empty" distinction is forced.
- **The `test_suite` `min_length=1` validator stays load-bearing for MC-5.** The bad-test-suite whole-Quiz `generation_failed` path (ADR-040 / ADR-037) is the project's MC-5 floor: a Question with no real test suite never ships. Extending that strictness to `preamble` would create a *new* `generation_failed` trigger for a case that is genuinely valid (no shared shapes needed), which is wrong by construction.

**Imports stay `ai_workflows.workflows.*` + `ai_workflows.primitives.tiers` + `pydantic` (+ stdlib `os`) only — MC-1's architecture portion preserved.** The new field is a Pydantic schema change; the new prompt wording is a string change; no LLM/agent SDK is touched; the forbidden-SDK boundary grep over `app/workflows/` stays clean.

### The processor wiring — read each Question's `preamble` from the `aiw run` artefact, default to `""` if absent, pass it verbatim to the persistence call; a missing/blank `preamble` is NOT a `generation_failed` trigger

`app/workflows/process_quiz_requests.py`'s parse step is extended: when it builds the list it passes to `add_questions_to_quiz` (ADR-040 / ADR-041 / ADR-046), each item now carries `preamble` read from the artefact's `questions[i].get("preamble", "")` (defensive default — `aiw run`'s JSON serialization may omit a field that holds the Pydantic default empty string, and a pre-prompt-change saved artefact won't have the key at all). The value is passed to the persistence call **verbatim** — no transformation, no regeneration, no truncation, mirroring ADR-036's "the workflow returns data, CS-300 code persists it" rule and ADR-040's `test_suite` posture.

The processor's lifecycle (`requested` → `generating` → `ready` / `generation_failed`) is **unchanged** (ADR-037). The bad-test-suite whole-Quiz `generation_failed` check (ADR-040) is **unchanged** — a Question with missing/empty/whitespace-only `test_suite` still makes the whole Quiz `generation_failed`, zero `questions` rows persisted, no placeholder synthesized. The `preamble` field is **NOT** added to that whole-Quiz failure check — a missing or empty `preamble` is a real and valid semantic, not a failure mode. This is the deliberate asymmetry between `test_suite` and `preamble`: `test_suite` is load-bearing (a Question without one cannot be tested, manifest §8); `preamble` is optional (a Question without shared shapes is a Question that doesn't need them).

### The ADR-040 / ADR-036 relationship — this is a new ADR citing them, not a supersedure

ADR-040's *core* decisions are all consumed unchanged:

- the test-suite *representation* (a single source-code string per Question — the new framing is "assertion-only", which is a *constraint on the contents* of that string, not a change of representation);
- the bad-test-suite whole-Quiz `generation_failed` path with no placeholder synthesized;
- the `min_length=1` validator on `GeneratedQuestion.test_suite`;
- the `extra="forbid"` posture on `GeneratedQuestion`;
- the "MC-1 imports stay `ai_workflows.*` + `pydantic` + stdlib only" constraint;
- the per-Section `ready`-entry caption unchanged;
- the take surface's read-only test-suite block (ADR-043) consumed unchanged in structure.

What changes is *only*: (a) the prompt wording (STRICT REQUIREMENTs 7 and 8); (b) `GeneratedQuestion` gains one additive optional field `preamble: str = Field(default="")`. This is exactly the kind of additive extension the declarative `WorkflowSpec` design anticipates — the same shape as ADR-040 extending ADR-036 with `test_suite`. ADR-036's *core* decisions are all consumed unchanged (the dependency, the module path, the authoring surface, the `aiw run` invocation, the forbidden-SDK list, the test seam, the cohabitation option).

So: **a new ADR citing ADR-040 and ADR-036**, not a supersedure of either. ADR-040's "the shape commitment is 'a single source-code string per Question'" is read in context as a *no-structured-cases / no-file-per-Question* commitment (ADR-040 §A and §B in Alternatives), not a *never-add-a-new-field* commitment; `preamble: str` (more shared coding-task source) does not change the representation of `test_suite` and does not make a non-coding Question expressible. A supersedure ADR would imply ADR-040 was *wrong*; it wasn't — it's being *extended additively*, in the same way ADR-040 itself extended ADR-036.

## Alternatives considered

**A. Inline the preamble in the `prompt` text (a code block in the prompt) instead of a first-class `GeneratedQuestion.preamble` field.**
Rejected. The runner would have to text-parse the `prompt` to extract the preamble for the splice — exactly the fragile pattern ADR-042 rejected for the body-replacement splice. (A code-block extractor by triple-backtick fence is plausible but only if the LLM always uses fenced code blocks for the preamble *and never elsewhere in the prompt*, a brittle assumption. Surfacing the preamble as a structured field on the Question is the architecturally clean form: separate concerns are separate fields, the runner splices them in the right order, the take page renders them in the right places.)

**B. Inline the preamble at the top of `test_suite` (a header section followed by the assertions).**
Rejected. This pushes the same parsing problem into the runner from a different angle: the runner now has to know "where does the preamble end and the asserts begin" inside one string, which it can't (and shouldn't try to — same MC-5-spirit concern as the body-replacement splice). Editorially, it also means the take page's read-only `<pre class="quiz-take-test-suite">` block shows the struct definitions mixed with the assertions, when the right framing is "the shared shapes are one thing, the assertions are another, your implementation is a third — three blocks". A first-class field separates the concerns cleanly.

**C. Option (b) from the project_issue — a two-translation-unit compile (`g++ solution.cpp test.cpp -o runner`) + `#include "solution.h"` header convention + the test suite `extern`-declares the target.**
Rejected. More machinery (a header file, two `g++` TUs, an `extern`-declaration convention the LLM must produce), and ADR-042's `run_test_suite` signature would change *shape* (two TUs, two source strings — `solution.cpp` and `test.cpp` — and the header) rather than gain an additive default argument. The single-TU concatenate already works for assertion-only test suites — the only thing missing is the *shared shapes*, which a first-class `preamble` field + a three-piece splice (`preamble + response + test_suite`) cleanly supplies. Editorially, "the struct/class shapes" are part of the Question's *specification*, not a hidden harness detail; surfacing them on the take page as a first-class block is editorially right; the two-TU header form would hide them in a generated `solution.h` the learner doesn't see. Recorded so a future `/design` cycle doesn't re-litigate without new evidence.

**D. Option (c) from the project_issue — leave `question_gen` as-is and tell the learner to write a complete program.**
Rejected by the project_issue itself: not viable (the test suite has its own `main`; two `main`s is a conflict). Recorded only to show why a generator-side fix is forced.

**E. Option (d) from the project_issue — runner-side textual strip of the reference implementation from the test suite.**
Rejected by ADR-042 and by the project_issue's option (d) analysis: too fragile (matching `void append(...) {` to its closing `}` is a real C++ parse, and a wrong strip silently corrupts the test — MC-5's spirit forbidden). Recorded so a future `/design` doesn't re-litigate without new evidence.

**F. Require `preamble` non-empty (`Field(min_length=1)`), or fail the whole Quiz on missing/blank `preamble` at the processor.**
Rejected. Many Questions need no shared shapes (a standalone function over standard-library types). Requiring `preamble` non-empty would force the LLM to invent shared shapes for Questions that don't need them — degrading prompt quality, adding cognitive noise on the take page (the learner sees a "Shared code" block that contains nothing meaningful), and creating a *new* MC-5-mirror `generation_failed` trigger for a case that is genuinely valid. The clean rule is: empty `preamble` is a real semantic ("this Question needs no shared shapes"); the take page omits the block when the preamble is empty (ADR-047); the runner's splice degrades cleanly to `"" + response + test_suite` = byte-equivalent-to-ADR-042's pre-task splice. The asymmetry with `test_suite` (`min_length=1`, missing → `generation_failed`) is deliberate: `test_suite` is what makes a Question testable (manifest §8); a Question without one is not shippable; `preamble` is a *helper* that some Questions need and some don't.

**G. Fold this ADR into ADR-040 (supersede ADR-040 in place).**
Rejected. ADR-040's *core* decisions are all consumed unchanged; only the prompt wording (additive STRICT REQUIREMENTs) and the schema (one additive optional field) extend.
