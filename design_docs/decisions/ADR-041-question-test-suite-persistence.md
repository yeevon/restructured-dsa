## Context

ADR-040 decides that a Question carries a test suite represented as **a single source-code string** (`GeneratedQuestion.test_suite: str` on the `question_gen` workflow output; the generation processor reads it from the `aiw run` artefact and passes it to the persistence layer). This ADR decides how that string is *stored* in `data/notes.db` and *exposed* to callers — the small persistence-layer changes ADR-040 forecast as "may be a section of [ADR-040] or a small separate ADR (`/design`'s call)". `/design`'s call: a small separate ADR — keeps ADR-040 about the *representation/workflow/processor/failure/surface* shape and this ADR about the *storage* shape; the two are tightly coupled but the split keeps each focused.

The established context:

- ADR-033's `questions` table: `question_id` PK AUTOINCREMENT, `section_id TEXT NOT NULL`, `prompt TEXT NOT NULL`, `topics TEXT NOT NULL DEFAULT ''` (`'|'`-delimited), `created_at TEXT NOT NULL` — created via `CREATE TABLE IF NOT EXISTS` in `app/persistence/connection.py`'s `_SCHEMA_SQL` block (under ADR-022's migration story). The `Question` dataclass in `app/persistence/quizzes.py` mirrors the columns (`topics` exposed as `list[str]`, split from the delimited column). `add_questions_to_quiz(quiz_id, questions: list[dict])` (ADR-037) takes a list of `{"prompt": str, "topics": list[str]}` dicts, INSERTs one `questions` row per dict (with `section_id` from the Quiz, `topics` `'|'`-joined) plus one `quiz_questions` row, all in one transaction. `list_questions_for_quiz(quiz_id) -> list[Question]` (ADR-039) is a `quiz_questions ⨝ questions` join returning `Question` dataclasses. `_row_to_question(row)` converts a `sqlite3.Row` to a `Question`. `app/persistence/__init__.py` re-exports `Question` and the public functions.
- ADR-022's migration story: `CREATE TABLE IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS` are additive (idempotent); a column that can't be expressed that way goes through `_apply_additive_migrations(conn)` — a `PRAGMA table_info(<table>)` check + a guarded `ALTER TABLE <table> ADD COLUMN <col> <type>` (nullable; no `NOT NULL` without a default that SQLite accepts on `ALTER`). ADR-037 added `quizzes.generation_error TEXT` via exactly that mechanic. A non-additive change (rename/re-constrain/drop) forces a follow-up ADR. ADR-033 flagged a "when the n-tables threshold makes the monolithic `_SCHEMA_SQL` block awkward → per-module schema-fragment refactor" forecast — not triggered by adding a *column*.
- MC-7: no `user_id` on any Quiz-domain table. MC-10: `import sqlite3` and SQL literals only under `app/persistence/`; callers receive dataclasses, never raw rows or `sqlite3.Connection`.

The decision space:

- **The storage form.** A nullable `questions.test_suite TEXT` column (additive `ALTER TABLE ADD COLUMN`) — vs. a `question_tests` table (a row per test case; only if ADR-040 had picked the structured-cases representation, which it didn't) — vs. a `NOT NULL DEFAULT ''` column (forces a default; complicates the "this Question has no test suite" failure-detection that ADR-040's processor relies on — see below).
- **Whether the column is in `_SCHEMA_SQL` or in `_apply_additive_migrations`.** A fresh DB created from `_SCHEMA_SQL` could have `test_suite` in the `CREATE TABLE questions` block directly; but an *existing* DB (one that already has the pre-TASK-016 `questions` table) needs the `ALTER TABLE ADD COLUMN` path — and putting it *only* in `_apply_additive_migrations` keeps the column-add logic in one place and mirrors ADR-037's `generation_error` precedent. (Alternatively: add it to *both* — the `CREATE TABLE` block for fresh DBs *and* the `ALTER` check for existing ones — which is what the codebase's existing pattern is for the *pre-TASK-016* columns. The `generation_error` column, added later, is only in the `ALTER` check.)
- **The `Question` dataclass field's type.** `test_suite: str` (always present)? `test_suite: str | None` (NULL for a Question that predates the column, or — by ADR-040's failure handling — for no Question, since a Question with no test suite never gets persisted)? — given the column is nullable (an existing-DB Question created before this task has `test_suite IS NULL`), `str | None` is the honest type; ADR-040's processor guarantees that *newly-persisted* Questions always have a non-empty `test_suite`, but the dataclass must model the column, which can be NULL for legacy rows.
- **The `add_questions_to_quiz` payload shape.** Each dict gains a `"test_suite"` key (`q["test_suite"]` or `q.get("test_suite")` — given ADR-040's processor whole-Quiz check has already rejected any Question missing it before calling `add_questions_to_quiz`, `q["test_suite"]` is safe; a `.get` with a fallback would be defensive but the contract is "the caller validated"). The function INSERTs it into the new column.
- **Whether a new accessor is needed.** A `get_question(question_id) -> Question | None` that returns the test suite? — nothing *reads* the test suite this task except a round-trip test (`list_questions_for_quiz` already returns `Question` dataclasses, which now carry `test_suite`); the in-app-runner slice (later) is the first thing that *consumes* the test suite at runtime, and *its* `/design` adds whatever accessor it needs. So: no new accessor this task — the existing `Question`-returning accessors carry the new field through; a round-trip test reads it via `list_questions_for_quiz`.

The manifest constrains the decision: §5 ("No multi-user features" — no `user_id` on the new column; "No non-coding Question formats" — the column holds only test *code*, never a non-coding artifact, by the same logic ADR-033's table has no `choices`/`correct_choice`/`answer_text` column), §7 ("Every Quiz Attempt, Note, and completion mark persists across sessions" — the test suite persists with the Question, which is part of the Bank, "never deleted" per §8), §8 (**Question** "carries a test suite … Persists with full Attempt history" — the column is the persistence; **Question Bank** "Never deleted" — there is no `questions`-delete path, and adding one is not in scope).

## Decision

### The storage — a nullable `questions.test_suite TEXT` column, added via the `_apply_additive_migrations` check (mirroring ADR-037's `generation_error`); also declared in the `CREATE TABLE questions` block in `_SCHEMA_SQL` for fresh databases

The `questions` table gains a column **`test_suite TEXT`** — nullable, no default (a NULL value means "this Question has no recorded test suite", which by ADR-040's failure handling can only be a Question that predates this task; a newly-persisted Question always has a non-empty `test_suite`). The column is added two ways, both idempotent:

1. **In `_apply_additive_migrations(conn)`** in `app/persistence/connection.py` — a `PRAGMA table_info(questions)` check + a guarded `ALTER TABLE questions ADD COLUMN test_suite TEXT`, mirroring the `quizzes.generation_error` precedent ADR-037 set. This is what brings an *existing* `data/notes.db` (one created before TASK-016) up to date — no migration-trigger fires (ADR-022's migration story: a nullable `ALTER TABLE ADD COLUMN` is additive).
2. **In the `CREATE TABLE IF NOT EXISTS questions` block in `_SCHEMA_SQL`** — the column is added to the `CREATE TABLE` statement so a *fresh* database (one created from scratch — e.g. a test's `NOTES_DB_PATH` monkeypatch) gets the column directly without needing the `ALTER` round-trip. (This is the codebase's existing pattern for the pre-TASK-016 `questions` columns — `prompt`, `topics`, etc. are in the `CREATE TABLE` block; the `generation_error` column, added later, is *only* in the `ALTER` check. Adding `test_suite` to *both* keeps a fresh DB clean and an existing DB upgraded; the `PRAGMA table_info` check makes the `ALTER` a silent no-op when the column already exists from the `CREATE TABLE` block. The implementer may, equivalently, put `test_suite` *only* in the `ALTER` check like `generation_error` — both are correct; "in both" is the more readable shape for a column that's part of the table's logical schema, "only in the ALTER check" is the more minimal diff. Implementer's call; not architecture.)

**Why a nullable column, not `NOT NULL DEFAULT ''`:** SQLite can `ALTER TABLE ADD COLUMN` with a `NOT NULL DEFAULT '<literal>'`, so it's *possible* — but a `DEFAULT ''` means a Question with no test suite reads back as `test_suite = ''` (an empty string), which is harder to distinguish from "the column exists but this row genuinely has no test suite" than a NULL is, and it would make a legacy Question look like it has an *(empty)* test suite rather than *no* test suite. Nullable is the honest model: NULL = "no recorded test suite" (legacy rows only); a non-empty string = "the test suite" (every newly-persisted Question, by ADR-040's failure handling). ADR-040's processor never persists a Question with a missing/empty test suite (the whole Quiz is `generation_failed` instead), so in practice every `questions` row created from this task forward has a non-empty `test_suite`; the nullable type is for the legacy-row case and for honest column modeling.

**Why a column, not a `question_tests` table:** ADR-040 picked the single-source-code-string representation (the in-app runner needs the test suite *runnable*, not *queryable*); a string is one TEXT column on `questions`, mirroring `prompt` and `topics` being columns. A `question_tests` table would only make sense for a structured-test-cases representation, which ADR-040 rejected. ADR-033's "n-tables threshold → per-module schema-fragment refactor" forecast is **not triggered** — this task adds a *column* to an existing table, not a table.

### The `Question` dataclass — gains `test_suite: str | None`

The `Question` dataclass in `app/persistence/quizzes.py` gains a field **`test_suite: str | None`** (placed after `topics`, before `created_at`, or wherever reads cleanly — implementer's call; not architecture). `_row_to_question(row)` carries it through: `test_suite=row["test_suite"]` (the `sqlite3.Row` yields `None` for a NULL column — exactly the `str | None` the field models). The dataclass continues to model the `questions` table columns; `test_suite` is `str | None` because the column is nullable (a legacy row has `test_suite IS NULL`; a newly-persisted row has a non-empty string). No `user_id` (MC-7). The docstring notes "the test suite holds runnable test source code (ADR-040); NULL only for a Question that predates TASK-016 — a Question persisted from TASK-016 forward always has a non-empty test suite, since ADR-040's generation processor fails the whole Quiz rather than persist a Question without one."

### `add_questions_to_quiz` — the per-Question payload dict gains a `"test_suite"` key

`add_questions_to_quiz(quiz_id, questions: list[dict])` is unchanged in signature; each dict in `questions` now carries a `"test_suite"` key (a non-empty string — ADR-040's processor has already done the whole-Quiz "any Question missing/empty in `test_suite` → `generation_failed`" check before calling this, so the contract is "every dict has a non-empty `test_suite`"; the function reads `q["test_suite"]`, or `q.get("test_suite")` for defence-in-depth — implementer's call). The function's INSERT gains the column:

```
INSERT INTO questions (section_id, prompt, topics, test_suite, created_at)
VALUES (?, ?, ?, ?, ?)
```

— `test_suite` from `q["test_suite"]`. The `quiz_questions` INSERT and the one-transaction discipline are unchanged. The docstring's "`questions` is a list of dicts: `[{"prompt": str, "topics": list[str]}, ...]`" becomes "`[{"prompt": str, "topics": list[str], "test_suite": str}, ...]`"; the "the caller is responsible for sanity-checking that questions is non-empty and each prompt is non-empty (ADR-037)" note becomes "… and each `test_suite` is non-empty (ADR-040 — the processor fails the whole Quiz otherwise)". MC-2 unchanged (every Question carries the Quiz's `section_id`); MC-10 unchanged (SQL stays here); no `user_id` (MC-7).

### No new accessor this task — the existing `Question`-returning accessors carry the new field through

Nothing *reads* the test suite at runtime this task — the in-app-runner slice (later) is the first thing that *consumes* it (to run it against the learner's code), and *its* `/design` adds whatever accessor it needs (e.g. a `get_question(question_id) -> Question | None`, or a `get_test_suite_for_question(question_id) -> str | None`). The existing `Question`-returning accessors — `list_questions_for_quiz(quiz_id) -> list[Question]` (ADR-039) — carry the new `test_suite` field through automatically (the dataclass gains the attribute; the join's `SELECT q.question_id, q.section_id, q.prompt, q.topics, q.created_at` gains `q.test_suite` so `_row_to_question` can read it). A round-trip test reads `test_suite` back via `list_questions_for_quiz` (after a fresh DB connection). So: **no new public function this task**; `list_questions_for_quiz`'s `SELECT` adds `q.test_suite` (and `_row_to_question` reads it); `add_questions_to_quiz`'s `INSERT` adds the column; `Question` gains the field; that's the whole persistence change. `app/persistence/__init__.py`'s re-exports are unchanged (no new symbol — `Question` is already re-exported; it just carries one more attribute).

## Alternatives considered

**A. A `question_tests` table (a row per test case), instead of a `questions.test_suite TEXT` column.**
Rejected — this would only make sense for a structured-test-cases representation, which ADR-040 rejected (the in-app runner needs the test suite *runnable* — a source-code string — not *queryable*; "input"/"expected output" don't generalize across DSA concepts). A source-code string is one TEXT column on `questions`, mirroring `prompt` and `topics`. A `question_tests` table is more surface (a join, a row-per-case, ADR-033's n-tables threshold closer) for a representation the runner can't use. (If a future runner slice's `/design` wants structured cases, ADR-040's supersedure path covers migrating the string column to a `question_tests` table — additive `CREATE TABLE IF NOT EXISTS`; bounded cost; but the runner doesn't need it.)

**B. A `NOT NULL DEFAULT ''` column instead of a nullable one.**
Rejected. SQLite *can* `ALTER TABLE ADD COLUMN test_suite TEXT NOT NULL DEFAULT ''` — but a `DEFAULT ''` means a Question with no test suite reads back as `test_suite = ''`, which is harder to distinguish from "this row genuinely has an empty test suite" than a NULL, and it makes a legacy Question look like it has an *(empty)* test suite rather than *no* test suite. Nullable is the honest model: NULL = "no recorded test suite" (legacy rows only); a non-empty string = "the test suite". ADR-040's processor never persists a Question with a missing/empty test suite, so in practice every row from this task forward has a non-empty `test_suite`; the nullable type is for legacy rows and honest column modeling. (And it matches `quizzes.generation_error`'s precedent — a nullable column for "this fact may or may not exist for this row".)

**C. Add `test_suite` only to `_SCHEMA_SQL`'s `CREATE TABLE questions` block, not to `_apply_additive_migrations`.**
Rejected — an existing `data/notes.db` (one created before TASK-016) already has a `questions` table; `CREATE TABLE IF NOT EXISTS` is a no-op on it, so the column would never get added to an existing DB. The `_apply_additive_migrations` `PRAGMA table_info` + `ALTER` path is what upgrades an existing DB (ADR-022's migration story; ADR-037's `generation_error` precedent). The column must be in the `ALTER` check; whether it's *also* in the `CREATE TABLE` block (for fresh DBs to skip the `ALTER` round-trip) is the implementer's call.

**D. Add a `get_question(question_id) -> Question | None` accessor this task.**
Considered (the task's forecast: "any new accessor — `get_question` returning the test suite, if one is wanted before the in-app-runner slice — likely not needed this task, since nothing reads the test suite yet except a round-trip test"). Rejected as premature — nothing *consumes* the test suite at runtime this task; the in-app-runner slice (later) is the first thing that does, and *its* `/design` adds the accessor it needs (which might be `get_question`, or `get_test_suite_for_question`, or something else — its call). Adding it now would be an accessor with no caller — the architecture-on-spec anti-pattern. The existing `Question`-returning accessors (`list_questions_for_quiz`) carry the new field through; a round-trip test reads it via that. (If the test-writer finds it genuinely needs a single-Question accessor for a unit test rather than going through `list_questions_for_quiz` + a Quiz, that's a small implementer addition the runner slice would have added anyway — not a re-decision; but the forecast and this ADR both say "likely not needed".)

**E. Fold this ADR into ADR-040 (one ADR, not two).**
Considered (the task: "one ADR or two is `/design`'s call — the persistence change is small and tightly coupled to the test-suite-representation decision, so folding it into the test-suite ADR's Decision section is reasonable; splitting is fine too"). The architect split it — ADR-040 stays about the *representation/workflow/processor/failure/surface* shape (the substantive decisions and their alternatives), this ADR about the *storage mechanics* (the column, the dataclass, the function payload, the converter — concrete, crisp, low-controversy). The split keeps each ADR focused and makes the storage decision independently citable (the in-app-runner slice will cite "ADR-041 — the `questions.test_suite` column" when it adds its accessor and the `attempt_questions` test-result columns). Either was fine; the split is the cleaner record.

The nullable column models "no recorded test suite" honestly (NULL = none, for legacy rows only) — it never *fabricates* a test suite (an empty default string would be closer to fabrication; nullable avoids it — Alternative B). **PASS.**
- **MC-6 (Lecture source read-only).** Honored — the `test_suite` column lives in `data/notes.db`; nothing under `content/latex/` is opened for write by `app/persistence/quizzes.py` or `connection.py`. **PASS.**
- **MC-7 (Single user).** Honored — no `user_id` on the new `questions.test_suite` column, the `Question` dataclass's new field, or any other new construct; no auth, no session, no per-user partitioning. Architecture portion (active per ADR-022): a future contributor cannot add a `user_id` to `questions` without superseding ADR-022 and ADR-033. **PASS.**
- **MC-8 (Reinforcement loop preserved).** Honored — the new column doesn't touch the loop's machinery (`quiz_questions` multi-membership, `attempt_questions.is_correct` wrong-answer history); it advances the loop's precondition (the test suite the grading slice needs to compute correctness, which the composition slice needs for Weak Topics — ADR-040). No fresh-only-post-first-Quiz path created. **PASS.**
- **MC-9 (Quiz generation is user-triggered).** Orthogonal — this ADR doesn't add a generation trigger; `add_questions_to_quiz` is called by the out-of-band processor (ADR-037) processing existing user-created `requested` rows; nothing scheduled/auto. **PASS.**
- **MC-10 (Persistence boundary).** Honored — `import sqlite3` and SQL literals stay under `app/persistence/` (`connection.py` for the `ALTER TABLE` / `CREATE TABLE` DDL; `quizzes.py` for the `INSERT` and `SELECT`); the `Question` dataclass and the typed public functions are the only surface callers see — the processor passes a `list[dict]` to `add_questions_to_quiz` and receives `Quiz` dataclasses; `list_questions_for_quiz` returns `Question` dataclasses (now carrying `test_suite`); no caller receives a `sqlite3.Connection` or a raw row tuple. **PASS** (the persistence package exists in code — the rule is at **blocker** severity; this ADR's change stays within the boundary).

No previously-dormant rule is activated by this ADR (MC-7 / MC-10 are already active; this ADR's change stays within both).

## Consequences

**Becomes possible:**

- A `questions` row carries its test suite (`questions.test_suite`) — readable back via the `Question` dataclass; a round-trip (write, fresh connection, read) returns it intact (the test suite persists with the Question, which is part of the Bank, "never deleted" — §8).
- The in-app-runner slice (later) to read a Question's test suite at runtime (it adds whatever accessor it needs — `get_question` / `get_test_suite_for_question` / etc. — citing this ADR's `questions.test_suite` column) and run it against the learner's submitted `response` code.
- The grading slice (later, keyed off test results) to know which Questions had which test suites (via `list_questions_for_quiz`'s `Question` dataclasses, now carrying `test_suite`).

**Becomes more expensive:**

- `connection.py` grows: the `test_suite TEXT` column in the `CREATE TABLE IF NOT EXISTS questions` block (for fresh DBs) and a `PRAGMA table_info(questions)` + `ALTER TABLE questions ADD COLUMN test_suite TEXT` in `_apply_additive_migrations` (for existing DBs) — mirroring ADR-037's `generation_error`. Mitigation: ~4 lines; the mechanic already exists; idempotent; no migration-trigger.
- `quizzes.py` grows: the `Question` dataclass's `test_suite: str | None` field; `_row_to_question` reading `row["test_suite"]`; `add_questions_to_quiz`'s `INSERT` gaining the column (reading `q["test_suite"]`); `list_questions_for_quiz`'s `SELECT` gaining `q.test_suite`. Mitigation: each is a one-line change following the existing `questions.topics` pattern; no signature changes.
- `app/workflows/process_quiz_requests.py`'s payload-build step includes `test_suite` per Question (ADR-040 owns this — listed here only as the upstream caller of the changed `add_questions_to_quiz` contract).

**Becomes impossible (under this ADR):**

- A `questions` row without a `test_suite` column (the column is added to every DB — fresh via `_SCHEMA_SQL`, existing via `_apply_additive_migrations`).
- A newly-persisted Question with an empty/missing test suite (ADR-040's processor fails the whole Quiz instead — `add_questions_to_quiz` is only ever called with non-empty `test_suite` payloads; the contract is enforced upstream).
- A `user_id` on `questions` (MC-7 preserved; ADR-022 + ADR-033 would have to be superseded to add one).
- The persistence layer leaking a raw row or `sqlite3.Connection` for the test suite (callers get the `Question` dataclass, which carries `test_suite`; SQL stays under `app/persistence/` — MC-10).

**Supersedure path if this proves wrong:**

- If the source-code-string column proves the wrong shape (the runner slice wants structured test cases) → ADR-040's supersedure path applies: a future ADR migrates `questions.test_suite` (a source string) to a `question_tests` table (parsed cases) — additive `CREATE TABLE IF NOT EXISTS`; bounded cost.
- If `test_suite` should carry metadata (language / test framework / entry-point name) → a future ADR adds those as additional nullable columns on `questions` (additive); the `test_suite` string stays the runnable code. Bounded.
- If a single-Question accessor (`get_question` / `get_test_suite_for_question`) is wanted before the runner slice → it's a small implementer addition (a `SELECT ... FROM questions WHERE question_id = ?` in `quizzes.py`, re-exported from `__init__.py`) — not a re-decision; the runner slice would add it anyway.

The supersedure path, in every case, runs through a new ADR. This ADR does not edit any prior ADR in place; it consumes ADR-033 (the `questions` schema — the new column is additive), ADR-022 (the persistence boundary / additive-migration story — preserved), ADR-037 (the `add_questions_to_quiz` function — a payload-key addition, not a signature change; the `_apply_additive_migrations` mechanic — extended), and ADR-039 (the `Question`-returning accessor — the dataclass carries one more attribute); the test-suite representation / workflow / processor / failure-handling / surface decisions are in the companion ADR-040.
