# ADR-032: Saving a Note must not relocate the reader — the Notes-form PRG round-trip needs a no-relocation mechanism (forecast: client-side JavaScript); recorded now, deferred for a future task

**Status:** `Proposed`
**Date:** 2026-05-11
**Task:** none (parked / deferred — recorded as a follow-up to TASK-012 `966b824`; no task scheduled, no implementation now)
**Resolves:** none (a thin tracking project_issue, `notes-save-relocates-reader.md`, is filed alongside this ADR; this ADR is the primary artifact)

## Context

### The empirical finding

TASK-012 (committed `966b824`) moved the Notes panel into a right-hand sticky rail (ADR-029) and fixed the section-completion `303` redirect from snapping the reader away — by anchoring it to `#section-{n}-end` on the `.section-end` wrapper plus a large `scroll-margin-top` on `.section-end` in `lecture.css`, **no JavaScript** (ADR-031, superseding ADR-030 §Decision; ADR-030's no-fragment "Option 1" was empirically refuted — the Playwright regression test it itself mandated proved Chromium resets `scrollY` to `0` on a fragment-less `POST → 303 → GET /lecture/{chapter_id}` same-URL navigation: `pre-click scrollY=3514px → post-reload scrollY=0px`, audit Run 006).

That empirical finding has a consequence ADR-030 / ADR-031 noted only for the completion toggle but that applies equally to the **Notes form**: the Notes-form save is *also* a fragment-less `POST → 303 → GET /lecture/{chapter_id}` navigation (ADR-023 §Route-shape: "On success, the route returns **HTTP 303 See Other** with `Location: /lecture/{chapter_id}`" — no fragment; carried unchanged by ADR-028 / ADR-029). Since Chromium resets `scrollY` to `0` on exactly that navigation shape (proven by the TASK-012 Playwright test), **saving a Note snaps the reader to the top of the chapter** — they lose their reading position in the main column. The sticky RHS Notes rail stays visible (it is `position: sticky`), so the *Notes panel* does not move; but the *reading position in `.page-main`* is lost. A reader scrolled to the middle of a 30-viewport chapter who pops a quick note into the RHS rail is teleported back to the chapter heading.

### What this ADR corrects

ADR-023, ADR-028, ADR-029, and ADR-030 all asserted — or built on the assertion — that the Notes-form PRG round-trip does **not** have the snap problem. Specifically:

- **ADR-030 §"Scope of this supersedure ADR" → "This ADR does not decide: ... Async (JS) treatment of the Notes form. Out of scope here — the Notes form's PRG round-trip does not have the snap problem."** And ADR-030 §"Option 2 contingency": "the Notes form is explicitly *not* given the async treatment in that ADR — the Notes form's PRG round-trip does not have the snap problem because the form is in the rail, not at the bottom of a 30-viewport Section."
- **ADR-031 §"This ADR does not decide: ... Async (JS) treatment of the Notes form. Out of scope here — the Notes form's PRG round-trip does not have the snap problem (the form is in the rail, not at the bottom of a tall Section)."** ADR-031 §Alternatives-considered A even spells out the (incorrect) reasoning: "the Notes form's *post-reload* scroll-to-top is not noticed because the form was near the top to begin with."
- **ADR-029 / ADR-023** carried the fragment-less `303 → GET /lecture/{chapter_id}` redirect forward without flagging it as a scroll concern.

**That assertion is wrong, and this ADR records the correction.** The reasoning in those ADRs ("the form is in the rail near the top, so the post-reload scroll-to-top is not noticed") confused *where the form is* with *where the reader is*. The form is in a sticky RHS rail — but the reader can be scrolled anywhere in the main reading column when they use it; the rail is sticky precisely so the reader can note something without scrolling back up. After the save, the rail stays put (sticky) but the main column snaps to the top, so the reader's *reading position* is exactly as disrupted as it was for the bottom-of-Section completion toggle before ADR-031. The "the form is near the top" reasoning held for the *redirect shape* (ADR-023's `303 → /lecture/{chapter_id}` is the same shape), but not for the *user-perceived effect* — which is exactly the distinction ADR-031 §Alternatives-considered A drew for the completion toggle, mis-applied to the Notes form in the opposite direction.

This is not a supersedure of ADR-023 / ADR-028 / ADR-029 — none of their *decisions* (route shape, validation, PRG, multiple-Note display order, the RHS-rail placement, the `rail_notes_context` template variable, the no-edit/delete/Markdown scoping, the textarea-sizing approach, the "visibility follows scroll-position-cost" principle) is changed by this ADR. What is corrected is a *factual side-note* those ADRs carried ("the Notes PRG doesn't have the snap problem"). The route, the template, the CSS, and the persistence layer are all untouched by this ADR.

### The requirement

The load-bearing principle ADR-030 encoded and ADR-031 implemented for the completion toggle —

> **The response to a reading-flow action should not relocate the user. An affordance placed at the natural reading position earns a response that leaves the user at that position — it is an annotation on what was just read, not a navigation event.**

— applies, by the same logic, to the Notes-form save: writing a Note about the chapter you are reading is an annotation on what you are reading, not a navigation event; the response (the page re-render with the new Note in the RHS list) must leave you where you were in the chapter. This ADR records that requirement. The *mechanism* that delivers it is the open question — and unlike the completion toggle, there is no clean no-JS mechanism (see "Alternatives considered" §A): the only ways to keep the reader where they were across the Notes save are (a) submit the form without a navigation at all (async — JS), or (b) record and restore the scroll position around the existing PRG round-trip (JS). Hence: forecast mechanism = client-side JavaScript, the project's first; recorded now, deferred for a future task.

## Decision

**The requirement (binding once a task implements it; recorded now):** saving a Note via the Notes form must not relocate the reader from their reading position in the main column. Writing a Note about the chapter you are reading is an annotation on what you are reading, not a navigation event; the response (the page re-render with the new Note in the RHS list) leaves the reader where they were in the chapter. This is ADR-030's load-bearing principle (retained by ADR-031), extended to the Notes-form save.

**The mechanism (forecast — client-side JavaScript; decided by a future task's `/design`, not here):** no clean no-JS / CSS-only mechanism exists for the general case (see "Alternatives considered" §A — the completion toggle's `#section-{n}-end` anchor + `scroll-margin-top` trick works only because `.section-end` is a static element at a known position near where the user clicked; the Notes form has no analogous static anchor, because the reader can be scrolled anywhere when they use the always-visible sticky rail). The mechanism is therefore client-side JavaScript. The future task's `/design` chooses between two candidate shapes (not chosen here):

- **(a) async `fetch()` submit** of the Notes form that updates the RHS Notes list in place with **no navigation at all** — cleanest, also zero scroll motion (the page never re-navigates), and could later extend symmetrically to the completion toggle (the ADR-030 Option-2 shape that ADR-031 rendered moot for completion specifically); or
- **(b) JS scroll-restore around the existing PRG round-trip** — record `window.scrollY` in `sessionStorage` before submit, restore it on the next `GET /lecture/{chapter_id}` load — minimal, leaves the PRG path intact, but still has a flicker (reload → scroll-jump-to-0 → restore).

This ADR does **not** pick between (a) and (b). That is the future task's `/design`, which will also decide: where the JS lives (`app/static/`), how it is loaded (a `<script>` tag in `base.html.j2`? a module?), whether the no-JS PRG path is retained as a progressive-enhancement fallback (forecast: yes — progressive enhancement, not JS-required), how it is tested (Playwright with JS-on / JS-off variants), and whether the same mechanism is extended to the completion toggle for symmetry (ADR-031's `#section-{n}-end` + `scroll-margin-top` mechanism stays the no-JS baseline regardless).

**Status / sequencing (binding now):**

- ADR-032 is `Proposed` and **stays `Proposed` / deferred** until a future task picks it up. It is **not** auto-accepted. No task is scheduled. No test or code change happens now.
- ADR-032's implementation will introduce the project's first client-side JavaScript — and that is **not effected by this ADR**; it happens when a future task actually implements the JS (likely as part of that task's `/design`). (Reframing note, post-ADR-035: this is *not* a "no-JS-posture supersedure" — there is no binding no-JS posture to supersede. The "no-JS commitments" the prior wording here referred to in ADR-023 / ADR-025 / ADR-027 / ADR-028 / ADR-029 / ADR-030 / ADR-031 were descriptions of what those surfaces happen to do, since reframed by ADR-035. The future task accepts *that JS itself* on its own merits — where it lives, how it loads, how it's tested-and-tested-without — not the lifting of a project-wide ban, because there is none.)
- This ADR corrects a mistaken assumption ADR-023 / ADR-028 / ADR-029 / ADR-030 / ADR-031 carried ("the Notes PRG does not have the snap problem"); it does **not** supersede those ADRs (their decisions stand). The correction is recorded in the "What this ADR corrects" context section above; no edit to those ADRs' decision bodies is made (per `AS-1` / Accepted-ADR content immutability — the correction lives in this new ADR, not as an in-place rewrite of theirs).

## Alternatives considered

### A. No-JS / CSS-only mechanism — honestly evaluated; none works for the general case

The human asked: *"unless there is another css hack."* This is the alternative that, if it existed, would let this ADR recommend a no-JS mechanism and skip the deferral. It does not exist for the general case. The candidates considered:

- **The ADR-031 trick — anchor the `303` redirect to a static element near where the user acted, plus `scroll-margin-top` to land it back where it was.** This worked for the completion toggle because `.section-end` is a *static element at a known position* (the bottom of the just-finished Section) *near where the user clicked* (the "mark complete" button inside `.section-end`). The Notes form has no analogous static anchor. The Notes form is in an always-visible sticky RHS rail; the reader can be scrolled to *any* position in the main column when they use it (that is the *point* of the sticky rail — note something without scrolling back). There is no element in the DOM at "wherever the reader happened to be scrolled to" — restoring that position requires *knowing* it, which requires the form to send it (→ JS) or the page to remember it across the navigation (→ JS). **A per-section anchor** (anchor the redirect to the Section the reader is currently looking at) would require JS to determine *which* Section that is at submit time — same problem. **Rejected:** no static anchor exists for "where the reader was."
- **CSS scroll-anchoring (`overflow-anchor: auto`).** Chrome's scroll-anchoring keeps the viewport anchored to on-screen content when content *above* the viewport changes height. It does **not** carry across a navigation — a `POST → 303 → GET` is a fresh document load; scroll-anchoring starts from scratch on the new document, which loads at `scrollY=0`. **Rejected:** wrong tool; scroll-anchoring is a same-document mechanism, not a cross-navigation one.
- **`autofocus`-based scroll** (give an element near the reader's position `autofocus` so the browser scrolls to it on load). Unreliable (browsers scroll focused elements into view inconsistently, and only if the element is focusable), and — same problem as the anchor — requires knowing *which* element is "near the reader's position," which requires JS. **Rejected.**
- **Replace the PRG with a `<form target="_blank">` / iframe trick** so the navigation happens in a hidden frame and the main document never reloads. This is a hack that *does* avoid the navigation — but it is fragile (the main document's Notes list does not update without JS to copy the result back in), it abuses `target` semantics, and it is strictly worse than an honest `fetch()`. **Rejected:** not actually no-JS-clean (still needs JS to reflect the new Note), and uglier than option (a).
- **`scroll-behavior` / `:target` CSS** — irrelevant; these affect *how* the browser scrolls to a fragment, not *whether the reader's prior position is preserved*. There is no fragment that names "where the reader was." **Rejected.**

**Conclusion: there is no clean no-JS mechanism.** The completion toggle was a special case (a static affordance at a known reading position); the Notes form is the general case (an always-visible affordance used from an arbitrary reading position). The mechanism is therefore client-side JavaScript, per the human's call.

### B. Async `fetch()` submit of the Notes form — no navigation, list updated in place (forecast mechanism shape (a))

The cleanest UX: the form submits via `fetch()`, the server returns the new Note (or the re-rendered Notes-list fragment), the JS prepends it to `.rail-notes-list` (and clears the textarea / swaps the empty-state for the list on the first Note). **No navigation at all → zero scroll motion, zero flicker.** Could later extend symmetrically to the completion toggle (this is exactly the ADR-030 "Option 2" shape that was rejected *for the completion toggle specifically* because ADR-031's no-JS mechanism solved it — but the Notes form, having no no-JS mechanism, is the natural place for this shape to land first). **Not chosen here** — that is the future task's `/design` (it has to decide the JSON-vs-HTML-fragment response shape, the `<script>` loading, the progressive-enhancement fallback, the test variants). Recorded as a candidate.

### C. JS scroll-restore around the existing PRG round-trip (forecast mechanism shape (b))

Minimal: a small `<script>` that records `window.scrollY` in `sessionStorage` on the Notes form's `submit` event, and on `DOMContentLoaded` for `GET /lecture/{chapter_id}` reads it back and calls `window.scrollTo(0, saved)` (then clears it). Leaves the entire PRG path intact (route, redirect, template all unchanged); the JS is purely additive. **Downside:** the reload still happens — the reader sees a flash to the top of the page and then a jump back to their position; it is "restore" not "never move," strictly worse than (B)'s "never move." **Not chosen here** — recorded as a candidate; the future task's `/design` weighs (B) vs (C).

### D. Do nothing — accept the snap (the status-quo non-choice the human is rejecting)

Leave the Notes-form PRG as-is: saving a Note snaps the reader to the top of the chapter. **Rejected by the human** — *"make new adr for notes forcing anchor on save."* The human explicitly wants this fixed (eventually); the status quo is the thing this ADR exists to schedule a fix for. Recorded as the rejected baseline so the ADR's reason-for-existing is on the record.

### E. Move the Notes form somewhere it does not have the problem (e.g., a separate `/lecture/{chapter_id}/notes` page)

Considered and rejected. ADR-023 explicitly rejected a separate Notes page ("No `GET /lecture/{chapter_id}/notes` partial route. The Notes UI lives inside the Lecture page"); ADR-028 / ADR-029 doubled down on the rail-resident placement for visibility-from-any-scroll-position reasons ("visibility follows scroll-position-cost"). Moving the form off the Lecture page would make the snap irrelevant *by removing the in-context affordance the project deliberately built* — the cure would be worse than the disease. **Rejected:** contradicts ADR-023 / ADR-028 / ADR-029, which this ADR is explicitly *not* superseding.

### F. Decide the Notes-save JavaScript now, as part of this ADR

Considered. The argument: this ADR forecasts JS, so it could also decide where the JS lives / how it loads / how it's tested now and have it on the books. **Rejected** because (a) the human deferred — *"we'll deal with it in the future not now"*; (b) those decisions (JSON-vs-HTML-fragment response, `<script>` loading, build-or-no-build, progressive-enhancement fallback, Playwright JS-on/JS-off variants) belong in the `/design` cycle of the task that *actually implements* the JS, with the implementation in hand. **Rejected:** this ADR records the *requirement* (Notes save must not relocate the reader) and the *forecast mechanism* (client-side JS), not the JS's architecture. (Reframing note, post-ADR-035: the original wording of this section spoke of "the no-JS-posture supersedure" / "flipping a project-wide posture" / "the no-JS posture stays in force" — that framing is corrected by ADR-035: there is no binding no-JS posture; the prior ADRs' "no-JS" phrasing was a description of what those surfaces do, not a rule. So there is nothing to "flip"; the future task simply decides this surface's JavaScript, the same way any surface's architecture gets its own ADR. Point (b) of the original — don't have a JS-architecture decision "Accepted" while no JS exists — still stands, restated above without the "posture" framing.)

### G. Write this as a project_issue instead of an ADR

Considered. A project_issue ("known unresolved architectural question") would be a defensible home for "the Notes PRG has the snap problem; how do we fix it?" **Rejected as the primary artifact** because the human asked specifically for an ADR — *"make new adr for notes forcing anchor on save"* — and because the *requirement* (Notes save must not relocate the reader) is a decision the human has made, not an open question; only the *mechanism between (a) and (b)* is open, and that is a future task's `/design`, not a standing project_issue. A thin project_issue pointer (`notes-save-relocates-reader.md`) is filed *alongside* this ADR purely as a tracking surface for the deferral (so `ls design_docs/project_issues/` shows the parked item), pointing at this ADR as the primary artifact. **The ADR is the decision; the project_issue is the tracking breadcrumb.**

- **§3 Primary Objective.** Drive consumption + retention. Notes is one of three named consumption pillars; a Notes save that throws the reader to the top of the chapter is friction against the consumption objective — the same kind of friction ADR-030/ADR-031 removed for the completion toggle. Recording the requirement to fix it (eventually) serves §3; the deferral is a sequencing call, not a §3 concession.
- **§5 Non-Goals.** "No mobile-first" — Chromium is the binding UI test target per ADR-010; other browsers' navigation-scroll behavior is out of scope; the future task's Playwright test (whatever the mechanism) locks the binding-target behavior in. "No LMS / no AI tutor / no remote deployment / no multi-user" — all orthogonal; nothing in this ADR touches them; the future JS, when it lands, is reader-facing scroll-behavior code, not an AI surface, not a multi-user feature.
- **§6 Behaviors and Absolutes.** "Single-user" — preserved; no `user_id` is involved; the Notes form and route are untouched by this ADR and will stay single-user when the future task implements the mechanism. "AI work asynchronous" — orthogonal; the Notes save is not AI work (manifest §8: a Note is "Never auto-generated"); a `fetch()`-based async submit (mechanism shape (a)) is async in the *network* sense but is not "AI work" — treating it as an MC-4 / §6-AI-async matter would be a misread. "Lecture source read-only" — preserved; the future implementation touches `app/main.py` / `app/templates/` / `app/static/`, never `content/latex/`. "Mandatory and Optional honored everywhere" — orthogonal; the Notes panel is per-Chapter regardless of designation.
- **§7 Invariants.** **"A Note is bound to exactly one Chapter and may optionally reference one Section within it."** — directly preserved and entirely untouched; this ADR is purely a scroll-behavior concern; the per-Chapter binding (encoded in the `/lecture/{chapter_id}/notes` route URL) is unchanged. "Every Note … persists across sessions" — preserved untouched (the persistence layer per ADR-022 is not involved). "Completion state lives at the Section level" — orthogonal (that's ADR-024 / ADR-031's surface). "Mandatory and Optional are separable in every learner-facing surface" — preserved untouched.
- **§8 Glossary.** Note, Chapter, Section — all stable; no entity definition changes; no new terms. "Notification" — unrelated; the Notes save is synchronous-from-the-user's-perspective (or async-via-`fetch()`-with-immediate-DOM-update under mechanism (a)); it is not an AI result, so no Notification is involved.

**Nothing forces or forbids JavaScript at the manifest level.** The no-JS posture is an *ADR-level* commitment (ADR-023's "The project has no JavaScript build/serve story" reasoning, carried by ADR-025 / ADR-027 / ADR-028 / ADR-029 / ADR-030 / ADR-031), not a manifest one. The manifest's only architectural commitment is §4's `ai-workflows`-is-the-AI-engine choice, which is orthogonal to client-side scroll-behavior JS. So the no-JS-posture supersedure that ADR-032's implementation will require is an ADR-vs-ADR matter, gated by the human at implementation time — not a `MANIFEST TENSION`. No manifest entry is flagged as architecture-in-disguise for this decision; the manifest is internally consistent with it.

## Conformance check

This ADR changes no code, so no conformance rule is *exercised* by it now; the check below is what the *future task that implements ADR-032* must satisfy, and confirms ADR-032 does not, by being recorded, violate any drift-critical guardrail.

- **MC-1 (No direct LLM/agent SDK use).** **MC-1 governs *AI* SDKs, not client-side JavaScript.** A `fetch()`-based Notes-form submit (mechanism shape (a)), or a `sessionStorage` scroll-restore (shape (b)), is client-side browser code with no AI surface — it is not an MC-1 matter. The future task's JS, whatever its shape, must not call an LLM SDK directly (it would not — it is scroll-behavior / form-submit code), so MC-1 stays satisfied. Recording this ADR does not touch MC-1.

- **MC-5 (AI failures surfaced).** Orthogonal — no AI in this surface. (The future task should surface a *Notes-save* failure to the reader as a failure — e.g., a "could not save note" message — but that is good form, not an MC-5 obligation, which is about AI-driven processing.)
- **MC-6 (Lecture source read-only).** Honored — this ADR changes nothing; the future implementation touches `app/main.py` / `app/templates/` / `app/static/`, never `content/latex/`. The pre-commit guard's "lecture source root never staged with application code" rule is unaffected.
- **MC-7 (Single user).** Honored — no `user_id`, no auth, no per-user partitioning is involved now or in the future implementation.
- **MC-8 (Reinforcement loop preserved).** Orthogonal — no Quiz machinery; saving a Note does not trigger Quiz generation.

- **MC-10 (Persistence boundary).** Honored — this ADR changes no DB code; the future implementation's `fetch()` (shape (a)) would still POST to the existing `POST /lecture/{chapter_id}/notes` route, which calls `create_note` / `list_notes_for_chapter` from `app/persistence/` — no `sqlite3` import or SQL literal outside `app/persistence/` is introduced.

The conformance skill has no MC rule about client JS; the no-JS posture lives in ADR-023 ff. (Accepted ADRs), so the threshold is an ADR-acceptance gate, not an MC-rule check.
