## Context

TASK-016 closes a gap the 2026-05-12 manifest amendment opened. The amendment tightened the Quiz pillar:

- §6 added "**Code is written, run, and tested within the application.** The learner answers a Question by writing code, running it, and running tests against it — all inside the application's own interface."
- §8 **Question** now reads "A Question is a hands-on coding task: it asks the learner to implement a concept from the Section in code, **and it carries a test suite the learner runs in-app to verify the implementation**. Has one or more Topic tags. Persists with full Attempt history."
- §8 **Grade** now defines per-Question correctness as "**determined by whether the learner's code passed the Question's tests**".

But the Questions the generation slice produces today (TASK-014, ADR-036) carry only `prompt: str` + `topics: list[str]` (`GeneratedQuestion`); the persisted `questions` rows have only `prompt`/`topics`/`section_id`/`created_at` (ADR-033). **No test suite anywhere.** That is load-bearing: until a Question carries a test suite, (a) the learner can't run tests in-app (§6), (b) the Quiz Attempt can't carry test results (§8 Quiz Attempt), and (c) a §8-conformant Grade — "correctness determined by whether the code passed the tests" — can't be computed. The reinforcement loop's next step (grading) is *defined* in terms of "the Question's tests", which don't exist yet.

This ADR makes them exist: the `question_gen` workflow emits a test suite per Question, and the generation processor persists it with the Question (which is part of the Question Bank — "never deleted" per §8). It deliberately does **not** ship the in-app *runner* (the sandbox for running arbitrary learner code, the "Run tests" affordance, the `attempt_questions` test-result columns, displaying test outcomes) — that is the next slice, with its own substantial sandboxing ADR; this ADR only makes the test *suite* exist.

The decision space has materially different alternatives:

- **The test-suite representation** the workflow emits and the Bank stores:
  - **(a) A single source-code string** — a runnable test module (e.g. a Python file with `unittest`/`pytest` cases against a named function/class the prompt asks the learner to implement). One string field on `GeneratedQuestion`; one TEXT column on `questions`; mirrors `questions.prompt` being a string.
  - **(b) A structured set of test cases** — `GeneratedQuestion.test_cases: list[TestCase]` where `TestCase` is `{name, input, expected_output, [harness]}`; persisted as a `question_tests` table (a row per case). More queryable; but the in-app runner doesn't need it *queryable*, it needs it *runnable*, and "what counts as a test case" varies wildly by language/concept (a graph algorithm's "input" isn't a string).
  - **(c) A file-per-Question on disk** — the workflow writes a `.py`/`.cpp` test file under some path; the `questions` row stores a path. Re-introduces a filesystem-write surface (and the question of where; `content/latex/` is read-only per MC-6, so it'd be a new `data/`-side tree) for no benefit over a TEXT column.
- **The storage form** (coupled to the representation choice): a `questions.test_suite TEXT` column (additive nullable `ALTER TABLE ADD COLUMN` per ADR-022) — vs. a `question_tests` table (additive `CREATE TABLE IF NOT EXISTS`, only if (b) is picked) — vs. a path column (if (c)).
- **Where the test-suite field is validated** — the `response_format=QuestionGenOutput` pydantic validator on the `LLMStep` (a `str` field is non-`None` by construction; a `min_length=1` constraint forces non-empty)? The CS-300 processor's post-parse sanity check (mirroring ADR-037's existing `q.get("prompt", "").strip()` filter)? Both?
- **The bad-test-suite failure handling** (MC-5): a Question whose LLM-supplied test suite is missing / empty / whitespace-only — does the whole Quiz become `generation_failed` (the strictest, simplest)? Or are only the bad Question(s) dropped and the Quiz proceeds with the survivors (if ≥1 survives)? Either way, **a Question without a real test suite must not ship, and the failure must be surfaced, never papered over with a placeholder test ("an empty test file", "a `pass` test", "a `# TODO` comment")** — that is the floor MC-5 / §6 set.
- **The per-Section `ready`-entry caption** (ADR-034's surface): unchanged from ADR-038's state ("Ready" + the "Take this Quiz" link)? Or a small honest caption ("Ready — N Questions, each with tests") that does *not* imply the learner can run the tests yet (the runner is the next slice)?
- **The ADR-036 relationship**: is this extension a **new ADR citing ADR-036** (the cleaner framing if ADR-036's *core* decisions are consumed unchanged and only the output *shape* extends) or a **supersedure ADR of ADR-036's `GeneratedQuestion` shape**?

The manifest constrains the decision: §5 ("No non-coding Question formats. No multiple-choice, no true/false, no short-answer, no describe-the-concept, no interview-style verbal Questions. Every Question is a hands-on coding task" — the test-suite field's type must be such that it can hold **only** test code, never an option list / true-false key / "describe" prompt / recall answer; "No live / synchronous AI results" — this task does no live AI, the existing out-of-band processor (ADR-037) produces the Questions-with-tests on the existing user trigger; "No in-app authoring of lecture content" — the test suite is *generated* and persisted to `data/notes.db`, nothing is written under `content/latex/`; "No multi-user features" — no `user_id` on the new column; "No coverage of CS-300 Chapters 1–2" / "No remote deployment" / "No LMS features" — none crossed), §6 ("Code is written, run, and tested within the application" — this task ships the precondition: the tests must *exist* before the learner can run them; "AI work is asynchronous … AI failures are visible … never fabricates a result" — a Question with a malformed/empty test suite from the LLM is a generation *failure*, not a "ship it with a placeholder"), §7 ("Every Question is a hands-on coding task. The learner writes code that implements a concept … Questions never ask the learner to describe, explain, recall, or choose among options" — a test suite verifies the *implementation*; it does not turn the Question into a non-coding one; "Every Quiz Attempt, Note, and completion mark persists across sessions" — the test suite persists with the Question, which is part of the Bank, "never deleted"), §8 (**Question** — "carries a test suite the learner runs in-app to verify the implementation" — *this is the entry this ADR implements*; **Grade** — "correctness determined by whether the learner's code passed the Question's tests" — the *downstream consumer* that needs the tests to exist; **Quiz** / **Question Bank** — the Bank now holds Questions with test suites).

## Decision

### The test-suite representation — a single source-code string per Question (`test_suite: str`)

A Question's test suite is **a single string of runnable test source code** — a self-contained test module that verifies an implementation of the Question's coding task. The `GeneratedQuestion` schema gains **one field, `test_suite: str`** (the implementer may name it `test_code` / `tests` — the shape commitment is "a single string of test source code, nothing else"); the `questions` table gains **one nullable column, `test_suite TEXT`** (ADR-041 owns the column DDL). The string holds, for a Question that asks the learner to "Implement a hash table with open addressing in C++", something like a C++ test file with `assert`-style cases (or a small test-harness `main()`) exercising the named class/methods the prompt asks for; for a Python-targeted Question, a `unittest`/`pytest` module. The exact test framework/language convention the LLM emits is the LLM's call within the prompt's framing (§The `question_gen` prompt extension below) — the *architectural* commitment is "one runnable test-code string per Question", not a fixed framework.

**Why a single source-code string, not a structured set of test cases (b) or a file-per-Question (c):**

- **It is what the in-app runner (the next slice) needs — runnable, not queryable.** The runner's job is "execute the test suite against the learner's submitted code, report pass/fail". A source-code string is *directly* what an interpreter/compiler+runner consumes; a structured `{input, expected_output}` list would need a CS-300-authored harness per language to *turn into* runnable tests, and "input"/"expected output" don't generalize across DSA concepts (a graph algorithm's input is a graph, not a string; a hash table's "test" is a sequence of operations with invariant checks, not one input→output pair). The runner needs the test suite *runnable*; it does not need it *queryable* (nothing aggregates across test cases). Picking (a) is picking the representation the consuming slice can defend.
- **It mirrors `questions.prompt` being a string** — a Question is `prompt` (what to implement) + `test_suite` (the tests that verify it) + `topics` (tags); all TEXT/string. Consistent, minimal, no new table.
- **It is the additive-cheapest storage** — one nullable `ALTER TABLE ADD COLUMN` (ADR-022's migration story; no migration-trigger), vs. a new `question_tests` table that would also be additive but is more surface (a join, a row-per-case, ADR-033's "n-tables threshold" inching closer) for a representation the runner can't use as well.
- **(c) file-per-Question is rejected** — it re-introduces a filesystem-write surface (where? `content/latex/` is read-only per MC-6, so a new `data/`-side tree) and a path-management problem (orphaned files when a Question is superseded by content reorganization) for zero benefit over a TEXT column — the test suite is small text, it belongs in the row.

The `test_suite: str` field's type **can hold only test code** — a string of source code. It **cannot** hold an option list, a true/false key, a "describe in a sentence" prompt, a recall answer, or any non-coding-Question artifact, by the same logic that ADR-036's `GeneratedQuestion` has no `choices`/`correct_choice`/`answer_text` field and ADR-033's `questions` table has no such column: the schema does not admit it, and the prompt instructs the LLM to put test *source code* there. (Manifest §5 / §7: every Question is a hands-on coding task; the test suite verifies the implementation, it does not make a non-coding Question expressible.)

### The `question_gen` output-schema extension — `GeneratedQuestion` gains `test_suite: str`; `QuestionGenOutput` is unchanged in shape

`GeneratedQuestion` in `app/workflows/question_gen.py` gains a `test_suite: str` field alongside `prompt: str` and `topics: list[str]` — three fields, `model_config = ConfigDict(extra="forbid")` unchanged (so no non-coding field can sneak in). `QuestionGenOutput`'s shape is **unchanged** — its first field is still `questions: list[GeneratedQuestion]` (the framework's `FINAL_STATE_KEY` convention; the *element* type gains a field, the list field is unchanged). The `LLMStep`'s `response_format=QuestionGenOutput` is unchanged in *binding* (still the required validator, KDR-004) — it now validates the new field as a non-`None` `str` (and the implementer adds a `min_length=1` / `pydantic.constr(strip_whitespace=True, min_length=1)` constraint so a literally-empty test suite is rejected at the validator layer too — belt and braces with the processor's sanity check below). **The module's imports stay `ai_workflows.workflows.*` + `ai_workflows.primitives.tiers` + `pydantic` (+ stdlib `os`) only — MC-1; ADR-036's architecture portion stays satisfied (the forbidden-SDK list — `openai`/`anthropic`/`google.generativeai`/`google.genai`/`cohere`/`mistralai`/`groq`/`together`/`replicate`/`litellm`/`langchain*`/`langgraph*`/raw-provider-HTTP — is not touched; AI *invocation* is still the `aiw run` CLI subprocess).**

### The `question_gen` prompt extension — instruct the LLM to emit a runnable test suite per Question, naming the implementation target

`_question_gen_prompt_fn` in `app/workflows/question_gen.py` is extended: the system prompt gains a STRICT REQUIREMENT that **each Question's `test_suite` is runnable test source code that verifies an implementation of that Question's `prompt`**, and that the `prompt` must **name the function/class signature the test suite tests** (so the learner's implementation has a target the tests can call) — e.g. "Implement a function `int* two_sum(const std::vector<int>& nums, int target)` that …" with the `test_suite` calling `two_sum`. How prescriptive the prompt is about the test framework/language is the implementer's call (a sensible framing: "emit a self-contained test file in the same language as the implementation — for C++, `assert`-based cases in a `main()`; for Python, a `unittest.TestCase`"; the exact wording is implementer-followable, not architecture). The user message is unchanged in structure (Section title + Section content). The prompt continues to forbid non-coding Question formats (the existing STRICT REQUIREMENTs 1–6 stay).

### The processor wiring — read each Question's `test_suite` from the `aiw run` artefact and pass it to the persistence call

`app/workflows/process_quiz_requests.py`'s "parse the `aiw run` artefact → persist the Questions" step (`process_pending`, step d) is extended: when it builds the list it passes to `add_questions_to_quiz`, each item now carries `test_suite` (read verbatim from the artefact's `questions[i]["test_suite"]` — not transformed, not regenerated). The `add_questions_to_quiz` payload shape changes accordingly (ADR-041 owns the signature: `questions: list[dict]` where each dict has `"prompt"`, `"topics"`, **and `"test_suite"`**). The processor's lifecycle (`requested` → `generating` → `aiw run` → `ready` / `generation_failed`) is otherwise unchanged (ADR-037).

### The bad-test-suite failure handling (MC-5) — a Question whose LLM-supplied test suite is missing/empty makes the whole Quiz `generation_failed`; no placeholder synthesized

The processor's existing post-parse sanity check (currently: "keep questions with a non-empty `prompt`") is extended to a **whole-Quiz check**: after parsing the artefact, **if any item in `questions` is missing `test_suite`, or its `test_suite` is empty / whitespace-only**, the processor calls `mark_quiz_generation_failed(quiz_id, error=<a descriptive message, e.g. "Generation produced a Question with no test suite (question N: '<prompt prefix>')">)` — `quizzes.status` → `generation_failed`, the detail in `quizzes.generation_error` (ADR-037's column), **zero `questions` rows persisted for that Quiz**, **no placeholder test suite synthesized anywhere** (no "empty test file", no "`pass` test", no "`# TODO`"). The existing "no valid coding-task Questions" check (empty list / all prompts empty → `generation_failed`) stays; the test-suite check is added alongside it (both are "this artefact is not shippable" conditions).

**Why the whole-Quiz check (reject the whole Quiz), not "drop the bad Question(s) and proceed with the survivors":**

- **It is the strictest reading of MC-5 / §6 and the simplest to implement and verify.** §8 now says a Question "carries a test suite" — a Question without one is not a Question this project produces. If the LLM returned a batch where *some* Questions have a test suite and some don't, the honest thing is "this generation didn't produce a clean batch" → `generation_failed`, the author re-clicks Generate (a new `requested` row — MC-9). "Drop the bad ones and ship the rest" introduces a partial-success path (how many is too few? if 5 of 6 have tests, ship 5? if 1 of 6, ship 1?) that the project doesn't need at single-user scale and that muddies the MC-5 line ("we shipped a Quiz, just a smaller one than the LLM tried to make" — vs. the clean "this generation failed, try again").
- **It mirrors ADR-037's existing posture.** ADR-037 already treats "no valid coding-task Questions" (empty list / all prompts empty) as a whole-artefact failure → `generation_failed`, not "ship whatever's valid". The test-suite field is now part of "what makes a generation artefact valid" — extending the same whole-artefact check to it is consistent, not new.
- **`generation_failed` is honest and recoverable.** The Quiz row stays as a historical record (ADR-033's "never deleted"); the `generation_error` detail tells the author *which* Question(s) lacked tests; the author re-clicks "Generate a Quiz" (a new user-triggered `requested` row — MC-9; not an automatic retry — MC-5) and gets a fresh attempt. The per-Section surface shows "Generation failed" (ADR-034's label, unchanged) — never a fabricated "Ready".
- **The validator backstop.** The `min_length=1` constraint on `GeneratedQuestion.test_suite` means the framework's `response_format` validation *also* rejects a literally-empty test suite *before* the artefact ever reaches the processor (a validation failure inside the workflow run → the `aiw run` subprocess exits non-zero with an `error:` on stderr → ADR-037's existing non-zero-exit path → `generation_failed`). The processor's whitespace-only check catches the "the LLM emitted `   \n  `" case the `min_length=1` constraint alone would let through. Both layers, same outcome: a Question without a real test suite never ships, and the failure is surfaced.

(`/design`'s call, recorded: the whole-Quiz reject. If a future task finds the strictness too coarse — the author tires of a whole batch failing because one Question's test suite was thin — a future ADR can relax it to "drop-and-proceed-if-≥N-survive", with the partial-success semantics spelled out then; this ADR's floor — "a Question without a real test suite does not ship; the failure is surfaced, never papered over" — is preserved under any relaxation.)

## Alternatives considered

**A. The test-suite representation as a structured set of test cases (`GeneratedQuestion.test_cases: list[TestCase]`; a `question_tests` table — a row per case).**
Considered (the task named it as `/design`'s call). Rejected: the in-app runner (the consuming next slice) needs the test suite *runnable*, not *queryable* — a structured `{name, input, expected_output}` list would need a CS-300-authored harness *per language* to turn into runnable tests, and "input"/"expected output" don't generalize across DSA concepts (a graph algorithm's input is a graph; a hash table's "test" is a sequence of operations with invariant checks, not one input→output pair). A source-code string is *directly* what an interpreter/compiler+runner consumes. The structured form is more queryable, but nothing aggregates across test cases (the Grade aggregates per-*Question* correctness, not per-test-case), so the queryability buys nothing; meanwhile a `question_tests` table is more surface (a join, a row-per-case, ADR-033's "n-tables threshold" closer) for a representation the runner can't use as well. (If a future runner slice's `/design` decides it genuinely wants structured cases — e.g. to show per-case pass/fail in the results panel — it can supersede this ADR and migrate `questions.test_suite` (a source string) to a `question_tests` table (parsed cases); the migration is additive (`CREATE TABLE IF NOT EXISTS`); the cost is bounded. But the runner doesn't need it, so don't pay it now.)

**B. A file-per-Question on disk (the workflow writes a `.py`/`.cpp` test file; the `questions` row stores a path).**
Rejected. Re-introduces a filesystem-write surface (where? `content/latex/` is read-only per MC-6, so a new `data/`-side tree) and a path-management problem (orphaned files when a Question is superseded by content reorganization — ADR-033's `questions` are "never deleted; only superseded", which a file tree doesn't track) for zero benefit over a TEXT column — the test suite is small text; it belongs in the row, alongside `prompt` and `topics`. (And it would complicate the in-app runner's job — it'd have to read from two places, the row for `prompt` and a file for `test_suite`.)

**C. Validate the test-suite field only at the processor (no `min_length=1` constraint on the pydantic model), or only at the pydantic model (no processor whitespace check).**
Rejected — both layers, belt and braces: the `min_length=1` constraint on `GeneratedQuestion.test_suite` makes the framework's `response_format` validation reject a *literally-empty* test suite *inside the workflow run* (so the `aiw run` subprocess fails fast with a clear `error:` rather than emitting a useless artefact the processor then has to reject); the processor's `.strip()` check catches the "the LLM emitted `   \n  `" / "the artefact's JSON is malformed enough that `test_suite` is missing entirely" cases (the artefact comes from stdout, which the framework controls but which CS-300 should still validate at its own boundary, mirroring ADR-037's existing `q.get("prompt", "").strip()` defensive filter). Same outcome under either failure (`generation_failed`, zero rows, no placeholder); two cheap layers are worth it for a load-bearing MC-5 invariant.

**D. Drop the bad Question(s) and ship the survivors (partial-success), instead of failing the whole Quiz.**
Considered (the task left the whole-Quiz-vs-bad-Question call to `/design`, with the floor "a Quiz that would otherwise ship a Question without a real test suite does not ship that Question, and the failure is surfaced"). Rejected for *this* slice: a partial-success path introduces a "how few is too few?" question (5 of 6 → ship 5? 1 of 6 → ship 1?) the project doesn't need at single-user scale, and it muddies the MC-5 line ("we shipped a Quiz, just smaller" vs. the clean "this generation failed, try again"). The whole-Quiz reject mirrors ADR-037's existing "no valid coding-task Questions → `generation_failed`, not ship whatever's valid" posture, is the simplest to implement and verify, and is recoverable (the author re-clicks Generate — a new `requested` row, MC-9). The floor is preserved (no Question without a real test suite ships; the failure is surfaced); a future ADR can relax to drop-and-proceed if the strictness proves coarse.

**E. Add a "Ready — N Questions, each with tests" caption to the per-Section `ready` entry this slice.**
Considered (the task named it as `/design`'s call — "either is fine"). Rejected: a caption that mentions tests *before* the in-app runner exists risks implying the learner can *run* them (they can't this slice), and it doesn't help the learner *do* anything (the structural fact "the Questions carry tests" is invisible-to-the-learner-useful only once there's a "Run tests" affordance). The honest move is to leave ADR-034/ADR-038's surface alone and let the *runner slice* decide the caption (when the test suite gets a learner-facing role). Keeps this task's UI footprint at zero — a non-UI task (workflow-schema + persistence + processor wiring).

**F. A supersedure ADR of ADR-036's `GeneratedQuestion` shape, rather than a new ADR citing ADR-036.**
Rejected. ADR-036's *core* decisions (the PyPI dependency, the module path, the authoring surface, the `aiw run` invocation path, the forbidden-SDK list, the test seam, the cohabitation option) are all consumed unchanged — only the output *shape* extends, additively, with one field, which is the kind of evolution ADR-036's declarative-`WorkflowSpec` design anticipated. A supersedure ADR would imply ADR-036 was *wrong*; it wasn't — it's being *extended*. ADR-036's "the shape commitment is 'a `prompt: str` and a `topics: list[str]`, nothing else'" clause is read in context (ADR-036 §The output schema's shape commitment) as a *no-non-coding-field* commitment ("no `option_*` / `correct_choice` / `answer_text` / `describe_*` / `recall_*` / `choices` field anywhere"), not a *never-add-any-field* commitment; `test_suite: str` (test *code*) does not make a non-coding Question expressible (§The test-suite representation). A new ADR citing ADR-036 is the right framing — the task forecast it, and it keeps ADR-036's record intact.

**G. A schema-only / `question_gen`-output-only change with no processor wiring or persistence.**
Rejected as a non-slice (the task flagged it): changing the workflow's `output_schema` without wiring the processor to persist the new field means the field is generated and discarded — nothing carries forward; the manifest's "the Question carries a test suite" isn't true (the Bank doesn't hold it). The slice is "the Question *carries* a test suite" — generated, persisted, round-trips — not "the workflow *emits* a test suite into the void". This ADR (with ADR-041) ships the whole vertical: schema → prompt → processor → storage → round-trip.

- The take surface or the `attempt_questions` schema being changed this task (the test suite lives on the `questions` row, not on `attempt_questions`; ADR-038/ADR-039 consumed unchanged; the `attempt_questions` test-result columns are the runner slice).

**Future surfaces this ADR pre-positions:**

- The in-app code-execution / test-runner slice — runs the Question's `test_suite` against the learner's submitted `response` code, in-app, reports pass/fail; with its own substantial sandboxing/isolation ADR; adds the `attempt_questions` test-result columns; adds a "Run tests" affordance to the take surface; decides whether the take surface displays the test suite read-only; decides the per-Section `ready`-entry caption (when the test suite gets a learner-facing role). Tracked in `design_docs/project_issues/in-app-test-runner-slice-shape.md` (filed by this `/design` cycle).
- The grading slice — now keyed off test results (the amended §8): the out-of-band grading processor reads the in-app test results (not a free AI judgment), the AI's job is the per-Question *explanation* + the Grade aggregate (score / Weak Topics / recommended Sections), not the *correctness* call (which the runner already computed). Tracked in `design_docs/project_issues/quiz-grading-slice-shape.md` (which this task does **not** edit — the grading slice's own `/design` updates it to reflect this reordering; the architect flags it but does not action it).
- The relational Topic vocabulary — `topics` table + `question_topics` join, backfilled from `questions.topics` — the Weak-Topic-identification slice; unchanged by this task.
- The replay-+-fresh composition for post-first Quizzes — replays Bank Questions (now including their test suites) + fresh Weak-Topic-targeted Questions; a later slice; the first-Quiz-only guard (ADR-037) is unchanged this task.

**Supersedure path if this proves wrong:**

- If the source-code-string representation proves the wrong shape (the runner slice's `/design` wants structured test cases — e.g. to show per-case pass/fail in the results panel) → a future ADR supersedes this one and migrates `questions.test_suite` (a source string) to a `question_tests` table (parsed cases); the migration is additive (`CREATE TABLE IF NOT EXISTS`); the cost is bounded. (But the runner doesn't need it, so don't pay it now.)
- If the whole-Quiz bad-test-suite reject proves too coarse (the author tires of a whole batch failing because one Question's test suite was thin) → a future ADR relaxes it to "drop-and-proceed-if-≥N-survive", with the partial-success semantics spelled out then; the floor ("a Question without a real test suite does not ship; the failure is surfaced") is preserved under any relaxation.
- If the per-Section surface needs a caption before the runner slice (the human wants "Ready — N Questions, each with tests" now) → that's a one-line `lecture.html.j2` change reusing the `section-quiz-*` namespace (ADR-008); bounded; this ADR records "unchanged" as the call, not a hard constraint.
- If `test_suite: str` should hold metadata alongside the code (a declared language, a declared test framework, a declared entry-point name — so the runner doesn't have to infer them) → a future ADR adds those as additional `GeneratedQuestion` / `questions` fields (additive — more nullable columns / pydantic fields); the `test_suite` string itself stays the runnable code. Bounded.

The supersedure path, in every case, runs through a new ADR. This ADR does not edit any prior ADR in place; it builds on ADR-036 (the integration shape — consumed unchanged, only the output shape extends additively), consumes ADR-033 (the `questions` schema — the new column is additive), ADR-037 (the generation processor — gains the test-suite payload + the bad-test-suite check; the lifecycle/failure-state/detail-column are unchanged), ADR-034 (the per-Section surface — unchanged), ADR-038 (the take surface — unchanged), ADR-039 (the Attempt-lifecycle persistence — unchanged), and ADR-022 (the persistence boundary / additive-migration story — preserved); the persistence mechanics are in the companion ADR-041.
