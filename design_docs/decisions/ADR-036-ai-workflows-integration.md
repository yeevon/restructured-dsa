## Context

TASK-014 ships slice 2 of the Quiz pillar: the *processor* — the `ai-workflows` integration that walks a `requested`-status Quiz (created by ADR-034's `POST .../quiz` route) through `generating` to `ready`, generating a fresh set of hands-on coding-task Questions for the Section, asynchronously. Manifest §4 names `ai-workflows` as "the only architectural commitment made at the manifest level; everything about *how* that integration is shaped lives in the architecture document" — this ADR is that shaping. It activates MC-1's currently-dormant architecture portion ("the specific list of forbidden SDK package names and the workflow module path come from the relevant Accepted ADRs").

What `ai-workflows` is (read from the co-located clone at `/home/papa-jochy/prj/ai-workflows` — Tier-4 reference; it documents a *different* repo's architecture, not CS-300's — and confirmed against the editable install in the dev `.venv`):

- The package is published on PyPI: distribution name **`jmdl-ai-workflows`** (`pip install jmdl-ai-workflows`), import package **`ai_workflows`**, current version **0.4.0**; `[project.scripts]` ships two console scripts — **`aiw`** (the CLI) and **`aiw-mcp`** (a FastMCP server). Its `pyproject.toml` requires Python `>= 3.12`.
- It is a heavy framework — runtime dependencies include `langgraph`, `langgraph-checkpoint-sqlite`, `litellm`, `fastmcp`, `pydantic`, `python-dotenv`, `pyyaml`, `structlog`, `typer`, `yoyo-migrations`. Vendoring it (copying the source into this repo) is not viable.
- It has a four-layer internal architecture (`primitives → graph → workflows → surfaces`, enforced by `import-linter`). Workflows are the top of the *workflow* stack; the two **surfaces** are the `aiw` CLI and the `aiw-mcp` MCP server — both thin wrappers over the workflow registry, both routing through the internal `ai_workflows.workflows._dispatch` core (a `_`-prefixed module — the framework's *shared internal*, not a documented downstream API; its `_dispatch.py` even carries a `FINAL_STATE_KEY = "questions" from CS-300` comment, i.e. the framework author co-develops both repos and has already accommodated this exact integration *internally* — but that comment lives in a private module and is not part of the framework's semantic-version contract; KDR-008 names the *MCP tool schemas* as the public v0.1.0 contract, and the `aiw` CLI's stdout/stderr is the documented scripted-use surface).
- It has a **declarative authoring surface** (M19 — "Declarative authoring surface"): a workflow is a Python module that calls `register_workflow(WorkflowSpec(...))` at import time; `WorkflowSpec` requires five fields (`name`, `input_schema`, `output_schema`, `steps`, `tiers`) and is a pure pydantic data object; the framework compiles the spec to a LangGraph `StateGraph` at registration time. **The author never `import langgraph`.** The five built-in step types are `LLMStep`, `ValidateStep`, `GateStep`, `TransformStep`, `FanOutStep`. `LLMStep` requires a `response_format` (a pydantic model) — this is how the framework enforces validator pairing by construction (KDR-004 — an unvalidated LLM step is unexpressible). `LLMStep` takes exactly one of `prompt_template` (Tier-1 sugar — a `str.format()` template; the framework's own docs flag a brace-escaping hazard when end-user-controlled field values may contain `{`-style syntax) or `prompt_fn` (Tier-2 — a callable `(state: dict) -> (system: str | None, messages: list[dict])` that builds the prompt explicitly, recommended exactly when the input may contain format syntax). The first field of `output_schema` becomes the framework's `FINAL_STATE_KEY` and is surfaced as the run's terminal artefact (`result.artifact`). The conventional tier-registry helper is `<workflow_name>_tier_registry()` — a module-level function returning a `dict[str, TierConfig]` whose name prefix must literally match the workflow name.
- It has an **external-module discovery mechanism** designed for exactly this case: a downstream consumer registers its own workflow modules via the **`AIW_EXTRA_WORKFLOW_MODULES`** environment variable (comma-separated dotted module paths) or the **`--workflow-module`** CLI flag (repeatable, on both `aiw` and `aiw-mcp`). Each entry is `importlib.import_module(...)`'d at startup; the module's top-level `register_workflow(...)` call fires as the import side effect. The module must already be importable via the running interpreter's `sys.path` (the typical layout is a pip-installable package, editable or otherwise). The framework's `docs/writing-a-workflow.md` §"External workflows from a downstream consumer" and §"Worked example — the CS-300 shape" walk through `AIW_EXTRA_WORKFLOW_MODULES=cs300.workflows.question_gen aiw run question_gen --input chapter_text=... --input num_questions=...` — i.e. the framework's own docs model this project as a consumer, using a `question_gen` workflow as the worked example.
- The **`aiw run` CLI's stdout contract** (read from `ai_workflows/cli.py`): on a completed run the CLI prints the workflow's terminal artefact as `json.dumps(artifact, indent=2)` on stdout, followed by a `total cost: $X.XXXX` line; on a `HumanGate` pause it prints the run id + an `awaiting: gate` line + a resume hint; on failure it prints `error: <message>` on **stderr** and exits non-zero. Structured logs go to **stderr** (`configure_logging(level="INFO")` in the CLI), so **stdout stays the machine-parseable surface**. `aiw run <workflow> --input KEY=VALUE` accepts repeatable `--input` pairs (pydantic v2 coerces string values to the declared types); `--run-id` is optional (auto-generates a ULID if omitted). The `aiw`/`aiw-mcp` console scripts auto-load a cwd-local `.env` at startup (shell-exported values win).
- It owns its **own persistence**: a `SQLiteStorage` run registry at `~/.ai-workflows/storage.sqlite3` and LangGraph `SqliteSaver` checkpoints at `~/.ai-workflows/checkpoint.sqlite3` (both env-overridable via `AIW_STORAGE_DB` / `AIW_CHECKPOINT_DB`). It does not need or want tables in CS-300's `data/notes.db`.
- It has a **hermetic test seam**: `StubLLMAdapter` (`ai_workflows.evals._stub_adapter`) — an in-process LLM adapter that returns a pre-armed canned response, monkeypatched in via `monkeypatch.setattr(ai_workflows.graph.tiered_node, "LiteLLMAdapter", StubLLMAdapter)` — so the *real* LangGraph state machine / validators / retry edges run end-to-end with no token spent. There is also `compile_step_in_isolation` (`ai_workflows.workflows.testing`) for unit-testing custom steps. **This seam only works when the workflow is invoked in-process** — a monkeypatch in the test process cannot reach a subprocess.
- Provider routing: tiers route through `LiteLLMRoute` (e.g. `LiteLLMRoute(model="gemini/gemini-2.5-flash")`, which needs `GEMINI_API_KEY` at runtime) or `ClaudeCodeRoute` (which routes to the `claude` CLI via OAuth — no `ANTHROPIC_API_KEY`, no `anthropic` SDK; the `claude` CLI must be installed and logged in). Real-provider tests are gated on `AIW_E2E=1` in the framework's own suite.

The framework's docs example uses `cs300.workflows.question_gen`, but the example's package name is illustrative; this project's package is `app`, so `app.workflows.question_gen` is the analogous path.

In no case does a CS-300 test make a real LLM/provider call (MC-1's spirit).
- **The forbidden-SDK list (MC-1's architecture portion):** enumerate every LLM/agent SDK + raw-provider-HTTP path that application code must not import — `openai`, `anthropic`, `google.generativeai` / `google.genai`, `cohere`, `mistralai`, `groq`, `together`, `replicate` (or any other LLM-provider SDK), `litellm` (the unified adapter `ai-workflows` wraps), `langchain*` / `langgraph*` (the orchestration layer it wraps — CS-300 authors a `WorkflowSpec`, never a `StateGraph`), plus a raw `httpx` / `requests` / `urllib` call to a provider's chat-completions / messages / generate endpoint. The list goes into the conformance skill's MC-1 entry — the **human's edit** (Tier-2 file).

The manifest constrains the decision through §4 (the `ai-workflows` commitment; the secondary objective being "ranked below primary" and "never justifies adding a feature" — this integration is *driven by* the generation feature, the §7 loop, and `ai-workflows` is the engine it uses), §5 ("No live / synchronous AI results … surfaced via Notification — not in real time" — the workflow runs out-of-band, ADR-037's mechanism; "no remote deployment / no hosted product" — no MCP server, no job-queue broker), §6 ("AI work is asynchronous from the learner's perspective"; "AI failures are visible … never fabricates a result" — a non-zero `aiw run` exit / `error:` on stderr → `quizzes.status = 'generation_failed'`, no fabricated Questions; "Lecture source read-only" — the workflow *reads* the Section's `.tex` content as prompt material, never writes it; "Single-user" — no `user_id` anywhere), §7 ("Every Question is a hands-on coding task" — the workflow's prompt constrains the output to coding tasks; "The first Quiz for a Section contains only fresh Questions" — this task ships exactly the first-Quiz case; "Quiz generation is always explicitly user-triggered" — the processor processes existing user-created `requested` rows, never creates one), §8 (Quiz, Question, Question Bank — consumed; the generated Questions go into the Bank, never deleted).

## Decision

### The project depends on the `jmdl-ai-workflows` PyPI package — a version pin in `pyproject.toml` `[project] dependencies`

The application takes a runtime dependency on the published package **`jmdl-ai-workflows`** (import package `ai_workflows`), declared in `pyproject.toml` `[project] dependencies` with a version pin of at least **`>= 0.4`** (the version with the M19 declarative-authoring surface and the M16 `AIW_EXTRA_WORKFLOW_MODULES` discovery mechanism this ADR's integration uses; the implementer picks the exact spec — `"jmdl-ai-workflows>=0.4"` is fine; a tighter `>=0.4,<1` or `~=0.4` is also acceptable). `pyproject.toml`'s `requires-python` moves from `>= 3.11` to **`>= 3.12`** to match the framework's floor (a no-op for the dev environment, which is Python 3.13). **These two `pyproject.toml` edits are the implementer's, made as part of TASK-014's implementation phase** — flagged here, not made by the architect.

The co-located clone at `/home/papa-jochy/prj/ai-workflows` is **read-only reference only** — it is NOT a path/editable dependency and must not be wired in as one (no `pip install -e ../ai-workflows`, no PEP 508 path requirement in `pyproject.toml`). It exists so the integration author can read the framework's docs and source while writing `app/workflows/question_gen.py` and the processor; CS-300's *dependency* is the PyPI package.

**Vendoring is rejected outright** — `jmdl-ai-workflows` is a ~10-runtime-dependency framework (`langgraph`, `litellm`, `fastmcp`, …); copying its source into this repo is unmaintainable and defeats the point (the manifest wants the project to *consume* `ai-workflows`, not fork it).

### Where the CS-300 workflow module lives — `app/workflows/`, a new package under `app/`

The CS-300-owned workflow module lives at **`app/workflows/question_gen.py`** (a new `app/workflows/` package — `app/workflows/__init__.py` plus the workflow module; ADR-037 adds the out-of-band processor module to the same package). Rationale: it is application code (it imports `ai_workflows.workflows.*` and `pydantic`, declares a `WorkflowSpec`, calls `register_workflow` at import time), so it lives under `app/` like everything else (parser, discovery, designation, persistence, templates). The framework's docs example uses `cs300.workflows.question_gen` — but the example's top-level package name is illustrative; this project's package is `app`, so `app.workflows.question_gen` is the analogous path, and introducing a second top-level package (`cs300/` or a bare `workflows/`) just to match an illustrative example would be gratuitous.

The module's shape (the implementer follows the framework's `docs/writing-a-workflow.md` §"Minimum module shape" + §"Worked example — the CS-300 shape", adapted for CS-300's needs):

- **Two (or three) pydantic schemas:**
  - `QuestionGenInput` — a `section_content: str` field (the Section's parsed LaTeX text — the prompt material) and any framing fields the prompt needs (e.g. `section_title: str`). **No `num_questions` controlled by the learner this slice** — ADR-034's "Generate a Quiz" affordance is a single submit button with no form fields, and the manifest does not require a learner-tunable count; the workflow's prompt picks a sensible count (a fixed framing constant in the prompt text, or the `prompt_fn` decides). (The framework's worked example has `num_questions: int`; CS-300 legitimately deviates — it would change ADR-034's surface to expose it, which is out of scope.)
  - `QuestionGenOutput` — its **first field** is the workflow's terminal artefact per the framework's `FINAL_STATE_KEY` convention: **`questions: list[GeneratedQuestion]`** where `GeneratedQuestion` is a nested pydantic model with **`prompt: str`** (the coding-task instruction) and **`topics: list[str]`** (Topic tags). This is a deliberate, endorsed deviation from the framework example's `questions: list[str]`: structured items let `add_questions_to_quiz` fill ADR-033's `questions.prompt` and `questions.topics` columns from the LLM's structured output rather than from a bare string. **No `option_*` / `correct_choice` / `answer_text` / `describe_*` / `recall_*` / `choices` field anywhere in `QuestionGenOutput` or `GeneratedQuestion`** — the schema makes a non-coding Question impossible to express, mirroring how ADR-033's `questions` table has no such column (manifest §5 / §7); the `LLMStep`'s prompt *also* constrains the LLM to hands-on coding tasks about the Section's content. (`GeneratedQuestion` may be named differently — `QuestionSpec`, etc. — implementer's call; the shape commitment is "a `prompt: str` and a `topics: list[str]`, nothing else".)
- A `question_gen_tier_registry() -> dict[str, TierConfig]` helper at module level (the framework's convention — the `question_gen_` prefix must literally match the workflow name `question_gen`; for a spec-API workflow `WorkflowSpec.tiers` is the authoritative registry and the framework's compiler synthesises a `<spec.name>_tier_registry` accessor from it, but following the helper-name convention keeps the module readable and matches the docs). One LLM tier; the route is `LiteLLMRoute(model="gemini/gemini-2.5-flash")` (the framework's worked-example route — needs `GEMINI_API_KEY` at runtime) or `ClaudeCodeRoute(...)` (OAuth via the `claude` CLI — no `anthropic` SDK, no `ANTHROPIC_API_KEY`); the *specific* provider/model is a tuning decision inside the tier registry, not an architectural commitment (the framework's tier-override mechanism makes it changeable without re-decision; the implementer picks one — `gemini/gemini-2.5-flash` is the documented default and a sensible pick). The architectural commitment is "one LLM tier, routed through `ai-workflows`'s tier machinery, never a raw SDK call". An optional `fallback` chain on the `TierConfig` (e.g. a second `LiteLLMRoute` or a `ClaudeCodeRoute` last-resort) is the implementer's discretion.
- A `register_workflow(WorkflowSpec(name="question_gen", input_schema=QuestionGenInput, output_schema=QuestionGenOutput, tiers=question_gen_tier_registry(), steps=[...]))` call at module top level (fires at import time when `AIW_EXTRA_WORKFLOW_MODULES` imports the module, or when the processor imports it directly). The `steps` list is:
  - one `LLMStep(tier="<the tier name>", prompt_fn=<a callable>, response_format=QuestionGenOutput, retry=RetryPolicy(...) optionally)` — **`prompt_fn`, not `prompt_template`**, because the Section's content is LaTeX and *will* contain `{`-style characters (`\frac{a}{b}`, `\texttt{...}`, etc.), which the framework's docs flag as the exact hazard that calls for `prompt_fn` (the callable receives raw state and builds the prompt explicitly, so escaping is fully under CS-300's control). The `prompt_fn` builds a system prompt that constrains the output to hands-on coding tasks about the Section's content — never describe/recall/choose items — and a user message carrying the Section's content; `response_format=QuestionGenOutput` is required (KDR-004 — an unvalidated `LLMStep` cannot be expressed) and is how the framework pairs a validator with the call;
  - optionally a paired `ValidateStep(target_field="questions", schema=QuestionGenOutput)` — illustrative; a runtime no-op when its schema matches the upstream `LLMStep.response_format`, but harmless and consistent with the framework's worked example (implementer's discretion to include it or not).
- **No `import langgraph`, no `import openai` / `import anthropic` / `import litellm` / `import google.generativeai` / etc.** — the module imports only `ai_workflows.workflows.*` (the spec API: `WorkflowSpec`, `LLMStep`, `ValidateStep`, `RetryPolicy`, `register_workflow`), `ai_workflows.primitives.tiers` (`TierConfig` + a route type — `LiteLLMRoute` / `ClaudeCodeRoute`), and `pydantic`. (MC-1.)

### How application code invokes the workflow — the out-of-band processor shells out to `aiw run question_gen` (a subprocess), reading the terminal artefact from stdout

ADR-037 decided the out-of-band processor is a **manual command** — `python -m app.workflows.process_quiz_requests`, a `__main__` module under `app/workflows/`, run by the author after clicking "Generate a Quiz" (or periodically). This ADR decides *how that processor invokes the workflow*: it **shells out to the `aiw` CLI as a subprocess** — the framework's *documented* downstream surface — not by importing the framework's internal `_dispatch.run_workflow`.

For each `requested` Quiz the processor needs to process (ADR-037 owns the loop), it runs roughly:

```python
# in app/workflows/processor.py (or wherever ADR-037's processor module lives) —
# this ADR owns the "how do we call the workflow" seam; ADR-037 owns the loop.
import json
import os
import subprocess
import sys

def _invoke_question_gen(*, section_content: str, section_title: str, run_id: str) -> dict:
    """Run the question_gen workflow via the documented `aiw run` CLI subprocess.

    Returns the workflow's terminal artefact dict — {"questions": [...]} — parsed
    from stdout (the CLI prints `json.dumps(artifact, indent=2)` on a completed run,
    followed by a `total cost: $X.XXXX` line; structured logs go to stderr).

    Raises QuestionGenFailed (a CS-300-defined exception) on a non-zero exit /
    an `error: ...` line on stderr — the processor maps that to mark_quiz_generation_failed
    (ADR-037).  No fabricated artefact on failure.
    """
    env = {**os.environ, "AIW_EXTRA_WORKFLOW_MODULES": "app.workflows.question_gen"}
    # `aiw run ...` — or, PATH-independent: [sys.executable, "-m", "ai_workflows.cli", "run", ...]
    proc = subprocess.run(
        [
            "aiw", "run", "question_gen",
            "--input", f"section_content={section_content}",
            "--input", f"section_title={section_title}",
            "--run-id", run_id,
            # "--budget", "<a CS-300 config constant>",  # optional
        ],
        env=env,
        capture_output=True,
        text=True,
    )
    if proc.returncode != 0:
        # The CLI prints `error: <message>` on stderr and exits non-zero on failure.
        raise QuestionGenFailed(proc.stderr.strip() or f"aiw run exited {proc.returncode}")
    # stdout for a completed run: the indented artefact JSON, then a `total cost: $X.XXXX` line.
    artefact = _parse_artefact_from_stdout(proc.stdout)   # strip the trailing `total cost:` line, json.loads the rest
    return artefact   # {"questions": [{"prompt": "...", "topics": [...]}, ...]}
```

Key points:

- **`AIW_EXTRA_WORKFLOW_MODULES=app.workflows.question_gen` in the subprocess env** is the framework's documented downstream-discovery mechanism — `aiw` imports the module at startup, its top-level `register_workflow(...)` fires, and `question_gen` is in the registry before `aiw run question_gen` resolves it. (The `--workflow-module app.workflows.question_gen` CLI flag is an equivalent alternative; the env var is the one the framework's CS-300 worked example uses.)
- **`app/workflows/question_gen.py` must be importable from the subprocess's `sys.path`.** It is — the processor runs in CS-300's virtualenv (`python -m app.workflows.process_quiz_requests`), `app` is on `sys.path`, and the subprocess inherits the env; `jmdl-ai-workflows` being a normal venv dependency means the `aiw` console script is in the same venv's `bin/`. (If `aiw` is somehow not on the subprocess's `PATH`, the PATH-independent form `[sys.executable, "-m", "ai_workflows.cli", "run", ...]` avoids the `PATH` dependency entirely — implementer's call.)
- **The artefact is read from stdout** — `json.dumps(artifact, indent=2)` followed by a `total cost: $X.XXXX` line; the processor strips the cost trailer and `json.loads` the rest. For `question_gen` the artefact is `{"questions": [...]}` (the first field of `QuestionGenOutput` is `questions`, so it is the `FINAL_STATE_KEY` / `result.artifact`). The processor validates each item has a non-empty `prompt`; an empty `questions` list (or stdout that doesn't parse as the expected shape) is treated as a failure (no Questions persisted — MC-5).
- **Failure is a non-zero exit / `error: ...` on stderr** — the framework's `RetryingEdge` does *bounded* retry *inside* the workflow run (the subprocess); if the run still fails, `aiw run` exits non-zero and prints `error: <descriptive message>` on stderr; the processor maps that to `mark_quiz_generation_failed(quiz_id, error=...)` (ADR-037). The processor does **not** wrap the subprocess call in its own retry loop, and does **not** re-process `generation_failed` rows (MC-5; a re-attempt is a new user-triggered `requested` row, MC-9 — ADR-037).
- **`--run-id`** — the processor passes a run id derived from the CS-300 `quiz_id` (e.g. `quiz-{quiz_id}` or a hash) so the `~/.ai-workflows/storage.sqlite3` run-registry row is traceable to the CS-300 Quiz (`aiw list-runs` shows it). Optional but recommended; ADR-037 owns the exact form.
- **Provider key at runtime** — the subprocess needs whatever the tier registry's route requires (`GEMINI_API_KEY` for `LiteLLMRoute(model="gemini/...")`, or a logged-in `claude` CLI for `ClaudeCodeRoute`). The processor's environment supplies it (an exported var, or a `.env` in the processor's cwd — `aiw` auto-loads cwd `.env`). This is a runtime/config concern, not architecture; ADR-037 notes it as a consequence. Tests need no key — the test seam (below) mocks the subprocess.

**Rejected invocation alternatives:** (i) importing `ai_workflows.workflows._dispatch.run_workflow` and calling it in-process — *rejected*: `_dispatch` is a `_`-prefixed module, the framework's *internal* shared core (the same core the `aiw` CLI and `aiw-mcp` server both wrap), not part of the documented downstream API; its return-dict shape and module path are not under the framework's semver contract (KDR-008 names the *MCP tool schemas* as the public v0.1.0 contract); the `FINAL_STATE_KEY = "questions" from CS-300` comment in `_dispatch.py` shows the framework author has accommodated this integration *internally*, but a comment in a private module is not a stable API surface, and CS-300 building on it would couple the project to framework internals that can churn. The `aiw run` CLI's stdout/stderr is the *documented* scripted-use surface (`docs/writing-a-workflow.md` §"Running your workflow" + §"External workflows from a downstream consumer"); consuming that is the honest demonstration the manifest's secondary objective wants. The trade-off — a subprocess spawn per Quiz generation, and stdout parsing — is real but tiny: this is a single-user dev command processing a small table, and the parsing is "strip one trailing line, `json.loads` the rest", which the framework's stdout contract makes deterministic. (iii) calling the `aiw-mcp` HTTP/stdio server — *rejected*: the framework's docs model `aiw-mcp` as the surface a *separate frontend* (an Astro/React/Vue app) talks to over HTTP; for a same-process consumer that already has `ai_workflows` and the `aiw` script in its venv, standing up an MCP server is the heavyweight-job-queue anti-pattern the task explicitly flagged, and the manifest's "no remote deployment / no hosted product" (§5) cuts against running an extra server. The `aiw-mcp` path is a future option only if CS-300 ever splits its frontend from its workflow runner.

### The orchestration boundary — the workflow returns data; CS-300 code persists it via `app.persistence.*`

The `ai-workflows` workflow produces *data* — a `QuestionGenOutput` artefact (`{"questions": [{"prompt": "...", "topics": [...]}, ...]}`), read from `aiw run`'s stdout. It does **not** touch CS-300's database (MC-10 — the workflow module imports no `sqlite3`, opens no connection, and never receives one; the `aiw` subprocess writes only to its own `~/.ai-workflows/*.sqlite3` stores). The CS-300 processor (ADR-037) adapts the returned data and persists it via the typed public functions in `app/persistence/quizzes.py`:

- New persistence functions in `app/persistence/quizzes.py` (re-exported by `app/persistence/__init__.py`; ADR-033 already forecast `add_questions_to_quiz(quiz_id, questions)` and the transition functions):
  - `mark_quiz_generating(quiz_id) -> None` — `UPDATE quizzes SET status='generating' WHERE quiz_id=?`.
  - `mark_quiz_ready(quiz_id) -> None` — `UPDATE quizzes SET status='ready' WHERE quiz_id=?`.
  - `mark_quiz_generation_failed(quiz_id) -> None` (and, per ADR-037, `mark_quiz_generation_failed(quiz_id, error: str | None = None)` writing a nullable `quizzes.generation_error TEXT` column — additive `ALTER TABLE ADD COLUMN` per ADR-022's migration story; ADR-037 owns that call).
  - `add_questions_to_quiz(quiz_id, questions) -> None` — for each generated question: `INSERT INTO questions (section_id, prompt, topics, created_at) VALUES (...)` with `section_id` taken from the Quiz's `section_id` (so the Question lands in the *Section's* Bank — MC-2: exactly one Section) and `topics` the `'|'`-joined Topic-tag list (the cheap delimited form ADR-033 ships — see §Topic tags below); then `INSERT INTO quiz_questions (quiz_id, question_id, position) VALUES (?, ?, ?)` with 1-based `position`. All within one transaction.
  - `list_requested_quizzes() -> list[Quiz]` (or similar — ADR-037 owns the processor's "find work to do" query shape; named here as a forecast).
  - `get_quiz(quiz_id) -> Quiz | None` (a single-row read the processor needs to know a Quiz's `section_id` and current `status`).
- The processor reads the Section's LaTeX content read-only (`content/latex/{chapter_id}.tex` — the `chapter_id` is the prefix of the Quiz's `section_id` per ADR-002 — via the existing `app.parser` path; the Section's text is the prompt material — MC-6: read-only, identical to what the route handlers already do), composes the `--input section_content=...` / `--input section_title=...` args, calls `_invoke_question_gen(...)`, and:
  - on success and the artefact non-empty (≥1 item with a non-empty `prompt`): `add_questions_to_quiz(quiz_id, ...)` then `mark_quiz_ready(quiz_id)`.
  - on `QuestionGenFailed` (non-zero `aiw run` exit / `error:` on stderr) — or an empty/invalid artefact / a `questions` list that fails CS-300's own sanity check that ≥1 coding-task Question came back: `mark_quiz_generation_failed(quiz_id[, error])`. **No fabricated Questions; no silent unbounded retry** (the framework's `RetryingEdge` does *bounded* retry inside the subprocess's workflow run; the CS-300 processor does not wrap that in an infinite loop — MC-5). The learner-facing signal is the honest "Generation failed" the per-Section surface already renders (ADR-034).

### `ai-workflows` persistence cohabitation — option (c): the framework owns its own run/checkpoint store

ADR-022 §`ai-workflows` cohabitation offered three options for where the integration's persistence lives: (a) tables in CS-300's `data/notes.db`; (b) a separate `ai-workflows`-owned store; (c) "consume `ai-workflows`'s own persistence configuration if the framework provides one". The framework *does* provide one — a `SQLiteStorage` run registry at `~/.ai-workflows/storage.sqlite3` and LangGraph `SqliteSaver` checkpoints at `~/.ai-workflows/checkpoint.sqlite3` (both env-overridable via `AIW_STORAGE_DB` / `AIW_CHECKPOINT_DB`). So this ADR picks **option (c)**: the framework owns its run/checkpoint state in its own store; CS-300's `data/notes.db` owns the Quiz domain only (the `quizzes` / `questions` / `quiz_questions` / `quiz_attempts` / `attempt_questions` tables per ADR-033); **no `ai-workflows` tables are added to `data/notes.db`**. The two stores are independent; the only data that crosses is the workflow's *terminal artefact* (a dict read from `aiw run`'s stdout), which the CS-300 processor persists into `data/notes.db` via `app.persistence.*`. (Because the `aiw` subprocess runs entirely outside CS-300's `app/` code, its SQLite access is trivially outside CS-300's persistence boundary — MC-10 stays clean — and the cross-store coupling ADR-022's option (a) would have introduced is avoided.)

### The test seam — CS-300's own processor boundary is the seam (mock the `aiw run` call); the framework's `StubLLMAdapter` is NOT usable here, because the workflow runs in a subprocess

Tests exercise the generation path *without* a real `ai-workflows` / LLM provider call (MC-1's spirit). Because this ADR chose the **subprocess** invocation (`aiw run question_gen ...`), the test seam **must be CS-300's own boundary** — not the framework's in-process `StubLLMAdapter` monkeypatch, which a test process cannot reach into a subprocess. Concretely:

- The processor's "run the workflow" call is a thin seam — `_invoke_question_gen(section_content, section_title, run_id) -> dict` (or, lower-level, `subprocess.run(["aiw", "run", "question_gen", ...])`). Tests **mock that seam**: a success-path test patches `_invoke_question_gen` to return a realistic-shaped artefact (`{"questions": [{"prompt": "<a coding task>", "topics": ["<a topic>"]}]}`) and asserts the processor calls `add_questions_to_quiz` then `mark_quiz_ready`, with the right `section_id`, `prompt`, and `topics` persisted; a failure-path test patches it to raise `QuestionGenFailed("...")` (or, lower-level, patches `subprocess.run` to return a `returncode != 0` with `error: ...` on stderr) and asserts the processor calls `mark_quiz_generation_failed(quiz_id, error=...)`, persists **zero** `questions` rows for that Quiz, and does not loop (MC-5); an empty-artefact test patches it to return `{"questions": []}` and asserts the same failure handling (no Questions persisted, status `generation_failed`).
- **The MC-1 boundary check** (a grep test): no `import openai` / `import anthropic` / `import litellm` / `import langgraph` / `from openai import …` / `import google.generativeai` / a raw `httpx`/`requests`/`urllib` call to a provider chat-completions endpoint — anywhere under `app/`, including `app/workflows/`. (The `app/workflows/question_gen.py` module imports only `ai_workflows.workflows.*`, `ai_workflows.primitives.tiers`, and `pydantic`; the processor imports `subprocess` / `json` / `os` / `app.persistence.*` / `app.parser`; neither imports a forbidden SDK.)
- A registration sanity test is optional but cheap: import `app.workflows.question_gen` and assert `register_workflow` succeeded (the module is importable, the `WorkflowSpec` is well-formed — no tier-typo `ValueError`, non-empty `steps`). This exercises the framework's spec-validation path without running the workflow or spending a token. (It does *not* exercise dispatch — that needs a real run, which is the human-only gate.)
- A real end-to-end run against the *actual* `ai-workflows` engine is a **human-only verification gate** (TASK-014's "`ai-workflows` integration sanity — a real end-to-end generation against the actual `ai-workflows` engine, run by the human once") — not a programmatic test (it depends on a real provider call and on human judgment of "is this a sensible hands-on coding-task Question about this Section"); `/auto` files a `pending human` row; the human fills it post-commit.

**Consequence stated plainly:** by choosing the documented `aiw run` CLI surface over the in-process private `_dispatch.run_workflow`, CS-300's test suite does *not* exercise the real `ai-workflows` dispatch path / validators / retry edges — those are the framework's own test suite's job (`ai-workflows`'s `tests/workflows/test_summarize.py` etc.), and the real CS-300↔`ai-workflows` integration is covered by the human-only verification gate. In exchange, CS-300's tests do *not* depend on the framework's internal import-site naming (the `ai_workflows.graph.tiered_node.LiteLLMAdapter` monkeypatch site), are not coupled to the `_dispatch.run_workflow` return-dict shape, and stay simple (mock one CS-300 seam). This is the right boundary: CS-300 has no business unit-testing `ai-workflows`'s LangGraph state machine, and the manifest's secondary objective is served by *consuming the documented surface*, which the human-only gate verifies end-to-end.

**Rejected test-seam alternatives:** the framework's `StubLLMAdapter` monkeypatch (`monkeypatch.setattr(ai_workflows.graph.tiered_node, "LiteLLMAdapter", StubLLMAdapter)` + `StubLLMAdapter.arm(...)`) — the framework's *documented* hermetic pattern for downstream consumers, and the highest-fidelity seam (it runs the real dispatch path with a canned LLM response) — but it **only works for in-process invocation**, which this ADR rejected in favour of the documented `aiw run` subprocess; a monkeypatch in the test process cannot reach a subprocess. (If a future ADR ever switches CS-300 back to in-process `run_workflow`, the `StubLLMAdapter` pattern becomes the right seam — but that ADR would have to re-argue the private-internal-dependency trade-off.) A `@pytest.mark.skipif(no_key)` real-call test — rejected: it makes a real provider call (against MC-1's spirit) and is exactly what the human-only verification gate covers.

### The forbidden-SDK list (MC-1's architecture portion) — enumerated; the conformance-skill update is the human's same-commit edit

Application code under `app/` (including `app/workflows/`) must **not** import any of:

- `openai`, `anthropic`, `google.generativeai`, `google.genai`, `cohere`, `mistralai`, `groq`, `together`, `replicate` (or any other LLM-provider SDK);
- `litellm` (the unified adapter `ai-workflows` wraps — CS-300 code goes through `ai-workflows`, not LiteLLM directly);
- `langchain`, `langchain_*`, `langgraph`, `langgraph_*` (the orchestration layer `ai-workflows` wraps — CS-300 authors a `WorkflowSpec`, never a `StateGraph`);
- a raw `httpx` / `requests` / `urllib` call to a provider's chat-completions / messages / generate endpoint (CS-300 already depends on `httpx` for `TestClient` and on `requests` for Playwright fixtures — those are fine; what is forbidden is a *direct LLM-API HTTP call* from `app/` code).

The **only** AI dependency `app/` code imports is `ai_workflows.*` (the `WorkflowSpec` API: `WorkflowSpec`, `LLMStep`, `ValidateStep`, `RetryPolicy`, `register_workflow`; `TierConfig` + a route type from `ai_workflows.primitives.tiers`). All AI *invocation* goes through the `aiw` CLI subprocess (which itself goes through `ai_workflows.workflows._dispatch` internally — but that is the framework's internal, not CS-300's import).

**This list goes into the conformance skill's MC-1 entry in the same commit that Accepts this ADR — the human owns that edit** (`.claude/skills/manifest-conformance/SKILL.md` is a Tier-2, human-owned file; the architect does not edit it). The architect flags it here so it is not missed: when this ADR moves from `Proposed` → `Accepted`, MC-1's architecture portion changes from `cannot evaluate (ADR pending)` to `blocker`, and the skill's MC-1 entry must enumerate the forbidden set above and name the workflow module path (`app/workflows/`). If the skill is *not* updated when the ADR lands, MC-1's architecture portion stays dormant and a forbidden import could slip through — a real coordination point, not a formality. ### Topic tags — deferred again to the Weak-Topic-identification slice; first-Quiz generation stores them in `questions.topics`

ADR-033 §Topic tags shipped `questions.topics` as a `'|'`-delimited TEXT column and deferred the relational `topics` table + `question_topics` join "to the generation task, whose `/design` decides the source shape … and the read path", noting the migration from the cheap form to the relational form is *additive* (a data migration, not a schema change — no migration-trigger). This is that task's `/design`, and the call is: **defer the relational form once more, to the Weak-Topic-identification slice** (the first thing that *queries across* Topics). Rationale:

- First-Quiz generation only *produces and stores* Topic tags (the `GeneratedQuestion.topics: list[str]` the workflow returns, `'|'`-joined into `questions.topics` by `add_questions_to_quiz`) — it does not query across them (no Weak-Topic identification this slice; the bank is empty; there are no Weak Topics yet). The `questions.topics` delimited column is sufficient for "store the tags the workflow returned". The relational form earns its keep when something *reads across* Topics (Weak-Topic identification, post-grading) — and that slice is the natural home for the Topic-vocabulary-source decision (`content/`-derived like the Lecture corpus? a Python module? a `topics` table seeded from a source?) and the backfill of `questions.topics` into `question_topics` rows.
- Shipping a `topics` table *now* (with no source, no curriculum-side maintenance path, and nothing reading it) is the architecture-on-spec anti-pattern the project's task discipline has rejected for Topics three times (TASK-009 → TASK-012). The §8 framing — Topics are "maintained alongside Chapter content", a *curriculum* artifact — reinforces this: the decision belongs with the slice that has a curriculum-side consumer, not the slice that merely emits free-text tags from an LLM.
- Deferring costs nothing: the migration is additive. The question is **not** "unresolved and homeless" — it has a named home (the Weak-Topic-identification slice's `/design`), so **no project_issue is filed** (consistent with how ADR-033 §Topic tags handled it).

So: the `GeneratedQuestion.topics: list[str]` the workflow returns is `'|'`-joined and written into `questions.topics` by `add_questions_to_quiz` (the cheap form, unchanged from ADR-033). When the Weak-Topic-identification slice lands, *its* `/design` decides the relational shape and the source.

## Alternatives considered

**A. Vendor the `ai-workflows` source into this repo.**
Rejected outright. `jmdl-ai-workflows` is a ~10-runtime-dependency framework (`langgraph`, `langgraph-checkpoint-sqlite`, `litellm`, `fastmcp`, `pydantic`, `python-dotenv`, `pyyaml`, `structlog`, `typer`, `yoyo-migrations`). Vendoring it means maintaining a fork — every framework change has to be re-vendored — and it defeats the manifest's secondary objective (which wants the project to *consume* `ai-workflows` as a real dependency, not absorb it). It is a 10-dependency framework — not a serious contender for vendoring.

**B. A local path / editable dependency to the co-located `../ai-workflows` clone, instead of a PyPI pin.**
Considered, and was the architect's original recommendation while the framework was only present as an editable clone in the dev `.venv`. **Rejected (by the human's resolution).** It matches the co-development reality (the author builds the framework and this consumer in lockstep; an editable install picks up a framework change without a republish; the framework's docs even cite "the first integration gap CS-300 hit in pre-flight smoke testing", i.e. integration friction flowing back into the framework). But the cost is non-reproducibility from PyPI alone — a clone of CS-300 would need the `../ai-workflows` clone too — and the human's call is that CS-300 should be a clean PyPI consumer: `jmdl-ai-workflows` is published (`pip install jmdl-ai-workflows`, current `0.4.0`), so a normal version pin in `[project] dependencies` is the right shape; the co-located clone stays as read-only reference. (If the author later wants to co-develop against an unpublished framework state, that's a per-occasion local override, not a committed dependency form.)

**C. Author the workflow as a Tier-3 custom `Step` subclass, or via the Tier-4 escape hatch (`register(name, build_fn)` with a hand-authored `StateGraph`).**
Rejected. First-Quiz generation is "one LLM call → validate the output schema → return" — exactly the shape `LLMStep` + its paired validator covers (Tier 1, the framework's recommended path; `docs/writing-a-workflow.md` says "most workflows do not need the escape hatch", and "ask first whether the pattern can be expressed as a custom step type"). A custom `Step` subclass is for when no built-in covers the need; the Tier-4 escape hatch is for topologies the linear step list cannot express (conditional routing, multiple terminal nodes). Neither applies. Reaching for them here would (a) couple CS-300 code to LangGraph (`import langgraph` — which the conformance skill's MC-1 entry will then forbid), (b) be more code for no benefit, (c) ignore the framework's explicit "use the declarative surface" guidance. The declarative `WorkflowSpec` is the right call.

**D. Invoke the workflow in-process by importing `ai_workflows.workflows._dispatch.run_workflow(...)`.**
Considered seriously — this was the *first draft's* invocation choice. `_dispatch.run_workflow` is the clean shared core both framework surfaces (`aiw` CLI, `aiw-mcp` server) wrap; it is an `async` function returning a tidy dict `{run_id, status, awaiting, artifact, plan, total_cost_usd, error, gate_context}` (`status` ∈ `pending` / `completed` / `aborted` / `errored`; `artifact` is the `model_dump()` of the workflow's first output-schema field, or `None` on error); and the framework author co-develops both repos — `_dispatch.py` even carries a `FINAL_STATE_KEY = "questions" from CS-300` comment, i.e. CS-300's exact integration is already anticipated in there. **Rejected** in favour of the documented `aiw run` CLI subprocess (Alternative — the chosen one — see §How application code invokes the workflow) because: (a) `_dispatch` is a `_`-prefixed module — the framework's *internal* shared core, **not part of the documented downstream API**; the framework's downstream contract is the `aiw` CLI's stdout/stderr (`docs/writing-a-workflow.md` §"Running your workflow" + §"External workflows from a downstream consumer", which use a `question_gen` workflow as the worked example) and the `aiw-mcp` tool schemas (KDR-008 names the MCP tool schemas, not `_dispatch.run_workflow`'s signature, as the public v0.1.0 contract); (b) a comment in a private module is not a stable API surface — `_dispatch.run_workflow`'s name, signature, module path, and return-dict shape can churn between framework versions without that being a semver break, and CS-300 building on it would couple the project to framework internals; (c) the manifest's secondary objective — "`ai-workflows` is a viable engine … in a real consumer application beyond its own examples" — is more honestly demonstrated by consuming the *documented downstream surface* (the `aiw` CLI + `AIW_EXTRA_WORKFLOW_MODULES`, which the framework's docs literally model with a `question_gen` example) than by reaching into a private module. The trade-off — a subprocess spawn per Quiz, and stdout parsing — is real but tiny for a single-user dev command processing a small table, and the framework's stdout contract (`json.dumps(artifact, indent=2)` then `total cost: $X.XXXX`; logs on stderr) makes the parsing deterministic ("strip one trailing line, `json.loads` the rest"). The accepted *consequence* is the test seam: in-process `run_workflow` would have let CS-300 use the framework's `StubLLMAdapter` monkeypatch (running the real dispatch path hermetically); the subprocess choice means CS-300's test seam is its own processor boundary (mock the `aiw run` call), and the real CS-300↔`ai-workflows` integration is verified by the human-only gate — which is the right division of labour (CS-300 should not unit-test `ai-workflows`'s LangGraph internals).

**E. Invoke the workflow through the `aiw-mcp` HTTP/stdio server.**
Rejected for this project. The framework's docs model `aiw-mcp --transport http` (or stdio) as the surface a *separate frontend* (an Astro / React / Vue app) talks to — i.e. when the consumer is a *different process* from the workflow runner. CS-300's processor *is* the consumer and *has* `ai_workflows` (and the `aiw` console script) in its venv; standing up an HTTP/stdio MCP server between the processor and the workflow is the heavyweight-job-queue / job-broker anti-pattern the task explicitly flagged ("don't reach for a heavyweight queue … wildly over-scoped for a single-user dev project"), and the manifest's "no remote deployment / no hosted product" (§5) cuts against running an extra server. The `aiw-mcp` path is a future option if CS-300 ever splits its frontend from its workflow runner — not now.

**F. `ai-workflows` persistence cohabitation option (a) — add `ai-workflows`-state tables to CS-300's `data/notes.db`.**
Rejected. ADR-022 §`ai-workflows` cohabitation pre-positioned option (a) as the *default* but did not foreclose (b) or (c) — and (c) is what the framework actually supports (it has its own `SQLiteStorage` + checkpoint store at `~/.ai-workflows/*.sqlite3`, env-overridable). Adding `ai-workflows`-owned tables to `data/notes.db` would (i) couple two stores' schemas (a framework upgrade that changes its run-registry schema would touch CS-300's DB), (ii) put `ai-workflows`'s SQL adjacent to CS-300's (it already runs entirely outside CS-300's `app/` code via the subprocess, so this would be a regression in cleanliness), (iii) buy nothing (the only data that crosses is the workflow's terminal artefact, which CS-300 persists into its own tables via `app.persistence.*`). Option (c) — the framework owns its store, CS-300 owns its store, the artefact crosses on stdout — is the clean separation.

**G. The framework's `StubLLMAdapter` monkeypatch as the CS-300 test seam.**
Rejected — *not because it's the wrong kind of seam* (it is the framework's documented hermetic pattern for downstream consumers, and the highest-fidelity one — it runs the real LangGraph state machine / validators / retry edges with a canned LLM response, no token spent) — but because it **only works for in-process invocation**, and this ADR chose the documented `aiw run` *subprocess* surface (Alternative D rejected; the subprocess chosen). A `monkeypatch.setattr(ai_workflows.graph.tiered_node, "LiteLLMAdapter", StubLLMAdapter)` in the CS-300 test process cannot reach into the `aiw` subprocess. So CS-300's test seam is its own processor boundary (mock `_invoke_question_gen` / `subprocess.run`). The trade-off — CS-300 tests don't exercise the real framework dispatch path — is accepted (it's the framework's own test suite's job; the real CS-300↔framework integration is the human-only gate). (If a future ADR ever switches CS-300 back to in-process `run_workflow`, `StubLLMAdapter` becomes the right seam — but that ADR re-opens the private-internal-dependency trade-off.)

**H. Let the learner control `num_questions` (a form field on the "Generate a Quiz" button → a `QuestionGenInput.num_questions` field).**
Rejected for this slice. The "Generate a Quiz for this Section" affordance ADR-034 ships is a single submit button — no form fields, by ADR-034's decision ("no form body is needed — the action is unambiguous"). Adding a `num_questions` field would change that surface (a `<input type="number">` on the form, route-handler validation of the value) — out of scope for a generation slice that's already the largest the project has attempted, and the manifest does not require a learner-tunable Question count. The workflow's prompt picks a sensible count (a fixed framing constant, or the `prompt_fn` decides); a future task can add learner control if there's a real need. (The workflow's `QuestionGenInput` has no `num_questions` field this slice — keeping the surface unchanged. This is a deliberate deviation from the framework's worked example, which has `num_questions: int`; the framework example's field is illustrative.)

- The grading-workflow slice (later) to reuse this ADR's integration shape unchanged — it authors `app/workflows/grade.py` as a sibling `WorkflowSpec`, the processor shells out to `aiw run grade --input ...`, the Grade is persisted via `app.persistence.*`; it does not re-decide the dependency, the module path, the invocation surface, or the test seam.

**Becomes more expensive:**

- `pyproject.toml` grows a runtime dependency (`jmdl-ai-workflows>=0.4` in `[project] dependencies`) and `requires-python` moves to `>= 3.12`. **The implementer makes both edits.** Mitigation: the framework is the manifest's named AI engine — this dependency is the secondary objective, not scope creep; the Python-version bump is a no-op for the dev environment (3.13).
- `app/` grows a new `app/workflows/` package (the `question_gen.py` `WorkflowSpec` module + the out-of-band processor module(s) — ADR-037's `process_quiz_requests.py` + any helper + `__init__.py`). Mitigation: localized; the workflow module is ~one `WorkflowSpec` declaration following the framework's documented pattern; the processor is a thin loop around `subprocess.run(["aiw", "run", "question_gen", ...])` + stdout parsing + the persistence transitions.
- `app/persistence/quizzes.py` grows the transition functions + `add_questions_to_quiz` + the processor's read accessors. Mitigation: ADR-033 already forecast `add_questions_to_quiz`; the transition functions are 1-line `UPDATE`s; the read accessors mirror the existing `list_quizzes_for_chapter` shape.
- The processor spawns an `aiw` subprocess per Quiz generation and parses its stdout. Mitigation: tiny for a single-user dev command processing a small table; the framework's stdout contract (`json.dumps(artifact, indent=2)` then `total cost: $X.XXXX`; logs on stderr) makes the parsing deterministic. If a very long Section's content ever approached the OS arg-length limit on `--input section_content=...`, the mitigation is to pass the content via a temp file path + a `--input section_content_path=...` field that the `prompt_fn` reads — a bounded change, deferred until it's a real problem (a single Section's LaTeX is a few KB, well under `ARG_MAX`).
- The processor (and a real generation run) needs `ai-workflows` importable in CS-300's venv (it is — a declared dependency), the `aiw` console script on the subprocess's `PATH` (it is — same venv's `bin/`; the PATH-independent `[sys.executable, "-m", "ai_workflows.cli", "run", ...]` form is the fallback), and a provider key in the processor's environment (`GEMINI_API_KEY` for a Gemini tier, or a logged-in `claude` CLI for `claude_code`). Mitigation: the framework's tier registry makes the provider choice changeable; tests need no key (they mock the subprocess); the real-engine run is a human-only gate (the human has the key).

**Becomes impossible (under this ADR):**

- Application code calling an LLM provider SDK directly (the forbidden-SDK list + the boundary grep test forbid it; all AI through `ai-workflows`).
- Application code importing the framework's `_dispatch.run_workflow` (or any `ai_workflows.*._`-prefixed internal) — CS-300 imports only the `WorkflowSpec` API (`ai_workflows.workflows.*` public names) and the tier types, and *invokes* the workflow via the `aiw` CLI subprocess; it does not couple to the framework's internal dispatch core.
- A non-coding Question being generated (the `QuestionGenOutput`/`GeneratedQuestion` schema has no choice/recall/describe field; the paired validator enforces the shape; the prompt constrains the LLM).
- A cross-Section Quiz being generated (the workflow operates on exactly one `section_id`; `add_questions_to_quiz` writes one Section's `section_id`).
- The "Generate a Quiz" click blocking on the AI call (the workflow runs out-of-band — ADR-037's manual command; the route returns 303 immediately).
- A fabricated Quiz/Question on workflow failure (the failure path sets `generation_failed` and persists zero `questions` rows; no silent unbounded retry).
- CS-300 importing `langgraph` (the framework wraps it; CS-300 authors a `WorkflowSpec`; the conformance skill's MC-1 entry forbids the import).

**Future surfaces this ADR pre-positions:**

- The grading workflow — `app/workflows/grade.py` as a sibling `WorkflowSpec`; the processor shells out to `aiw run grade --input ...`; the Grade persisted via `app.persistence.*`. A later slice.
- The Weak-Topic-identification slice — the first thing that *queries across* Topics; *its* `/design` decides the relational `topics` table + `question_topics` join shape and the Topic-vocabulary source, and backfills `questions.topics` into `question_topics` rows (additive). The named home for the deferred Topic-vocabulary decision.
- The Quiz-taking surface — adds a "take this Quiz" affordance to ADR-034's `.section-quiz` block for `ready` Quizzes; renders the generated Questions; persists the Attempt. A later slice (no take-button this task — ADR-034's commitment stands).
- The `aiw-mcp` server path — a future option if CS-300 ever splits its frontend from its workflow runner; not this project.

**Supersedure path if this proves wrong:**

- If the PyPI dependency proves awkward (e.g. the author needs to co-develop against an unpublished framework state for a stretch) → the author uses a *per-occasion local override* (a temporary editable install in the dev venv) without committing a path dependency; if that becomes the norm, a future ADR re-decides. Cost: a one-line `pip install -e ../ai-workflows` in the dev shell, not a repo change.
- If the `aiw run` CLI's stdout contract changes (the framework alters the artefact-printing format or moves the cost trailer) → the processor's `_parse_artefact_from_stdout` is the one seam to update; or, if the change is large, a future ADR switches to the `aiw-mcp` tool surface (KDR-008's semver-contracted MCP tool schemas) or — re-opening Alternative D's trade-off — to in-process `run_workflow`. Cost: bounded — one parsing function, or one ADR.
- If the subprocess spawn proves a real bottleneck (it won't, at single-user scale) → a future ADR switches to in-process invocation (re-opening the private-internal-dependency trade-off in Alternative D) or to a long-lived `aiw-mcp` stdio session. Cost: bounded.
- If the declarative `WorkflowSpec` surface proves insufficient (first-Quiz generation needs a topology `LLMStep` + `ValidateStep` cannot express — unlikely for "one call → validate → return") → a future ADR upgrades to a Tier-3 custom `Step` subclass or the Tier-4 escape hatch (which would then need `import langgraph` allowed for the workflow module — a conformance-skill carve-out the human owns). Cost: bounded — the workflow module's authoring surface, not the integration shape.
- If option (c) for `ai-workflows` persistence proves wrong (the framework's store conflicts with CS-300's, or CS-300 needs to query the framework's run registry) → a future ADR moves to option (a) or (b) per ADR-022 §`ai-workflows` cohabitation; the env-overrides (`AIW_STORAGE_DB` / `AIW_CHECKPOINT_DB`) give a non-ADR escape hatch in the meantime. Cost: bounded.

The supersedure path, in every case, runs through a new ADR. This ADR does not edit any prior ADR in place; it fills in ADR-022 §`ai-workflows` cohabitation's deferred integration mechanics (picking option (c) for persistence) and consumes ADR-033 / ADR-034 / ADR-037 unchanged (ADR-037 being Proposed alongside it in the same `/design` cycle).
