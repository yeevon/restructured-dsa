## Context

§8 (post-2026-05-12 amendment) defines a **Grade** as the result of grading a Quiz Attempt and enumerates four facets: **per-Question correctness** (determined by whether the learner's code passed the Question's tests), **per-Question explanation** (of why the learner's approach worked or didn't), **an aggregate score**, **identified Weak Topics**, and **recommended Sections to re-read**. ADR-033 reserved the `quiz_attempts.status` enum `submitted → grading → graded` / `grading_failed` for exactly this slice; ADR-039 left `attempt_questions.is_correct` and `.explanation` NULL on Attempt start with the explicit forecast "the grading slice writes them"; ADR-042–ADR-044 added the in-app test runner that produces `attempt_questions.test_passed` (the verdict the §8 amendment makes load-bearing for correctness); ADR-045–ADR-047 made that verdict *meaningful* on real generated Questions (the assertion-only test-suite + `preamble` extension makes `passed=True`/`passed=False` reachable, not universally `compile_error`). TASK-019 is the slice that consumes those inputs and produces the §8 Grade.

This ADR owns the *workflow-authoring* half of the slice — the CS-300-owned `WorkflowSpec` module that the out-of-band grading processor (ADR-049) invokes via the `aiw run` CLI subprocess (ADR-036's downstream surface). The companion ADRs own the rest: **ADR-049** the out-of-band processor and lifecycle/failure-handling; **ADR-050** the Grade aggregate's persistence (the `grades` table, the `quiz_attempts.grading_error` column, the new persistence functions and dataclasses); **ADR-051** the graded-state and grading-failed-state rendering on the take page.

The decision space for the workflow:

- **The Pydantic input/output schemas.** `GradeAttemptInput` carries the Quiz's Section grounding (title + content) and a list of per-Question records (the Question's `prompt`, `preamble`, `test_suite`, the learner's `response`, and the persisted test result — `test_passed`, `test_status`, `test_output`). `GradeAttemptOutput` carries `per_question: list[QuestionGrade]` (one `explanation` per Question by `question_id`) plus the aggregate (`score: int`, `weak_topics: list[str]`, `recommended_sections: list[str]`). `extra="forbid"` everywhere makes a non-coding / chat / tutor-style field unexpressible by construction (manifest §5 / §7); `min_length=1` on `QuestionGrade.explanation` so an empty explanation cannot satisfy the schema (a missing explanation is a grading failure, MC-5).
- **The prompt framing — load-bearing.** §8 says "correctness determined by whether the learner's code passed the Question's tests". The architecturally honest implementation is that the workflow **does not free-form-judge correctness** — it reads `test_passed` and produces the *explanation* and the *aggregate*. The grading persistence (ADR-050) sets `attempt_questions.is_correct` from the persisted `test_passed`, not from anything the workflow says about it. The workflow ADR's job is to commit to that framing in the prompt and the Pydantic shape (no `is_correct` field on `QuestionGrade` — the workflow doesn't produce one; the persistence derives it from the runner's verdict).
- **The `LLMStep` invocation surface.** `prompt_fn` (not `prompt_template`) because the Section content + the test-suite source + the learner's response may contain `{` `}` characters that collide with `str.format` placeholders (the same hazard `question_gen` addressed). `response_format=GradeAttemptOutput` so the framework validates the artefact end-to-end. A single `LLMStep` over a multi-step shape (one step per Question for explanations + a second step for the aggregate) — the multi-step shape adds N LLM calls and inter-step state machinery for no benefit (the explanations don't need to be independent; an LLM seeing the whole Attempt produces *more* coherent explanations and a *more* honest aggregate, not less).
- **The `RetryPolicy`.** Mirroring `question_gen`'s default: `max_attempts=3` with exponential backoff. Transient provider failures get bounded retry inside the workflow run (the subprocess); a permanent failure surfaces as a non-zero exit and the processor sets `grading_failed` (ADR-049). No silent unbounded retry (MC-5).
- **The tier registry.** A single tier named `grade-attempt-llm` via `grade_attempt_tier_registry()` (a sibling of `question_gen_tier_registry()`); the specific provider/model is a tuning decision (env-var-overridable, same pattern as `question_gen`), not an architectural commitment. Default forecast: the same `gemini/gemini-2.5-flash` as `question_gen` (one provider key, one observed-cost line item; the grading workflow is comparable in shape to question generation — one prompt, one structured response, a few thousand tokens of context).
- **Loading and registration.** The module imports the `WorkflowSpec` authoring API + `pydantic` + stdlib `os` only (MC-1's enumeration of the forbidden-SDK list — `openai`, `anthropic`, `google.generativeai`, `google.genai`, `cohere`, `mistralai`, `groq`, `together`, `replicate`, `litellm`, `langchain*`, `langgraph*`, raw provider HTTP — is honored, mirroring `question_gen.py`); a module-level `register_workflow(_spec)` call fires at import time; the processor's `aiw run` subprocess env carries `AIW_EXTRA_WORKFLOW_MODULES=app.workflows.question_gen,app.workflows.grade_attempt` (extending ADR-036's existing single-module value — the framework documents this as a comma-separated list of module dotted paths).

The manifest constrains the decision: **§4** ("`ai-workflows` is the AI engine the project commits to" — the workflow is authored under that engine), **§5** ("No live / synchronous AI results" — the workflow runs out-of-band; the processor calls it from outside any FastAPI request; "No AI tutor / chat interface for the learner" — the workflow produces one structured Grade output, not a chat), **§6** ("AI work is asynchronous from the learner's perspective" — orthogonal to the workflow itself, which runs in the processor's subprocess; "AI failures are visible … never fabricates a result" — the workflow either produces a well-formed `GradeAttemptOutput` or the framework raises and the processor sets `grading_failed`; the schema's `min_length=1` on `QuestionGrade.explanation` prevents a "looks like a grade but isn't" near-empty output), **§7** ("The reinforcement loop is the reason this project exists. Quiz performance MUST drive next-Quiz generation" — the workflow produces the Weak Topics list the composition slice consumes; "Every Question is a hands-on coding task" — the Grade's `explanation` is *commentary on coding*, not a non-coding artifact), and **§8** (the Grade glossary — the workflow ADR realizes three of the four §8 facets directly: explanation, score, weak topics, recommended sections; the fourth — per-Question correctness — is read from the runner's verdict, not re-judged here).

## Decision

### The workflow module — `app/workflows/grade_attempt.py` (a `WorkflowSpec` mirroring ADR-036's `question_gen.py` shape)

A new module at `app/workflows/grade_attempt.py` defines and registers a single `WorkflowSpec` named **`grade_attempt`**:

- **Imports:** `ai_workflows.workflows.{WorkflowSpec, LLMStep, register_workflow}`, `ai_workflows.primitives.tiers.{LiteLLMRoute, TierConfig}`, `ai_workflows.workflows.RetryPolicy` (or the appropriate `RetryingEdge`/`RetryPolicy` path the framework exposes; the implementer follows the existing `question_gen.py` import set), `pydantic.{BaseModel, ConfigDict, Field}`, stdlib `os`. **No** `openai` / `anthropic` / `google.generativeai` / `google.genai` / `cohere` / `mistralai` / `groq` / `together` / `replicate` / `litellm` / `langchain*` / `langgraph*` / raw provider-HTTP imports — MC-1's architecture portion (active per ADR-036) preserved.
- **Schemas:** three Pydantic models, every one with `model_config = ConfigDict(extra="forbid")`:
  - `GradeAttemptInput` — the Attempt to grade. Fields:
    - `section_title: str`, `section_content: str` — the Section grounding (same provenance as `question_gen`'s inputs; the processor reads `content/latex/{chapter_id}.tex` and the parsed Section body, MC-6 read-only — ADR-049).
    - `questions: list[QuestionForGrading]` — one record per Attempt question.
  - `QuestionForGrading` — one Attempt-question record (named distinctly from `question_gen`'s `GeneratedQuestion` because the *grading* input also carries the learner's response and the persisted test result, which are not part of a generated Question). Fields:
    - `question_id: int` — so the workflow's per-Question output can be matched back; the processor reads `attempt_questions ⨝ questions` (ADR-049).
    - `prompt: str` (the coding-task instruction the learner answered), `preamble: str` (ADR-045 — may be empty; preserved verbatim), `test_suite: str` (ADR-040 — the spec the learner's code had to satisfy).
    - `response: str` (the learner's code as submitted), and:
    - `test_passed: bool | None` (`True` / `False` / `None` — the runner's verdict; `None` when the learner submitted without running the tests, or when the run did not complete — `test_status != 'ran'`).
    - `test_status: str` (`'ran'` | `'timed_out'` | `'compile_error'` | `'setup_error'` | `'not_run'` — the persisted ADR-042/ADR-044 column value; `'not_run'` is a workflow-input-level synthetic for the NULL-`test_run_at` case so the LLM sees a single enum vocabulary, set by the processor at input-marshalling time).
    - `test_output: str` (the persisted runner output / diagnostic; may be empty if not run; truncated by the sandbox per ADR-042 so the full Grade's input stays bounded).
  - `QuestionGrade` — one per-Question grade record:
    - `question_id: int` — matches an input `QuestionForGrading.question_id` (a `QuestionGrade.question_id` not present in the input is a workflow-output-validation failure handled by the processor — ADR-049 — and produces `grading_failed`).
    - `explanation: str = Field(min_length=1)` — non-empty English commentary on why the learner's approach worked or didn't, referencing the test result (the prompt instructs the LLM to ground the explanation in `test_status` / `test_output`, MC-5's spirit). **No** `is_correct` field — correctness is the runner's verdict and is written to `attempt_questions.is_correct` from `test_passed` by the persistence layer (ADR-050), not free-form-judged by the LLM.
  - `GradeAttemptOutput` — the terminal artefact, first field per ADR-036's `FINAL_STATE_KEY` convention:
    - `per_question: list[QuestionGrade]` — one record per Question in the input (the processor validates the question_id set matches the input's; missing/extra → `grading_failed`, ADR-049).
    - `score: int = Field(ge=0)` — the aggregate score (forecast convention: **points correct**, i.e. the count of Questions whose `test_passed = True`, expressible as `score / len(per_question)`; the prompt instructs the LLM to compute it as that count — but the persistence layer cross-checks against the actual `test_passed` set per ADR-050 to keep §8 honest if the LLM miscounts).
    - `weak_topics: list[str]` — Topic tags from the failing Questions' `topics` (the prompt instructs the LLM to draw from each failing Question's tags; an empty list is valid for a perfect Attempt — `min_length=0`).
    - `recommended_sections: list[str]` — `{chapter_id}#section-{n-m}` Section IDs the learner should re-read (ADR-002's composite form). The prompt steers toward the Section the Attempt is from for now (the relational Topic vocabulary is deferred — ADR-050 — so the workflow cannot yet recommend Sections outside the Attempt's Section based on Topic-similarity; recommending "this Section" / "" / a subset is acceptable, and the composition slice will tighten the recommendation surface when the relational Topic vocabulary lands).
- **The `LLMStep`'s prompt_fn** — `_grade_attempt_prompt_fn(state: dict) -> tuple[str | None, list[dict]]` — builds a system + user message pair, mirroring `_question_gen_prompt_fn`:
  - **System prompt** commits architecturally to:
    1. *Do not re-judge correctness.* Each Question's per-Question test result is provided (`test_passed` + `test_status` + `test_output`); the explanation must be consistent with that verdict — if `test_status == 'ran'` and `test_passed = True`, explain why the implementation works; if `test_status == 'ran'` and `test_passed = False`, explain what the test caught; for `compile_error` / `timed_out` / `setup_error` / `not_run`, explain the failure mode in plain English (referencing `test_output` where useful). Never claim a Question was correct when its `test_passed` is `False`, and never claim it was incorrect when `test_passed` is `True`.
    2. *Be concrete.* Each `explanation` references the Question's prompt and the learner's response specifically; generic platitudes ("good job" / "consider reviewing the material") alone are not the §8 explanation a Grade is defined in terms of.
    3. *Weak Topics come from failing Questions.* Aggregate the Topic tags of Questions whose `test_passed != True` (failures and not-runs both count — a learner who skips a Question is signaling a knowledge gap on the Topic, MC-5's spirit). Empty list is valid (a perfect Attempt has no Weak Topics).
    4. *Recommended Sections.* For now, recommend the Attempt's own Section when any Question failed (the relational Topic vocabulary is deferred; cross-Section recommendations are the composition slice's concern). Empty list valid.
    5. *The score is the count of Questions whose `test_passed = True`.* Not a free-form estimate. The persistence layer cross-checks (ADR-050).
    6. *The output is JSON conforming to the `GradeAttemptOutput` schema.* The `response_format` validator will reject malformed output; the framework's `RetryingEdge` will retry transient validation failures within bounded `max_attempts=3`.
    7. *Strict non-fields.* No `is_correct` field on `QuestionGrade`; no chat / tutor / conversational fields anywhere; the prompt does not ask the LLM to advise on study strategy or to converse with the learner (manifest §5 "No AI tutor / chat interface for the learner").
  - **User message** carries: `section_title`, `section_content` (the Section grounding the Questions came from), and for each `QuestionForGrading` a structured block listing `question_id` / `prompt` / `preamble` / `test_suite` / `response` / `test_status` / `test_passed` / `test_output`. The implementer formats this as a clearly delimited human-readable structure (the `aiw run` CLI passes the dict through `json.dumps`; the LLM sees structured English).
- **The `LLMStep`** itself: `tier="grade-attempt-llm"`, `prompt_fn=_grade_attempt_prompt_fn`, `response_format=GradeAttemptOutput`. A single step — no multi-step shape (rejected as Alternative B below).
- **The `RetryPolicy`** attached to the `LLMStep`: `max_attempts=3` with exponential backoff (mirroring the documented `question_gen` default). The retries happen inside the workflow run (the `aiw run` subprocess); the CS-300 processor does **not** wrap the subprocess call in a retry loop (ADR-049, mirroring ADR-037).
- **The tier registry** — `grade_attempt_tier_registry() -> dict[str, TierConfig]`, mirroring `question_gen_tier_registry()`: one tier named `"grade-attempt-llm"`, with model resolved via a `_resolve_model()` helper that reads (in order) `GRADE_ATTEMPT_MODEL`, `OLLAMA_MODEL_GRADE_ATTEMPT`, defaulting to `"gemini/gemini-2.5-flash"` (mirroring `question_gen`'s precedence chain — env-overridable, with the same Ollama-prefix convenience). The specific model is a tuning decision; the architectural commitment is "one tier, one model string at a time, resolved via the same env-var precedence as `question_gen`".
- **Registration** — `register_workflow(_spec)` at module bottom; `_spec = WorkflowSpec(name="grade_attempt", input_schema=GradeAttemptInput, output_schema=GradeAttemptOutput, tiers=grade_attempt_tier_registry(), steps=[LLMStep(...)])`.

### `is_correct` is the runner's verdict, not the LLM's — committed architecturally in the schema

`QuestionGrade` does **not** carry an `is_correct` field. The §8 amendment makes correctness a function of the test result; the schema makes the LLM unable to express a correctness disagreement with the runner; the persistence layer (ADR-050) derives `attempt_questions.is_correct` from `attempt_questions.test_passed` (the runner's verdict) — not from the workflow output. This is the architectural realization of §8's "correctness determined by whether the learner's code passed the Question's tests": the LLM produces the explanation and the aggregate, the runner produced the correctness, and the two cannot disagree because the LLM is not given a field through which to disagree.

(Where the runner's verdict is `None` — the learner submitted without running the tests, `test_status='not_run'` — the persistence layer's mapping is **`is_correct = 0`** with the explanation noting the test never ran, rather than `is_correct = NULL` or `grading_failed`; ADR-050 makes that call. The workflow does not need to know — it just produces the explanation; the persistence handles the mapping.)

### `extra="forbid"` everywhere; non-coding fields unexpressible by construction

`model_config = ConfigDict(extra="forbid")` on `GradeAttemptInput`, `QuestionForGrading`, `QuestionGrade`, and `GradeAttemptOutput`. The LLM cannot add a `correct_choice` / `option_list` / `recall_answer` / `describe_prompt` / `chat_turn` field through the schema gap; if it tries (a non-conformant artefact), the `response_format` validator rejects it and the framework either retries (within `RetryPolicy`'s bound) or the run errors out, at which point the processor (ADR-049) flips the Attempt to `grading_failed`. Manifest §5 "No non-coding Question formats" and §5 "No AI tutor / chat interface" are honored architecturally — by the shape of the schema, not by trust in the LLM's behavior.

## Alternatives considered

**A. Free-form-judge correctness in the workflow — a `QuestionGrade.is_correct: bool` field set by the LLM independently of `test_passed`.**
Considered (this is the obvious shape for someone reading §8 without the §6 amendment in mind). Rejected on §8 grounds: the post-2026-05-12 §8 amendment makes "the in-app test results" the correctness signal — "correctness determined by whether the learner's code passed the Question's tests". An LLM that re-judges correctness can disagree with the runner ("the test failed but I think the code is morally correct" / "the test passed but the code has a subtle bug"); either disagreement is the §8 violation MC-5's spirit forbids — a fabricated correctness verdict (in the case where it disagrees with the runner) or a free-form override of the architecturally-binding signal. Removing `is_correct` from the schema makes the disagreement unexpressible; the persistence layer (ADR-050) reads `test_passed` and writes `is_correct`, and the LLM produces the *explanation* that grounds the test result in plain English. This is the architecturally honest shape; the alternative is a hope about LLM behavior, not a guarantee.

**B. A multi-step workflow — one `LLMStep` per Question for explanations, a second `LLMStep` for the aggregate score / Weak Topics / recommended Sections.**
Considered. Rejected: a multi-step shape adds N+1 LLM calls per Attempt (one per Question + one aggregate), N+1 prompt-construction surfaces, N+1 schema validators, and inter-step state — for no benefit. The per-Question explanations don't need to be independent (an LLM seeing the *whole* Attempt produces more coherent, less repetitive explanations and a more honest aggregate, because the explanations and the score/weak-topics naturally cross-reference); the latency and cost both go up; the chance of a partial-failure (some steps succeed, others fail) goes up, which makes the `grading_failed` discipline (ADR-049) harder. A single `LLMStep` over the whole Attempt with a structured `GradeAttemptOutput` is the simpler and more honest shape; matches `question_gen`'s precedent (one `LLMStep` produces the whole `questions: list[GeneratedQuestion]`); the framework's bounded retry handles transient failures inside the step.

**C. `prompt_template` instead of `prompt_fn`.**
Rejected on the same grounds as `question_gen` (ADR-036 / ADR-040): the inputs contain LaTeX-derived Section content, free-form learner responses, and test-suite source code — all of which may contain `{` `}` characters that would collide with `str.format` placeholders. `prompt_fn` builds the system+user messages programmatically and avoids the placeholder hazard. No reason to introduce a different invocation pattern from `question_gen`.

**D. No `RetryPolicy` (let transient failures fall straight through to `grading_failed`).**
Rejected mildly: the framework's `RetryingEdge` is built for exactly this — bounded retry of transient provider failures (rate-limits, transient 5xx, malformed responses that the schema rejects but a re-prompt fixes) — and `question_gen` uses it; using it here too gets the same robustness for free, MC-5 still honored because the retry is **bounded** (`max_attempts=3`) and falls through to `grading_failed` after that. Without it, a single rate-limit blip becomes a Grade the author has to manually re-grade by running the processor again; that's friction for no offsetting benefit.

**E. A separate `grade-attempt-llm` tier from `question-gen-llm` with a different (e.g. higher-end) model by default.**
Considered (grading is in some sense harder than generation — the LLM has to read code, reason about test results, and produce useful commentary). Rejected as a default: the two workflows have comparable shape and context size; the same `gemini/gemini-2.5-flash` default is a reasonable starting point; the env-var precedence (`GRADE_ATTEMPT_MODEL` → `OLLAMA_MODEL_GRADE_ATTEMPT` → default) means swapping in a stronger model for grading is a one-env-var change without touching the workflow code. The architectural commitment is "one tier per workflow, env-var-resolved model"; the specific model is a tuning decision.

**F. Fold `grade_attempt` into `question_gen` as a second step (one workflow handles both generation and grading).**
Rejected outright. The two workflows have different inputs, different outputs, different lifecycles, and are invoked from different processors at different times for different `quiz_attempts` / `quizzes` rows. The `question_gen` workflow runs when a `requested` Quiz needs Questions; `grade_attempt` runs when a `submitted` Attempt needs a Grade. Conflating them into one module would obscure the two-call shape and make the `aiw run` invocations contradictory (the processor would have to pick a "mode" via inputs). Two separate, parallel workflow modules is the clean shape; mirrors how `question_gen.py` is already structured under `app/workflows/`.

## My recommendation vs the user's apparent preference

Aligned with the user's apparent preference, captured in TASK-019's task file (the "Architectural decisions expected" section forecasts this ADR explicitly, with the architecturally-load-bearing commitment "the per-Question correctness signal is the persisted test result; the workflow's job is the *explanation* + the *aggregate*, NOT re-judging whether the code is correct"), `quiz-grading-slice-shape.md` (which forecasts "a CS-300-owned declarative `WorkflowSpec` module at `app/workflows/grade_attempt.py` (mirroring ADR-036's `question_gen.py`) — one `LLMStep` + a `response_format` validator; `GradeAttemptInput` carrying the Questions + the learner's code per Question; `GradeAttemptOutput` carrying per-Question correctness + explanation + the aggregate's components"), the manifest §8 amendment (which makes correctness a function of the test result, not the LLM's judgment), and ADR-044 §`is_correct`'s source (which forecast "`is_correct` is set by the grading slice (reading the persisted test result), not by the runner — keeps the Grade coherent; this is the grading slice"). The architect's call removes `is_correct` from the workflow output entirely — the user's forecast had `GradeAttemptOutput` carrying "per-Question correctness + explanation", but on reflection the only way to *guarantee* §8 is honored is to make correctness unexpressible by the LLM. The persistence layer (ADR-050) writes `is_correct` from `test_passed`; the workflow produces the explanation alone. This is a tightening of the user's forecast in the direction the user's framing was already pointing (the task file's "Architectural concerns #2: §8 Grade-correctness must be an architectural property of the workflow, not a hope about LLM behavior" *is* this call). No disagreement to surface — the architect is implementing the architecturally honest shape the user's framing forecast.

The other calls — `prompt_fn` over `prompt_template`, single `LLMStep` over multi-step, `extra="forbid"` everywhere, `min_length=1` on `explanation`, one tier with env-var-resolved model, `RetryPolicy(max_attempts=3, exponential)` — all mirror `question_gen`'s precedent exactly and are the conservative direction the user's framing endorses ("mirroring ADR-036's `question_gen.py` exactly in shape").

I am NOT pushing back on:

- **The §8 amendment's "correctness determined by … the Question's tests"** — load-bearing for this ADR; the schema realizes it; the persistence layer (ADR-050) writes it; the workflow does not contradict it.
- **ADR-036's `ai-workflows` integration shape** — consumed unchanged: same `WorkflowSpec` authoring surface, same `aiw run` CLI-subprocess invocation pattern (ADR-049's concern), same `AIW_EXTRA_WORKFLOW_MODULES` registration (extended), same test seam at CS-300's processor boundary, same forbidden-SDK list (honored), same option (c) for `ai-workflows` persistence.
- **ADR-040's `question_gen.py` shape** — consumed as the precedent; this ADR's module mirrors it (the `_resolve_model()` precedence chain, the `prompt_fn` discipline, `ConfigDict(extra="forbid")` on every model, `register_workflow` at module bottom).
- **ADR-044's `is_correct` source** — consumed as the architectural commitment this ADR implements (the runner produces `test_passed`; the grading slice's persistence reads it and writes `is_correct`; the workflow output schema has no `is_correct` field through which to disagree).
- **The single-user posture (MC-7)** — preserved: no `user_id` anywhere in the schemas or in any column the workflow writes (the workflow writes nothing — the processor / persistence layer write).

## Manifest reading

Read as binding for this decision:

- **§4 Secondary Objective.** `ai-workflows` is the project's AI engine. This ADR adds a second CS-300-owned `WorkflowSpec` module under the same engine — no new orchestration shape, no new AI-engine call surface, just one more workflow loaded by the same `AIW_EXTRA_WORKFLOW_MODULES` mechanism ADR-036 established.
- **§5 Non-Goals.** "No live / synchronous AI results … arrives when ready and is surfaced via Notification — not in real time at the moment of request" — the workflow runs in the out-of-band processor (ADR-049), never inside a FastAPI request. "No AI tutor / chat interface for the learner" — the workflow produces one structured `GradeAttemptOutput` per Attempt; the schema has no `chat_turn` / `dialog` / `follow_up_question` field; `extra="forbid"` blocks the LLM from inventing one. "No non-coding Question formats" — the workflow grades coding Questions only (the input `QuestionForGrading` carries `prompt` / `preamble` / `test_suite` / `response`, all of which are coding-task fields); the output `QuestionGrade.explanation` is commentary on the learner's *code*, not a non-coding artifact.
- **§6 Behaviors and Absolutes.** "AI work is asynchronous from the learner's perspective" — the workflow's invocation is the processor's concern (ADR-049); this module is loaded into the framework's registry and runs only inside the `aiw run` subprocess. "AI failures are visible. If AI-driven processing fails, the failure is surfaced to the learner as a failure. The system never fabricates a result" — the schema's `min_length=1` on `explanation` and `extra="forbid"` everywhere prevent a "looks like a grade but isn't" near-empty or malformed artefact from passing validation; an invalid artefact triggers bounded retry then errors out, at which point ADR-049 flips the Attempt to `grading_failed`. "Code is written, run, and tested within the application" — the test result the workflow reads is produced by the in-app test runner (ADR-042) — the §6 loop's "tested within the application" arrow flows from runner → persisted column → workflow input. "Single-user" — no `user_id` in any schema.
- **§7 Invariants.** "The reinforcement loop is the reason this project exists. Quiz performance MUST drive next-Quiz generation" — the workflow's `weak_topics` output is what the composition slice consumes to drive the fresh-Question portion of the next Quiz. "Every Quiz Attempt … persists across sessions" — the workflow itself doesn't persist (the processor / persistence layer do), but its output drives the persisted Grade. "Every Question is a hands-on coding task" — the workflow's input carries coding-task fields only; the output explanations are coding commentary.
- **§8 Glossary.** **Grade** — "Includes per-Question correctness (determined by whether the learner's code passed the Question's tests), per-Question explanation of why the learner's approach worked or didn't, an aggregate score, identified Weak Topics, and recommended Sections to re-read." This ADR realizes three of the four facets directly (explanation, score, weak topics, recommended sections); the fourth (correctness) is read from the runner's verdict and written by ADR-050's persistence. The workflow schema has no `is_correct` field, by construction. **Weak Topic** — "Drives the fresh-Question portion of subsequent Quizzes for the same Section" — the `weak_topics: list[str]` output is the per-Attempt list; the relational vocabulary is deferred to the composition slice (ADR-050).

No manifest entries flagged as architecture-in-disguise for this decision. The schema-design choices (`min_length=1`, `extra="forbid"`, `Field(ge=0)`), the `prompt_fn`-over-`prompt_template` choice, the single-`LLMStep`-over-multi-step choice, and the tier-registry shape are operational architecture that the manifest delegates to "the architecture document" (manifest scope: "Technology choices, frameworks, data models, integration patterns, and implementation policies live in the architecture document").

## Consequences

**Becomes possible:**

- A `submitted` Attempt can be graded by invoking `aiw run grade_attempt --input ... --run-id attempt-{attempt_id}` (the processor's concern — ADR-049). The terminal artefact carries the per-Question explanations and the aggregate (`score`, `weak_topics`, `recommended_sections`); the persistence layer (ADR-050) maps the runner's verdict into `is_correct` and stores the aggregate.
- §8 Grade-correctness is honored architecturally — the LLM cannot disagree with the runner's verdict because the schema has no field through which to disagree. A reviewer can verify this by reading the schema and the prompt; it's not a hope about LLM behavior, it's a guarantee about the data model.
- The reinforcement loop's downstream half can close — the Weak Topics list is produced per Attempt; the composition slice (next) reads it and drives the fresh-Question portion of the next Quiz (MC-8); the wrong-answer-replay history (ADR-050's `is_correct` writes) drives the replay portion.
- The same `aiw run` CLI invocation pattern from ADR-036 extends to a second workflow without architectural novelty — the processor (ADR-049) shells out exactly as `process_quiz_requests` does for `question_gen`.

**Becomes more expensive:**

- `app/workflows/` grows a second `WorkflowSpec` module (`grade_attempt.py`). Mitigation: mirrors `question_gen.py`'s structure section-for-section; one prompt to author, three Pydantic models, one tier registry, one `WorkflowSpec` registration.
- The `AIW_EXTRA_WORKFLOW_MODULES` env var the processors pass becomes a comma-separated list (`app.workflows.question_gen,app.workflows.grade_attempt`); both processors set the same value so either workflow can be invoked from either subprocess. Mitigation: trivial — a string concatenation in two places (the existing `process_quiz_requests` invocation gets the new value; ADR-049's `process_quiz_attempts` invocation gets the same value).
- One more LLM tier in the project's cost line item (the grading model — defaulting to the same `gemini/gemini-2.5-flash` as `question_gen`, so likely shared key, but conceptually a separate cost stream). Mitigation: the env-var precedence (`GRADE_ATTEMPT_MODEL`) makes the model trivially swappable without touching the workflow code.

**Becomes impossible (under this ADR):**

- The grading LLM disagreeing with the runner's correctness verdict (the schema has no field through which to do so).
- A workflow output with an empty `QuestionGrade.explanation` (the `min_length=1` validator rejects it; the framework retries within `RetryPolicy`'s bound; on exhaustion the processor flips `grading_failed` — ADR-049).
- A workflow output with extra non-coding fields (`extra="forbid"` rejects them; the LLM cannot inject a `correct_choice` / `recall_answer` / `chat_turn` field).
- A silent unbounded retry of a permanently-failed grade (the `RetryPolicy` is bounded; on exhaustion the processor produces `grading_failed`; MC-5).
- A direct LLM/agent SDK import in `app/workflows/grade_attempt.py` (the import surface is `ai_workflows.*` + `pydantic` + stdlib; the boundary grep enforces it).

**Future surfaces this ADR pre-positions:**

- **The composition slice** — reads `attempt_questions.is_correct` (per-Question wrong-answer-replay history, written by ADR-050 from the runner's `test_passed`) and the per-Attempt `weak_topics` list (written by ADR-050 from this workflow's output) to compose the next Quiz's replay + fresh portions (MC-8). The relational Topic vocabulary migration (deferred from this slice) lands then.
- **A second-Attempt feedback loop** — a Grade for an Attempt now exists in `data/notes.db`; a future surface could show "your Grade across Attempts" (a trend). Not in scope; the data is there.
- **A `provenance` field on `GradeAttemptOutput`** — e.g. `grader_model: str`, `grader_run_id: str` — for the author to know which LLM produced which Grade. Not in scope; can be added additively to the schema later (with `extra="forbid"` honored — the schema migration is a new ADR if it lands).
- **A different model per tier** — easy via the env-var precedence chain; no architectural change.

**Supersedure path if this proves wrong:**

- If the LLM's `score` consistently miscounts and ADR-050's cross-check overrides too often, a future ADR could drop `score` from `GradeAttemptOutput` entirely (the persistence layer computes it from `test_passed`); the schema migration is additive (remove a field, tighten `extra="forbid"`). Bounded.
- If single-`LLMStep` proves to produce inconsistent or low-quality explanations, a future ADR could split it into a per-Question step + an aggregate step (Alternative B); the `WorkflowSpec` shape is replaceable in place. Bounded.
- If `gemini/gemini-2.5-flash` proves underpowered for grading, swap the model via `GRADE_ATTEMPT_MODEL` — no architectural change; a tuning decision.
- If a future need for a stricter per-Question correctness signal surfaces (e.g. an LLM-side "did the test fail because the test was wrong?" affordance), that's a future ADR — and it adds a *new* field with a clear semantic, not a re-judgment of the runner's verdict; the architectural commitment "correctness = test result" stays.

The supersedure path runs through new ADRs. This ADR does not edit any prior ADR in place; it consumes ADR-036 (the integration shape), ADR-040 (the precedent), ADR-042–ADR-044 (the runner's persisted output the workflow input reads), ADR-033 (the lifecycle states the persistence transitions over, ADR-050's concern), and ADR-039 (the Attempt-lifecycle persistence the persistence layer extends) unchanged.
