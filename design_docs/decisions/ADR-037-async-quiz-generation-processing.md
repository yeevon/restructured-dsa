## Context

TASK-014 ships the *processor* for the Quiz pillar's generation half. ADR-036 decided the `ai-workflows` integration shape — a CS-300-owned `WorkflowSpec` module (`app/workflows/question_gen.py`), invoked by the out-of-band processor **shelling out to `aiw run question_gen --input section_content=... --input section_title=... --run-id ...`** as a subprocess (with `AIW_EXTRA_WORKFLOW_MODULES=app.workflows.question_gen` in the subprocess env — the framework's documented downstream surface), reading the terminal artefact (`{"questions": [{"prompt": "...", "topics": [...]}, ...]}`) from the CLI's stdout; the CS-300 processor adapts that artefact and persists via `app.persistence.*`. This ADR decides the *CS-300 side*: where the processing runs (it must be out-of-band — never inside a request handler — MC-4 / §6), how it is triggered, how the learner is notified the result is ready, and the failure-handling discipline.

The relevant established context: ADR-034 ships a real `POST /lecture/{chapter_id}/sections/{section_number}/quiz` route that inserts one `quizzes` row with `status = 'requested'` (no AI call, no background job) and PRG-redirects (303) to `GET /lecture/{chapter_id}#section-{section_number}-end` (ADR-031's `#section-{n-m}-end` + `scroll-margin-top` no-relocate recipe). ADR-033 reserves the `quizzes.status` enum `requested` → `generating` → `ready` → `generation_failed`, all stored as TEXT, additive-extensible; the `requested` status is explicitly "the user clicked, we recorded the request — no AI call has been made". ADR-034's per-Section Quiz surface already renders all four states in plain language (`requested` → "Requested — generation pending"; `generating` → "Generating…"; `ready` → "Ready" with no takeable affordance; `generation_failed` → "Generation failed") — this task makes the `generating` / `ready` / `generation_failed` labels actually appear by walking the lifecycle. ADR-022's migration story: `CREATE TABLE IF NOT EXISTS` / nullable `ALTER TABLE ADD COLUMN` are additive; a non-additive change forces a follow-up ADR. The project is single-user, single-machine, no CI, locally run, no remote deployment (§5).

The decision space has materially different alternatives:

- **Where the processing runs (the out-of-band mechanism):**
  - **A manual command** (`python -m app.workflows.process_quiz_requests` — the author runs it after clicking Generate; or runs it periodically). Simplest; honest ("you submitted; the processing happens when you run the processor; then it's ready"); zero concurrency surface; no "the app must stay running" coupling; no worker lifecycle.
  - **A background worker started at app launch** (a thread or `asyncio` task that polls `requested` rows; more "automatic" — the Quiz appears without the author running a command — but adds: concurrency on the SQLite connection, an "the FastAPI process must stay running for generation to happen" coupling, a worker that must not crash the app, and the question of what happens to in-flight work on shutdown).
  - **A separate long-running process** (a dedicated daemon). All the worker's complexity plus a second process to manage. Over-scoped.
  - **A heavyweight job queue / broker** (Celery, RQ, a message broker). Wildly over-scoped for a single-user, no-CI, locally-run project — the task and the manifest's "no remote deployment / no hosted product" (§5) both cut against it. Not a serious contender; named only to be rejected.
- **How the trigger handoff works:** does `POST .../quiz` *enqueue* the workflow (signal the processor — start a thread, write to a queue, fire-and-forget an `asyncio` task)? Or does the processor poll `requested` rows independently (the route just records the row; the processor finds it on its next pass)? Either is acceptable so long as the route does not block on the AI call (MC-4).
- **How the learner is notified the result is ready:** (a) the per-Section Quiz status-list display flipping to "Ready" (already shipped, ADR-034) *is* the "notified" mechanism for this slice — the learner sees "Ready" on their next visit to the Section; or (b) ship the Notification entity (the `notifications` table ADR-022 forecast + ADR-033 deferred, plus a learner-visible Notification surface — a rail decoration / a "you have N new Quizzes" indicator / a page).
- **Failure-detail persistence:** persist the failure detail (a `quizzes.generation_error TEXT` column — additive); or don't (the learner-facing signal is just "Generation failed" regardless, and a single-user dev project can read the processor's stderr/logs).
- **The first-Quiz-only guard (MC-8):** reject a second `requested` request at the route level (the row never exists); leave a second `requested` row unprocessed for the future composition slice; assert in the processor that the Section's bank is empty before generating (process-side guard). Any prevents a fresh-only post-first Quiz; *processing a second request through the fresh-only path* is the bug to avoid.

The manifest constrains the decision through §5 ("No live / synchronous AI results … arrives when ready and is surfaced via Notification — not in real time"; "no remote deployment / no hosted product" — bounds the simplicity ceiling: no broker, no daemon-farm), §6 ("AI work is asynchronous from the learner's perspective. Submission, processing, and result delivery are decoupled in time. The learner submits, continues working, and is notified when results are ready"; "AI failures are visible … never fabricates a result"; "single-user"), §7 ("Every post-first Quiz for a Section contains both replayed wrong-answer Questions and freshly-generated Questions … The first Quiz for a Section contains only fresh Questions" — the first-Quiz-only guard is load-bearing here; "Quiz generation is always explicitly user-triggered. The system never auto-generates a Quiz in response to background events" — the processor *processes* user-created `requested` rows, it never *creates* one), §8 (Notification — "a learner-visible indication that an async AI result has become available"; the architect must reason whether the passive status-flip satisfies this for this slice).

## Decision

### The out-of-band mechanism — a manual command, `python -m app.workflows.process_quiz_requests`

The Quiz-generation processing runs in a **manual command**: `python -m app.workflows.process_quiz_requests` — a module under the `app/workflows/` package (ADR-036's package) with a `__main__` entry point. The author runs it (after clicking "Generate a Quiz", or periodically, or as part of a dev-loop habit). It is **not** a background thread/task in the FastAPI process; it is **not** a separate long-running daemon; it is **not** a job queue.

What the command does, each invocation:

1. Query `app.persistence.list_requested_quizzes()` (the read accessor ADR-036 forecast) — the `quizzes` rows with `status = 'requested'`. (Optionally bounded — "process up to N this run" — implementer's call; for a single-user dev project the set is small.)
2. For each such Quiz:
   a. `app.persistence.mark_quiz_generating(quiz_id)` — `status` → `'generating'`.
   b. Read the Section's LaTeX content read-only (`content/latex/{chapter_id}.tex` — the `chapter_id` is the prefix of the Quiz's `section_id` per ADR-002; via the existing `app.parser` path — MC-6: read-only). Compose the `--input section_content=...` / `--input section_title=...` args (the `QuestionGenInput` fields — ADR-036).
   c. Invoke the workflow via the documented `aiw run` CLI subprocess (ADR-036's revised §How application code invokes the workflow): `subprocess.run(["aiw", "run", "question_gen", "--input", f"section_content={...}", "--input", f"section_title={...}", "--run-id", f"quiz-{quiz_id}"], env={**os.environ, "AIW_EXTRA_WORKFLOW_MODULES": "app.workflows.question_gen"}, capture_output=True, text=True)` — or the PATH-independent `[sys.executable, "-m", "ai_workflows.cli", "run", ...]` form. On exit-0, parse the terminal artefact from stdout (`json.dumps(artifact, indent=2)` then a `total cost: $X.XXXX` trailer; structured logs on stderr) → `{"questions": [{"prompt": "...", "topics": [...]}, ...]}`. On a non-zero exit / an `error: ...` line on stderr, treat as failure (step e). **Not** an in-process `run_workflow` import (ADR-036 rejected that). (`asyncio.run(...)` is *not* needed — the dispatch happens inside the subprocess.)
   d. On exit-0 and the artefact non-empty (≥1 item with a non-empty `prompt`): `app.persistence.add_questions_to_quiz(quiz_id, ...)` then `app.persistence.mark_quiz_ready(quiz_id)`.
   e. On a non-zero `aiw run` exit / an `error: ...` on stderr (or the artefact is empty / fails CS-300's sanity check): `app.persistence.mark_quiz_generation_failed(quiz_id, error=<the stderr `error:` message, or a brief description>)`. **No fabricated Questions** (zero `questions` rows inserted on this path). **No silent unbounded retry** — the framework's `RetryingEdge` already does *bounded* retry inside the workflow run (the subprocess); the CS-300 command does *not* wrap the `aiw run` call in a retry loop or a "keep retrying `generation_failed` rows forever" pass (a `generation_failed` row stays `generation_failed`; the author re-clicks "Generate a Quiz" if they want another attempt — which is a *new* `requested` row, the only way generation re-fires, MC-9). (MC-5.)
3. Log a one-line summary per Quiz (processed → `ready`, or failed → `generation_failed` with the error) to stderr. (Single-user dev: the author reads it.)

**Why a manual command, not a background worker:**

- **Simplicity, and it's honest.** A single-user, no-CI, locally-run project does not need a production-grade async pipeline. The manifest *forbids* real-time AI (§5) and *requires* async (§6) — it does not require automatic delivery. A manual command is the simplest thing that satisfies MC-4 (the `POST .../quiz` route does not block on the AI call — it returns 303 immediately; the processing happens when the author runs the command) and MC-9 (the command *processes* existing `requested` rows; it never *creates* one — only the user's explicit click does that). The semantics are honest: "you submitted; the processing happens when you run the processor; then the Quiz is ready and you'll see 'Ready' next time you visit the Section."
- **No concurrency surface, no app-lifetime coupling, no worker-crash blast radius.** A background thread/task in the FastAPI process would add: a second writer on the SQLite connection (ADR-022 explicitly chose stdlib `sqlite3` with "no concurrency story — single user, single process"); a coupling where generation only happens if the FastAPI process is running and stays running (the author might run the app, click Generate, stop the app, and the Quiz never generates — a confusing failure mode); a worker that must not crash the app (an uncaught exception in the worker thread, or a hung `aiw run` subprocess, becomes a question about the app's reliability); and a shutdown-handling question (what happens to a Quiz in `generating` when the app stops mid-workflow?). The manual command sidesteps all of it — it runs in its own process, against its own short-lived SQLite connections (the existing `app.persistence.get_connection()` open/use/close pattern), and if it crashes, the app is unaffected (the Quiz it was processing is left in `generating` — a future "stale `generating` reset" affordance can be added if that proves a problem; for now, the author re-clicks Generate, getting a new `requested` row).
- **The architecture-on-spec test applies** (the task flagged it): build the *simplest* out-of-band mechanism that satisfies MC-4 / MC-9, not the most "scalable" one. A background worker is "more automatic" but adds real surface for a single-user dev project; a job queue/broker is over-scoped by orders of magnitude. The manual command is the right level.
- **CLAUDE.md's "Commands" section may gain a line** — "Process pending Quiz requests: `python -m app.workflows.process_quiz_requests`" — but **the architect does not edit CLAUDE.md** (it is human-owned, Tier 2). The architect flags it here so it is not missed; the human owns that edit, in the same commit or a follow-up. (This is a process-gap fill, not architecture — the same kind of edit as filling the lint/type-check placeholders.)

### The trigger handoff — the `POST .../quiz` route is unchanged; the processor polls `requested` rows independently; no enqueue

The `POST /lecture/{chapter_id}/sections/{section_number}/quiz` route (ADR-034) is **unchanged**: it validates, calls `app.persistence.request_quiz(section_id)` (inserts the `requested` row — no AI call, no background job), and PRG-redirects (303) to `#section-{section_number}-end`. It does **not** enqueue the workflow, signal the processor, start a thread, or fire-and-forget an `asyncio` task. The processor (`python -m app.workflows.process_quiz_requests`) polls `requested` rows independently when the author runs it.

**Why no enqueue:** an enqueue from the route (start a thread, write to an in-process queue, fire an `asyncio` task) is the seed of a background worker — it re-introduces the concurrency surface and app-lifetime coupling the manual-command decision avoids, for the dubious benefit of "the processor doesn't have to poll" (it polls a small table — trivial at single-user scale). The route's job is to record the user's request honestly and return; the processor's job is to find and process recorded requests. Keeping them decoupled (the route writes; the processor reads, independently, on the author's trigger) is the clean separation, and it keeps the route's "returns 303 immediately, no AI processing inside the request" guarantee (MC-4) trivially true — there is nothing in the route that could block.

### Failure-handling discipline — `generation_failed` on failure; the detail persisted in a nullable `quizzes.generation_error TEXT` column

On the `ai-workflows` call failing (`result["status"]` ∈ `{"errored", "aborted"}`, or an empty/invalid artifact): `app.persistence.mark_quiz_generation_failed(quiz_id, error=result.get("error"))` — sets `quizzes.status = 'generation_failed'` (ADR-033 reserves it) **and** writes the error string into a new nullable column **`quizzes.generation_error TEXT`** (added via `ALTER TABLE quizzes ADD COLUMN generation_error TEXT` — nullable, additive per ADR-022's migration story; no migration-trigger fires; the column is `NULL` for every non-failed Quiz). The `mark_quiz_generation_failed` signature gains an optional `error: str | None = None` parameter; SQL stays in `app/persistence/quizzes.py` (MC-10).

**Why persist the failure detail:** for a single-user dev project, "Generation failed" on the surface is the honest learner-facing signal (MC-5 — never fabricated, never dressed up as a success or a transient), but the *author* (who is also the learner) wants to know *why* it failed — was it a provider rate-limit, a malformed LLM response that failed schema validation, a timeout? Persisting the `error:` message from `aiw run`'s stderr (the framework's descriptive error string) means the author can read it from the SQLite shell (or a future "show me failed generations" affordance) instead of having to correlate with the processor's stderr log (which they might not have kept). It is additive (a nullable column), so it costs nothing in migration pain. The **learner-facing signal stays "Generation failed"** — the `generation_error` detail is a debugging aid, not surfaced as the learner-facing message (a future task could add a "details" expander, but that's not this slice; ADR-034's "Generation failed" label is unchanged).

**No silent unbounded retry** (MC-5: "silent retries that mask permanent failure"): the framework's `RetryingEdge` does *bounded* retry *inside* the workflow run (the subprocess — the workflow's `RetryPolicy`: transient cap, semantic cap, backoff); a route's worth of CS-300 code does *not* wrap the `aiw run` subprocess call in a retry loop, and the `process_quiz_requests` command does *not* re-process `generation_failed` rows on a later run (a `generation_failed` row stays `generation_failed`). If the author wants another attempt, they re-click "Generate a Quiz for this Section" — a *new* `requested` row, processed afresh; the surface then shows a new Quiz row alongside the failed one. (A future task could add a "retry this generation" affordance on a `generation_failed` Quiz — but that's a *user-triggered* retry, MC-9-compliant; an *automatic* retry of a permanently-failed generation is what MC-5 forbids, and this ADR introduces none.)

### How the learner is notified — for this slice, the per-Section status-flip to "Ready" *is* the "notified" mechanism; the active Notification entity ships with the grading slice

**For TASK-014, the per-Section Quiz surface's status-list display flipping to "Ready" (already shipped, ADR-034) is the "notified when results are ready" mechanism — no Notification entity ships this task.** When the processor walks a Quiz to `ready`, the per-Section surface (rendered on the learner's next `GET /lecture/{chapter_id}`) shows that Quiz's status as "Ready" instead of "Generating…" / "Requested — generation pending". That *is* "a learner-visible indication that an async AI result has become available" (§8's Notification glossary): it is learner-visible; it indicates the result (a `ready` Quiz with its Questions) is available; and it is *not real-time* — which is exactly what the manifest mandates (§5: "Output … arrives when ready and is surfaced via Notification — not in real time at the moment of request").

**Reasoning — is the passive status-flip honestly "notified"?** This is the subtlest call in the task. Two readings:

- **Yes, for this slice.** The learner just clicked "Generate a Quiz for this Section" on *that Section* — they know exactly where to look for the result (the same `.section-quiz` block they clicked the button in). When they next visit that Section (or refresh the page they're on), they see "Ready". The §8 glossary's "learner-visible indication that an async AI result has become available" is satisfied by a learner-visible status the learner will see when they return to the surface they initiated the action from. The manifest *forbids* real-time delivery — so the obligation cannot be "the learner is told the instant it's ready"; it must be "there is a learner-visible signal the learner will encounter". The status-flip is that signal.
- **Not quite — a "Notification" connotes something more active** (a "you have 1 new Quiz" badge surfaced *where the learner is*, not *where the result lives*). True — but a Notification surface *earns its keep* when (a) there is more than one *kind* of async result to notify about (a Grade *and* a Quiz), and (b) the learner is plausibly *elsewhere* when the result lands (browsing a different Chapter, not the Section they requested generation for). For first-Quiz generation, neither holds yet: there is one kind of result (a `ready` Quiz), and the learner just acted on *that Section* so they're plausibly still near it. A Notification surface now would be a single-purpose badge with one thing to show — and the moment the *grading* slice lands (a Grade is another async result, and the learner may well be elsewhere after submitting an Attempt and continuing to work), a Notification surface has *two* kinds of result to project and a real "the learner is elsewhere" case — that is when it earns its keep.

So the call: **the per-Section status-flip is the sufficient "notified" mechanism for the first-Quiz-generation slice; the active Notification entity** (the `notifications` table ADR-022 forecast + ADR-033 deferred — a `notification_id`, a `kind` describing what became available, a `payload`/reference, a `created_at`, **no `user_id`** — additive DDL; plus a learner-visible Notification surface — a rail decoration or a "you have N new …" indicator, with CSS scoped per ADR-008) **ships with the grading slice**, where there is a Grade to notify about too and a real "the learner is elsewhere" case. This is recorded here as the call, with the reasoning, per the task's instruction ("name the call in 'My recommendation vs the user's apparent preference'"). It is *not* escalated to `> NEEDS HUMAN` — the task's framing grants the architect the authority ("a defensible 'the status-flip is sufficient for this slice, the Notification entity ships with the grading slice when there's a Grade to notify about too' is fine"), and that is the defensible call.

**In no case is a stand-in Notification fabricated** (MC-5): the per-Section surface shows "Ready" *only* when `quizzes.status = 'ready'` actually holds (which the processor sets only after `add_questions_to_quiz` succeeds) — never a "your Quiz is ready" when it is not; and when the Notification entity does land (grading slice), its rule is the same — a Notification row for a Quiz is produced *only* when that Quiz reaches `ready`, never speculatively.

### The first-Quiz-only guard (MC-8) — reject a second `requested` request at the route level

The "Generate a Quiz for this Section" route (`POST .../quiz`) gains a guard: **before inserting a new `requested` row, check whether the Section already has a non-failed Quiz** (a Quiz with `status` ∈ `{'requested', 'generating', 'ready'}` — i.e. any Quiz that isn't `generation_failed`). If it does, **reject the request** — return an HTTP status (e.g. `409 Conflict`) with a message ("This Section already has a Quiz request in progress or completed; the post-first-Quiz composition surface is not yet available") rather than inserting a second `requested` row. The check uses a new persistence accessor (e.g. `section_has_nonfailed_quiz(section_id) -> bool`, or it can be derived from the existing `list_quizzes_for_section(section_id)`) — SQL in `app/persistence/quizzes.py` (MC-10); validation at the route handler (ADR-024's split — the persistence layer trusts the caller).

**Why the route-level reject (not "leave a second `requested` row unprocessed", not "process-side bank-empty assertion"):**

- **It is the cleanest place to enforce "this task ships only the first-Quiz case".** The route is where a Quiz request is born; rejecting a second request at birth means the `requested` table only ever holds first-Quiz requests this task — the processor doesn't have to know about a "process only if the bank is empty" rule, and there's no orphan `requested` row sitting around confusing the per-Section surface (which would render it as "Requested — generation pending" with no processor that will ever process it correctly this task).
- **It honors MC-8 by construction.** A fresh-only *post-first* Quiz is a bug (MC-8); the only Quiz this task's generation path produces is fresh-only; so the only way to keep MC-8 clean is to ensure the generation path never runs for a Section that already has a Quiz — and rejecting the second request at the route does exactly that. The post-first composition slice (later) builds the replay-+-fresh path and *then* a second request gets accepted and processed correctly (the composition slice's `/design` decides whether the route guard relaxes to "allow a second request if the composition path exists" or whether a different affordance — "request another Quiz" with the composition logic behind it — replaces the simple button).
- **A `generation_failed` Quiz does *not* count against the guard** — the author can re-click "Generate a Quiz" on a Section whose only Quiz is `generation_failed` (a fresh attempt; the failed row stays as a historical record per ADR-033's "never deleted" posture). So the guard is "the Section has a Quiz that isn't `generation_failed`" — not "the Section has any Quiz". (This means a Section's `.section-quiz` list can show a `generation_failed` Quiz *and* a new `requested`/`generating`/`ready` Quiz — which is correct: the failed one is history; the new one is the live attempt.)
- **Rejected alternatives:** "leave a second `requested` row unprocessed for the composition slice" — leaves an orphan row the per-Section surface renders misleadingly ("Requested — generation pending" forever, this task); "process-side bank-empty assertion" — pushes the guard into the processor (which then has to skip rows it can't handle, and the route still let a doomed row in). Route-level reject is the clean call. (TASK-014's AC says "the only acceptable outcomes are 'rejected at request' or 'left in `requested`'" — this ADR picks "rejected at request".)

### Scope of this ADR

This ADR fixes only:

1. The out-of-band mechanism: a manual command `python -m app.workflows.process_quiz_requests` (a `__main__` module under `app/workflows/`) — poll `requested` rows, walk each `requested` → `generating` → run the workflow (ADR-036's dispatch) → `ready` (persist Questions + `quiz_questions`) or `generation_failed`; log a per-Quiz summary to stderr. **Not** a background worker; **not** a daemon; **not** a job queue.
2. The trigger handoff: the `POST .../quiz` route (ADR-034) is unchanged — no enqueue, no signal; the processor polls `requested` rows independently on the author's trigger.
3. The failure-handling discipline: `generation_failed` on failure; the detail persisted in a new nullable `quizzes.generation_error TEXT` column (additive `ALTER TABLE ADD COLUMN`); `mark_quiz_generation_failed` gains an optional `error` parameter; **no fabricated Questions; no silent unbounded retry** (the framework's `RetryingEdge` does bounded retry inside the workflow; the CS-300 command does not re-process `generation_failed` rows); the learner-facing signal stays "Generation failed" (ADR-034's label unchanged).
4. The "notified" mechanism for this slice: the per-Section Quiz status-flip to "Ready" (already shipped, ADR-034) — **no Notification entity ships this task**; the active Notification entity (the `notifications` table + a learner-visible surface) ships with the grading slice (when there's a Grade to notify about too and a real "the learner is elsewhere" case). In no case is a stand-in Notification fabricated (the surface shows "Ready" only when `quizzes.status = 'ready'` actually holds).
5. The first-Quiz-only guard (MC-8): the `POST .../quiz` route rejects (HTTP 409) a second request for a Section that already has a non-failed Quiz (`status` ∈ `{'requested', 'generating', 'ready'}`); a `generation_failed` Quiz does not count against the guard (the author can re-click after a failure); a new persistence accessor backs the check (SQL in `app/persistence/quizzes.py`).
6. The flag (not an edit — the architect does not own CLAUDE.md): CLAUDE.md's "Commands" section should gain "Process pending Quiz requests: `python -m app.workflows.process_quiz_requests`" — the human owns that edit.

This ADR does **not** decide:

- The `ai-workflows` integration shape (the workflow module, the `aiw run` CLI-subprocess invocation path, the test seam, the forbidden-SDK list, the PyPI dependency form) — owned by ADR-036.
- The Notification entity's schema or surface — deferred to the grading slice's `/design` (this ADR records that it ships *then*, with the reasoning, not *now*).

## Alternatives considered

**A. A background worker started at app launch (a thread or `asyncio` task that polls `requested` rows).**
Considered — it's "more automatic" (the Quiz appears without the author running a command). Rejected for a single-user, no-CI, locally-run project because it adds real surface for marginal benefit: (a) a second writer on the SQLite connection (ADR-022 chose stdlib `sqlite3` with explicitly "no concurrency story"); (b) an app-lifetime coupling — generation only happens if the FastAPI process is running and stays running, so "run the app, click Generate, stop the app" silently never generates the Quiz (a confusing failure mode); (c) a worker that must not crash the app — an uncaught exception or a hung `aiw run` subprocess in the worker thread becomes a question about the app's reliability; (d) a shutdown-handling question — what happens to a Quiz in `generating` when the app stops mid-workflow. The manual command sidesteps all of it (own process, own short-lived connections, crash-isolated from the app). The task explicitly flagged "don't over-build the async mechanism … build the simplest out-of-band mechanism that satisfies MC-4 / MC-9, not the most 'scalable' one" — the manual command is that. (If, post-TASK-014, the manual-command UX proves too friction-heavy — the author tires of running the command — a future ADR can add a thin background worker *then*, with the concurrency/lifetime tradeoffs understood; building it now is premature.)

**B. The `POST .../quiz` route enqueues the workflow (fire-and-forget an `asyncio` task / start a thread / write to an in-process queue).**
Rejected. An enqueue from the route is the seed of a background worker (Alternative A's complexity, introduced via the route instead of at app launch) — and it makes the route's "returns 303 immediately, no AI processing inside the request" guarantee (MC-4) less obviously true (a fire-and-forget task started from the request *is* AI processing kicked off by the request, even if the request doesn't await it — and a hung or crashing fire-and-forget task is a request-handler reliability question). Keeping the route purely "record the request, return 303" and the processor purely "poll and process, on the author's trigger" is the clean separation that makes MC-4 trivially satisfied. The benefit of an enqueue ("the processor doesn't have to poll") is nil at single-user scale (it polls a small table).

**C. A heavyweight job queue / broker (Celery, RQ, a message broker).**
Rejected outright. Wildly over-scoped for a single-user, no-CI, locally-run project. The manifest's "no remote deployment / no hosted product" (§5) cuts directly against running a broker; the task explicitly named this anti-pattern ("don't reach for a heavyweight queue (Celery, RQ, a broker) — that is wildly over-scoped"). Named only to be rejected.

**D. Ship the Notification entity (the `notifications` table + a learner-visible Notification surface) this slice.**
Considered — the task names it as a defensible option. Rejected for *this* slice: a Notification surface earns its keep when (a) there's more than one *kind* of async result to notify about (a Grade *and* a Quiz) and (b) the learner is plausibly *elsewhere* when the result lands. For first-Quiz generation, neither holds yet — one kind of result (a `ready` Quiz), and the learner just acted on *that Section* so they're plausibly still near it; the per-Section status-flip ("Ready" appears where they clicked Generate) is a sufficient "learner-visible indication" (§8). The grading slice — where a Grade is a second kind of result and "the learner submitted an Attempt and continued to work elsewhere" is a real case — is when the Notification entity earns its keep; this ADR records that it ships *then*. Shipping it now would be a single-purpose badge with one thing to show, and would push TASK-014 (already the largest slice the project has attempted) over the edge for no offsetting benefit. (The task's "Architectural concerns" explicitly endorses this: "the defensible middle: the status-flip is sufficient *for this slice* … the active Notification entity ships with the *grading* slice".)

**E. Don't persist the failure detail (no `quizzes.generation_error` column).**
Considered — the learner-facing signal is just "Generation failed" regardless, and a single-user dev project can read the processor's stderr. Rejected mildly: persisting `result["error"]` in a nullable additive column costs nothing (no migration-trigger, NULL for every non-failed Quiz) and saves the author from having to correlate a failure with a stderr log they might not have kept — they can read it from the SQLite shell (or a future "show me failed generations" affordance). The learner-facing message stays "Generation failed" (the `generation_error` is a debugging aid, not surfaced); MC-5 is honored (no fabrication; the failure is surfaced as a failure). Cheap, useful, additive — ship it.

**F. The first-Quiz-only guard as a process-side bank-empty assertion (the processor skips `requested` rows whose Section already has Questions), instead of a route-level reject.**
Rejected. A process-side assertion still lets the route insert a doomed `requested` row, which the per-Section surface then renders as "Requested — generation pending" forever (the processor will keep skipping it). The route-level reject is the clean place — a second request never becomes a row; the `requested` table only ever holds first-Quiz requests this task; the processor doesn't need a "skip if bank non-empty" rule; the per-Section surface never shows an orphan request. (TASK-014's AC permits "rejected at request" or "left in `requested`" — the route-level reject is the "rejected at request" branch, which is the cleaner one.)

**G. Allow a second `requested` row but leave it unprocessed (the "left in `requested`" branch).**
Considered (the task's AC permits it). Rejected as worse than the route-level reject: an unprocessed `requested` row sits in the per-Section list rendered as "Requested — generation pending" with no processor that will handle it correctly this task — a misleading state. The route-level reject (HTTP 409 with a clear message) tells the author *at the moment they click* "this Section already has a Quiz; the post-first composition surface isn't available yet" — honest and immediate, vs. an orphan row that looks like it's queued but isn't.

- The composition slice's relaxation of the first-Quiz-only guard — when the replay-+-fresh path exists, a second request for a Section with a non-failed Quiz is accepted (or a different affordance — "request another Quiz" with the composition logic behind it — replaces the simple button); the composition slice's `/design` owns that.

**Supersedure path if this proves wrong:**

- If the manual-command UX proves too friction-heavy (the author tires of running the command) → a future ADR adds a thin background worker (a thread/task that polls `requested` rows on a timer), accepting the concurrency/app-lifetime tradeoffs; the persistence transitions and the `aiw run` subprocess call are reused unchanged. Cost: bounded — the worker is a wrapper around the same processing loop.
- If the per-Section status-flip proves an insufficient "notified" mechanism even for this slice (the human disagrees at the gate; the author wants a Notification *now*) → ship the `notifications` table + a minimal Notification surface this task instead of deferring to the grading slice. Cost: an additive table + a small surface (CSS scoped per ADR-008) — bounded; this ADR records the deferral as the architect's call, not a hard constraint.
- If the route-level first-Quiz-only guard proves too strict (the composition slice lands and the guard needs relaxing, or a Section needs to allow a second request before composition exists for some reason not foreseen) → the composition slice's `/design` (or a future ADR) relaxes it. Cost: bounded — the guard is one persistence check + a 409 branch.
- If `quizzes.generation_error` proves the wrong shape (a structured error object is wanted instead of a string; or it should be a separate `generation_failures` table with a history) → a future ADR refines; the additive column stays as a denormalized last-error cache, or a new table is added (`CREATE TABLE IF NOT EXISTS` — additive). Cost: bounded.

The supersedure path, in every case, runs through a new ADR. This ADR does not edit any prior ADR in place; it builds on ADR-036 (the integration shape), consumes ADR-033 (the `generation_failed` state; the additive-migration story for the new column) and ADR-034 (the per-Section surface; the `requested`-row trigger route, which gains only the first-Quiz-only guard) unchanged, and reuses ADR-031's no-relocate redirect recipe on the successful path unchanged.
