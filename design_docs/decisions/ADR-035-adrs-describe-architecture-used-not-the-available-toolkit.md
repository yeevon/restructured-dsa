# ADR-035: ADRs describe the architecture *used*, not a restriction of the available toolkit — JavaScript is part of the toolkit; no-JS / CSS-only solutions are preferred where clean and sufficient, never mandatory; "no JavaScript" is not a project invariant

**Status:** `Accepted`
Accepted by the human on 2026-05-11 (human-directed cleanup arising from TASK-013 review).
**Date:** 2026-05-11
**Task:** none (human-directed architecture-hygiene cleanup arising from TASK-013's implementer PUSHBACK and the human's explicit instruction — see "Context")
**Resolves:** none (no project_issue was filed; this ADR closes an architecture smell the human identified directly)
**Supersedes:** none — this ADR does **not** reverse any prior decision. The decisions in ADR-006 / ADR-023 / ADR-025 / ADR-027 / ADR-028 / ADR-029 / ADR-030 / ADR-031 all stand exactly as made (synchronous form POST + PRG redirect for Notes and section-completion; `#section-{n-m}-end` + `scroll-margin-top` for the no-relocate redirect; `field-sizing: content` + `resize: vertical` for the textarea; etc.). What this ADR corrects is the *framing* in those ADRs that treated "no JavaScript" as a standing prohibition / "the project's posture" / a constraint future work must honor. The accreted restrictive language in those ADRs has been softened in place to descriptions of what each surface actually does, each with an inline note pointing here; this ADR establishes the principle those notes point to.

> **Read this first.** No binding document establishes a no-JavaScript rule for this project. `design_docs/MANIFEST.md` does not. `CLAUDE.md` does not. `.claude/skills/manifest-conformance/SKILL.md` does not. Yet "no JavaScript" / "the project's no-JS posture" / "synchronous form posting throughout" accreted across ADR-006 / ADR-023 / ADR-025 / ADR-027 / ADR-028 / ADR-029 / ADR-030 / ADR-031 as if it were an invariant — with ADR-034 (TASK-013) inheriting the same framing and citing a chain of those ADRs for an "invariant" none of them had the authority to establish. That is the architecture smell this ADR closes. ADRs record the architecture the project **uses**; they do not amputate the toolkit. JavaScript is part of the toolkit. A no-JS / CSS-only solution is **preferred where it is clean and sufficient** — and it has been, for every surface shipped so far — but it is never *mandatory*, and choosing JavaScript when a surface genuinely needs it is a normal architectural step, decided in its own ADR, not the breach of a project-wide ban.

## Decision

1. **ADRs describe the architecture the project uses.** An ADR's job is to record what was chosen and why. It is not to declare what is forbidden in general. A sentence in an ADR of the form "the project's posture is X" / "X is the project-wide commitment" is appropriate only when X is genuinely a binding invariant — which, for "no JavaScript", it is not.

2. **JavaScript is part of the available toolkit.** Client-side JavaScript (and the asset infrastructure to ship it — a `<script>` tag in `base.html.j2`, a flat static file under `app/static/`, ES modules, whatever the surface needs) is available to this project the same way HTML, CSS, Jinja2, FastAPI, and SQLite are. Using it is a normal architectural step.

3. **No-JS / CSS-only solutions are preferred where they are clean and sufficient — not where they are merely possible.** Every surface shipped so far (Notes form, section-completion toggle, the no-relocate redirect, the textarea auto-resize) reached a no-JS solution that was *clean and sufficient*, so it used one. That is the right outcome and those decisions stand. The preference is for the *cleaner* solution at the *appropriate cost* — when that is a CSS trick (ADR-031's `scroll-margin-top`; ADR-028's `field-sizing: content`), great; when a surface genuinely needs JavaScript (ADR-032's Notes-save anchor-on-save, by ADR-032's own honest evaluation), JavaScript is the right tool and gets its own ADR deciding where it lives and how it's tested (and, for progressive enhancement, tested-without).

4. **"No JavaScript" is not a project invariant, and reviewers must not enforce it as one.** A future surface that needs client-side JS is not "breaking the project's posture." It is decided in its own ADR — same as any other architectural choice — on cost-vs-benefit grounds for that surface. The reviewer / conformance skill does not (and must not) reject a decision merely because it introduces JavaScript; there is no manifest, CLAUDE.md, or conformance backing for such a rejection. (What the reviewer *does* still enforce, unchanged, are the actual invariants: MC-1's no-direct-LLM-SDK rule, MC-6's read-only-Lecture-source rule, the persistence-boundary rule, the single-user posture, the placement-quality principles from ADR-027/028/030, etc. Those are real; "no JS" was not.)

5. **This ADR introduces no restriction in the other direction.** It does not mandate JavaScript anywhere. It does not deprecate the no-JS surfaces shipped so far — they are fine as-is and are not to be rewritten to use JS. It does not require any new task. It changes nothing executable. It establishes a framing principle and records that the restrictive framing has been corrected in the ADRs that carried it.

## Alternatives considered

**A. Leave the framing as-is; fix only ADR-034's line 89.**
Rejected. ADR-034 line 89 cited "the project's established posture (ADR-023/025/027/028/029/030/031)" — fixing only line 89 would leave the chain of ADRs it cites still asserting that posture, so the next ADR would re-inherit it. The human's instruction was explicit: *"adr cleanup don't leave that till later"* — the leak is across the set, not just at the leaf. Fixing only the leaf would be a spot-fix that leaves the smell.

**B. Supersede each affected ADR individually with a fresh "this decision's framing was over-stated" ADR.**
Rejected. This would be eight or nine new supersedure ADRs for nothing but a framing correction — the *decisions* in those ADRs are all fine and stand. Supersedure is for changing a decision; this is not a decision change. The human directed an in-place softening ("soften in place to a description of what this surface actually does, with a brief inline note that the change is a framing correction — not a decision reversal"), which is the right mechanism for correcting over-stated *scope* of decisions that otherwise stand. One principle ADR (this one) plus in-place softening of the framing language is the proportionate fix.

**C. Do nothing — "no JavaScript" framing is harmless.**
Rejected. It is not harmless: it already caused ADR-034 to assert an invariant that does not exist, and it already made ADR-032 (the corrective) read as if it were "breaking a posture." Left alone, it would keep biasing future `/design` cycles toward contorted no-JS solutions when a clean JS solution exists, and would keep mis-shaping reviewer expectations. The human identified it as "architecture smell which I have been intentionally designing around" — leaving it would countermand an explicit design intent.

**D. Write the principle into `CLAUDE.md` or the manifest instead of an ADR.**
Rejected — and not the architect's call anyway. `CLAUDE.md` and the manifest are human-owned; the architect does not edit them. More to the point, this *is* an architectural framing principle (it governs how architecture decisions are scoped and what the toolkit is), which is exactly what ADRs are for. If the human later wants a one-line pointer in `CLAUDE.md`'s conventions ("ADRs describe architecture used, not the toolkit — see ADR-035"), that is a human edit; this ADR is the substance it would point to.

## My recommendation vs the user's apparent preference

Aligned with — and directly requested by — the user. The human explicitly asked for this cleanup, named the smell ("a javascriptless project ... would have been architecture smell which I have been intentionally designing around"), specified the mechanism (soften in place + a new principle ADR), and said not to defer it. This ADR records exactly that. There is no disagreement to surface. (If anything, the architect's own prior `/design` cycles are the source of the over-reach being corrected here — the restrictive framing accreted through ADR-025 → 027 → 028 → 029 → 030 → 031 → 034, each cycle citing the prior; the human caught it and corrected the trajectory. The lesson for future `/design` cycles: when an ADR uses a tool sparingly, say "this surface didn't need X," not "the project's posture is no-X" — the former is what happened, the latter is a claim about authority the ADR does not have.)

## Consequences

**What this commits us to:**
- Future ADRs phrase tool *non-use* as "this surface does not need X" / "a no-X solution was clean and sufficient here", not "the project's posture is no-X" — unless X is a genuine binding invariant (manifest / CLAUDE.md / conformance backed).
- A future surface that needs client-side JavaScript gets its own ADR deciding where the JS lives (`app/static/`), how it's loaded (`<script>` in `base.html.j2` or a module), how it's tested (Playwright; and tested-without, for progressive enhancement) — same as any other architectural choice. It is not gated as "crossing the no-JS threshold."
- Reviewers / the conformance skill do not reject a decision for introducing JavaScript. (No conformance rule does today — see "Conformance check"; this ADR makes that explicit so no one adds one by reflex.)
- The no-JS surfaces shipped so far stay as-is. This ADR does not schedule any rewrite.

**What it makes easier:** future `/design` cycles can pick the genuinely-cleanest solution for a surface without rationalizing around a non-existent prohibition; ADR-032's eventual implementation is unblocked of the "this breaks a posture" framing overhead.

**What it makes harder:** nothing. It removes a constraint that was never authorized, without adding one.

**Supersedure path if this proves wrong:** if the human ever *does* want a binding no-JavaScript invariant, that belongs in `design_docs/MANIFEST.md` (it is a product-shape constraint), added by the human — at which point a new ADR would record the architecture decisions that follow from it, and the conformance skill would gain a rule. As of this ADR, no such manifest entry exists, so no such invariant exists. If instead the *principle* ("ADRs describe used architecture, not the toolkit") proves too loose in some way, a future ADR refines it — but the burden there is "name the case where the looseness bites," not "restore the blanket prohibition."

## Manifest reading

Read as binding for this decision:

- **`design_docs/MANIFEST.md` in full** — no entry establishes a no-JavaScript posture, a "synchronous-form-posting-only" rule, or any client-side-technology restriction. §3 (Primary Objective), §5 (Non-Goals — "no mobile-first", "no remote deployment", "no LMS features", "no AI tutor / chat", "no live / synchronous AI results", "no non-coding Question formats"), §6 (Behaviors and Absolutes), §7 (Invariants), §8 (Glossary) — none of them mentions JavaScript or constrains the client-side toolkit. §5's "no mobile-first" and "no remote deployment" bound *operational shape* (local, desktop-tuned), not *which client technologies are allowed* — a local desktop app may use JavaScript. §6's "AI work is asynchronous from the learner's perspective" governs *AI* work specifically (and is, if anything, an argument *for* client-side JS in the eventual Notification surface, not against it). So: nothing in the manifest establishes or forbids a no-JS posture. This ADR is the *inverse* of the usual "architecture-in-disguise" finding — it removes an architecture-in-disguise (a no-JS rule) that crept into the ADR set without ever appearing in a binding document.
- **No manifest entry flagged as architecture-in-disguise for this decision.** (The smell being corrected here lived in the ADRs, not the manifest. The manifest is clean on this point — that cleanliness is precisely the evidence that the ADRs over-reached.)

If the human reads this ADR and *wants* a no-JS invariant after all, that is a manifest edit for the human to make — not something this ADR pre-empts. As written, the manifest does not have one, and this ADR records that fact.

(If the reviewer ever finds an ADR backsliding into "the project's no-JS posture" framing, the corrective is to cite this ADR — not a new MC-N.)
