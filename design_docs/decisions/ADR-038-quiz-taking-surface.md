## Context

The Quiz pillar so far: TASK-013 shipped the schema (ADR-033) + a per-Section Quiz read surface inside the `.section-end` action zone (ADR-034) + a `POST /lecture/{chapter_id}/sections/{section_number}/quiz` route that records a `requested`-status Quiz row; TASK-014 shipped the generation processor (ADR-036, ADR-037) — `python -m app.workflows.process_quiz_requests` walks a `requested` Quiz to `ready`, generating fresh hands-on coding-task Questions (`questions` rows + `quiz_questions` links). So a Section can now have a `ready` Quiz with Questions — but ADR-034 explicitly shipped the `ready` state with **no takeable affordance**: "the Quiz-taking surface is a later slice." TASK-015 is that slice. It makes a generated Quiz *takeable* — the learner opens it, sees each Question's coding-task `prompt` with a code-entry field, writes their implementation, and submits, and the submission persists as a Quiz Attempt (`quiz_attempts` row `in_progress` → `submitted` + one `attempt_questions` row per Question carrying the learner's `response`). No grading happens this slice: `attempt_questions.is_correct` / `.explanation` stay NULL; no Grade aggregate; the submit route does **not** invoke grading synchronously (MC-4 — grading is a later out-of-band slice, like the generation processor).

The established surrounding structure this ADR composes against:

- **Routes** (`app/main.py`): the per-Section routes are `/lecture/{chapter_id}/sections/{section_number}/complete` (ADR-025/027/031) and `/lecture/{chapter_id}/sections/{section_number}/quiz` (ADR-034); the Notes route is `/lecture/{chapter_id}/notes` (ADR-023/028/029). Each validates path parameters at the route handler (the ADR-024 split — `chapter_id` via `tex_path.exists()`; `section_number` by composing `section_id = f"{chapter_id}#section-{section_number}"` and checking it against `{s["id"] for s in extract_sections(...)}`); the persistence layer trusts the caller. Form-post-plus-`303`-PRG is the established round-trip shape (no JavaScript — because a no-JS recipe was clean and sufficient for each surface, not because "no JS" is a project rule; see ADR-035).
- **Templates** (`app/templates/`): `base.html.j2` is the three-column shell (LHS `.lecture-rail` chapter rail | centered `.page-main` | RHS `.notes-rail`), loading `/static/base.css` then `/static/lecture.css`; the RHS Notes rail is rendered only when `rail_notes_context` is truthy. `lecture.html.j2` extends `base.html.j2` and renders per-Section blocks, each `<section>` carrying `<div class="section-end" id="{{ section.fragment }}-end">` with the completion form, then the `.section-quiz` block (ADR-034).
- **CSS** (`app/static/`): ADR-008's prefix split — `page-*` / `nav-*` / `index-*` / `rail-*` → `base.css`; `lecture-*` / `section-*` / `callout-*` / `designation-*` → `lecture.css`; both flat files, loaded flat from `base.html.j2`, no preprocessor, no build step. `section-quiz-*` falls on the `lecture.css` side (ADR-034). ADR-033's `quiz_attempts` / `attempt_questions` tables already exist in `connection.py`'s `_SCHEMA_SQL`; this surface is the first thing that *writes* them, via the functions ADR-039 decides.

The decision space has materially different alternatives:

- **The route URL shape:** keyed under the existing per-Section path (`GET/POST /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take`); a flatter `/quizzes/{quiz_id}/take`; keyed on `attempt_id` once an Attempt is started (`/lecture/.../quiz/{quiz_id}/attempts/{attempt_id}`).
- **The take surface's template / layout:** its own `quiz_take.html.j2` extending `base.html.j2` (three-column shell, rails present); its own template with a slimmer single-column shell (no rails — focused-work mode); a section of an existing template.
- **The Attempt-start trigger:** a `quiz_attempts` row created when the `GET .../take` surface first loads (opening the take page = starting an Attempt); an explicit "Start this Quiz" `POST` that creates the Attempt then renders the Questions; a `GET`-creates-but-reuses-the-latest-`in_progress`-Attempt-for-the-Quiz approach (no orphan rows from idle reloads).
- **When `attempt_questions` rows are created:** at Attempt start (one row per Question, `response` NULL — so `save_attempt_responses` is an `UPDATE`); at submit (the rows are created with the responses in one `INSERT` pass — no `attempt_questions` row until submit).
- **The take affordance's shape on the `.section-quiz` block:** a "Take this Quiz" link; a button-styled link; inline next to "Ready", or below it.
- **The submit-route behavior:** PRG-redirect back to the Lecture page (anchored at `#section-{n-m}-end` per ADR-031's recipe); PRG-redirect back to the take surface (`GET .../take`), which re-renders showing the submitted state; re-render the take surface inline on the `POST` response (no redirect — re-submit-on-refresh hazard).
- **The take-page CSS file:** a new `app/static/quiz.css`; reuse `lecture.css` (a take *page* is arguably not Lecture-body content); reuse `base.css` (not a fit — it is not page-chrome/rail).
- **The honest "submitted — not yet graded" copy:** "Submitted — grading not yet available"; "Submitted — awaiting grading"; "Submitted (not yet graded)".

The learner writes code that implements a concept from the Section under study. ## Decision

### The route pair — `GET/POST /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take`

Two new route handlers in `app/main.py`, keyed under the existing per-Section path so the URL space is consistent with the per-Section completion and Quiz-trigger routes:

- **`GET /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take`** — renders the Quiz-taking surface. Validation at the route handler (the ADR-024 split):
  - `chapter_id` valid (the corpus `.tex` exists) — else `404`; `chapter_id` malformed (no valid chapter number per `chapter_designation()`) — else `422` (mirroring `render_chapter`).
  - `section_number` corresponds to a known Section in the Chapter (compose `section_id = f"{chapter_id}#section-{section_number}"`; check against the parsed Section set) — else `404`.
  - `quiz_id` exists (`get_quiz(quiz_id)` is not `None`) and the Quiz's `section_id` matches the composed `section_id` (the Quiz belongs to the Section in the URL — a Quiz of another Section is not takeable from this URL) — else `404`.
  - The Quiz's `status` is `ready` — if it is `requested` / `generating` / `generation_failed`, the surface does **not** present a takeable form: it renders an honest "this Quiz is not ready to take" state (HTTP `200` with a non-takeable rendered state, *or* a `409`/`404` — implementer's call; the architectural commitment is that no takeable form is presented for a non-`ready` Quiz, not the exact status code).
  - On the happy path: call `start_attempt(quiz_id)` (ADR-039 — reuses the latest `in_progress` Attempt for this Quiz if one exists, else inserts a `quiz_attempts` row `in_progress` + one `attempt_questions` row per Question with `response` NULL), fetch the Attempt's per-Question rows (`list_attempt_questions(attempt_id)` → `list[AttemptQuestion]` with `question_id` / `prompt` / `response` / `position`, ordered by `quiz_questions.position`), and render `quiz_take.html.j2` with: the Chapter's designation context, the Section's title/number, the Quiz, the Attempt, the ordered `AttemptQuestion` list, and a "back to this Section" link (`/lecture/{chapter_id}#section-{section_number}-end`). If the Attempt is already `submitted` (the learner has submitted and the page is re-loaded — including via the submit route's PRG redirect), render the surface in the "Submitted — grading not yet available" state instead of a takeable form (the responses are shown read-only; no re-submit form).
- **`POST /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take`** — records the learner's responses and submits the Attempt. Same path-parameter validation as `GET`. Body: one form field per Question, named by `question_id` (e.g. `response_{question_id}`) — the implementer picks the field-name scheme; the contract is "the body carries the learner's code for each Question, keyed so the route can map it to a `question_id` of the Quiz." The handler:
  - Resolves the Attempt to submit — the latest `in_progress` Attempt for `quiz_id` (the one `GET .../take` created/reused). If there is no `in_progress` Attempt (e.g. it was already submitted), the handler treats the `POST` as a no-op submit and redirects to `GET .../take` (which renders the submitted state) — submitting twice is harmless and idempotent-ish.
  - Calls `save_attempt_responses(attempt_id, responses)` (ADR-039 — `responses` is a `dict[question_id -> code-string]` built from the form fields; only `question_id`s that belong to this Quiz are accepted; the stored `response` is the submitted code verbatim — the response *is* the code, not a transformation of it).
  - Calls `submit_attempt(attempt_id)` (ADR-039 — flips `quiz_attempts.status` → `submitted`, sets `submitted_at`).
  - **Does not invoke grading.** No `ai-workflows` call, no grading workflow, no LLM SDK, no out-of-band job kicked off from the request. The Attempt sits in `submitted` until the (later) grading slice's out-of-band processor picks it up — exactly as ADR-037's generation processor picks up `requested` Quizzes. The submit route is written so that adding that processor later requires **no change to the submit route** (it already does the right thing — record and return).
  - Returns `303 See Other` → `GET /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take` (the take surface, which now finds the Attempt `submitted` and renders the "Submitted — grading not yet available" state). PRG: a browser refresh re-`GET`s the take surface, it does not re-`POST` (no double-submit).

**Why this route shape and not the alternatives:**

- **Keyed under `/lecture/{chapter_id}/sections/{section_number}/...` (not a flat `/quizzes/{quiz_id}/take`).** The project's URL space is per-Chapter/per-Section: every Quiz-related action so far lives under `/lecture/{chapter_id}/sections/{section_number}/...`. Keeping the take route there is consistent, makes the back-link trivial (`/lecture/{chapter_id}#section-{section_number}-end` is already in scope), and lets the route validate the Section ID the same way the sibling routes do. A flat `/quizzes/{quiz_id}/take` would be defensible (a Quiz has its own `quiz_id`) but would break the established per-Section URL pattern and would still need to look up the Quiz's Section to render "the Section this Quiz tests" and to compose the back-link — no real saving. Including `quiz_id` *in addition to* `section_number` (rather than only `section_number`) is right because a Section can have more than one Quiz over time (the loop's later slices); the take URL must name *which* Quiz.
- **`GET` creates/reuses an Attempt; not an explicit "Start" `POST` first.** An explicit-Start step is one more click for an idle benefit (no orphan `in_progress` rows from a load that's never submitted); the chosen "GET creates, but reuses the latest `in_progress` Attempt for the Quiz" approach gets most of that benefit (an idle reload reuses the existing `in_progress` Attempt rather than spawning a new one) with no extra click. A "resume an `in_progress` Attempt across sessions" affordance beyond this is *not* required this slice — re-opening `GET .../take` on a Quiz with an `in_progress` Attempt simply lands the learner back on it with their (saved-on-submit, not yet saved-mid-edit) responses; richer mid-edit autosave is a future task. (This slice has no mid-edit save — `save_attempt_responses` is called once, by the submit route. The `in_progress` Attempt holds `attempt_questions` rows with `response` NULL until submit.)
- **`attempt_questions` rows created at start, not at submit.** Creating them at start (one row per Question, `response` NULL) makes `save_attempt_responses` a clean `UPDATE` keyed on `(attempt_id, question_id)`, makes the per-Question-state structure (`response` / `is_correct` / `explanation`) exist from the moment the Attempt exists (the grading slice `UPDATE`s `is_correct`/`explanation` on the same rows; the replay-composition slice `SELECT`s `question_id` where `is_correct = 0` — neither has to handle "the row might not exist yet"), and keeps the take template's render path uniform (it always iterates `AttemptQuestion`s, never "Questions, some of which have an `attempt_questions` row"). Creating-at-submit would mean an `in_progress` Attempt has no `attempt_questions` rows — also workable, but a less uniform structure for no offsetting benefit at this scope. The AC permits either; this ADR commits to at-start (and ADR-039 encodes it).

### The take surface's template and layout — its own `quiz_take.html.j2` extending `base.html.j2` (the three-column shell, rails present)

A new template `app/templates/quiz_take.html.j2` extending `base.html.j2`. It renders, inside `{% block main %}`:

- A header naming the Quiz's Section ("Quiz — Section N.M: <Section title>"), with the Chapter's Mandatory/Optional designation badge (reusing the `.designation-badge designation-{m|o}` markup from `lecture.html.j2`) so the M/O context is visible (manifest §6).
- For a `ready` Quiz with an `in_progress` Attempt: a `<form method="post" action=".../take">` containing, for each `AttemptQuestion` in `position` order, a block with the Question's coding-task `prompt` (rendered as text — the prompt is a plain-language instruction, not LaTeX; `{{ aq.prompt }}` autoescaped) and a code-entry `<textarea name="response_{{ aq.question_id }}">` (pre-filled with `aq.response` if non-NULL — empty on a fresh Attempt), plus a submit button ("Submit Quiz"). The `<textarea>` uses ADR-029's no-JS auto-grow recipe (`rows` attribute + CSS `field-sizing: content` with a `min-height` fallback + `resize: vertical`) and a monospace font (it is a code field).
- For a Quiz whose latest Attempt is `submitted`: the **"Submitted — grading not yet available"** state — the prompts and the learner's submitted `response` for each Question shown read-only (no editable form, no submit button), and an honest status line ("Submitted — grading not yet available." — minor wording implementer-tunable; it must not imply a Grade exists, must not show a score, must not say "all correct" or any invented result; `attempt_questions.is_correct` / `.explanation` are NULL and nothing is rendered in their place). A "back to this Section" link.
- For a non-`ready` Quiz reached via the URL: an honest "this Quiz is not ready to take" state (no takeable form), with the Quiz's status in plain language and a "back to this Section" link.
- A "back to this Section" link (`/lecture/{chapter_id}#section-{section_number}-end`) in all states, so the learner can return to the Lecture page where the per-Section Quiz block reflects the Quiz's status.

The shell is `base.html.j2` **unchanged** — the take page reuses the three-column layout (LHS chapter rail, centered `.page-main`, RHS Notes rail). `render` is called with a truthy `rail_notes_context` (the same `_RailNotesContext` namespace `render_chapter` builds — `chapter_id` + the Chapter's Notes list) so the RHS Notes rail renders (jotting a note while taking a Quiz is genuinely useful, and the rail is already there). The LHS chapter rail (`_nav_rail.html.j2` + the `nav_groups` from `discover_chapters()` with `_attach_progress_counts`) provides "back to any chapter" navigation. No `base.html.j2` change; no grid change; the take page does not touch the Lecture-page layout (so the parked ADR-032 — Notes-save scroll — does **not** activate; that issue's "Decide when: a future task that touches the Notes form or the page layout" is not triggered — this task touches neither the Notes form nor the page-layout grid).

**Why its own template, the three-column shell, rails present:**

- **Its own template** — a Quiz-taking page is a distinct surface from the Lecture page (it shows Questions + code fields + a submit form, not Chapter content); folding it into `lecture.html.j2` would bloat that template with a conditional render mode it does not otherwise have. ADR-023 set the precedent that per-Section/per-Chapter *read* surfaces (Notes, completion, the `.section-quiz` block) live inside the Lecture page — but the take *action* is a focused interaction the learner navigates *to*, not an annotation on a page they are reading; a separate route + template is the right shape for it (it is the first non-Lecture, non-landing HTML surface, and that is fine — `base.html.j2` was built to be reusable).
- **The three-column shell, rails present** — reusing `base.html.j2` unchanged is the cheapest path (no new shell template, no grid change) and the rails are genuinely useful on the take page (the Notes rail for jotting; the chapter rail for "back to any chapter"). A slimmer single-column "focused-work" shell is the closest competitor — it would emphasize "do the Quiz, nothing else" — but it would mean a second base template (or a `base.html.j2` variant with a `{% if %}` to suppress the rails), more surface area, for a UX claim ("focus") that the existing shell does not actually undermine (the take form is in the centered `.page-main`, where the eye goes; the rails are peripheral). Reuse wins on cost and consistency; if the rails prove distracting on the take page, a future ADR can introduce the slimmer shell — bounded cost.

### The take affordance on the `.section-quiz` block — a "Take this Quiz" button-styled link on the `ready` list entry, below the "Ready" label

ADR-034's `.section-quiz` block renders, for a `ready` Quiz, a `<li class="section-quiz-item section-quiz-item--ready">Ready</li>`. This ADR adds inside that `<li>` (only for `ready`), below the "Ready" label, a link to the take surface styled as a button:

```html
<li class="section-quiz-item section-quiz-item--ready">
  Ready
  <a class="section-quiz-take-link"
     href="/lecture/{{ chapter_id }}/sections/{{ section.section_number }}/quiz/{{ quiz.quiz_id }}/take">Take this Quiz</a>
</li>
```

(Exact markup — element nesting, whether the link is `inline-block` next to "Ready" or on its own line — is implementer-tunable; the architectural commitments are: the affordance appears **only** on `ready` Quizzes; it is a link to the take route keyed by `quiz_id`; it is in the `section-quiz-*` namespace; it is visually clear it is a take affordance.) The `requested` / `generating` / `generation_failed` entries get **no** such affordance (they are not takeable) — their labels are unchanged from ADR-034. The empty-state caption ("No quizzes yet for this Section.") and the "Generate a Quiz for this Section" button are unchanged.

Whether the `.section-quiz` block also surfaces *Attempt* state (e.g. "Submitted — grading not yet available" alongside the Quiz once the learner has submitted) is **not required this slice** — the `.section-quiz` block keeps showing Quiz status only; the Attempt's "submitted — not yet graded" state surfaces on the take surface. Keeping the per-Section block to Quiz-status-only and the take surface to Attempt-status is the simplest split; a richer per-Section rendering (past Attempts + their Grades) is a later slice's addition (the grading slice produces the Grades to render). The take affordance being shown on a `ready` Quiz even after it has been taken is fine — re-opening the take surface lands the learner on the submitted state honestly.

### The submit-route behavior — PRG-redirect back to the take surface (`GET .../take`), which re-renders in the "Submitted — grading not yet available" state

The `POST .../take` handler returns `303 See Other` → `GET /lecture/{chapter_id}/sections/{section_number}/quiz/{quiz_id}/take`. The take surface's `GET` handler, finding the Quiz's latest Attempt is now `submitted`, renders the "Submitted — grading not yet available" state (responses shown read-only, no editable form, honest status line, "back to this Section" link). This is a proper PRG round-trip (a refresh re-`GET`s, does not re-`POST`), the learner sees their submission confirmed *right where they were* (on the take page they just submitted from), and there is no relocation concern (the take page is its own URL — there is no "where the reader was in the Lecture page" to preserve).

**Why this and not "redirect to the Lecture page":** redirecting to `GET /lecture/{chapter_id}#section-{section_number}-end` (reusing ADR-031's recipe) would be acceptable (the AC permits it), but it would take the learner *away* from the take context the moment they submit — they would have to navigate back to see what they submitted. Re-rendering the take surface in the submitted state keeps the confirmation in context (the same surface, now showing "you submitted this, here's your code, grading isn't available yet"). The learner returns to the Lecture page via the "back to this Section" link when *they* choose to — a deliberate navigation, not an automatic relocation. (And re-rendering inline on the `POST` response — no redirect — is **rejected**: a refresh would re-submit; PRG is the right shape, as everywhere else in the project.)

### The take-page CSS — a new `app/static/quiz.css` (`quiz-take-*` namespace), loaded flat from `base.html.j2`; the take affordance reuses `section-quiz-*` in `lecture.css`

Per ADR-008's class-name-prefix posture (two flat files, non-overlapping by prefix, loaded flat, no preprocessor — and "future learner-facing surfaces … get a clean naming convention: `base.css` for chrome shared across surfaces, per-surface CSS for surface-specific content"; ADR-008 §Future-surfaces explicitly forecasts "a future `quiz.css` if the surface grows enough to warrant its own file"):

- **A new file `app/static/quiz.css`** owns the take-*page*'s classes — the `quiz-take-*` namespace: `.quiz-take` (the page wrapper inside `.page-main`), `.quiz-take-header`, `.quiz-take-form`, `.quiz-take-question`, `.quiz-take-prompt`, `.quiz-take-response` (the code `<textarea>` — monospace, ADR-029's auto-grow recipe), `.quiz-take-submit`, `.quiz-take-submitted` (the "Submitted — grading not yet available" state wrapper), `.quiz-take-submitted-status`, `.quiz-take-response-readonly` (the read-only response display in the submitted state), `.quiz-take-not-ready` (the non-`ready`-Quiz state), `.quiz-take-back-link`. ## Alternatives considered

**A. Route keyed flat — `GET/POST /quizzes/{quiz_id}/take` (not under the per-Section path).**
Considered — a Quiz has its own `quiz_id`, so a flat URL is defensible. **Rejected** because the project's URL space is per-Chapter/per-Section (`/lecture/{chapter_id}/sections/{section_number}/...` for every Quiz/completion action so far); keeping the take route there is consistent, makes the back-link trivial (`/lecture/{chapter_id}#section-{section_number}-end` is already in scope), and lets the route validate the Section ID the same way the sibling routes do. A flat `/quizzes/{quiz_id}/take` would still need to look up the Quiz's Section to render "the Section this Quiz tests" and compose the back-link — no real saving — while breaking the established per-Section pattern.

**B. Route keyed on `attempt_id` once an Attempt is started — `GET /lecture/.../quiz/{quiz_id}/attempts/{attempt_id}`.**
Considered. **Rejected** as a more complex URL for no benefit at this scope: keying on `quiz_id` and resolving the Attempt server-side (latest `in_progress` for the Quiz) is simpler — the learner navigates to "take this Quiz", not "resume Attempt 17". A future "see all my Attempts" / "resume Attempt N" surface might want an `attempt_id`-keyed URL; that surface owns it.

**C. An explicit "Start this Quiz" `POST` step before the take surface shows the Questions.**
Considered (avoids orphan `in_progress` Attempts from idle loads). **Rejected** — one extra click for an idle benefit the chosen "GET creates, but reuses the latest `in_progress` Attempt for the Quiz" approach already mostly delivers (a reload reuses the existing `in_progress` Attempt). Simpler is better at this scope; if orphan `in_progress` Attempts ever become a real problem (single-user dev DB — unlikely), a future task can add an explicit-Start step or a "discard this Attempt" affordance.

**D. Create `attempt_questions` rows at submit (one `INSERT` pass), not at start.**
Considered — the AC permits it. **Rejected** in favor of at-start: creating the rows at start (one per Question, `response` NULL) makes `save_attempt_responses` a clean `UPDATE` keyed on `(attempt_id, question_id)`, makes the per-Question-state structure exist from the moment the Attempt exists (the grading slice and the replay-composition slice both operate on these rows and never have to handle "the row might not exist yet"), and keeps the take template's render path uniform (always iterate `AttemptQuestion`s). At-submit is workable but less uniform for no offsetting benefit. (Encoded in ADR-039.)

**E. Submit route PRG-redirects to the Lecture page (`#section-{n-m}-end` per ADR-031), not back to the take surface.**
Considered — the AC permits it; ADR-031's recipe would deliver it for free. **Rejected** in favor of redirecting back to `GET .../take`: redirecting to the Lecture page takes the learner *away* from the take context the moment they submit (they'd have to navigate back to see what they submitted). Re-rendering the take surface in the "submitted — grading not yet available" state keeps the confirmation in context; the learner returns to the Lecture page via the "back to this Section" link when *they* choose. (And re-rendering inline on the `POST` response — no redirect — is rejected: refresh would re-submit. PRG is the right shape.)

**F. The take page uses a slimmer single-column shell (no rails), not `base.html.j2`'s three-column shell.**
Considered — "focused-work" mode. **Rejected** in favor of reusing `base.html.j2` unchanged: a second base template (or a `base.html.j2` `{% if %}` to suppress the rails) is more surface area for a UX claim ("focus") the existing shell does not actually undermine (the take form is in the centered `.page-main`; the rails are peripheral and the Notes rail is genuinely useful while taking a Quiz). Reuse wins on cost and consistency; a future ADR can introduce the slimmer shell if the rails prove distracting on the take page — bounded cost.

**G. Fold the take surface into `lecture.html.j2` (a conditional render mode on the Lecture page) rather than its own template.**
Considered — ADR-023 set the precedent that per-Section/per-Chapter *read* surfaces live inside the Lecture page. **Rejected** — the take *action* is a focused interaction the learner navigates *to*, not an annotation on a page they are reading; folding it into `lecture.html.j2` would bloat that template with a conditional mode it does not otherwise have. A separate route + template is the right shape (and `base.html.j2` was built to be reusable — the take page is its first non-Lecture, non-landing consumer, which is fine).

**H. Take-page CSS in `lecture.css` (extend it with `quiz-take-*` rules), not a new `quiz.css`.**
Considered — fewer files. **Rejected** — a take *page* is not Lecture-body content; putting its rules in `lecture.css` would conflate two surfaces in one file, exactly the conflation ADR-008 split `base.css` out to avoid. ADR-008 §Future-surfaces explicitly forecasts "a future `quiz.css` if the surface grows enough to warrant its own file"; the take page is that surface. (The `.section-quiz` take-link, by contrast, *is* a per-Section Lecture-body element, so its rule stays in `lecture.css` under the existing `section-quiz-*` namespace — that part does not get a new file.)

**I. Take-page CSS in `base.css`.**
Rejected — `base.css` is page-chrome/rail (`page-*` / `nav-*` / `index-*` / `rail-*`); the take page's content classes are neither. A new `quiz.css` is the right home.

**J. Defer the take affordance on the `.section-quiz` block; ship only the take route + template (no link to it).**
Rejected — a route that returns a page no affordance links to is a half-implementation by construction (UI-1 forbids shipping layout-bearing DOM, or a route that returns it, without the surface being a real, *reachable*, styled thing). The "Take this Quiz" link on the `ready` entry is the deliverable's reachability.

**K. The take surface presents a non-code input for some Questions (a "describe your approach" field, an "explain the complexity" toggle).**
Rejected — manifest §5 / §7: every Question is a hands-on coding task; the learner *writes code*. The take surface renders the Question's coding-task `prompt` and a code-entry `<textarea>`, nothing else; the `attempt_questions.response` column holds the learner's *code*. A non-code input would violate the manifest.

**L. Run the learner's code (a test-runner / sandbox) on submit.**
Rejected — running the learner's code is the *grading* concern (or a future one), and the manifest's "every Question is a hands-on coding task" requires the learner to *write* code, not that the system *runs* it in this surface; recording the text and grading it later (the §6 async decoupling) is the manifest-aligned shape. A code-runner also pulls in a sandboxing question (running arbitrary learner code) that is a significant decision in its own right — out of scope for the take surface.

**M. The submit route kicks off grading (synchronously, or by enqueuing).**
Rejected — MC-4 / manifest §6: AI work is asynchronous from the learner's perspective; submission, processing, and result delivery are decoupled in time. The submit route records the responses, flips to `submitted`, and returns; the (later) grading slice's out-of-band processor picks up `submitted` Attempts — exactly as ADR-037's generation processor picks up `requested` Quizzes. The submit route is written so that adding that processor later requires no change to it.

## Manifest reading

Read as binding for this decision:

- **§3 Primary Objective.** Per-Section Quizzes are one of the three named consumption surfaces; a `ready` Quiz the learner can't take is a dead end and the reinforcement loop never starts. This surface is the missing link — it makes a generated Quiz takeable, the Attempt persists, and that Attempt is the raw material the grading slice (then the composition slice) turns into the loop. Placing the take action on its own surface the learner navigates *to* (with a clear "Take this Quiz" affordance on the per-Section block) makes the flow read as "this Quiz is ready — take it here."
- **§5 Non-Goals.** "No non-coding Question formats. No multiple-choice, no true/false, no short-answer, no describe-the-concept, no interview-style verbal Questions. Every Question is a hands-on coding task" — the take surface renders each Question's coding-task `prompt` and a code-entry `<textarea>`; it renders **no** non-code input for any Question; the stored `response` is the learner's *code*. "No live / synchronous AI results" — no AI runs in this surface; the submit route does not trigger grading inside the request. "No multi-user features. No accounts, no auth, no sharing, no social, no roles" — no `user_id`, no auth, no session, no per-user partitioning; the single user owns every Attempt. "No mobile-first product; usable layout in a desktop browser is sufficient" — the take page is desktop-tuned (it reuses the desktop-tuned three-column shell). "No AI tutor / chat interface for the learner. The Quiz reinforcement loop is the only AI-driven surface the learner interacts with" — the take surface *is* part of the loop (it is where the learner does the Quiz) and it is a *form* (prompts shown, code written, submit), not a chat. "No in-app authoring of lecture content" — the surface writes the Attempt to `data/notes.db`, never to `content/latex/`. "No LMS features / No remote deployment / No coverage of CS-300 Chapters 1–2" — none crossed (Quizzes are offered only on Sections of in-scope Chapters, same as every other per-Section surface).
- **§6 Behaviors and Absolutes.** "AI work is asynchronous from the learner's perspective. Submission, processing, and result delivery are decoupled in time. The learner submits, continues working, and is notified when results are ready" — this slice ships the *submission*; the *processing* (grading) and *result delivery* (the Grade, the Notification) are later slices and explicitly **not** triggered by this slice's submit route — the submit route records the responses, flips `quiz_attempts.status` → `submitted`, sets `submitted_at`, and PRG-redirects; the Attempt then sits in `submitted` until the grading slice's out-of-band processor ships. "AI failures are visible … never fabricates a result" — not implicated (no AI call), but the surface must not *fake* a Grade — a `submitted` Attempt is shown honestly as "Submitted — grading not yet available", never with a fabricated score; the `is_correct`/`explanation` columns stay NULL; the surface renders the absence of a Grade honestly. "Quizzes scope to Sections; Lectures and Notes scope to Chapters" — the Quiz being taken is bound to one Section (ADR-033's `quizzes.section_id`). "Mandatory and Optional are honored everywhere" — the take page renders the Chapter's designation badge; the entry point inherits the designation via ADR-004; the surface surfaces no chrome that hides the split. "Single-user" — no multi-tenant data path. "A Lecture has a single source" / "Consumption modes of a Lecture remain consistent" — orthogonal; Quizzes are a separate entity from Lectures. "Lecture source read-only" — the take route reads `content/latex/{chapter_id}.tex` read-only (to validate the Section ID and get the Section title); nothing under `content/latex/` is written.
- **§7 Invariants.** **"The reinforcement loop is the reason this project exists. Quiz performance MUST drive next-Quiz generation. A Quiz feature without the loop is not the project."** — this surface is the missing link that makes the loop *possible*: a Quiz can now be *taken*, an Attempt *persists*, and that Attempt is the raw material the grading slice turns into a Grade with Weak Topics, which the composition slice turns into the next Quiz's replay-+-fresh content; without a takeable surface there is no Attempt history and the loop is inert. **"A Quiz is bound to exactly one Section."** — the take surface and the Attempt reference exactly one Quiz, which references exactly one Section (MC-2); the Attempt does not span Quizzes or Sections. **"Every Question is a hands-on coding task. The learner writes code that implements a concept from the Section under study. Questions never ask the learner to describe, explain, recall, or choose among options."** — the take surface renders each Question's coding-task `prompt` (an instruction to implement something) and a code-entry `<textarea>` for the learner to write that implementation in; it never renders a non-code input; the `attempt_questions.response` column holds the learner's *code*. **"Every post-first Quiz for a Section contains both replayed wrong-answer Questions and freshly-generated Questions … The first Quiz for a Section contains only fresh Questions (the bank is empty)."** — orthogonal to *taking* a Quiz; this slice makes whatever Questions a `ready` Quiz contains (currently only first-Quiz fresh ones, per TASK-014) takeable; the composition that builds post-first Quizzes is a later slice; the take surface creates no fresh-only-post-first-Quiz path (it composes the Attempt strictly from one Quiz's `quiz_questions` rows). **"Quiz generation is always explicitly user-triggered."** — not implicated by *taking* a Quiz; the take surface generates nothing. **"A Note is bound to exactly one Chapter"** / **"Completion state lives at the Section level"** — orthogonal, unchanged. **"Every Quiz Attempt, Note, and completion mark persists across sessions and is owned by the single user."** — the central invariant this slice implements: a Quiz Attempt (a `quiz_attempts` row + its `attempt_questions` rows carrying the learner's `response` for each Question) is created on "start the Quiz", updated on "submit", and persists across sessions; owned by the single user (no `user_id` — there is only one). **"Mandatory and Optional are separable in every learner-facing surface."** — the take page renders the Chapter's M/O designation badge; the entry point (the `.section-quiz` block) already respects the designation; the take surface surfaces no Question-format chrome that would obscure it.
- **§8 Glossary.** **Quiz** — "a graded assessment scoped to exactly one Section, composed of one or more Questions" — the take surface displays the Quiz's Questions and lets the learner respond to each. **Question** — "A single graded item belonging to a Section's Question Bank. A Question is a hands-on coding task: it asks the learner to implement a concept from the Section in code … Persists with full Attempt history (which Attempts it appeared in, correctness in each)" — the take surface renders the Question's coding-task prompt and a code field; the `attempt_questions` row records this Question's appearance in this Attempt and (later, when grading ships) its correctness. **Question Bank** — consumed unchanged (the take surface reads Questions; it does not add or delete any). **Quiz Attempt** — "A single submission of a Quiz by the user. Carries the user's responses, a progress status through grading, and — once graded — a Grade" — this slice creates the Quiz Attempt entity in the running system: a `quiz_attempts` row (status `in_progress` while the learner is taking it → `submitted` when they submit) plus one `attempt_questions` row per Question carrying the learner's `response`; the *progress status through grading* (`grading`, `graded`, `grading_failed`) and the *Grade* are later slices — this slice ships the Attempt up to `submitted`, no further. **Grade** — "the result of grading a Quiz Attempt … per-Question correctness, per-Question explanation, an aggregate score, identified Weak Topics, recommended Sections to re-read" — explicitly out of scope this slice; no Grade is computed, displayed, or stored beyond the `attempt_questions.is_correct`/`explanation` columns staying NULL. **Weak Topic** — out of scope (no grading → no Weak Topics this slice). **Notification** — out of scope (no async result this slice; submitting an Attempt produces no Notification — the Notification entity ships with the grading slice per ADR-037). No new glossary terms forced — every entity this surface touches (Quiz, Question, Quiz Attempt) is already named in §8.

No manifest entries flagged as architecture-in-disguise for this decision. The take-surface placement and shape are operational UI design applying the project's encoded patterns (ADR-008's CSS-file posture, ADR-023's "surfaces inside the Lecture page" / "the take action is a navigated-to surface" distinction, ADR-024's validation split, ADR-027/031's per-Section route shape, ADR-029's textarea recipe, ADR-035's "ADRs describe what is used"); no manifest-level change. (The standing flag from MEMORY.md — "affordance placement follows the reading flow, not template scope" — is *applied* here: the "Take this Quiz" affordance is placed where the cognitive sequence puts it, on the `ready` Quiz's entry in the bottom-of-Section action zone after the learner has read the Section, not at the top of the Section or in a rail.)

## Consequences

**Becomes possible:**

- The learner can *take* a generated Quiz: open it from the per-Section block's "Take this Quiz" affordance, see each Question's coding-task prompt with a code-entry field, write their implementation, submit — and the submission persists as a Quiz Attempt (`quiz_attempts` row `in_progress` → `submitted` + one `attempt_questions` row per Question carrying the learner's `response`), across sessions (§7).
- The reinforcement loop has data: a `submitted` Attempt with the learner's `response` in each `attempt_questions` row is the raw material the (later) grading slice turns into `is_correct` + a Grade with Weak Topics, which the (later) composition slice turns into the next Quiz's replay-+-fresh content. Without this slice there is no Attempt history and the loop is inert; with it, the loop's later slices have a concrete input.
- The take surface is the project's first non-Lecture, non-landing HTML surface — validating `base.html.j2` as a reusable shell and establishing the `quiz.css` per-surface-CSS pattern ADR-008 forecast.
- The per-Section `.section-quiz` block validated as the standard place a learner reaches Quiz actions — "Generate a Quiz" (ADR-034) and now "Take this Quiz" (this ADR) both live there, in the bottom-of-Section action zone after reading.
- The submit route validated as the "record and return" shape the (later) out-of-band grading processor builds on — exactly mirroring the generation processor / `requested`-row pattern (ADR-037).

**Becomes more expensive:**

- A new template (`quiz_take.html.j2`), a new CSS file (`quiz.css`), a new route pair, one `<link>` in `base.html.j2`, a `.section-quiz-take-link` rule in `lecture.css`, a `ready`-entry take link in `lecture.html.j2`, and (via ADR-039) the Attempt-lifecycle persistence functions + dataclasses + a new index. Mitigation: each piece mirrors an established pattern (the route pair mirrors the per-Section routes' validation; the template extends `base.html.j2`; `quiz.css` extends ADR-008's posture; the persistence functions mirror ADR-033's dataclass-returning pattern).
- `quiz.css` is loaded on every page (via `base.html.j2`), including the landing page that does not need it. Mitigation: trivially cheap at local single-user scale (no CDN, no build step — manifest §5); consistent with the existing pattern (`lecture.css` is already loaded on the landing page).
- An idle `GET .../take` that is never submitted leaves an `in_progress` Attempt (reused on a subsequent reload, but not garbage-collected). Mitigation: harmless in a single-user dev database; a future task can add a "discard this Attempt" affordance or an explicit-Start step if `in_progress` Attempts ever pile up.

**Becomes impossible (under this ADR):**

- A take surface that presents a non-code input for a Question (option radios, a true/false toggle, a describe field). The "render the coding-task prompt + a code `<textarea>`" commitment forbids it (manifest §5/§7).
- A submit route that triggers grading synchronously or by enqueuing inside the request. MC-4 / §6 + the "record and return" commitment forbid it.
- A take surface that presents a `submitted` Attempt with a fabricated Grade, score, or "all correct". The "Submitted — grading not yet available" commitment + MC-5's spirit forbid it.
- A takeable form presented for a non-`ready` Quiz. The `GET .../take` route's `ready`-status check forbids it.
- A take surface or Attempt composed from Questions of more than one Section. MC-2 + the "compose the Attempt strictly from one Quiz's `quiz_questions` rows" commitment forbid it.
- A `user_id` on the Attempt or any new row/dataclass. MC-7 forbids it.
- The take-page CSS leaking into `lecture.css` or `base.css`. The `quiz-take-*` → `quiz.css` commitment (and ADR-008's prefix posture) keeps it in its own file.

**Future surfaces this ADR pre-positions:**

- The grading slice — a new `ai-workflows` workflow run out-of-band (a processor, like the generation processor) that picks up `submitted` Attempts, grades the learner's code, sets `attempt_questions.is_correct` / `.explanation`, computes the Grade aggregate (the deferred schema), identifies Weak Topics, sets `quiz_attempts.status` → `graded` / `grading_failed`, and (with the Notification entity) notifies the learner. The submit route this ADR ships requires no change for it. The next slice.
- The Grade display — once Grades exist, the take surface's "submitted" state (and possibly the `.section-quiz` block) gains a Grade rendering (per-Question correctness + explanation, aggregate score, Weak Topics, recommended Sections). The grading slice's `/design` decides the rendering shape.
- The replay-+-fresh composition for post-first Quizzes — reads `attempt_questions.is_correct = 0` (the wrong-answer-replay history this slice starts recording) + the grading slice's Weak Topics to compose the next Quiz's replay portion + fresh portion. A later slice.
- A "see all my Attempts" page / a "retake this Quiz" affordance / mid-edit autosave on the take page — future surfaces; the `quiz_attempts` / `attempt_questions` tables (ADR-033) and the route shape this ADR ships support them additively.
- A slimmer single-column "focused-work" shell for the take page — a future ADR's call if the three-column shell proves distracting on the take page; bounded cost (a new shell template, no data change).

**Supersedure path if this proves wrong:**

- If the take route shape proves awkward (e.g. a flat `/quizzes/{quiz_id}/take` would be cleaner once a "see all my Attempts" page exists) → a future ADR refines the URL; bounded cost (the route handlers + the back-link + the `.section-quiz` take-link href).
- If the three-column shell proves distracting on the take page → a future ADR introduces the slimmer single-column shell; bounded cost (a new shell template, no data change).
- If creating `attempt_questions` rows at start proves wrong (e.g. a Quiz with many Questions makes the start-of-take INSERT batch noticeable — unlikely at this scale) → a future ADR (and an ADR-039 supersedure) switches to create-at-submit; bounded cost (the persistence functions + the take template's render path).
- If "stay on the take surface after submit" proves worse than "redirect to the Lecture page" → a future ADR switches the submit redirect target; bounded cost (one line in the submit route + the take template's "submitted" state, which the Lecture page's `.section-quiz` block would then need to surface).
- If the `quiz.css` third-file split proves over-engineered (e.g. the take-page rules end up tiny and could live in `lecture.css`) → a future ADR collapses it; bounded cost (concatenate, drop the `<link>`, re-test).
- If `start_attempt`'s "reuse the latest `in_progress` Attempt" proves wrong (e.g. the learner wants a fresh Attempt each time they open the take surface) → a future ADR (and an ADR-039 supersedure) switches to "always a new Attempt" + adds an "Attempt history" surface; bounded cost.

The supersedure path, in every case, runs through a new ADR. This ADR does not edit any prior ADR in place; it adds the Quiz-taking surface, consuming ADR-033's schema, ADR-034's `.section-quiz` block, ADR-008's CSS posture, ADR-024's validation split, ADR-029's textarea recipe, ADR-031's no-relocate recipe (for the back-link target only), and ADR-039's Attempt-lifecycle functions, all unchanged.
